# v0.3.0 Deferred Plan - Phase 6B（保留）

- 状態: 保留（v0.3.X リリース対象外）

## この文書の役割
- v0.3.0 本線から外した 6B の詳細監査・保留候補を、ここへ集約する。
- `v0.3.0_master_plan.md` と `v0.3.0_phase6_retrieval_policy_and_freshness_plan.md` には詳細本文を残さず、リンク参照だけに留める。

## 目的
- 6A の運用ログを見たうえで、必要な拡張だけを最小差分で追加する。
- 検索精度改善と可観測性改善を、既存挙動を壊さず段階導入する。

## 6A から引き継ぐ判断材料
- `matchedBy` の分布（`semantic` / `lexical` / `both`）
- `fallbackReason` の傾向
- empty result の発生率
- strict topics fallback の実発生率

## 保留対象（6B 候補）
1. 常時並列 hybrid（lexical と HNSW を毎回併走）
2. RRF の導入
3. 正規化線形結合の拡張
4. lexical freshness メトリクス追加（lag/seq 可視化）
5. empty result 時 top-level diagnostics 拡張

## 監査サマリ（現時点）
- 常時並列 hybrid: 効果はあるが差分波及が広く、`matchedBy` / `fallbackReason` の意味崩れリスクが高い。
- RRF: rank 融合には有効だが、現行 `finalScore`（semantic/lexical/freshness/surprise/usefulness/exploration）との設計再整理が必要。
- 正規化線形結合: RRF より現実的だが、正規化対象を絞らないと式の意味がぼやける。
- freshness メトリクス: 有用だが新規観測基盤の追加が必要。
- top-level diagnostics: empty result の死角を埋める効果が高いが、Go -> RPC -> TS の shape 変更が必要。

## 実装優先順（保留解除時）
1. empty result 時 top-level diagnostics の設計確定（必要なら最小 shape 拡張）
2. 常時並列ではなく、正規化線形結合の限定拡張を先に比較
3. 常時並列 hybrid / RRF は最後に再評価

## 非対象
- Phase 6A で確定した backfill ロジックの再変更
- Anchor compaction / prepend 注入の責務変更
- `ep-save` 系仕様変更
