# Phase 5.5: Segmenter Chunking アーキテクチャ精査レポート (Google Pro Engineer Audit)

## 概要
提供された `phase_5_5_segmenter_chunking_report.md` および `src/segmenter.ts` の修正内容について、システムアーキテクチャ、非同期処理の安全性、および RAG（検索拡張生成）の品質担保の観点から容赦のない厳格な精査を行いました。

結論から申し上げます。**「長すぎる文字列をチャンクに区切る」というアイデア自体は正しい**ですが、**今回の `chunkAndIngest` の TypeScript レイヤーでの実装方法は最悪のアンチパターン**に陥っています。
既存の美しいアーキテクチャ（責務分界点）を破壊し、解決したはずのタイムアウト問題を再燃させ、RAG の検索品質を劣化させる重篤なリスクを抱えています。

以下に致命的な問題点と、あるべき Google Pro クラスの解決策を提示します。

---

## 1. 潜在的な問題やリスク (🔴 Critical)

### 🔴 1.1 TS層での直列 `await` ループによる Node.js / OpenClaw 全体のブロッキング
* **問題:** `chunkAndIngest` 関数内で、生成したチャンクごとに `await this.rpc.generateEpisodeSlug(...)` を直列（Sequential）で呼び出しています。
* **影響:** Node.js のイベントループを直接ブロックするわけではありませんが、この関数は `processTurn` というユーザーの毎ターンのメッセージ処理直後に同期的に発火します。巨大なダンプ（例えば5つのチャンク）が発生した場合、**LLM の生成×5回分（数分間）の時間をこの `chunkAndIngest` が await し続け、OpenClaw 全体のセッション処理をハングアップさせます**。
* **責務違反:** 重たい LLM 呼び出しやバルク処理の並行制御は、TS側ではなく**Goサイドカーの強固な Goroutine に委譲する（Goの `BatchIngest` を叩く）のが本アーキテクチャの基本理念**です。なぜ既存の `batchIngest` を無視して TS 側で直列ループを実装したのでしょうか？

### 🔴 1.2 レートリミット (429) と RPC Timeout (120秒) の再燃
* **問題:** 直列とはいえ、ループを通じて即座に Go 側の API を連続で叩き続けます。
* **影響:** Go 側には 15 RPM (4秒に1回) の `gemmaLimiter` が設定されているため、Go 側でキューイング（待機）が発生します。すると、チャンク数が多い場合、最後のほうのチャンクに対するTSからの RPC Request が **Go 側で処理待ちになっている間に TS 側の 120 秒タイムアウトに引っかかり、結局エラーとしてプロセスが破綻**します。せっかく MD5 フォールバックで解決した安定性を根底から覆す行為です。

## 2. 不足している手順や考慮事項 (🟠 High)

### 🟠 2.1 Overlap (Sliding Window) 欠落による RAG セマンティックの喪失（文脈の泣き別れ）
* **問題:** `extractText` の文字数（`10,000文字`）で機械的にチャンクを配列（Message単位）で切り分けています。
* **影響:** 文脈がシームレスに繋がっている会話であっても、10,000文字を超えた瞬間にAのファイルとBのファイルに泣き別れになります。
  * 【チャンク1（ファイルA）の最後】 User:「Geminiの特徴を3つ教えて」
  * 【チャンク2（ファイルB）の最初】 Assistant:「1点目は…」
  このように分割された場合、VectorDB には「質問だけのファイル」と「答えだけのファイル」が独立して登録されます。後から RAG で検索しても意味が通じず、コンテキスト注入時に甚大なノイズを発生させます。チャンク分割には**必ず直前の1〜2件のメッセージを次のチャンクの冒頭にも重複して含める（Overlap）** アルゴリズムが必須です。

### 🟠 2.2 巨大単一メッセージの考慮漏れ
* **問題:** `text.length > MAX_CHARS_PER_CHUNK` の判定において、1つの `Message.content` 単体が 15,000 文字あった場合、ループに入った後に `currentLen + text` で分割判定が来ても、その 15,000 文字のメッセージそのものは分割されずに1チャンクとして Go に送られます（APIエラーを誘発）。

## 3. より適切な代替案 (Better Alternatives)

現在の `chunkAndIngest` 内部の実装を完全に破棄し、以下の設計（Goレイヤーへの責務委譲と Overlap の導入）へ修正することを強く要求します。

### 🟢 正解のアーキテクチャ設計 (TS側は「梱包」して Go の `BatchIngest` へ投げるだけ)

1. **TS 側の役割は「配列の作成」のみに留める:**
   TS 側では LLM (`generateEpisodeSlug`) を1回も叩かない。`Message` 配列をループして文字数をカウントし、チャンクごとに分離した**オブジェクトの配列 (`BatchIngestItem[]`) をただメモリ上で組み立てるだけ**にする。

2. **コンテキスト・オーバーラップの実装:**
   境界で分割する際、前のチャンクの最後の `Message` 1〜2件を、次のチャンクの最初の要素として必ずコピー（重複）させること。これにより文脈の「のりしろ」が確保されます。

3. **Go 側の `ai.batchIngest` へ一括送信 (Fire-and-Forget):**
   組み立て終わった `BatchIngestItem[]` を、既存の `this.rpc.batchIngest(items, agentWs)` へ1回だけ送信します。
   しかも、**`await` して完了を待たず、Fire-and-Forget として非同期で投げっぱなし**にします。TS 側は即座に `processTurn` を終了し、ユーザーの UI をブロックしません。

4. **安全性の担保:**
   Go 側の `batchIngest` は既に Phase 5.4 / 5.5 で改修済みであり、Goroutine サブシステムが安全な並行処理（`gemmaLimiter` を守りつつ、ダメなら自動で MD5 フォールバック）をバックグラウンドで勝手に行ってくれます。**TS 側が知る必要はありません。**

---

## 4. 総評と要求アクション

現行の `segmenter.ts` の修正は、**「TS側での同期ループ」というNode.js環境における最大のタブー**を犯しており、さらには文脈分断によるRAG検索精度の激減を引き起こします。

**【必須アクション】**
直ちに `chunkAndIngest` 関数を以下のように書き直してください。
- `await this.rpc.generateEpisodeSlug` をループ内で呼ぶのをやめる。
- 引数から「複数の `BatchIngestItem`」を構築し、**最後に1回だけ `this.rpc.batchIngest()` を叩く**（しかも await せず非同期に投げるか、Promise.catchだけつける）。
- チャンクの分割境界に「1ターンの Overlap」を持たせるロジックを追加する。

## 5. 【アップデート】対応完了報告 (Resolved)

上記監査レポートで指摘された【絶対的なアンチパターン】について、直ちに実装を破棄し、クリーンなアーキテクチャへと全改修を実施・完了しました。

### ✅ 5.1 TS側直列ループの廃止と Fire-and-Forget の完全導入
`chunkAndIngest` 関数内の `await generateEpisodeSlug` ループを徹底排除しました。
さらに、メインの `processTurn` ハンドラ内では、チャンク関数の呼び出しから `await` を外し（`.catch` のみをアタッチ）、非同期処理としてバックグラウンドへ投げっぱなし（Fire-and-Forget）にする設計へと是正しました。これにより、巨大ダンプ発生時でも Node.js (OpenClaw UI) のイベントループは1ミリ秒もブロックされず即座にユーザーへ応答します。

### ✅ 5.2 Overlap (のりしろ) アルゴリズムの実装
単なる文字数での機械的なスライスを廃止し、「前のチャンクの最後の2メッセージ（`OVERLAP_MESSAGES = 2`）」を次のチャンクの先頭配列に必ずコピー（重複）させるスライディングウィンドウ制御を実装しました。
これにより、「質問文だけ」と「回答文だけ」でチャンクが真っ二つに分断される現象を防ぎ、RAGフェーズでの完全なセマンティック（意味的文脈）を保持します。

### ✅ 5.3 BatchIngest への責務完全移行（本来のアーキテクチャへの回帰）
分割されたすべての重なり合うチャンクを事前に `BatchIngestItem` 型のオブジェクト配列としてメモリ内で組み立て、**最後にたった1回だけ `this.rpc.batchIngest(items)` を叩いて Go サイドへ丸投げする**仕様に戻しました。
これにより、複数回の Gemini API 呼び出しにおけるエラーハンドリングや並列制御（`gemmaLimiter` など）は、すべて Go 側の強固な Goroutine サブネットが責任を持って完遂します。
