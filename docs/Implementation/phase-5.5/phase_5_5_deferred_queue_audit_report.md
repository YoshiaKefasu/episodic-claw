# Phase 5.5: API Retry & Deferred Queue アーキテクチャ精査レポート (Google Pro Engineer Audit)

## 概要
提示された `go_sidecar_api_retry_and_deferred_queue_implementation_plan.md` および `episode_generation_fix_report.md` について、システム全体の安定性、一貫性、およびベストプラクティスの観点から厳格なソースコードレベルの精査を実施しました。

結論から申し上げると、**エピソード生成バグの修正（`episode_generation_fix_report.md`）は妥当で理にかなっていますが、API リトライおよび Deferred Queue の実装プラン（`go_sidecar_api_retry_and_deferred_queue_implementation_plan.md`）には、アーキテクチャレベルでの致命的な欠陥が3点存在します。このまま実装した場合、OpenClaw 全体のプロセスハングやデータロストなどの中核的な障害を引き起こします。**

---

## 1. 潜在的な問題やリスク (🔴 Critical)

### 🔴 1.1 RPC タイムアウトの確実な誘発によるプロセス間通信の破綻
* **問題:** Go の `handleIngest` などの非同期であるべき処理の中で、同期的に `time.Sleep(60 * time.Second)` をリトライのたびに実行していますが、これは致命的です。
* **根拠:** TypeScript 側の `rpc-client.ts` の `request` メソッドのタイムアウトはデフォルトで **120,000ms (120秒)** に設定されています。Go 側で 429 エラーが起きて 60秒待機を 2回（120秒）行った時点で、**TypeScript 側は確実にタイムアウトエラーとしてリクエストを Reject し、強制終了**します。Go 側が裏で処理を継続して成功したとしても、クライアントへの応答は失われる「孤児プロセス」を意図的に生み出す設計になっています。

### 🔴 1.2 `handleSurprise` の同期ブロッキングによる OpenClaw 全体のフリーズ
* **問題:** ユーザーからメッセージを受け取るたびに発火する「軽量であるべき」 `handleSurprise` の中で `deferredQueue` の Drain（最大5件）を行っています。
* **根拠:** Drain 処理中に再び 429 エラーが発生した場合、1件のタスクにつき 60秒×2回 の同期 Sleep が発生します。これが 5件連続すると、最大 **10分間** `handleSurprise` がブロックされ応答を返さなくなります。これは Episodic-Claw プラグインだけでなく、後続の会話モデルへの通信も含め **OpenClaw (Gateway) 全体のレスポンスを完全にフリーズさせる** 地雷です。

### 🔴 1.3 インメモリキューによる永続化データの完全ロスト
* **問題:** `deferredQueue` が `var deferredQueue []DeferredIngest` として単なるメモリ上のグローバルスライスとして定義されています。
* **根拠:** Save に失敗したエピソードを救済するためのキューであるにもかかわらず、ユーザーが OpenClaw を再起動した際、あるいはプラグインがリロードされた際に、この **メモリ上のキューは完全に揮発してロスト** します。データの永続性を司るストレージプラグインとして、揮発性メモリに重要な State を持つことは許容されません。

---

## 2. 不足している手順や考慮事項

### 🟠 2.1 Exponential Backoff と Jitter の欠如
* **問題:** 429 Too Many Requests に対するリトライ戦略として、「一律 60秒」という固定値は不適切です。
* **影響:** キューに溜まった複数のタスクが同時に Drain された際、正確に60秒後に一斉にリトライを行い、ふたたび同時に 429 エラーを引き起こす **Thundering Herd（サージング問題）** が確実に再発します。

### 🟠 2.2 エラー種別の判別の欠如 (Fatal Errorへの無差別リトライ)
* **問題:** `genErr != nil` だけで無差別にリトライを行っています。
* **影響:** 429 エラーだけでなく、401 (Unauthorized) や 400 (Bad Request) のように「何度繰り返しても絶対に成功しない致命的エラー」に対しても無駄に 60 秒×2回待機し、リソースと時間を浪費します。

---

## 3. より適切な代替案 (Better Alternatives)

現在のプランを破棄し、以下のいずれかのアプローチを採用することを強く推奨します。

### 🟢 代替案A (推奨): 遅延キューを廃止し、即時 MD5 フォールバック＋非同期リネーム
複雑な Go 側のキュー管理を手放す、最も堅牢でシンプルな手法です。
1. **即時完了**: API エラー発生時は **一切 Sleep せず、即座に MD5 ハッシュ（`episode-xxxx`）でファイルを保存**してクライアントに正常応答（200 OK に相当）を返す。これでデータのロストは 100% 防げます。
2. **非同期解決**: ファイルが保存されてさえいれば、既存のバックグラウンドインデクサー（`background.go` 等）が暇なときに読み込み、「MD5のままになっているファイルを開いて、スラッグを裏で再生成してリネームする」タスクとして処理します。これなら Gateway を再起動してもファイルシステム上に情報が残ります。

### 🟡 代替案B: 堅牢な Worker Pool ＋ 永続化キュー (どうしてもキューを作る場合)
1. **Goroutine による非同期 Worker**: キューの処理は `handleSurprise` のようなエンドポイントに相乗りさせるのではなく、Go の `init()` または起動時に生成する専用の Background Ticker ゴルーチンで定期的に回すこと。（これで RPC ハンドラのブロッキングを回避）
2. **Persistent Storage (SQLite 等)**: `deferredQueue` はスライスではなく、SQLite のローカルテーブルなどに永続化し、プロセス再起動後もキューを再装填できるようにすること。
3. **Exponential Backoff**: Sleep 値は固定ではなく `(base * 2^attempt) + jitter` のアルゴリズムを導入すること。

---

## 4. 総評と次のステップ

現状の **`go_sidecar_api_retry_and_deferred_queue_implementation_plan.md` は破棄してください。** 
一時的な API レート制限に対処するために、システム全体を停止させるリスクのある「同期 Sleep」と「揮発性キュー」を導入することは、エンジニアリングとして本末転倒です。

**Next Action:**
「代替案A（即時フォールバック＋既存の非同期インデクサーの拡張）」の設計に切り替え、ファイルシステムを単一の信頼できる情報源（Single Source of Truth）として利用する方向にプランを書き直すことを要求します。
