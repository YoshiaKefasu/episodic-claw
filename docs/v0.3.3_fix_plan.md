# v0.3.3 修正プラン: ep-anchor ツールの登録とCLIモードログの重複防止

## 1. 問題の概要
1. **`ep-anchor` ツールの登録漏れ:** 
   `src/index.ts` に `ep-anchor` ツールが登録されておらず、エージェントが自発的にセッションアンカーを保存できない状態になっている。
2. **CLIモード判定ログの重複出力:** 
   OpenClawのプラグインシステムが `register()` を複数回呼び出す仕様のため、CLIコマンド実行時に `[Episodic Memory DEBUG] Starting register()...` （または `[Episodic Memory] Registering plugin...`）や `[Episodic Memory] CLI mode detected. Skipping...` といったログが無駄に連続出力されている。

## 2. 提案する修正内容と監査対策（Audit Action）

### 2.1. `src/index.ts` への `ep-anchor` ツールの追加
- ツールの引数（`anchorText` と `summaryText`）を定義する `EpAnchorSchema` を追加する。
- **【監査対策 (HIGH)】** ツールの `description` を `ep-save` と明確に区別し、LLMが通常の記憶保存（`ep-save`）とセッションの文脈維持（`ep-anchor`）を混同しないよう用途を限定する。
- `ep-save` や `ep-recall` の登録箇所に続いて、`api.registerTool(...)` で `ep-anchor` を登録する。
- `execute` メソッド内の処理:
  - **【注意点: gateway_start 前ガード】** 既存ツール（`ep-recall`, `ep-save`, `ep-expand`）と同様に、`resolveAgentWorkspaces()` 後に `!agentWs` の場合は早期リターンする。ゲートウェイ起動前にRPC発行を防ぐため。
  - エージェントのワークスペースを解決する。
  - **【監査対策 (HIGH)】** 渡された `anchorText` と `summaryText` に対して空文字チェック（バリデーション）を行い、両方が空の場合は処理を中断（Early Return）する。
  - `_singleton` 状態から `anchorStore` を取得する。
  - **【修正】 `anchorStore.write()` メソッドを呼び出し、アンカーを保存する。
    - プラン初稿では `saveAnchor()` と記載していたが、実際の `AnchorStore` クラスには `write()` が存在する。
    - シグネチャ: `async write(params: { content: string; agentWs: string; agentId: string; topics?: string[] }): Promise<AnchorWriteResult>`
    - `anchorText` と `summaryText` を結合して `content` として渡す（例: `[anchorText, summaryText].filter(Boolean).join("\n\n")`）。
    - `agentId` は `extractAgentId(ctx)` で取得可能。
    - `topics` は任意。指定しない場合 `AnchorStore.write()` 内部がデフォルトトピック `["anchor", "session-state", "compaction-bridge"]` を付与する。
  - 保存後、その直後に `invalidateRecallCacheForWorkspace` を呼び出すことで、次のリコール時に確実に最新のキャッシュが使われるようにする。
  - **【注意点: コンテンツ結合】** `anchorText` と `summaryText` を `write()` に渡す前に結合する。結合形式: `[anchorText.trim(), summaryText.trim()].filter(Boolean).join("\n\n")`。
  - **【注意点: アンカーサイズ上限】** アンカーはコンパクション後のシステムプロンプトに注入されるため、`ep-save` の3600文字制限と同様に上限を設ける（推奨: 4000文字程度）。
   - **【注意点: 挿入位置】** `ep-expand` ツール登録ブロックの直後（L1038の次行）、`catch` ブロック（L1045）の前に挿入する。

### 2.2. `register()` ログの重複出力の防止
- **【監査対策 (BLOCKER)】** プラグインが再評価・ロードされる際にファイルスコープ変数が揮発してしまう問題を回避するため、モジュールレベル（ファイル上部）ではなく、`global` オブジェクトのプロパティとしてログ出力の判定用フラグ変数を保持・追加する。
  ```typescript
  const CLI_SKIP_FLAG = Symbol.for("episodic.cli.skipped");
  const CLI_REGISTER_FLAG = Symbol.for("episodic.cli.registered");
  ```
- `register()` メソッドの先頭でこの `global` のフラグをチェックし、ログ出力を制御する。
  - デーモンプロセスではない場合（`!isDaemon`、つまりCLIモード）: `(global as any)[CLI_SKIP_FLAG]` をチェックし、初回のみスキップのログを出力してフラグを `true` にし、`return` で終了する。
  - デーモンプロセスである場合（`isDaemon`）: `(global as any)[CLI_REGISTER_FLAG]` をチェックし、初回のみ「Registering plugin...」のログを出力してフラグを `true` にし、初期化処理を続行する。

## 3. 実装手順
1. `src/index.ts` を編集し、`global` オブジェクトを利用したログ重複防止フラグを実装する（L388-398付近）。
2. `src/index.ts` を編集し、ツール登録ブロックの最後に空文字チェックを含めた `ep-anchor` ツールの定義を追加する（L1037の次行、L1039の直前 → 修正後: L1038の次行、L1045の直前）。
   - 挿入前に既存の `ep-expand` ブロック終了位置を確認
   - `gateway_start` 前ガード、コンテンツ結合、サイズ上限（4000文字）、`invalidateRecallCacheForWorkspace` 呼び出しを全て含める

### ⚠️ Impact on Related Features *(new only)*
- **LLMのツール誤用のリスク**: `ep-anchor` と `ep-save` のツール定義（description）の境界が曖昧だと、エージェントが通常の記憶保存とセッション文脈の維持（アンカー）を混同し、不必要なトークン消費や誤った記憶注入を引き起こす可能性があります。

### 🚨 Potential Problems & Risks *(new only)*
- **[BLOCKER] モジュールキャッシュ無効化によるフラグの揮発**: `src/index.ts` の既存コードには、「モジュールキャッシュが無効化されている環境では変数が毎回初期化されるため、`_singleton` を `global` オブジェクトに保持する」という修正（`BUG-1 修正`）が既に施されています。ファイルスコープのフラグ変数（`let hasSkippedForCli = false;`）というアプローチでは、プラグインが再ロードされるたびにフラグが `false` にリセットされるため、**CLIログの重複抑制が全く機能しない**という致命的な盲点があります。

### 📋 Missing Steps & Considerations *(new only)*
- **[HIGH] `anchorStore` の保存APIの不確実性**: 「`_singleton` から `anchorStore` を取得してアンカーを保存」と抽象的に書かれていますが、実際のメソッドシグネチャ（非同期呼び出しや引数の形式）の確認手順が欠落しています。 
- **[HIGH] 入力値のバリデーション**: `ep-save` には空文字チェックなどが実装されていますが、`ep-anchor` に対する空文字や極端に短い文字列のハンドリング手順がプランに記載されていません。

### 🕳️ Unaddressed Edge Cases *(new only)*
- `anchorText` と `summaryText` の両方が空でツールが呼び出された場合のエラーハンドリングや早期リターン（Early Return）が考慮されていません。
