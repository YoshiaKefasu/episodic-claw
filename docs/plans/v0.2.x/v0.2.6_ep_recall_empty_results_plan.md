# v0.2.6 個別プラン 1 — `ep-recall` 空結果の調査

## 目的
- `ep-recall` が `Nothing came back. I don't have any memories matching that.` を返した時に、同じ `agentWs` 配下の実ファイルには該当語がある、という食い違いの根を絞る。
- `ep-recall` と `prependSystemContext` を、同じ recall 契約の別経路として見直し、差分要因を分けて記録する。
- まだ修正は入れない。今回は調査結果の文書化だけに寄せる。

## すでに確認できた事実
- `Y:\\kasou_yoshia\\.openclaw\\workspace\\episodes` 配下には、`EJU特訓` を含む `.md` / `.json` が多数ある。
- `/tmp/episodic-core.log` では、問題発生時刻帯に `ai.recall` が呼ばれている。
- ローカル repo の Go 実装で、現行ストア `Y:\\kasou_yoshia\\.openclaw\\workspace\\episodes` を直接開き、`RecallWithQuery("EJU特訓", zero-vector)` を実行すると 5 件ヒットする。
- したがって、データが無いわけではない。少なくとも「ローカルのライブ store に対して検索すると見える」ことは確認済み。

## コードから確認できた事実

### `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts`
- `resolveAgentWorkspaces()` は `agentId` を解決したうえで、`agentWs = path.join(agentRoot, "episodes")` を返す。
- `ep-recall` は `resolveAgentWorkspaces()` で得た `agentWs` を使い、`k` と `topics`、`strictTopics` をそのまま `rpcClient.recall(...)` に渡している。
- `ep-save` / `compact` / `assemble()` は同じ `agentWs` を前提に動く。
- `assemble()` は recall の結果を `prependSystemContext` として返す。
- `assemble()` では `queryHash = sha1(fullQuery)` を作り、`agentId::agentWs::maxEpisodicTokens::k::queryHash` をキャッシュキーにしている。
- `ep-recall` と自動注入は、同じ store 由来でも入力の作り方が違う。`ep-recall` は明示クエリ、`assemble()` は会話メッセージから作った `fullQuery` を使う。
- `resolveAgentWorkspaces()` の workspace cache は、今は「参照元」ではなく「同期先」として扱う。`cached` と `resolved agentWs` がズレていれば警告を出し、返す値は常に現在の解決結果に揃える。

### `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts`

- つまり、自動注入は topics / strictTopics を渡さない。
- `queryHash` はここでも作っているが、用途は feedback 側の識別であり、tool 出力そのものの空振り判定とは別。

### `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\rpc-client.ts`

- `request()` は socket が無いとき、このファイルから addr を復元しようとする。
- `startWatcher()` と `recall()` は、同じ request 経路を通る。
- したがって、`ep-recall` が空でも socket の復元先が違えば、別の sidecar に投げている可能性を切り分ける必要がある。

### `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\main.go`

## `ep-recall` と `prependSystemContext` の差分要因
- 共通点:
  - どちらも最終的には recall を使う。
  - `agentWs` は同じ解決関数から来る。
- 差分:
  - `ep-recall` は人が明示的に指定した query を使う。
  - `retrieveRelevantContext()` は会話文脈から組み立てた query を使う。
  - `ep-recall` は `topics` と `strictTopics` を持ちうる。
  - `prependSystemContext` 側は topics を渡さない。
  - `ep-recall` は tool の戻り値として空かどうかが見える。
  - `prependSystemContext` 側は system prompt への注入結果としてしか見えない。

## 切り分け観点
- `ep-recall` の `agentWs` が、実ファイルを見ている `Y:\\kasou_yoshia\\.openclaw\\workspace\\episodes` と一致しているか。
- `SOCKET_ADDR_FILE` が示す addr が、今動いている sidecar と一致しているか。
- `handleRecall()` に入った `AgentWs` が、期待している workspace と同じか。
- `topics` / `strictTopics` の有無で、explicit tool だけ結果が落ちていないか。
- auto 注入側の query と explicit tool の query が、同じ語を含んでいるか。

### ⚠️ Impact on Related Features *(new only)*
- `src/index.ts:146-159` の `resolveAgentWorkspaces()` は `agentWs = path.join(agentRoot, "episodes")` を返す。`ep-recall` の空結果を調べるとき、まず見るべきなのは socket ではなく、`handleRecall()` に渡った実際の `agentWs` と `topics` / `strictTopics` の組み合わせだ。
- `src/retriever.ts:19-86` の自動注入は `topics` を送らない。一方で `ep-recall` は `topics` と `strictTopics` を持ちうる。したがって、2つは同じ「recall」でも失敗面が違う。ここを曖昧にすると、explicit tool の空結果を auto injection の問題と取り違える。
- `go\main.go:1810-1897` の `handleRecall()` は `params.Query` を受け取ったあと、`RecallWithQuery(...)` に渡す前に embedding と lexical の両系統を経由する。`Nothing came back` が出るのは result formatting の前なので、整形層は空結果の根本原因ではない。

### 🚨 Potential Problems & Risks *(new only)*
- [HIGH] この文書は `SOCKET_ADDR_FILE` を観測項目に入れているが、`ep-recall` 空結果の直接原因としては順位が高すぎる。`request()` が socket を復元できないなら通常は RPC 失敗になるので、空結果の主因候補はまず `agentWs` と `topics` / `strictTopics` の不一致だ。
- [MED] `resolveAgentWorkspaces()` には `WORKSPACE_CACHE_PREFIX` / `workspaceCachePath()` のローカルキャッシュがある。`cfgAgents` が無い場合、古い `agentWs` を再利用しうる。今の観測項目にはこのキャッシュが入っていないため、workspace mismatch を socket mismatch と誤認するリスクが残る。
- [LOW] `result 整形` を候補に残したままだが、`handleRecall()` では空結果判定が整形より前に行われる。ここを原因候補として扱うと、調査が余計に広がるだけだ。

### 📋 Missing Steps & Considerations *(new only)*
- `SOCKET_ADDR_FILE` の中身だけでは足りない。`workspaceCachePath(agentId)` の中身も確認しないと、`agentWs` の実体がずれているのに socket 側を疑い続けることになる。
- `ep-recall` の `params.Query`、`params.Topics`、`params.StrictTopics`、`params.AgentWs` をひとまとめに記録しないと、同じ query でも explicit tool だけ空になる理由を追えない。
- `prependSystemContext` は比較対象としては有効だが、auto injection 側は topics を送らない。つまり「同じ recall 契約」と言っても payload は一致していない。そこを前提にした観測順にしておかないと、比較が崩れる。

### 🕳️ Unaddressed Edge Cases *(new only)*
- `agentWs` が正しくても、`topics` が誤って付いた `strictTopics=true` の tool call なら空になりうる。`ep-recall` の空結果を store 側の不具合と決め打ちするのは危険。
- workspace キャッシュが古いと、`ep-recall` は「データはあるのに見えない」ではなく「別 workspace を見ている」だけになる。これは socket とは独立した失敗モードだ。
- `handleRecall()` の `strings.ToLower(params.Query)` は embedding 用だけで、lexical/score 以前に空結果を説明する主因ではない。そこを先に追うと遠回りになる。

### ✅ Recommended Actions
| Priority | Action | Reason | Is New? |
|----------|--------|--------|---------|
| HIGH | `handleRecall()` の実 payload を最初に記録する。`agentWs`、`query`、`topics`、`strictTopics` を1件の観測として揃える | 空結果を最短で潰すには、socket より先に request そのものを確定させるべきだから | ✅ New |
| MED | `workspaceCachePath(agentId)` の内容を観測項目に追加する | `resolveAgentWorkspaces()` のキャッシュ再利用で stale workspace を掴む可能性があるから | ✅ New |
| LOW | `result整形` は原因候補から外す | 空結果の判定は formatting より前に終わっているから | ✅ New |

### 修正方針
1. まず `handleRecall()` の request payload を固定して見る。`agentWs`、`query`、`topics`、`strictTopics` を同時に記録する。
2. 次に `workspaceCachePath(agentId)` と `SOCKET_ADDR_FILE` を比較し、どの解決層でズレたかを切る。
3. そのうえで、`topics` / `strictTopics` が空結果を作っていないかを確認する。ここは `ep-recall` と `prependSystemContext` で差分が出やすい。
4. `result整形` と `lexical.go` は後段に回す。空結果の一次原因ではない。

### 結論
このプランは方向性は合っているが、調査順の優先度がまだ広い。  
**最優先は socket ではなく、`handleRecall()` の payload と `agentWs` の実体確認**。  
その次に workspace キャッシュ、最後に lexical / 整形の切り分け、がいちばん効率がいい。
