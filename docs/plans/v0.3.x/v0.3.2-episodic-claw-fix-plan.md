# Episodic-Claw 汚染 & 生成停止 修正プラン

## 問題の確定

### 問題1: エピソードファイル汚染

**根本原因**: `processTurn()` がメッセージの `role` によるフィルタリングを一切行っていない。`toolResult` ロールのメッセージ（ツール実行結果）がそのままバッファに取り込まれ、以下の経路でエピソードに伝播する:

1. **`segmenter.ts` L103**: `currentMessages.slice(this.lastProcessedLength)` -- role 無差別に全メッセージを取得
2. **`segmenter.ts` L115-122**: dedup フィルタが `role:text` ペアで動作するが、`toolResult` を除外するロジックがない
3. **`segmenter.ts` L143-150**: surprise スコア計算用の `oldSlice`/`newSlice` にもツール出力が混入
4. **`summary-escalation.ts` L22-30**: `buildNormalSummary` が全ロールの `normalizeMessageText()` を呼ぶ
5. **`large-payload.ts` L17-37**: `extractPlainText()` もロール判別なし -- 純粋にテキスト抽出のみ

**証拠**: `plugin-runtime-environment.md` (48KB) の14行目に OpenClaw バンドル JS の `export` 列挙が丸ごと記録されている。`memory-recall-and-indexing.md` は 359KB で ep-recall の生ベクトル JSON が含まれている。

### 問題2: エピソード生成停止 (2026-04-01 以降)

**根本原因**: 汚染データが Go サイドカーの Gemini API（タイトル生成 / トピック抽出）に送られ、以下のいずれかで失敗:

- Gemini API がゴミ入力で有意なタイトルを生成できず `batchIngest` が 0 slugs を返す
- 巨大ペイロード (360KB) が RPC 経路でタイムアウトする
- 汚染コンテンツが API の safety filter に引っかかる

`episodic-core.log` で 4/1 以降の ingest 成功ログが消失しているのはこれが理由。

## 修正方針

### Fix 1: メッセージロールフィルタリングと tool_use の要約（最優先 -- 汚染の根本遮断）

**対象**: `src/segmenter.ts`

`processTurn()` の冒頭で `toolResult` ロールのメッセージを完全に除外する。これが全経路（surprise 計算、バッファ蓄積、サマリ生成、ingest）の上流にあるため、1箇所の修正で汚染を完全遮断できる。
`tool_use` (ツール呼び出しリクエスト) については、巨大な引数による汚染を防ぎつつ「どのツールを使ったか」という文脈だけを残すため、内容をツール名のみの短いテキストに変換して残す。複数ツールの並行呼び出し時はカンマ区切りで列挙する。

```typescript
// processTurn() 内、L103 の直後に追加
// ---- Fix 1: ツール出力の除外と tool_use の要約 ----
// toolResult / tool_result ロールは内部ツール実行結果であり、
// エピソードメモリに混入するとベクトル検索精度を著しく劣化させるため完全に除外。
export const EXCLUDED_ROLES = new Set(["toolResult", "tool_result"]);

const filteredNewMessages = newMessages
  .filter(m => !EXCLUDED_ROLES.has(m.role))
  .map(m => {
    if (m.role === "tool_use") {
      // 複数ツールの並列呼び出しに対応するため、すべてのツール名を抽出してカンマ区切りにする
      let toolNames: string[] = [];
      if (Array.isArray(m.content)) {
        toolNames = m.content
          .filter((b: any) => b.type === "tool_use" && b.name)
          .map((b: any) => b.name);
      }
      
      const namesStr = toolNames.length > 0 ? toolNames.join(", ") : "unknown_tool";
      return { ...m, content: `[Tool Used: ${namesStr}]` };
    }
    return m;
  });
```

以降の処理で `newMessages` の代わりに `filteredNewMessages` を使用する。
また、`EXCLUDED_ROLES` を export し、他のファイルでも利用できるようにする。

### Fix 2: extractText のサニタイゼーション（多層防御）

**対象**: `src/large-payload.ts`

`extractPlainText()` に明示的なツール出力パターン除去を追加する。Fix 1 が正しく動作すれば到達しないが、防御層として配置する。

```typescript
// extractPlainText() の戻り値に対して適用
function stripToolOutputPatterns(text: string): string {
  // "toolResult:" プレフィックスで始まるブロックを除去
  if (text.startsWith("toolResult:") || text.startsWith("tool_result:")) {
    return "";
  }
  return text;
}
```

## 既存汚染エピソードのクリーンアップ方針

**対象**: Kasouメインエージェント等の過去の汚染エピソード（3月分の `archive-*` 約838件など）

これ以降の新規生成エピソードはクリーンに保たれるため、永続的なパージアーキテクチャは実装しない。
ただし、既存の汚染データを取り除くため、**手動実行用の単発クリーンアップスクリプト** (`scripts/cleanup-polluted-episodes.ts`) を作成する。
GoサイドカーのDB、またはAPI経由で「サイズが異常に大きい」「`toolResult` が含まれている」エピソードを検索し、削除（パージ）を実行できるようにする。
