# v0.3.6 修正プラン: Cold-Start Ingestion (Day 0 Memory)

**Date:** 2026-04-07
**Severity:** MEDIUM (User Experience)
**Target Version:** v0.3.6

## 1. 問題の概要

### 問題: 初回インストール時のメモリ空白
ユーザーが既存のエージェント（例: 600kトークンの会話履歴を持つ）に初めて episodic-claw をインストールしても、**過去の会話履歴は一切メモリとして認識されない**。

プラグインは `gateway_start` 時に `.jsonl` セッションファイルを読み取らず、既存の `.md` エピソードファイルが存在しないため、HNSW/Lexical 両方のインデックスが空のまま動作を開始する。

---

## 2. 修正内容

### 2.1. 設計方針: 既存パイプラインの活用 (Option B 修正版)

**API呼び出しは必要最低限に抑えつつ、コードの重複を避けるため、既に実装済みの `processBacklogFile` ロジックを流用する。**

| 項目 | Option A (APIゼロ) | Option B (既存流用) |
|------|-------------------|-------------------|
| **API呼び出し** | 0回 | **必要最低限** (APIキーがある場合のみEmbedding) |
| **コード重複** | あり (Go側に新規実装) | **なし** (既存 `processBacklogFile` を使用) |
| **メモリ品質** | 低 (Surprise/Topicsなし) | **高** (既存ロジックにより自動計算) |
| **実装コスト** | 中 | **低** (TS側の変換のみ) |

**API制御ロジック:**
1.  **APIキーあり:** `processBacklogFile` がEmbeddingを実行し、高品質なメモリ（ベクトル+Surpriseスコア+Topics）を作成。
2.  **APIキーなし:** Embeddingをスキップし、`.md` ファイル（テキストのみ）を作成。インデックスはLexicalのみ。v0.3.6のPass 5やHealingWorkerが後でフォロー。

### 2.2. 実装詳細

**ファイル:** `src/rpc-client.ts` (TypeScript側)

Cold-Start Ingestion 用のヘルパー関数を実装。

1.  **`parseJsonlToMessages()`**: `.jsonl` セッションファイルを読み込み、`{role, content}` 配列に変換。マルチモーダル（文字列/配列の両方）に対応。
2.  **`ingestColdStartSession()`**: メッセージを50件単位でチャンク分割し、APIキーの有無に応じて2つのパスに分岐。
    *   **APIキーあり (高品質):** 一時JSONファイルを作成し、既存の `triggerBackgroundIndex` RPCを呼び出す。`processBacklogFile` がEmbeddingを実行し、高品質なメモリ（ベクトル+Surpriseスコア+Topics）を作成。
    *   **APIキーなし (ゼロAPI):** `.md` ファイルを直接作成（テキストのみ）。インデックスはLexicalのみ。API呼び出しはゼロ。
3.  **`resolveSessionFile(agentId)`**: `~/.openclaw/agents/<agentId>/sessions/` から最新の `.jsonl` セッションファイルを自動検出。**他エージェントのセッション混入を防ぐため、`agentId` で厳密に特定する。**

**ファイル:** `src/index.ts` (`gateway_start` ハンドラ内)

起動時にCold-Start Ingestionをトリガーする。

```typescript
// gateway_start ハンドラ内
(async () => {
  const { agentWs, agentId } = resolution;
  if (!agentWs || !agentId) return;
  const sessionKey = `${agentId}`; // agentIdベースで厳密に管理
  
  // 1. メモリチェック: 現在のセッションで既に実行済みか？
  if (ingestedSessions.has(sessionKey)) return;

  // 2. 永続チェック: 既に .md ファイル（年次フォルダ）が存在する場合はスキップ
  // これにより、再起動後の重複インジェスト（二重登録）を防ぐ
  try {
    if (fs.existsSync(agentWs)) {
      const entries = fs.readdirSync(agentWs, { withFileTypes: true });
      const hasExistingEpisodes = entries.some(e => e.isDirectory() && /^\d{4}$/.test(e.name));
      if (hasExistingEpisodes) {
        console.log("[Episodic Memory] Cold-Start: episodes already exist, skipping.");
        ingestedSessions.add(sessionKey);
        return;
      }
    }
  } catch { /* Ignore errors */ }

  // 3. セッションファイルチェック (agentIdで厳密に特定)
  const sessionFile = resolveSessionFile(agentId);
  if (!sessionFile) return;

  // 4. 実行 (Fire-and-forget)
  const hasApiKey = !!process.env.GEMINI_API_KEY;
  try {
    const msgCount = await ingestColdStartSession(sessionFile, agentWs, rpcClient, hasApiKey);
    console.log(`[Episodic Memory] Cold-Start: ingested ${msgCount} messages.`);
    ingestedSessions.add(sessionKey);
  } catch (err) {
    console.error("[Episodic Memory] Cold-Start ingestion failed:", err);
  }
})();
```

## 🔧 Pro Engineer Review — Round 3 — 2026-04-07
> Perspective: Google / IBM Production Engineering
> Focus: Option B (Reuse Existing Pipeline) — Pseudocode vs. Reality Audit
> Source code verified: ✅ (as of 2026-04-07)

### 📍 Assessment

**The intent is correct but the pseudocode has a fatal interface mismatch.** The plan assumes `triggerBackgroundIndex` accepts structured ingest items, but it actually accepts file paths. This is a fixable gap — not a design flaw — but it must be addressed before implementation.

### 🔍 Code Verification

| # | Item | Status | Detail |
|---|------|--------|--------|
| 1 | **`triggerBackgroundIndex` signature** | ❌ **MISMATCH** | `handleTriggerBackgroundIndex` (main.go:2123) expects `filePaths []string`, NOT `BatchIngestItem`-shaped objects. It passes paths to `ProcessBackgroundIndexing` → `processBacklogFile` which reads a JSON file from disk containing `[]BacklogMessage`. The plan's pseudocode passes inline objects — this will fail at runtime. |
| 2 | **`processBacklogFile` API key behavior** | ⚠️ **PLAN OVERCLAIM** | The plan says "APIキーなし: Embeddingをスキップし、`.md` ファイルを作成" — but `processBacklogFile` (background.go:340) calls `provider.EmbedContent()` unconditionally. If the API key is empty, the call fails and the chunk is **skipped entirely** — no `.md` file is created. The "text-only fallback" does NOT exist in `processBacklogFile`. |
| 3 | **`processBacklogFile` handles large inputs** | ✅ | 50MB file size guard (background.go:272). Chunks internally at 10 messages (background.go:290). Safe for large `.jsonl` files. |
| 4 | **Fire-and-forget safety** | ✅ | `handleTriggerBackgroundIndex` sends response immediately (main.go:2146) then spawns goroutine (main.go:2149). Will NOT block TS startup. |
| 5 | **Race with `runAutoRebuild`** | ⚠️ **LOW RISK** | `runAutoRebuild` (main.go:112) only triggers when `vstore.Count() == 0`. Cold-start via `processBacklogFile` adds records to vstore, so subsequent `getStore()` calls won't re-trigger rebuild. If rebuild runs first, it finds no `.md` files yet (cold-start hasn't written them). No collision. |
| 6 | **`.jsonl` → JSON array conversion** | ✅ | Straightforward. `.jsonl` is one JSON object per line; `processBacklogFile` expects `json.Unmarshal` of a JSON array of `{role, content}`. TS just needs to wrap lines in `[]`. |
| 7 | **Chunk size mismatch** | ⚠️ **MINOR** | Plan pseudocode uses 50-message chunks. `processBacklogFile` re-chunks at 10 internally (background.go:290). The 50-message summary becomes a single string that `processBacklogFile` treats as one `BacklogMessage` — its internal chunking won't split it further. Result: fewer, larger `.md` files than intended. Not a bug, but changes output granularity. |

### 🛤️ Final Recommendation

**Go/No-Go: ✅ GO — with two mandatory fixes before implementation.**

#### Fix 1: Use the correct interface (CRITICAL)

The TS side must write a temporary JSON file (array of `{role, content}`) and pass its path to `triggerBackgroundIndex`, NOT pass structured objects inline.

```typescript
// Correct approach:
const tempBacklog = path.join(os.tmpdir(), `cold-start-${Date.now()}.json`);
fs.writeFileSync(tempBacklog, JSON.stringify(messages, null, 0));
await rpcClient.triggerBackgroundIndex([tempBacklog], agentWs);
// Optionally clean up temp file after completion
```

#### Fix 2: Correct the "no API key" behavior claim (DOCUMENTATION)

When no API key is set, `processBacklogFile` will **fail to create any episodes** (embed fails → continue at background.go:344). This is acceptable for the "minimal API" requirement — if the user has no API key, cold-start is simply skipped. The HealingWorker won't help either (it also needs an API key for Pass 1). **Update the plan's API control logic table to reflect this reality.**

#### Fix 3 (Optional): Align chunk sizes

If the plan wants 50-message episodes, bypass `processBacklogFile` entirely and write `.md` files directly with `frontmatter.Serialize`, then call `triggerBackgroundIndex` with the `.md` file paths. This routes through `ProcessMDFileIndex` which has Smart Dedup and handles missing API keys gracefully (creates `.md` without embedding). This is the **true** "text-only fallback" path.

### ✅ Pro Recommendation

> **Choose Fix 1 (temp file) for immediate implementation.** It's the smallest change that makes the plan work. Estimated effort: 15 minutes.
>
> **Choose Fix 3 (direct `.md` write) if text-only fallback is a hard requirement.** This routes through `ProcessMDFileIndex` which already handles the no-API-key case correctly. Estimated effort: 30 minutes.
>
> **Rollback plan**: Delete `genesis-archive` tagged episodes, remove temp files, clear `meta:ingested_session` flag.
