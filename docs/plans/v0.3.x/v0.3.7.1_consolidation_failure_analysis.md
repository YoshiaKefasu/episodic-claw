## 2. 根本原因の特定

### 2.1 Consolidation トリガーの仕組み

```
startSleepTimer() ── 2分周期ticker ──▶ checkSleepThreshold()
                                            │
                                            ├─ GetRawMeta("meta:last_activity")
                                            │    ├─ err != nil → 即return（ログなし）★
                                            │    └─ len(val) == 0 → 即return（ログなし）★
                                            │
                                            ├─ lastActivity == 0 → 即return（ログなし）★
                                            │
                                            ├─ now - lastActivity <= 3h → return（ログなし）
                                            │
                                            └─ lastConsolidation < lastActivity
                                                 ├─ true → EmitLog("Sleep Timer triggered...")
                                                 │         → RunConsolidation()
                                                 └─ false → return（ログなし）
```

★ = **完全なサイレントリターン（ログ出力なし）**

### 2.2 原因: `checkSleepThreshold` のサイレントリターンがブラックホール化

**Root Cause:** `checkSleepThreshold` の**最初の3つのガード条件**がすべてログ出力なしで即returnする。ログに "Sleep Timer" の文字が一行もない = この関数が**一度も発火条件に到達していない**。

```go
func checkSleepThreshold(agentWs string, vstore *vector.Store, apiKey string) {
    val, closer, err := vstore.GetRawMeta([]byte("meta:last_activity"))
    if err != nil || len(val) == 0 {    // ← ここで落ちている可能性大
        return                          // ← ログなし
    }
    ...
}
```

### 2.3 なぜ手動ep-saveは動くのか

```
手動 ep-save → ai.consolidate RPC → handleConsolidate()
                                         │
                                         ├─ CAS(&isConsolidating, 0, 1)
                                         ├─ RunConsolidation() ← 直接呼び出し
                                         └─ SetMeta("last_consolidation", ...)

Sleep Timer → startSleepTimer() → checkSleepThreshold()
                                     │
                                     ├─ GetRawMeta("meta:last_activity") ← ここで失敗
                                     └─ return ← 到達しない
```

**手動ep-saveは `checkSleepThreshold` を経由しない。** `handleConsolidate()` が直接 `RunConsolidation()` を呼ぶため、metaストアの読み取りチェックをスキップする。

### 2.4 3つの仮説（確率順）

| # | 仮説 | 確率 | 検証方法 |
|---|---|---|---|
| **A** | `meta:last_activity` がPebbleに書き込まれていない（`handleIngest` の実行パス問題） | 60% | `SetMeta` の呼ばれる位置を確認 |
| **B** | `GetRawMeta` がエラーを返す（Pebble初期化/マイグレーション問題） | 30% | デバッグログ追加 |
| **C** | `last_activity` が常に現在時刻に近い値（3時間未満） | 10% | 01:35 ingest → 04:35 で明らかに3時間超 |

### 2.5 仮説Aの深掘り: `handleBatchIngest` の実行パス

`ai.batchIngest` は `handleBatchIngest()` で処理される。この関数が `handleIngest()` を内部で呼び、その中で `SetMeta("last_activity", ...)` が呼ばれる。

**問題の可能性:** `handleBatchIngest` はループで複数アイテムを処理する。各アイテムの `handleIngest` 内で `SetMeta` が呼ばれるが、**最後のアイテムでエラーが発生した場合、最終的な `SetMeta` が呼ばれない可能性がある。**

ただし、今回のログでは14件すべて正常に作成されている。この場合 `SetMeta` は呼ばれているはず。

### 2.6 仮説Bの深掘り: Pebble GetRawMeta の失敗

```go
// store.go:978-987
func (s *Store) GetRawMeta(key []byte) ([]byte, io.Closer, error) {
    s.mutex.RLock()
    defer s.mutex.RUnlock()
    val, closer, err := s.db.Get(key)  // ← key = "meta:last_activity"
    return val, closer, err
}
```

Pebbleの `db.Get()` がキーを見つけられない場合、`pebble.ErrNotFound` を返す。この場合 `err != nil` となり、`checkSleepThreshold` はサイレントリターンする。

**SetMeta は正常に動作している可能性が高い**（`db.Set()` は成功）。問題は **読み取り側** の可能性がある。

## 3. 修正プラン

### 3.1 Diagnostic Logging の追加（最優先）

**ファイル:** `go/main.go` — `checkSleepThreshold()`

原因追及と運用可観測性のため、3箇所のサイレントリターン（エラー系）に `EmitLog` を追加する。正常系の no-op リターン（3時間未満の待機時など）にはログは不要。

```go
func checkSleepThreshold(agentWs string, vstore *vector.Store, apiKey string) {
    val, closer, err := vstore.GetRawMeta([]byte("meta:last_activity"))
    if err != nil {
        EmitLog("[SleepTimer] WARN: failed to read last_activity for %s: %v", agentWs, err)
        return
    }
    if closer != nil {
        defer closer.Close()
    }
    if len(val) == 0 {
        EmitLog("[SleepTimer] WARN: last_activity is empty for %s", agentWs)
        return
    }

    var lastActivity int64
    fmt.Sscanf(string(val), "%d", &lastActivity)

    if lastActivity == 0 {
        EmitLog("[SleepTimer] WARN: last_activity is zero for %s", agentWs)
        return
    }

    idleSeconds := time.Now().Unix() - lastActivity
    if idleSeconds <= 3*3600 {
        // Not yet idle enough — normal, no log needed
        return
    }

    // 3 hours threshold crossed — always log
    EmitLog("[SleepTimer] Idle for %d seconds (%.1fh), checking consolidation for %s",
        idleSeconds, float64(idleSeconds)/3600.0, agentWs)

    // ... rest of consolidation logic
}
```

### 3.2 `last_activity` 設定の堅牢化

**ファイル:** `go/main.go` — `handleIngest()`

`SetMeta("last_activity", ...)` を **必ず実行する** ように、defer 或者は try-finally 的に配置。

```go
// Before: SetMeta only in success path
if embedding succeeded {
    vstore.Add(...)
    vstore.SetMeta("last_activity", ...)
}

// After: Always update last_activity regardless of embed result
defer func() {
    _ = vstore.SetMeta("last_activity", []byte(fmt.Sprintf("%d", time.Now().Unix())))
}()
```

### 3.3 `handleBatchIngest` での `last_activity` 保証

`handleBatchIngest` の最後でも明示的に `SetMeta` を呼ぶ:

```go
func handleBatchIngest(conn net.Conn, req RPCRequest) {
    // ... batch loop ...

    // Ensure last_activity is set even if individual items failed
    _ = vstore.SetMeta("last_activity", []byte(fmt.Sprintf("%d", time.Now().Unix())))

    sendResponse(...)
}
```

### 3.4 Pebble メタストア初期化の確認

gateway_start / watcher.start 時に、`meta:last_activity` キーが存在しない場合は初回値を設定する。
