# v0.3.5.1 修正プラン: 結果ベースの再注入防止（Turn-Based Guard）

**Date:** 2026-04-06
**Severity:** HIGH
**Target Version:** v0.3.5.1 (hotfix)

## 1. 問題の概要

### 問題: 同一メモリブロックの毎ターン再注入

`before_prompt_build` フックは毎ターン `retrieveRelevantContext()` を呼び出し、取得したメモリを `prependSystemContext` として注入している。同じトピックの会話が数ターン続くと、**同じメモリブロックが毎ターン注入され続ける**。

**影響:**
- トークン浪費（同じテキストが毎ターンシステムプロンプトに入る）
- 遅延（毎ターンGoサイドカーへのRPC呼び出し）
- LLMの混乱（古いコンテキストが繰り返し提示される）

### 死にコードの存在

`src/index.ts` に以下の機構が定義されているが、**一度も使用されていない**:
- `state.recallCache` (L87-91): キャッシュ状態
- `RECALL_DEBOUNCE_MS` (L448): 5000ms デバウンス定数
- `clearRecallCache()` (L491): キャッシュクリア関数

## 2. 修正内容

### 2.1. 設計方針: ターン数ベースの結果ガード（Option C 採用）

**時間ベースのデバウンスではなく、ターン数（メッセージ数）ベースのガードを採用する。**

| 項目 | 時間ベースデバウンス（却下） | ターン数ベースガード（採用） |
|------|---------------------------|--------------------------|
| 比較対象 | 経過時間（分/秒） | 経過ターン数（メッセージ数） |
| RPC呼び出し | 毎ターン実行 | 毎ターン実行（検索結果の変化を検知） |
| 注入判定 | 時間経過で再注入 | ターン数経過で再注入 |
| ユーザー制御 | `recallReInjectionCooldownMinutes` | 固定値（推奨: 10ターン） |
| 信頼性 | ⚠️ 低い（チャット速度に依存） | ✅ 高い（LLMのコンテキスト消費に直結） |

**理由:** LLMのコンテキストウィンドウは「時間」ではなく「メッセージのやり取り（Turns）」で消費される。時間ベースでは、高速チャット時に記憶が消失したり、放置後に重複注入されたりするリスクがある。ターン数ベースであれば、LLMがどれだけ文脈を消費したかに基づいて正確にガードできる。

**Option D（サーバーサイド除外）について:**
技術的には優れているが、RPCインターフェースの変更コストが高く、今回のホットフィックスには過剰であるため、`docs/future_consideration_plan.md` に記載し将来の検討課題とする。

### 2.2. 定数の定義

**ファイル:** `src/index.ts`

```typescript
// 同一メモリ結果の再注入を防止するためのクールダウンターン数
// 10ターン = 約20往復の会話。LLMが十分な文脈を消費した後に再評価する。
const RECALL_REINJECTION_COOLDOWN_TURNS = 10;
```

### 2.4. `before_prompt_build` フックの修正

**ファイル:** `src/index.ts`
**位置:** `retrieveRelevantContext()` 呼び出し後

```typescript
const recallOutcome = await retriever.retrieveRelevantContext(msgs, agentWs, k, maxRecallTokens);

// 結果ハッシュの計算（エピソードIDのソート済み組み合わせ）
const resultHash = recallOutcome.episodes
  .map((ep: any) => ep.id || "")
  .filter(Boolean)
  .sort()
  .join("|");
const currentHash = resultHash ? createHash("sha1").update(resultHash).digest("hex") : "";

const currentMsgCount = msgs.length;
const turnsSinceLastInjection = currentMsgCount - state.lastInjectionMessageCount;
const isSameResult = currentHash && currentHash === state.lastInjectedResultHash;
const isWithinCooldown = turnsSinceLastInjection < RECALL_REINJECTION_COOLDOWN_TURNS;

if (isSameResult && isWithinCooldown) {
  console.log(`[Episodic Memory] recall re-injection guard: same ${recallOutcome.episodes.length} episode(s), skipping (${turnsSinceLastInjection} turns since last injection)`);
  const prependSystemContext = anchorPrependText.trim();
  return prependSystemContext ? { prependSystemContext } : {};
}

// 注入実行
if (currentHash) {
  state.lastInjectedResultHash = currentHash;
  state.lastInjectionMessageCount = currentMsgCount;
}
```

### 2.5. `ingest()` での状態リセット

新しいエピソードが保存された場合、再注入ガードをリセットして最新のメモリを即座に反映できるようにする。

```typescript
// ingest() 内
state.lastInjectedResultHash = "";
state.lastInjectionMessageCount = 0;
```

### 🎯 Core Problem (1 sentence)
> 取得されたメモリの結果が同じでも毎ターン注入されるため、同じ話題が続く間にトークン浪費とLLM混乱が発生する。

### 🔍 Principle Filter
| Check | Result | Note |
|-------|--------|------|
| YAGNI — Is this actually needed now? | ✅ Yes | 毎ターン同一注入は明らかな無駄 |
| KISS — Is there a simpler solution? | ✅ Simple | エピソードIDのハッシュ比較 + ターン数チェックのみ |
| DRY — Any duplication to eliminate? | ✅ None | 単一のガードロジック |
| SOLID — Any violation causing real problems? | ✅ None | 単一責任は保たれている |

### 🛤️ Solution Options

#### Option A — 結果ベース再注入防止 (時間依存)
**Approach**: エピソードIDの組み合わせハッシュを計算し、クールダウン期間内かつ同一結果なら注入をスキップ。
**Implementation cost**: Low (~30行追加)
**Risk**: Medium
**Why rejected**: 「時間」をベースにしたデバウンスは、LLMのコンテキストウィンドウの消費（メッセージ数）と現実時間が無関係であるというアーキテクチャのレイヤー違反を引き起こす。高速でチャットすると記憶喪失になり、長期間放置すると重複注入されるため。

#### Option B — クエリベースデバウンス（却下済み）
**Approach**: 入力クエリのハッシュでデバウンス
**Why rejected**: クエリが微妙に変化しても結果が同じ場合、5秒ごとに再注入される。根本問題を解決できない。

#### Option C — ターン数 / コンテキストベースのガード *(Pro 推奨・採用)*
**Approach**: 時間（Minutes）の代わりに、「前回の注入から何ターン（メッセージの配列長）経過したか」で同一結果のスキップ判定を行う。
**Implementation cost**: Low
**Risk**: Very Low
**Why recommended**: LLMの記憶は「メッセージのやり取り（Turns）」で消費されるため、ガードもメッセージ数を基準にするのが最も堅牢。「時間」に依存したバグが消滅し、新たな設定項目を増やす必要（YAGNI違反）もなくなる。

#### Option D — ハイブリッド（RPCデバウンス + 結果ベース注入防止）
**Approach**: レイヤー1で同一クエリが5秒以内ならRPCをスキップし、レイヤー2で同一エピソードかつターン数基準を満たさなければ注入をスキップする2段構え。
**Why rejected (YAGNI/KISS)**: RPC呼び出しの節約（レイヤー1）は、Goサイドカーがローカルプロセスである現状では数msの利益しかなく、最適化の時期尚早（Premature Optimization）。2つの異なるレイヤーのキャッシュ状態を管理・クリアする複雑さ（KISS違反）が、得られる利益を上回る。

#### Option D — サーバーサイド除外フィルタリング (Exclusion-based Recall)
**Approach**: TypeScript側が直近注入したエピソードIDを保持し、RPCコール時に `excludeIds` としてGo側に渡し、Go側で検索結果から弾くアプローチ。
**Implementation cost**: High（RPCインターフェースの変更、Go側とTypeScript側の両方の修正が必要）
**Risk**: High
**Why rejected**: 解決したい問題（トークン浪費とLLM混乱）に対して、通信プロトコル（RPC層）とバックエンド（Go層）にまで手を入れるのは明らかに大掛かりすぎる。純粋なフロント（TypeScript）側のフィルタリング（Option C）で100%解決できる問題に対し、システム全体に変更を波及させるのはSOLID原則（関心の分離）およびKISS原則に違反する。

### ✅ Pro Recommendation
> **Choose Option C because**: LLMのプロンプト構築ロジックにおいて、時間（Time）に依存したデバウンスは状態の不一致（Race Conditionや記憶喪失）を生む最大の原因になります。設定項目を追加する手間を省きつつ、どんなチャットのペースにも完璧に追従する堅牢なガードになります。
> サーバーサイド除外（Option D）は技術的には美しいですが、「トークン消費を抑える」という目的のためだけにRPCインターフェースを変更するのはやりすぎ（YAGNI/KISS違反）です。TypeScript側でのフィルタリング（Option C）で十分に目的は達成されます。
