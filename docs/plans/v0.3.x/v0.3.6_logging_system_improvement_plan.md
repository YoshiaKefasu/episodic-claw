# v0.3.6 修正プラン: ロギングシステムの整理と自動クリーンアップ（Option C）

**Date:** 2026-04-07
**Severity:** MEDIUM
**Target Version:** v0.3.6

## 1. 問題の概要

### 問題1: 単一ログファイルへの無制限書き込み
現在、Goサイドカーは `os.TempDir()/episodic-core.log` に**1つのファイル**へ全ログを永久に追記している。ローテーションもサイズ制限もないため、長期間運用するとファイルが肥大化する。

### 問題2: 構造化ログが不完全
`EmitLog()` を通したログのみがJSONファイルに書き込まれる。`go/internal/` パッケージ内の `fmt.Fprintf(os.Stderr, ...)` や `log.Printf(...)` は**stderrにのみ出力され、ログファイルには記録されない**。

### 問題3: ログレベルの欠如
全てのJSONログエントリが `"level": "info"` にハードコードされている。warn/error/debug の区別がない。

### 問題4: 日付による整理がない
全ての日付のログが1つのファイルに混在しており、特定の日のログを探すのが困難。

---

## 2. OpenClaw プラグインログシステムとの相互性

### 2.1. OpenClawのログアーキテクチャ

OpenClawはプラグインを**インプロセス**（`jiti` 経由の動的インポート）で読み込む。子プロセスとして実行するわけではない。

| 項目 | OpenClawの仕様 |
|------|---------------|
| **ログ形式** | JSON Lines (JSONL) — 1行1JSONオブジェクト |
| **ログファイル** | `/tmp/openclaw/openclaw-YYYY-MM-DD.log`（日次ローリング） |
| **サイズ上限** | 500 MB（デフォルト、`logging.maxFileBytes` で変更可） |
| **ログレベル** | `trace`, `debug`, `info`, `warn`, `error`, `fatal`, `silent` |
| **プラグイン用SDK** | `api.runtime.logging.getChildLogger({ plugin: "episodic-claw" })` |
| **RuntimeLogger インターフェース** | `{ debug?, info, warn, error }` — 各メソッドは `(message: string, meta?: Record<string, unknown>) => void` |
| **ログ保存先** | プラグイン固有のファイルはない。メインのOpenClawログファイルに統合 |
| **ログローテーション** | 日次ローリング + 24時間経過した旧ファイル自動削除 |

### 2.2. Goサイドカーの位置づけ

episodic-clawのGoサイドカーは**子プロセス**としてTypeScriptプラグインからspawnされる。OpenClawのプラグインローダーからは見えない存在。

```
OpenClaw (インプロセス)
  └─ TypeScriptプラグイン (src/index.ts)
       └─ Goサイドカー (子プロセス: episodic-core)
            ├─ stderr → TypeScriptの console.warn に流れる
            └─ 独自ログファイル → os.TempDir()/episodic-core.log（現在）
```

**重要な制約:**
- GoサイドカーのログはOpenClawのログファイルに**自動的には統合されない**
- Goサイドカーのstderr出力はTypeScript側で `console.warn` として再出力され、OpenClawのログシステムに流れる
- Goサイドカーが独自にファイルに書き込むログは、OpenClawの管理外

### 2.3. 相互性に基づく設計判断

| 判断 | 理由 |
|------|------|
| **Goサイドカーは独自ログファイルを維持** | OpenClawのログシステムはインプロセス前提。子プロセスのGoサイドカーから直接利用するのは不可能 |
| **日次ローリング + カテゴリJSONフィールド** | OpenClawと同じパターン（`YYYY-MM-DD.log`）に揃えることで、ユーザーがログを探す際の認知負荷を低減 |
| **3日間保持** | OpenClawの24時間保持より長めに。Goサイドカーは子プロセスなので、クラッシュ時のデバッグに古いログが必要な場合がある |
| **stderr出力は継続** | TypeScript側の `console.warn` パイプラインを維持するため。OpenClawのログにもこの経路で到達する |

## 3. 修正内容（Option C: 単一日次ファイル + カテゴリJSONフィールド）

### 3.1. 新しいディレクトリ構造

```
os.TempDir()/episodic-claw/
├── 2026-04-06.log    # 全カテゴリのログ（JSONL形式）
├── 2026-04-07.log
├── 2026-04-08.log
└── ...（3日間保持、4日目は自動削除）
```

各ログ行はJSONオブジェクト:
```json
{"timestamp":"2026-04-06T12:00:00Z","level":"info","category":"core","message":"Starting Go Sidecar..."}
{"timestamp":"2026-04-06T12:00:01Z","level":"warn","category":"store","message":"Vector store initialized with 0 vectors"}
{"timestamp":"2026-04-06T12:00:02Z","level":"error","category":"ai","message":"Embedding API returned 429"}
```

### 3.2. Go側: 単一日次ロガーの実装

**ファイル:** `go/internal/logger/logger.go`（新規作成）

```go
package logger

import (
	"encoding/json"
	"fmt"
	"os"
	"path/filepath"
	"sync"
	"time"
)

type LogLevel string

const (
	LevelDebug LogLevel = "debug"
	LevelInfo  LogLevel = "info"
	LevelWarn  LogLevel = "warn"
	LevelError LogLevel = "error"
)

type LogCategory string

const (
	CatCore          LogCategory = "core"
	CatBackground    LogCategory = "background"
	CatConsolidation LogCategory = "consolidation"
	CatStore         LogCategory = "store"
	CatLexical       LogCategory = "lexical"
	CatWatcher       LogCategory = "watcher"
	CatAI            LogCategory = "ai"
)

var (
	mu        sync.Mutex
	logFile   *os.File
	encoder   *json.Encoder
	currentDate string
	logDir    string
)

type LogEntry struct {
	Timestamp string      `json:"timestamp"`
	Level     LogLevel    `json:"level"`
	Category  LogCategory `json:"category"`
	Message   string      `json:"message"`
}

// Init sets up the daily log file and returns the base directory path
func Init() string {
	logDir = filepath.Join(os.TempDir(), "episodic-claw")
	os.MkdirAll(logDir, 0755)
	rotateIfNeeded()
	return logDir
}

func rotateIfNeeded() {
	today := time.Now().Format("2006-01-02")
	if today == currentDate && logFile != nil {
		return
	}

	// Close old file
	if logFile != nil {
		logFile.Close()
		logFile = nil
		encoder = nil
	}

	currentDate = today
	path := filepath.Join(logDir, today+".log")
	f, err := os.OpenFile(path, os.O_APPEND|os.O_CREATE|os.O_WRONLY, 0644)
	if err != nil {
		fmt.Fprintf(os.Stderr, "[logger] Failed to open log file %s: %v\n", path, err)
		return
	}
	logFile = f
	encoder = json.NewEncoder(f)

	// Cleanup old directories（日次ファイルなのでディレクトリはないが、旧形式の互換性のため残す）
	cleanupOldFiles(logDir, 3)
}

func cleanupOldFiles(dir string, keepDays int) {
	entries, err := os.ReadDir(dir)
	if err != nil {
		return
	}
	cutoff := time.Now().AddDate(0, 0, -keepDays)
	for _, e := range entries {
		if e.IsDir() {
			// 旧形式のディレクトリ（YYYY-MM-DD/）も削除
			dirDate, err := time.Parse("2006-01-02", e.Name())
			if err == nil && dirDate.Before(cutoff) {
				os.RemoveAll(filepath.Join(dir, e.Name()))
			}
			continue
		}
		if filepath.Ext(e.Name()) != ".log" {
			continue
		}
		name := e.Name()[:len(e.Name())-4] // remove .log
		fileDate, err := time.Parse("2006-01-02", name)
		if err != nil {
			continue
		}
		if fileDate.Before(cutoff) {
			os.Remove(filepath.Join(dir, e.Name()))
		}
	}
}

// Emit writes a log entry to the daily log file and stderr
func Emit(level LogLevel, cat LogCategory, format string, a ...interface{}) {
	msg := fmt.Sprintf(format, a...)
	ts := time.Now().Format(time.RFC3339)

	// Always write to stderr (captured by TypeScript plugin)
	prefix := fmt.Sprintf("[%s] [%s] %s", cat, level, msg)
	fmt.Fprintln(os.Stderr, prefix)

	// Write to daily log file
	mu.Lock()
	defer mu.Unlock()

	rotateIfNeeded()
	if encoder == nil {
		return
	}
	encoder.Encode(LogEntry{
		Timestamp: ts,
		Level:     level,
		Category:  cat,
		Message:   msg,
	})
}

// Convenience wrappers
func Info(cat LogCategory, format string, a ...interface{})  { Emit(LevelInfo, cat, format, a...) }
func Warn(cat LogCategory, format string, a ...interface{})  { Emit(LevelWarn, cat, format, a...) }
func Error(cat LogCategory, format string, a ...interface{}) { Emit(LevelError, cat, format, a...) }
func Debug(cat LogCategory, format string, a ...interface{}) { Emit(LevelDebug, cat, format, a...) }
```

### 3.4. 自動クリーンアップ（3日間保持）

`cleanupOldFiles()` が日付ローテート時に自動的に呼び出される:
- 現在日より3日以上前の `.log` ファイル（例: `2026-04-03.log`）を削除

- 初回呼び出しは `Init()` 時、その後は日付が変わるたびに実行

## 🔧 Pro Engineer Review — Round 2 — 2026-04-07
> Perspective: Google / IBM Production Engineering
> Focus: Option C Design & OpenClaw Compatibility
> Source code verified: ✅ (as of 2026-04-07)

### 📍 Design Validation (Option C vs. OpenClaw Constraints)

**✅ Option C is sound.** A single daily JSONL file (`YYYY-MM-DD.log`) with a `"category"` JSON field is the right call for this codebase:

1. **OpenClaw alignment**: The `os.TempDir()/episodic-claw/YYYY-MM-DD.log` pattern mirrors OpenClaw's own `/tmp/openclaw/openclaw-YYYY-MM-DD.log`. Users looking for logs will intuitively check the same temp directory.
2. **stderr pipeline is correct**: `rpc-client.ts:229` captures Go stderr via `this.child.stderr.on("data", ...)` and pipes it to `console.warn(...)`. This means every Go log line reaches OpenClaw's log system indirectly. The proposed stderr format (`[Cat] [Level] msg`) is compatible — `console.warn` accepts arbitrary strings.
3. **3-day retention > OpenClaw's 24h**: Appropriate tradeoff. Go sidecar crashes are harder to debug without historical logs, and the sidecar is ephemeral (spawns per session).
4. **JSONL file + category field vs. 7 files**: Eliminates file descriptor overhead, simplifies cleanup, and keeps `grep`/`jq` usable. The category field is queryable without file-level partitioning.

**⚠️ One gap in the stderr format**: The proposed `[Cat] [Level] msg` uses space-separated brackets. If OpenClaw ever adds structured log parsing for plugin stderr, this format is parseable but not self-describing (no timestamp, no JSON). This is acceptable for now since stderr goes through `console.warn` which is unstructured anyway. No action needed — just a documented limitation.

### 🔍 Code Review of Proposed `logger.go`

After verifying against the actual codebase, here are the specific issues in the proposed `logger.go`:

#### BUG 1 — Race condition in `rotateIfNeeded()` (MUST FIX)
**Severity**: HIGH

`rotateIfNeeded()` is called from two contexts:
- `Init()` — single-threaded at startup (safe)
- `Emit()` — under `mu.Lock()` (safe)

However, `Init()` calls `rotateIfNeeded()` **without holding `mu`**. If `Init()` and `Emit()` race during startup (e.g., a background goroutine logs before `Init()` completes), `currentDate` and `logFile` can be corrupted.

**Fix**: `Init()` must acquire `mu` before calling `rotateIfNeeded()`:
```go
func Init() string {
    logDir = filepath.Join(os.TempDir(), "episodic-claw")
    os.MkdirAll(logDir, 0755)
    mu.Lock()
    rotateIfNeeded()
    mu.Unlock()
    return logDir
}
```

#### BUG 2 — `os.OpenFile` failure causes silent log loss (MUST FIX)
**Severity**: HIGH

If `os.OpenFile` fails in `rotateIfNeeded()`, the function returns early with `encoder = nil`. All subsequent `Emit()` calls will silently drop file writes (the `if encoder == nil { return }` guard). The stderr fallback exists, but the JSONL file will never be created — even after the transient error clears.

**Fix**: Set a retry flag or clear `currentDate` on failure so the next `Emit()` re-attempts rotation:
```go
if err != nil {
    fmt.Fprintf(os.Stderr, "[logger] Failed to open log file %s: %v\n", path, err)
    currentDate = "" // Force retry on next Emit
    return
}
```

#### BUG 3 — `json.Encoder.Encode()` can write partial lines (LOW)
**Severity**: LOW

`json.Encoder.Encode()` is not atomic. If the process crashes mid-write, the JSONL file can contain a truncated line. This is inherent to any logging system and not a blocker, but worth noting. The `os.O_APPEND` flag ensures writes go to EOF, and Go's stdlib `json.Encoder` writes in a single syscall for small objects, so the risk is minimal.

#### BUG 4 — `cleanupOldFiles` runs on every date change (NIT)
**Severity**: LOW

`cleanupOldFiles()` is called inside `rotateIfNeeded()`, which runs on every `Emit()` call (under lock). The `today == currentDate` guard prevents repeated cleanup within the same day, but the first log after midnight will block all other loggers while scanning the directory. For a temp dir with 3-5 files this is negligible, but if the directory accumulates non-log files, it could add latency.

**Fix**: Not urgent. If it becomes an issue, move cleanup to a background goroutine or run it only once per process startup.

#### BUG 5 — `fmt.Printf` on stdout in `store.go:264` (MISSED IN PLAN)
**Severity**: MEDIUM

The plan correctly identified `store.go:264` as a `fmt.Printf` (stdout) call, but did not include it in the migration map. This line:
```go
fmt.Printf("[Episodic-Core] Vector store initialized: loaded %d vectors into HNSW\n", count)
```
goes to **stdout**, not stderr. It will NOT be captured by `rpc-client.ts` (which only captures stderr at line 229). It will also NOT appear in OpenClaw's logs.

**Fix**: Add `CatStore` migration for this line. Convert to `logger.Info(logger.CatStore, "Vector store initialized: loaded %d vectors into HNSW", count)`.

#### BUG 6 — `log.Printf` calls in `store.go` and `lexical.go` (MISSED IN PLAN)
**Severity**: MEDIUM

The plan mentions replacing `fmt.Fprintf` / `log.Printf` but the migration map only lists stderr prefixes. There are **21 `log.Printf` calls** in `store.go` and `lexical.go` that use the standard library logger (which writes to stderr by default). These ARE captured by `rpc-client.ts`, but they won't get JSONL file entries or category tags unless migrated.

Affected files:

#### BUG 7 — Total log site count is higher than stated
**Severity**: LOW

The plan says "約50+" log calls. Actual count from source verification:

**Go, with 3 critical pre-implementation fixes:**

1. **Fix `Init()` race** — wrap `rotateIfNeeded()` in `mu.Lock()`/`mu.Unlock()` in `Init()`
