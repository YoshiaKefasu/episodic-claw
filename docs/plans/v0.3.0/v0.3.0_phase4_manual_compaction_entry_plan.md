# v0.3.0 Phase 4 プラン - manual compaction entry を整理する

- 状態: 収束済み（`/compact` 経路を正本化、`/write-anchor` は非採用で確定）

## 目的
- compaction 起点の Anchor flow にだけ関係する手動経路を整理する。
- `ep-save` のような manual save 入口とは混ぜない。

## このフェーズで決めること
- `/compact` を入れるか
- それぞれを chat command と tool のどちらで受けるか
- 内部 service をどこまで共通化するか

## 前提
- manual save の入口整理は別件であり、このフェーズには含めない。
- ここで扱う手動経路は「compaction を明示発火したい」「Anchor だけ先に作りたい」という用途だけ。

## 実装タスク
- `/compact` の入口案を決める。
- `/write-anchor` を非採用として明記する。
- `ep-save` とは責務境界を文書で明記する。

## 設計方針
- 手動経路は compaction flow の補助に留める。
- save flow と共通化しすぎない。
- UI や host API の制約に合わせて、command か tool かを最小追加で決める。

## 受け入れ条件
- `/compact` の設計が compaction flow と整合する。
- `/write-anchor` を実装しない判断理由が明記される。
- `ep-save` と責務が混ざらない。
- 必要な場合にだけ手動入口を追加する方針が定まる。

## 非対象
- `ep-save` の仕様変更
- manual registration の正本化
- semantic / lexical 更新
- compaction 後の注入寿命

## Research Addendum — 2026-04-03

### Findings
- `/compact` は OpenClaw host に**すでにある native chat command**です。`commands-registry.shared.ts` で `textAlias: "/compact"` として登録され、reply command handler は `handleCompactCommand` です。  
  根拠: `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\auto-reply\commands-registry.shared.ts:619-633`, `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\auto-reply\reply\commands-compact.ts:79-177`
- `handleCompactCommand` は `/compact` メッセージを受けると、既存の embedded run を止めたうえで `compactEmbeddedPiSession(...)` を呼びます。plugin 側の slash command ではなく、**host の built-in compaction path** が先に走ります。  
  根拠: `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\auto-reply\reply\commands-compact.ts:98-143`
- `compactEmbeddedPiSession(...)` は、`contextEngine.info.ownsCompaction === true` のとき `contextEngine.compact(...)` を直接呼びます。その前後で `before_compaction` / `after_compaction` を発火しますが、hook event は通知用で、payload carrier ではありません。  
  根拠: `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\agents\pi-embedded-runner\compact.ts:1036-1145`, `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\plugins\hooks.ts:567-583`
- `ContextEngine.compact(...)` の public contract には `force`, `currentTokenCount`, `compactionTarget`, `customInstructions`, `runtimeContext` が含まれます。manual `/compact` のとき host は `force: true` と `compactionTarget: "threshold"` を渡します。  
  根拠: `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\context-engine\types.ts:194-208`, `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\agents\pi-embedded-runner\compact.ts:1101-1111`
- `episodic-claw` はすでに `registerContextEngine("episodic-claw", ...)` を持ち、`info.ownsCompaction = true` と `compact(ctx)` を実装しています。つまり host の `/compact` は、現状でも plugin の `compact()` へ着地します。  
  根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:577-585`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:773-790`
- OpenClaw plugin API には `registerCommand(...)` も存在しますが、plugin command は「built-in commands より前」に処理されます。`/compact` と同名 command を plugin 側で登録すると、責務衝突や二重経路の原因になります。  
  根拠: `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\plugins\types.ts:1951-1957`, `D:\GitHub\OpenClaw Related Repos\openclaw-source\src\plugins\registry.ts:823-876`

### `/compact` を plugin 側で扱う場合の制約
- host には native `/compact` があるので、plugin が同名 slash command を新設すると command ownership が二重になります。
- `after_compaction` は通知用 hook であり、manual entry の payload や custom result を運ぶ場所には使えません。
- plugin 側で触れる最小境界は、**既存 host `/compact` が最終的に呼ぶ `contextEngine.compact()` の中身**です。

### 二重実行や責務衝突を避ける方針
- Phase 4 では plugin 独自の `/compact` slash command は追加しない。
- manual compaction entry は、まず **host built-in `/compact` → `compactEmbeddedPiSession(...)` → `contextEngine.compact()`** の既存経路を正本として扱う。
- plugin 側で新しい入口を足す場合でも、`/compact` と同名にせず、host command と競合しない別 surface か、将来の別トラックへ分離する。

### 修正や実装方針
- Phase 4 の最小実装案は、**新しい slash command を増やさず**、既存 host `/compact` が `episodic-claw` の `compact()` を呼ぶ経路を前提に、plugin 側では `compact(ctx)` とその結果処理だけを整えることです。
- もし manual compaction 専用の plugin surface が必要になっても、まず検討すべきは host `/compact` の customInstructions を活かす方向で、plugin 側 duplicate command を作らないことです。
- `before_compaction` / `after_compaction` は観測・通知に留め、manual compaction の主制御フローは `compact()` 本体に集約するのが最小です。

### 非対象
- plugin 独自の `/compact` slash command 追加
- `/write-anchor` の実装（v0.3.0 では非採用）
- `ep-save` / manual save との共通化
- compaction 以外の tool / command surface 拡張

### Resolution
- `/compact` は host native command として既に存在し、`episodic-claw` は context engine `compact()` でその終点に接続済みです。
- したがって、Phase 4 で最小に収めるなら「plugin 側で `/compact` を再実装しない」が最重要方針です。
- 実装では host の `/compact` 正規経路をそのまま使い、plugin 側では `compact()` 入口の観測ログと検証だけを追加しました。独自 slash command は追加していません。
- `/write-anchor` は、compaction ごとに内部で Anchor を生成する現在方針と重複し、既存正規経路もないため **v0.3.0 では非採用** とします。
- この前提で、Phase 4 は `/compact` 正本化と command ownership 境界の固定まで **収束済み** とします。
