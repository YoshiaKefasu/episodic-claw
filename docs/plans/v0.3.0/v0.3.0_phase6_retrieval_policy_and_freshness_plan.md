# v0.3.0 Phase 6A プラン - retrieval policy の最小改善（候補不足 backfill）

- 状態: 収束済み（v0.3.0 スコープ）

## 目的
- lexical-first の候補取得で recall を取り逃がすリスクを下げる。
- 非同期 lexical 更新を維持しつつ、鮮度ギャップの実害を軽くする。
- Phase 5 で可視化した diagnostics を前提に、検索 policy を別差分で改善する。
- v0.3.0 では最小差分（6A）のみを確実に入れて閉じる。

## このフェーズの進め方（確定）
- Phase 6A: `len(candidates) < N` のときだけ HNSW 候補を backfill する。
- v0.3.0 では、最初から常時並列へ飛ばず、`候補不足時だけ補完する` 最小差分を先に入れる。
- 6B（常時並列 hybrid / RRF / freshness 追加 / top-level diagnostics 拡張）は v0.3.0 から分離し、保留プランで管理する。
- 保留プラン: [v0.3.X_phase6b_deferred_plan.md](/D:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/docs/v0.3.X_phase6b_deferred_plan.md)

## このフェーズで決めること
- `len(candidates) < N` の段階的 backfill を採るか
- lexical 0 件時の既存 HNSW fallback を維持するか
- `embed_fallback_lexical_only` 中の抑止を入れるか

## 現行コードとの関係
- [go/internal/vector/store.go](/D:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/go/internal/vector/store.go) は lexical hit が 1 件以上あると HNSW 候補取得へ進まない lexical-first 構造を持つ。
- 逆に lexical hit が 0 件なら、現行コードでも `len(candidates) == 0` 条件で HNSW semantic search に落ちる。
- [go/internal/vector/lexical.go](/D:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/go/internal/vector/lexical.go) は lexical index を非同期 worker で更新している。
- [src/retriever.ts](/D:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/src/retriever.ts) と [src/index.ts](/D:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/src/index.ts) は、Phase 5 で diagnostics を受け取る前提に寄せる。
- degraded fallback の auto inject は `autoInjectGuardMinScore`（default `0.86`）で絞る。score 比較は 0.0-1.0 スケール。

## Phase 5 からの引き継ぎ
- `matchedBy` は「candidate source」の純粋指標ではなく、現行実装では「current scoring provenance」に近い意味で出る。Phase 6 で定義を固定し、運用ログの読み方を揃える。
- `fallbackReason` は `ScoredEpisode` 単位で返るため、empty result 時は diagnostics が残らない。Phase 6 で top-level diagnostics を追加するかどうかを判断する。

## 実装タスク
- lexical-first を壊さず、候補不足時の semantic backfill を設計する。
- lexical 0 件時の既存 HNSW fallback を維持したまま、少数ヒット時だけ backfill する。
- `embed_fallback_lexical_only` 中は backfill を抑止する。
- `idStr` 単位 dedupe を候補収集・後段評価の両方で維持する。
- docs / tests / build を 6A 実装にそろえる。

## 設計方針
- まずは `len(candidates) < N` の段階的 hybrid を優先し、いきなり常時並列へ行かない。
- score 式を一気に変えず、candidate collection と ranking policy を別差分で分離する。
- freshness 改善は observability とセットで入れる。
- Anchor と recall の責務境界は、このフェーズでも崩さない。
- lexical 0 件時の `HNSW fallback` は現行挙動として維持し、その上で `lexical が少数ヒットのときだけ取り逃がしを補完する` 方向で広げる。

## 受け入れ条件
- lexical hit が少ないときの recall 欠落を減らせる。
- policy 変更後も原因追跡ができる。
- strict topics fallback の挙動がログと diagnostics で読める。
- compaction 側の Anchor flow には影響を広げない。

## 非対象
- `matchedBy` / `fallbackReason` の surface 固定そのもの
- `rpc-client.ts` の最小型起こし
- `retriever.ts` / `index.ts` の diagnostics 透過だけの差分
- `ep-save` や manual save 系の仕様
- 常時並列 hybrid
- RRF / 正規化線形結合の導入
- lexical freshness 新規メトリクス
- empty result 時 top-level diagnostics 拡張

## 初手の候補
1. `len(candidates) < N` のときだけ HNSW 候補を追加取得する
2. 既存の重み付き和は維持し、候補集合だけ広げる
3. diagnostics を見てから、必要なら RRF / 正規化線形結合へ進む

## lexical が当たらないとき
- 現在のコードでは、lexical search が 0 件ならそのまま終わらず、HNSW semantic search にフォールバックする。
- したがって今の弱点は「lexical 0 件で完全停止すること」ではなく、「lexical が 1〜少数件だけ当たったときに semantic 候補を拾いに行かないこと」。
- Phase 6A はこの隙間を埋めるための差分で、`0 件なら従来どおり HNSW`、`少数件なら不足分だけ HNSW backfill` へ寄せる。

## Research Addendum — 2026-04-03

### Findings

#### BLOCKER
- なし。Phase 6 は retrieval policy と freshness の改善段階で、いま直ちに壊れている単一点は見つかっていない。

#### HIGH
- [HIGH-1] 現行の候補取得は lexical-first で固定されており、`len(candidates) < N` backfill を入れるなら変更点は `baseRecall(...)` の候補収集部だけに集中する。現状は lexical hit が 1 件でもあれば HNSW に進まず、HNSW に進むのは `len(candidates) == 0` のときだけ。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1106-1157`
- [HIGH-2] 常時並列 hybrid に進むと差分範囲は `baseRecall(...)` だけで閉じず、lexical result / HNSW result の union、重複排除、候補 provenance、最終順位計算、fallback diagnostics の全部が巻き込まれる。現行は `finalScore = semantic + lexical + freshness + surprise + usefulness + exploration` の線形式で、候補集合だけを増やす差分と ranking 自体の変更を分けられている。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1122-1231`
- [HIGH-3] strict topics fallback は「reverse index が冷えているときだけ」ではない。実際には `allowedIDsForTopics(..., true)` が空なら `strictTopics=false` に落として soft mode に戻るため、cold index / legacy data / sparse topics をまとめて含む運用問題になっている。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1106-1115`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1354-1383`

#### MED
- [MED-1] `len(candidates) < N` backfill の最小差分は、lexical hit 後に不足分だけ HNSW 候補を追加し、`idStr` 単位で union する形。リスクは 2 つで、(a) lexical 候補と semantic 候補の重複排除、(b) `CandidateRank` / `matchedBy` の意味が変わること。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1134-1157`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1290-1310`
- [MED-2] 常時並列 hybrid へ進むなら、現行の `matchedBy` は「候補 source」ではなく「現行 scoring provenance」に近い意味で出ているため、定義を作り直す必要がある。lexical 候補でも `semanticScore` は後付け計算されるので、今の `matchedByForCandidate(...)` をそのまま使うと誤読を招く。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1213-1231`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1290-1348`
- [MED-3] empty result 時の fallback 観測は、現状の `results[]` 要素だけでは返せない。TS 側は `recall_empty` になると diagnostics を空で返しており、embed fallback や topics fallback があっても結果ゼロなら見えない。top-level diagnostics を足すなら `ai.recall` の RPC contract 変更になる。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\main.go:1839-1901`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\rpc-client.ts:491-499`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts:128-138`
- [MED-4] freshness 改善を本当にやるなら lexical worker の観測点が足りない。現行 `lexicalWorker(...)` は非同期 WAL 消費で動き、queue age や sequence lag を response に載せる仕組みはまだ無い。これは policy 変更と別のメトリクス追加差分になる。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\lexical.go:50-202`

#### LOW
- [LOW-1] RRF と正規化線形結合は、現行重み式との相性が違う。いまは lexical / semantic だけでなく freshness / surprise / usefulness / exploration も同じ `finalScore` に乗っているので、RRF に行くなら「候補 source の統合」に限って使い、属性系のスコアは後段で別加点に残す方が自然。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:1228-1231`
- [LOW-2] strict topics fallback の運用は、まず diagnostics とログの読み方を固定するだけでも改善できる。TS 側では auto recall が topics を渡していない一方、manual `ep-recall` は `strictTopics` を明示入力できるので、Phase 6 実装時もこの責務差は残る。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts:128-138`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:847-879`

### 修正や実装方針
- 初手は `len(candidates) < N` backfill に留めるのが最小。変更点を `baseRecall(...)` の候補収集に閉じ込められ、既存の重み付き和 `finalScore` はそのまま維持できる。
- 常時並列 hybrid は 1 差分でやらない。候補集合、重複排除、`matchedBy` 定義、fallback diagnostics、最終順位づけが一気に動くため、Phase 6 の最初の一発としては広すぎる。
- Phase を増やさず、Phase 6A / 6B の内部段階で進める。6A は backfill 限定、6B は運用ログ確認後の拡張判断とする。
- RRF を採るなら「候補 source の統合」だけに使い、freshness / surprise / usefulness / exploration は別の線形補正として残す方が現行コードとの整合が良い。正規化線形結合を採るなら、現行の `finalScore` を延長しやすい。
- strict topics fallback は「cold index / legacy / sparse topics の包括フォールバック」として扱い、理由コードの意味を先に固定する。ここで冷えた reverse index だけに原因を狭めない。
- empty result 時 fallback 観測は、最小差分なら今回は見送る方が安全。top-level diagnostics を足すと `ai.recall` の戻り shape が変わるため、Phase 6 の第 1 差分には重い。

### Resolution
- Phase 6 の主戦場は Go の `baseRecall(...)` と lexical worker 周辺で、TS 側は diagnostics 受け取りの最小追従だけで済む。
- いまのコードは「候補取得 policy」と「最終スコア式」をまだ分離できているので、最小差分の初手は backfill だけに絞るべき。
- empty result 時の top-level diagnostics 拡張、常時並列 hybrid、freshness メトリクス新設は、どれも次の差分へ送った方が枝が増えない。

### Recommendation
- まずは `len(candidates) < N` backfill のみを入れ、`matchedBy` / `fallbackReason` の出方がどう変わるかを観測する。
- strict topics fallback は、Phase 6 実装時に「理由コードの定義」と「ログの読み方」を一緒に固定する。
- top-level diagnostics 拡張は、backfill の効果を見たあとで必要なら別差分に分ける。
- 常時並列 hybrid は 6A のあとに本当に必要かを判断する。v0.3.0 の最初の実装には混ぜない。

### 今すぐ実装すべき最短順
1. `baseRecall(...)` に `len(candidates) < N` backfill を追加し、候補不足時だけ HNSW 候補を補完する。
2. 追加候補を `idStr` 単位で dedupe し、`matchedBy` の意味を「lexical / semantic / both」の最小再定義に合わせる。
3. strict topics fallback の理由説明をログと docs で固定し、cold index / legacy / sparse topics を同一方針で扱う。
4. 既存の `finalScore` は維持し、RRF / 常時並列 hybrid / freshness メトリクス新設は次差分へ送る。
5. そのうえで、empty result 時 top-level diagnostics が本当に必要かを運用ログから判断する。

## Implementation Update — 2026-04-03（6A）

### Findings
- 6A は実装済み。`baseRecall(...)` は lexical hit が少ないときだけ HNSW 候補を backfill し、lexical 0 件時の既存 HNSW fallback はそのまま維持している。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go`
- 候補 dedupe は `idStr` 単位で前段・後段の両方に入っており、lexical 候補と HNSW 候補の重複で同一 episode が二重に並ぶのを防いでいる。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go`
- lexical-only embed fallback (`embed_fallback_lexical_only`) 中は sparse-hit backfill を抑止しつつ、lexical 0 件時の既存 HNSW fallback は維持している。そのうえで auto inject は `autoInjectGuardMinScore`（default `0.86`）で絞る。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\main.go`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts`

### 修正や実装方針
- 6A の差分は Go の候補収集に閉じた。`finalScore` の重み式、`rpc-client.ts`、`retriever.ts`、`index.ts` の RPC shape は変えていない。
- `matchedBy` は 6A 時点では現行の最小意味を維持する。lexical 候補は `both` または `lexical`、HNSW backfill 候補は `semantic` のまま扱う。
- 6B は research-only のまま維持する。常時並列 hybrid / RRF / freshness メトリクス / top-level diagnostics 拡張は別文書へ分離し、本線には持ち込まない。

### Resolution
- 6A の目的だった「lexical 少数ヒット時の取り逃がし補完」は最小差分で反映済み。
- diagnostics / RPC surface は Phase 5 の形を壊していない。
- strict topics が有効なときは、候補収集段階でも `allowedIDs` を見て除外するため、`len(candidates) < candidateK` 判定は topic-eligible な候補数ベースで動く。
- 6B は未実装で、引き続き調査・分割判断の段階に留める。

### Recommendation
- 次は運用ログで `matchedBy` と `fallbackReason` の出方を見て、6B で本当に常時並列 hybrid が必要かを判断する。
- strict topics fallback の運用整理は、6B の初手候補として残す。
