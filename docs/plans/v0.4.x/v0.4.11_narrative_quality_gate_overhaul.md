## 1. 問題定義

### 1.1 発生した事象

2026-04-14 の実運用で、244メッセージ（80,932トークン）の会話を2チャンクに分割して物語化したところ、以下の結果になった：

| チャンク | 入力トークン | 出力トークン | 圧縮比 | 保存された内容 |
|---|---|---|---|---|
| #1 | 47,960 | 129 | **0.27%** | エージェントの生返答3行 |
| #2 | 32,972 | 212 | **0.64%** | エージェントの生返答4行 |

正常な物語化（4/12の成功例）は512トークン。失敗例は57・33トークン。**圧縮比が2桁違う。**

### 1.2 なぜv0.4.9 sanitizerが検知できなかったか

v0.4.9の品質ゲートは以下のみ：

```
成功 = API例外なし + 出力10トークン以上 + タグなし
```

今回のオウム返しは：
- API例外なし ✓
- 129 > 10 ✓（MIN_NARRATIVE_TOKENSを通過）
- `<final>`や`[analysis]`タグなし ✓（sanitizeNarrativeOutputを通過）

**→ すべて通過して「成功」として保存される**

### 1.3 根本原因

OpenRouter free モデルが system prompt の指示を無視し、入力テキストの一部（エージェントの生返答）をそのまま出力した。現行コードには「出力が物語化されているか」の判定がない。

## 6. Phase C: 品質ゲート強化（Layer 1）

### 6.1 変更対象

`src/narrative-worker.ts` — `narrativizeWithRetry()` 内の品質ゲート

### 6.2 定数定義

既存の `MIN_NARRATIVE_TOKENS` (L35) の直後に追加：

```typescript
const MIN_COMPRESSION_RATIO = 0.01;  // Output must be >= 1% of input tokens
const ECHO_SAMPLE_LENGTH = 80;       // Characters to check for verbatim echo
const MIN_ECHO_LENGTH = 20;          // Minimum length to bother checking
```

### 6.3 ゲート1: 圧縮比チェック

```typescript
// [NEW] Compression ratio gate: output must be at least 1% of input
const MIN_COMPRESSION_RATIO = 0.01;
const compressionRatio = tokens / item.estimatedTokens;
if (compressionRatio < MIN_COMPRESSION_RATIO) {
  console.warn(
    `[NarrativeWorker] Attempt ${attempt + 1}/${MAX_RETRIES}: compression ratio too low (${(compressionRatio * 100).toFixed(2)}% < ${(MIN_COMPRESSION_RATIO * 100)}%). Retrying...`
  );
  continue;
}
```

今回のケース: 129/47960 = 0.27% < 1% → **Retry発動**

正常な物語化の圧縮比:
- 512/47960 ≈ 1.07% → 通過
- 入力が小さい場合（例: 2000トークン入力 → 80トークン出力 = 4%）→ 通過

### 6.4 ゲート2: オウム返し検出

出力の先頭N文字が入力テキストにそのまま含まれていたら「オウム返し」と判定：

```typescript
// [NEW] Echo detection: output must not be a verbatim copy of input
const normalizedOutput = text.replace(/\s+/g, " ").trim();
const echoSample = normalizedOutput.substring(0, ECHO_SAMPLE_LENGTH);
const normalizedInput = conversationText.replace(/\s+/g, " ");
if (echoSample.length >= MIN_ECHO_LENGTH && normalizedInput.includes(echoSample)) {
  console.warn(
    `[NarrativeWorker] Attempt ${attempt + 1}/${MAX_RETRIES}: output appears to be verbatim echo of input (first ${echoSample.length} chars match). Retrying...`
  );
  continue;
}
```

今回のケース: 「おう、バッチリ聞こえてるぜ！(≧∇≦)b Telegramからでもメッセージ届いてるって」は入力にそのまま含まれる → **Retry発動**

### 6.5 ゲート3: 既存のMIN_NARRATIVE_TOKENS（変更なし）

```typescript
if (tokens < MIN_NARRATIVE_TOKENS) {
  continue;  // 既存ロジック
}
```

### 6.6 ゲートの実行順序

```
1. MIN_NARRATIVE_TOKENS（最安値・即判定）
2. 圧縮比ゲート（構造的判定）
3. オウム返し検出（文字列比較・やや重い）
```

短い出力は1で弾けるから2と3に到達しない。2を通ったものだけ3を実行する。無駄な文字列比較を避けるため。

## 7. Phase D: ターミナルFallback（Layer 3）

### 7.1 変更対象

`src/narrative-worker.ts` — `processNextFromCache()` の null ハンドリング

### 7.2 現行

```typescript
const result = await this.narrativizeWithRetry(item);
if (result) {
  await this.saveNarrative(result, item);
  await this.rpcClient.cacheAck(item.id, "narrative-worker");
} else {
  await this.rpcClient.cacheRetry(item.id, "narrative-worker", "Narrativization returned empty", MAX_CACHE_ATTEMPTS);
}
```

全Retry失敗 → `null` → `cacheRetry` でキューに戻す。同じ結果を何度も繰り返す可能性が高い。

### 7.3 変更後

```typescript
const result = await this.narrativizeWithRetry(item);
if (result) {
  await this.saveNarrative(result, item);
  await this.rpcClient.cacheAck(item.id, "narrative-worker");
  console.log(`[NarrativeWorker] Successfully narrativized chunk [${item.id}]. (Output: ${result.tokens} tokens)`);
} else {
  // [NEW] All retries exhausted → generate deterministic fallback summary
  // instead of re-queueing the same bad output forever
  const fallbackText = this.buildDeterministicFallback(item);
  const fallbackResult: NarrativeResult = {
    text: fallbackText,
    tokens: estimateTokens(fallbackText),
    model: "fallback-summary",
  };
  // [NEW] Add "fallback-summary" tag for later manual re-narrativization
  await this.saveNarrative(fallbackResult, item, true);
  await this.rpcClient.cacheAck(item.id, "narrative-worker");
  console.warn(
    `[NarrativeWorker] LLM narrativization failed for [${item.id}] after ${MAX_RETRIES} attempts. Saved deterministic fallback summary (${fallbackResult.tokens} tokens).`
  );
}
```

### 7.4 buildDeterministicFallback()

`NarrativeWorker` クラスに追加：

```typescript
/**
 * Generate a deterministic fallback summary when all LLM attempts fail.
 * Uses simple text extraction — no API call, guaranteed to produce output.
 */
private buildDeterministicFallback(item: CacheItem): string {
  // Extract role:content lines from rawText (same format as buildNormalSummary)
  const lines = item.rawText
    .split(/\r?\n/)
    .map((line) => line.replace(/\s+/g, " ").trim())
    .filter((line) => line.length > 0)
    .slice(0, 20);  // First 20 non-empty lines

  const summary = lines.join("\n");
  const maxChars = 800;
  if (summary.length <= maxChars) return summary;
  return summary.substring(0, maxChars) + "\n[truncated]";
}
```

### 7.5 設計判断

- **cacheRetryをやめてcacheAckにする**: 全Retry失敗したものをキューに戻しても、同じモデルが同じゴミを出すだけ。確実なFallbackで保存して終わらせる
- **LLMを使わない**: `buildDeterministicFallback` は純粋なテキスト処理。APIコストゼロ、確実に成功
- **品質は下がるが「何もない」よりマシ**: Fallback保存されたMDには `model: "fallback-summary"` とタグ `"fallback-summary"` が付くので、後から手動で物語化し直すことも可能
- **saveNarrative のシグネチャ変更**: `isFallback: boolean = false` パラメータを追加し、`true`の場合はタグに `"fallback-summary"` を含める

## 15. 評価メモ（実装には不要・背景理解用）

### なぜ few-shot（Layer 0）だけで済ませないか

プロンプト改善だけでもオウム返し率は下がるが、**free モデルの品質は保証されない**。プロンプトを無視するモデルに対してはコード側の防御が必須。3層にすることで、どの層が効いてもゴミ保存を防げる。

### なぜ圧縮比ゲート（Layer 1）の閾値を1%にするか

- 正常な物語化の圧縮比: 1-5%（入力2K-50K → 出力50-512トークン）
- 今回の失敗: 0.27%
- 閾値1%は正常例の下限より低い。偽陽性リスクは極低
- ただし入力が極端に小さい場合（例: 200トークン入力 → 5トークン出力 = 2.5%）は通過するが、これは MIN_NARRATIVE_TOKENS=10 が別途カバーする

### buildFallbackSummary() と buildDeterministicFallback() の違い

- `buildFallbackSummary()`（summary-escalation.ts）: `SummaryMessageLike[]` を受け取る。Message オブジェクトの配列が必要
- `buildDeterministicFallback()`（新規）: `CacheQueueItem.rawText` を受け取る。文字列だけで動作。CacheItem には Message 配列がなく、rawText しかないため、別メソッドが必要
