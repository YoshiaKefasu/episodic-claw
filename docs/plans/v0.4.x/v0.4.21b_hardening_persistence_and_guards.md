### 3.2 Fix 2: Segmenter cursor 永続化（state.db）

**方針**: cursor は `state.db` の `segmenter-cursor` キーへ保存。`Watermark`（watcher用）には混ぜない。

**TS側変更** (`rpc-client.ts` helper):

```typescript
async getSegmenterCursor(agentWs: string, agentId: string): Promise<{ lastProcessedLength: number }> {
  const wsHash = hashWs(agentWs);
  const key = `agent:${agentId}:ws:${wsHash}:segmenter-cursor`;
  const raw = await this.stateGet(agentWs, agentId, key).catch(() => "");
  if (!raw) return { lastProcessedLength: 0 };
  const parsed = JSON.parse(raw);
  return { lastProcessedLength: Number(parsed?.lastProcessedLength) || 0 };
}

async setSegmenterCursor(agentWs: string, agentId: string, cursor: { lastProcessedLength: number }): Promise<void> {
  const wsHash = hashWs(agentWs);
  const key = `agent:${agentId}:ws:${wsHash}:segmenter-cursor`;
  await this.stateSet(agentWs, agentId, key, JSON.stringify(cursor));
}
```

**TS側変更** (`segmenter.ts`):

```typescript
private async persistCursor(agentWs: string): Promise<void> {
  await this.rpc.setSegmenterCursor(agentWs, { lastProcessedLength: this.lastProcessedLength })

}
```

## 4. 設計判断

### Fix 1: なぜ `state.db` を新設するのか

要件は「記憶本体DB（`vector.db`）をクリーンに維持する」こと。`recentSaveHashes` と cursor は制御状態であり、記憶本体でもキューでもないため、`state.db` に分離するのが責務として最も自然。

### Fix 2: なぜ global `state.db`（workspace外）にするのか

`cache.db` は queue専用、`vector.db` は記憶本体専用という既存分離に合わせ、制御状態は第三のDBへ集約する。配置は `~/.openclaw/episodic-claw/state.db`（Windowsでは `%USERPROFILE%\\.openclaw\\episodic-claw\\state.db`）。全エージェント状態はキー名前空間で分離する。

### Fix 2: cursor 永続化で warm-start 閾値は不要になるか

**ならない**。PebbleDB 読み取り失敗時（IO エラー、初回起動でDB未初期化等）のフォールバックとして閾値判定は残存する。ただし主経路は PebbleDB 復元になり、閾値はセーフネットに格下げ。

### Fix 3: 日本語パターンの誤検知リスク

日本語の日常会話で「では、整理しましょう」等が冒頭に来ることは叙事文では稀。叙事文は三人称・過去形で書かれるため、一人称・命令形の CoT パターンとの区別は明確。Gate 3 と同様に first-100-chars 制約で誤検知を抑止。

### Fix 4: なぜ handleIngest に same-request guard が不要か

`handleIngest` は単一アイテムの RPC ため、同一リクエスト内重複が発生しない。DB重複ガードのみで十分。
