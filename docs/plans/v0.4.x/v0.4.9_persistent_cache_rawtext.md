# v0.4.9 (追加): CacheDB rawText 永続保持 — Ack=削除 → Ack=スタンプのみ

---

## 目的

NarrativeWorker が物語化に成功した後、CacheDB から rawText（生の会話記録）を物理削除しない。代わりに `status: "done"` のスタンプを付けるだけにして、元のチャンクを永続保持する。

### 解決する問題

1. **再生成不能**: 現行 v0.4.9 では `cache.ack` が `deleteAfter=true` で PebbleDB から物理削除するため、物語化が失敗した（品質が低い）episode の rawText を使った再生成ができない
2. **デバッグ困難**: 物語化結果がおかしい場合、入力の rawText が消えているため原因調査ができない
3. **耐障害性**: OpenRouter free モデルの品質不安定に対して、「再処理」という回復手段が存在しない

### ダイアグラム（ユーザー提供アーキテクチャ）

```
Ingest Messages
       │
       ▼
64K以内？ ──→ 超えたら（512K）→ 複数の64Kブロックに分割
       │
       ▼
┌─────────────────────────────┐
│  64K Token ブロック          │
│  ・YYYY-MM-DD HH:MM タイムスタンプ │
│  ・順番のスタンプ (orderKey)       │
│  ・処理のスタンプ (status)         │
└─────────────────────────────┘
       │
       ▼
┌─────────────────────────────┐
│  PebbleDB (cache.db)        │
│  永続保存                    │
└─────────────────────────────┘
       │
       ▼
YYYY-MM-DD HH:MM スタンプ順番による Sequential 物語化
       │
       ▼
OpenRouter 物語化 → 成功まで Retry → 物語化の .md ファイル
       │
       ▼
status: "done" スタンプを CacheDB に反映
（rawText は削除しない）
```

## Phase 1: Ack の物理削除を停止

### 変更対象: `go/internal/cache/queue.go`

#### 1-A: `Ack()` の `deleteAfter` を `false` に変更

**現在のコード (L202-213):**

```go
// Ack marks an item as done and removes it from the queue.
func (q *Queue) Ack(id string) error {
	q.mu.Lock()
	defer q.mu.Unlock()

	// Find the item by scanning (simple approach; could be optimized with index)
	return q.updateItemByScan(id, func(item *QueueItem) bool {
		item.Status = StatusDone
		item.UpdatedAt = time.Now().UTC().Format(time.RFC3339)
		return true
	}, true) // delete after update    ← ★ ここ
}
```

**変更後:**

```go
// Ack marks an item as done (preserves rawText for potential re-narrativization).
func (q *Queue) Ack(id string) error {
	q.mu.Lock()
	defer q.mu.Unlock()

	return q.updateItemByScan(id, func(item *QueueItem) bool {
		item.Status = StatusDone
		item.LeaseOwner = ""
		item.LeaseUntil = ""
		item.UpdatedAt = time.Now().UTC().Format(time.RFC3339)
		return true
	}, false) // ★ deleteAfter=false: rawText を永続保持
}
```

**変更点:**
- `deleteAfter` を `true` → `false` に変更
- `LeaseOwner` と `LeaseUntil` をクリア（done 状態でリース情報は不要）
- コメントを更新

#### 1-B: `LeaseNext()` への影響確認

```go
if item.Status != StatusQueued {
    continue    // ← StatusDone はスキップされる。変更不要
}
```

`LeaseNext` は `StatusQueued` のアイテムのみをリースするため、`StatusDone` のアイテムは自然にスキップされる。**変更不要**。

#### 1-C: `Stats()` に `done` カウントを追加

**現在のコード (L374-399):**

```go
func (q *Queue) Stats() (queued, leased, deadLetter int, err error) {
```

**変更後:**

```go
func (q *Queue) Stats() (queued, leased, done, deadLetter int, err error) {
```

`StatusDone` のカウントも返すようにする。done アイテムが蓄積する想定なので、監視用。

**`Stats()` のシグネチャ変更は必須。** 現在 `Stats()` は Go 側でどこからも呼ばれていないため、シグネチャ変更はコンパイルエラーを起こさない。呼び出し元がいない今が最も安全な変更タイミング。`handleCacheStats` は現在存在しないため main.go 側の対応は不要。

## Phase 2: Requeue RPC の追加

`StatusDone` のアイテムを `StatusQueued` に戻して再物語化するための RPC。

### 2-A: `Requeue()` メソッドを `queue.go` に追加

```go
// Requeue moves a "done" item back to "queued" for re-narrativization.
// Returns error if item not found or not in "done" status.
func (q *Queue) Requeue(id string) error {
	q.mu.Lock()
	defer q.mu.Unlock()

	return q.updateItemByScan(id, func(item *QueueItem) bool {
		if item.Status != StatusDone {
			return false // Only requeue "done" items
		}
		item.Status = StatusQueued
		item.Attempts = 0
		item.LeaseOwner = ""
		item.LeaseUntil = ""
		item.BackoffUntil = ""
		item.LastError = ""
		item.UpdatedAt = time.Now().UTC().Format(time.RFC3339)
		return true
	}, false)
}
```

**設計判断:**
- `Attempts` をリセット（再生成は新しい試行として扱う）
- `StatusDone` 以外のアイテムに対しては `false` を返して no-op
- `dead-letter` からの復活も将来的には「Requeue」メソッドで拡張可能だが、今回は `done` のみ（YAGNI）

### 2-B: `handleCacheRequeue` RPC ハンドラーを `main.go` に追加

```go
func handleCacheRequeue(conn net.Conn, req RPCRequest) {
	var params struct {
		ID string `json:"id"`
	}
	if err := json.Unmarshal(req.Params, &params); err != nil {
		sendResponse(conn, RPCResponse{JSONRPC: "2.0", Error: &RPCError{Code: -32602, Message: "Invalid params"}, ID: req.ID})
		return
	}

	// Note: iterates cacheQueues without lock, consistent with handleCacheAck
	for agentWs, q := range cacheQueues {
		if err := q.Requeue(params.ID); err == nil {
			EmitLog("[Episodic DB] Requeued cache job [%s] from %s for re-narrativization", params.ID, agentWs)
			sendResponse(conn, RPCResponse{JSONRPC: "2.0", Result: "requeued", ID: req.ID})
			return
		}
	}

	sendResponse(conn, RPCResponse{JSONRPC: "2.0", Error: &RPCError{Code: -32000, Message: "Item not found or not in done status"}, ID: req.ID})
}
```

**ルーター登録 (main.go L2965付近):**

```go
case "cache.requeue":
	go handleCacheRequeue(conn, req)
```

### 2-C: TS 側に `cacheRequeue()` を追加

## Phase 3: ストレージ管理（将来の考慮事項 — 今回は実装しない）

rawText を永続保持すると PebbleDB のサイズが膨らむ。概算:

| 項目 | 値 |
|---|---|
| 1チャンクの rawText 平均サイズ | ~30KB (64K tokens ≈ 30KB UTF-8) |
| 1日あたりのチャンク数（見込み） | ~5-20 |
| 1ヶ月のストレージ増分 | ~5-20MB |
| 1年のストレージ増分 | ~60-240MB |

2010年代のハードウェアでも許容範囲内。ただし将来的に以下を検討:

- **TTL ベースの自動パージ**: 90日以上前の `StatusDone` アイテムの rawText を削除
- **Comet DB への移行**: ユーザーのダイアグラムにも記載あり。将来バージョンで検討

**今回は YAGNI で実装しない。**

### 🔍 Detailed Code Review Findings

#### F-1: Phase 2 (Requeue) は Phase 1 より優先度が低い（重要度: HIGH）

Phase 1 の `deleteAfter=false` だけで、本プランの3つの問題のうち2つ（「デバッグ困難」「耐障害性の回復手段欠如」）が解決する。デバッグは PebbleDB を直接覗けばよい。再生成も Go側で直接ステータスを書き換えれば可能。

Phase 2 の `cacheRequeue()` RPC は「TS側から再生成をトリガーする」ためのUIが存在しない現状では、呼び出し元がない死にコードになる。実際に再生成をトリガーするのは、将来の OpenClaw UI や手動操作スクリプト。

**当初判断**: Phase 2 は YAGNI。
**→ ユーザー判断で却下（Option B 採用）**: 永続保持がアーキテクチャの核心であり、Requeue RPC は再物語化の正式な手段として Phase 1 と一緒に実装する。

#### F-2: `Stats()` の `done` カウント追加はコンパイルブレークのリスク（重要度: MEDIUM）

`Stats()` のシグネチャを変えると、戻り値の分解位置がずれる。現在 `Stats()` はどこからも呼ばれていないのでコンパイルエラーは起きないが、将来誰かが `q.Stats()` を呼んだ時に3変数から4変数に変わっていることに気づかないリスクがある。

Go の多値返却は位置で受け取るため、シグネチャ変更は暗黙の互換性破壊になりがち。

**判断**: やるなら今（呼び出し元がいない今）が最も安全なタイミング。ただし実装時は呼び出し元が生まれないことをコメントで明記すること。

#### F-3: PebbleDB のストレージ増加に対応する運用監視がない（重要度: LOW）

Phase 3 の概算（1年で60-240MB）は妥当。しかし `done` アイテムが蓄積することを確認する手段が、`Stats()` だけでは不十分。`cache.db` のファイルサイズをログ出力する仕組みも YAGNI で今回は不要だが、Phase 3 を実装する時に必要になる。

**判断**: ✅ 今は不要。Phase 3 の時に対処する。

#### F-4: `handleCacheRequeue` のクエリスキャンにロックがない（重要度: LOW）

プランの `handleCacheRequeue` は `for agentWs, q := range cacheQueues` で全キューをスキャンするが、`cacheQueuesMu` のロックを取得していない。既存の `handleCacheAck` (L721) もロックなしでスキャンしているので一貫性はあるが、`cacheQueues` マップへの read アクセスと concurrent な write（新規 agentWs 追加）の競合可能性がある。

**判断**: 既存パターンと同じなので一貫性を優先。ただし別の修正でロックを入れるならここにも入れる。

#### F-5: `Requeue()` メソッドの `Attempts` リセット（重要度: LOW）

Attempts=0 にリセットしている。これは再生成を「完全に新しい試行」として扱う設計。一方で、何回再生成されたかの履歴は失われる。`Attempts` をリセットしないと `MAX_CACHE_ATTEMPTS` (20) に達して dead-letter に落ちるが、再生成コンテキストではリセットが自然。

**判断**: ✅ 正しい設計。再生成は別コンテキスト。
