# v0.4.2 Redundant Config Cleanup Plan

**Parent:** [v0.4.0 Narrative Architecture Roadmap](../v0.4.0_narrative_architecture_roadmap.md)  
**Date:** 2026-04-09  
**Target Version:** v0.4.2  
**Status:** 完了（契約整理 Phase 1 完了 / 実削除は v0.4.3 以降）

> 本プランは、「Cache-and-Drain」モデルの堅牢な64K分割アーキテクチャが実装されることに伴い、不要になる「バッファ文字数」や「チャンク上限」といったレガシーな設定項目の扱いを整理するための計画です。**YAGNI（You Aren't Gonna Need It）** および **KISS（Keep It Simple, Stupid）** の原則に基づき、設定の複雑さを削ぎ落とします。v0.4.2 では staged deprecation（段階的非推奨化）を採用し、実削除は v0.4.3 以降に分離します。

---

## 1. 排除対象の設定項目（3点） — 初期案（非推奨）

> **注:** 以下の「完全排除」案は検討の結果、v0.4.2 では見送り。`maxBufferChars` / `maxPoolChars` はまだ live path で使用中。`maxCharsPerChunk` も legacy path で使用中。詳細は後述の「Pro Engineer Review」を参照。

新アーキテクチャでは、物理リミット（定数ハードコードによる64K分割）と意味論的境界（Surprise、TimeGap）だけでエピソード化を制御するため、ユーザーが文字数を微調整する以下の設定は不要ではないか、という初期仮説があった。

1. `maxBufferChars` (Segmenterバッファの強制保存文字数)
2. `maxCharsPerChunk` (v0.3.x時代のチャンク分割ルール)
3. `maxPoolChars` (NarrativeWorker向けのプール容量)

> **注:** 実際には `maxBufferChars` / `maxPoolChars` は live path でサイズガードとして機能しており、`maxCharsPerChunk` も legacy path で使用中。「完全に無意味」という初期仮説は誤りだった。

## 2. 修正対象ファイルと詳細手順 — 初期案（非推奨）

> **注:** 以下の「削除手順」は実行済みではなく、検討のための初期案。実際の v0.4.2 での対応は後述の「Pro Engineer Review」に定める staged deprecation 方式。

### ① `openclaw.plugin.json` (設定UIの定義)
ユーザーの設定画面から完全に消し去ります。
- `properties` 内から `maxBufferChars` (Line: 48-52) を削除。
- `properties` 内から `maxCharsPerChunk` (Line: 53-57) を削除。
- `properties` 内から `maxPoolChars` (Line: 194-199) を削除。

### ② `src/index.ts` (TypeBox スキーマ)
バリデーションスキーマからも排除します。
- `EpisodicPluginConfigSchema` 内の `maxBufferChars`, `maxCharsPerChunk`, `maxPoolChars` 定義を削除。
- `Segmenter` および `NarrativePool` 初期化時に渡している `cfg.maxXXX` の参照を削除。

### ③ `src/types.ts` & `src/config.ts` (型とデフォルトパース)
- `EpisodicPluginConfig` インターフェースから該当3項目を削除。
- `config.ts` の `loadConfig()` 関数内にあるフォールバック代入 (`Math.max(500, rawConfig?.maxBufferChars ?? 7200)` 等) を全削除。

### ④ `src/segmenter.ts` (境界線の強制分割ロジック)
内部の物理ガードライン（ハードリミット）を定数化します。
- クラスプロパティ `this.maxBufferChars`, `this.maxCharsPerChunk` を削除。
- 巨大バッファ判定をユーザー設定値ではなく、ハードリミット定数に置換：
  ```typescript
  const HARD_LIMIT_CHARS = 64000;
  const sizeLimitExceeded = estimatedChars > HARD_LIMIT_CHARS;
  ```
- レガシーの `chunkAndIngest` 関数内の定数も内部ハードリミット（`64000`など）で固定化。

### ⑤ `src/narrative-pool.ts` (プールの文字数あふれ処理)
- `this.maxPoolChars` を設計から削除。
- コンストラクタ引数から削除。
- 代わりに、Step 1（Segmenter側の64K分割ロジック）で整理されたチャンクをそのまま引き受けるか、内部定数 `64000` 文字でガードをかけるのみのシンプルな構造に変更。

## 3. 期待される効果 — 初期案に基づく想定

> **注:** 以下は「3設定を完全削除した場合」の期待効果であり、実際の v0.4.2 は staged deprecation を採用。効果の多くは「説明の整理」レベルで達成済み。
- **ユーザー体験の向上**: 専門的で難解な文字数設定が消え、ユーザーは「物語化モデル」「サプライズ感度」といった本質的な設定のみに集中できる。
- **アーキテクチャの単純化**: 3つの変数が絡み合う予測困難な挙動（バグの温床）が消滅し、コードの保守性が劇的に向上する。
- **v0.4.2設計との合致**: Sequential Cache-and-Drain モデルが「64Kチャンク分割」という固定ルールに依存しやすくなり、実装の一貫性が担保される。

## 🔧 Pro Engineer Review — 2026-04-10
> Perspective: Google / IBM Production Engineering  
> Principles applied: YAGNI · KISS · DRY · SOLID  
> Source code verified: ✅ (as of 2026-04-10)

### 📍 Current Reality (Source Code vs. Document)

- ⚠️ `maxBufferChars` はまだ死んでいない。`src/segmenter.ts` で `sizeLimitExceeded = estimatedChars > this.maxBufferChars` として live path の boundary 判定に使われている。
- ⚠️ `maxPoolChars` もまだ死んでいない。`src/narrative-pool.ts` で pool flush 条件として使われている。
- ⚠️ `maxCharsPerChunk` は narrative 主経路では薄くなったが、`src/segmenter.ts` の legacy `chunkAndIngest()` ではまだ使用中。
- ⚠️ `openclaw.plugin.json`、`src/index.ts`、`src/types.ts`、`src/config.ts` の4層すべてでこの3設定がまだ正式契約として露出している。
- ❌ したがって、「v0.4.2 で3設定を完全排除」は現行コードと一致しない。**削除より先に責務の再分類が必要**。

### 🎯 Core Problem (1 sentence)
> 問題は「3設定が完全に不要」なことではなく、**旧設計の名前と説明のまま、別の責務で生き残っていること** でユーザーに誤解を与えている点にある。

### 🔍 Principle Filter

| Check | Result | Note |
|---|---|---|
| YAGNI — 今すぐ完全削除は必要か? | ❌ No | 3設定とも runtime 上まだ参照が残る。削除は尚早 |
| KISS — より単純な進め方は? | ✅ Yes | まず「削除」ではなく「再分類 + 非推奨化」に寄せる |
| DRY — 重複責務はあるか? | ⚠️ Yes | `maxBufferChars` と `maxPoolChars` はどちらもサイズガードだが、段階が違う |
| SOLID — すぐ大改修が必要か? | ❌ No | v0.4.2 では UI/契約整理、実削除は後続が妥当 |

### 🛤️ Solution Options

#### Option A — Staged Deprecation *(推奨)*
**Approach**: v0.4.2 では 3設定を一気に削除しない。まず役割を再定義し、ユーザー向けには「非推奨 / advanced / legacy-only」として整理する。runtime 互換は維持する。  
**Implementation cost**: Low  
**Risk**: Low  
**Why recommended**: 既存ユーザー設定を壊さず、実コードとドキュメントのズレだけ先に閉じられるため。  
**Concrete steps**:
1. `maxBufferChars` を **segmenter live flush guard** として再説明する。
2. `maxPoolChars` を **narrative pool flush guard** として再説明する。
3. `maxCharsPerChunk` を **legacy path only** に格下げする。
4. `openclaw.plugin.json` / `src/index.ts` / `src/types.ts` / `src/config.ts` の説明文を揃える。
5. README ではこの3つを「通常ユーザーは触らない advanced knobs」に落とす。

#### Option B — Hard Removal in v0.4.2
**Approach**: このプランの元案どおり、3設定を schema / parser / runtime から即削除する。  
**Implementation cost**: Medium  
**Risk**: High  
**When to choose this instead**: narrative path 以外を同時に廃止し、既存ユーザー設定の breakage も許容できる場合のみ。  
**Why risky**:
1. `maxBufferChars` を消すと live path の boundary 制御を内部定数へ再設計する必要がある。
2. `maxPoolChars` を消すと pool flush の責務を別レイヤへ移し替える必要がある。
3. `maxCharsPerChunk` を消すには legacy `chunkAndIngest()` の扱いを先に決める必要がある。

### ✅ Pro Recommendation
> **Choose Option A because**:  
> いま切るべきなのは「旧設計の名前による誤解」であって、runtime の足場そのものではない。  
> v0.4.2 では **責務の再分類 + 非推奨化** までをやり、実削除は削除条件が揃った後続リリースに分離するのが最も安全。

### ✂️ どう糸を切るか

この3設定は、同じ「サイズ系ノブ」に見えて実際は別の層にいます。  
雑にまとめて切ると事故るので、以下の順序で切るのがよいです。

#### 1. まず「名前の誤解」を切る

- `maxBufferChars`
  - 旧認識: 生ログ時代の buffer 文字数
  - 現実: **segmenter が surprise/time-gap を待たずに flush するための上限**

- `maxPoolChars`
  - 旧認識: NarrativeWorker 用の曖昧なプール容量
  - 現実: **narrative pool が flush item を返すトリガー**

- `maxCharsPerChunk`
  - 旧認識: 現行主経路の chunk size
  - 現実: **legacy `chunkAndIngest()` 側の互換設定**

この切り分けを文書で先にやるだけでも、だいぶ紛らわしさが減る。

#### 2. 次に「ユーザーに見せる契約」を切る

v0.4.2 での扱いは次のように分けるのが現実的。

| 設定 | v0.4.2 での扱い | 理由 |
|---|---|---|
| `maxBufferChars` | **残すが advanced 扱い** | live path でまだ使用中 |
| `maxPoolChars` | **残すが advanced 扱い** | pool flush にまだ必要 |
| `maxCharsPerChunk` | **legacy-only / deprecated** | narrative 主経路では本質でない |

つまり、**3つ全部を同じ濃さで UI に見せるのをやめる** のが先。  
削除より、露出の濃さを変えるほうが v0.4.2 向きです。

#### 3. 最後に「runtime の糸」を切る

本当に削除してよい条件は、次のように考えると安全です。

**`maxCharsPerChunk` を消せる条件**
1. legacy `chunkAndIngest()` を完全に使わなくなる
2. narrative 無効時の fallback 契約を別途固定する

**`maxPoolChars` を消せる条件**
1. pool 自体を廃止する  
   または
2. pool flush を内部定数に固定して、ユーザー調整を不要と判断できる

**`maxBufferChars` を消せる条件**
1. segmenter の size-limit boundary を内部定数で十分運用できる
2. `maxPoolChars` との責務競合を整理済み

### 🧭 v0.4.2 の現実的な着地点

v0.4.2 でやるべき cleanup は「3設定を即削除」ではない。  
現実的な着地点は次のとおり。
