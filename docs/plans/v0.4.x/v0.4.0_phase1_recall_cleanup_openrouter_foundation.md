1. **Recall クエリ汚染の根本解決** — v0.3.8 の 3層フィルタ (全て no-op) を user-only include list に置き換え

**File:** `src/retriever.ts`

### Step 3: `stripReasoningTagsFromText` の recall 呼び出しを残すかどうか

**判断: 防御的に残す。** user メッセージに thinking タグが入ることは通常ないが、OpenClaw が将来 user メッセージにメタデータを埋め込む可能性がゼロとは言い切れない。`stripReasoningTagsFromText` の呼び出しは no-op でも害がない（処理コストは QUICK_TAG_RE の test() 1回だけ）。

ただし将来の Phase で `reasoning-tags.ts` ごと削除する場合のための NOTE コメントを追加:


### Step 4: `src/openrouter-client.ts` 新規作成

```typescript
// 最小限の OpenRouter Chat Completion API クライアント
// KISS: fetch ベース、外部 HTTP ライブラリ不要

export interface OpenRouterConfig {
  apiKey: string;
  model: string;           // default: "openrouter/free"
  maxTokens?: number;      // 省略推奨 (OpenRouter 任せ)。指定時のみキャップ
  temperature: number;     // default: 0.4
  baseUrl: string;         // default: "https://openrouter.ai/api/v1"
  timeoutMs: number;       // default: 30000
}

export class OpenRouterClient {
  private config: OpenRouterConfig;

  constructor(config: Partial<OpenRouterConfig> & { apiKey: string });
  async chatCompletion(params: {
    systemPrompt: string;
    userMessage: string;
  }): Promise<string>;
}
```

**実装の要点:**
- `fetch()` (Node 18+ built-in) を使用。`axios` 等の追加依存は不要
- 429 レスポンスの場合、`Retry-After` ヘッダを尊重して待機
- タイムアウトは `AbortController` で実装
- レスポンスから `choices[0].message.content` を抽出

### Step 5: `src/narrative-worker.ts` 新規作成

```typescript
// 物語化の非同期ワーカー
// KISS: シンプルな FIFO キュー + 指数バックオフ

export class NarrativeWorker {
  private queue: PoolFlushItem[] = [];
  private isProcessing: boolean = false;
  private lastNarrativeTail: string = "";

  constructor(
    private client: OpenRouterClient,
    private rpcClient: EpisodicCoreClient,
    private config: EpisodicPluginConfig,
  );

  async enqueue(item: PoolFlushItem): Promise<void>;
  private async processNext(): Promise<void>;
  private async narrativize(item: PoolFlushItem): Promise<string>;
  private async saveRawFallback(item: PoolFlushItem): Promise<void>;
  async drain(): Promise<void>;  // graceful shutdown 用
}
```

**実装の要点:**
- `enqueue()` はキューに追加して `processNext()` を fire-and-forget
- `processNext()` は `isProcessing` フラグで排他制御（同時 1リクエスト）
- Retry: 最大 5 回、指数バックオフ (1s, 2s, 4s, 8s, 16s)
- 全失敗時: `saveRawFallback()` で `buildFallbackSummary` を呼んで生ログ保存
- `drain()`: キューが空になるまで待機（session_end / gateway_stop 用）

### Findings

#### [F1] `Retry-After` ヘッダの取得が不安定 — 中 severity

**場所:** `openrouter-client.ts` L118-124

**問題:** `extractRetryAfter()` が `err.message` への正規表現マッチで `Retry-After` を取得しようとしているが、`Retry-After` は HTTP レスポンスヘッダであり `err.message` に含まれる保証がない。

**修正案:**
```typescript
// doCompletion() 内:
if (!response.ok) {
  const retryAfter = response.headers.get("Retry-After");
  const errorText = await response.text().catch(() => "");
  const err = new Error(`OpenRouter HTTP ${response.status}: ${errorText.slice(0, 200)}`);
  (err as any).retryAfterSeconds = retryAfter ? parseInt(retryAfter, 10) : 0;
  throw err;
}

// chatCompletion() の retry ループ内:
if (lastError.message.includes("429")) {
  const retryAfter = (lastError as any).retryAfterSeconds as number ?? 0;
  if (retryAfter > 0) {
    await this.sleep(retryAfter * 1000);
    continue;
  }
}
```

#### [F2] `PoolFlushItem` / `NarrativeResult` が `types.ts` にない — 低 severity

**場所:** `narrative-worker.ts` L20-33

#### [F3] `resolvePrompt()` がファイル読み込みを実装していない — 中 severity

**場所:** `config.ts` L52-55

**問題:** ロードマップで決定した `.md`/`.txt` ファイルパス対応が未実装。現在は `value.trim()` のみ。

**修正案:**
```typescript
import * as fs from "fs";
import * as path from "path";

function resolvePrompt(value: string | undefined): string {
  if (!value) return "";
  const trimmed = value.trim();
  if (trimmed.endsWith(".md") || trimmed.endsWith(".txt")) {
    try {
      return fs.readFileSync(path.resolve(trimmed), "utf8").trim();
    } catch {
      console.warn(`[Episodic Memory] Failed to read prompt file: ${trimmed}`);
      return "";
    }
  }
  return trimmed;
}
```
