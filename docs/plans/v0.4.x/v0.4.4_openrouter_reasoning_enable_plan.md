## 1. 目的

`openclaw.plugin.json` の `openrouterConfig` 経由で reasoning/thinking を制御し、Narrative の出力品質を上げる。  
本件の方針はシンプルに固定する。

- **Option A 採用**
- **デフォルト: reasoning ON + effort=high**
- **ユーザーは `openrouterConfig` で自由に調整可能**

参照（公式）:
- [OpenRouter API Overview](https://openrouter.ai/docs/api/reference/overview)
- [OpenRouter Reasoning Tokens Guide](https://openrouter.ai/docs/guides/best-practices/reasoning-tokens)

## 3. 最終仕様（Option A）

### 3.1 `openrouterConfig.reasoning` を正式導入する

`openclaw.plugin.json` / `src/index.ts` / `src/types.ts` へ次を追加する。

- `openrouterConfig.reasoning.enabled?: boolean`
- `openrouterConfig.reasoning.effort?: "none" | "minimal" | "low" | "medium" | "high" | "xhigh"`
- `openrouterConfig.reasoning.maxTokens?: integer`
- `openrouterConfig.reasoning.exclude?: boolean`

### 3.2 デフォルト値（本件の断言）

- `reasoning.enabled = true`
- `reasoning.effort = "high"`

ユーザーが何も指定しない場合でも、v0.4.4 は上記デフォルトで `reasoning` を送る。

### 3.3 送信ルール（正規化ルール）

`normalizeOpenRouterReasoning()` を `config.ts` に追加し、設定の解釈を1か所に集約する。

1. `enabled === false` のときは `reasoning` 自体を送らない  
2. `maxTokens` と `effort` が両方あるときは `maxTokens` を優先し、`effort` は送らない  
3. `maxTokens` は `reasoning.max_tokens` にマッピングする  
4. `exclude === true` のときのみ `reasoning.exclude=true` を送る  
5. `maxTokens` が無効値（0以下、非整数など）の場合は未設定扱いに落とす

### 3.4 今回やらないこと

- `includeReasoning` / `include_reasoning` 互換レイヤーの導入
- Responses API への移行
- provider別の細かい最適化

## 4. 実装差分（対象ファイル）

### 4.1 `openclaw.plugin.json`

- `openrouterConfig.reasoning` schema を追加
- `enabled=true` / `effort=high` を default として明記
- description に「出力品質優先。必要なら low/medium/none へ調整可」を追記

### 4.2 `src/index.ts`（plugin schema）

- `openrouterConfig` の `Type.Object` に `reasoning` を追加
- `effort` は enum 制約
- `maxTokens` は正整数制約
- `exclude` / `enabled` は boolean

### 4.3 `src/types.ts`

- `OpenRouterReasoningConfig` 型を新規追加
- `EpisodicPluginConfig["openrouterConfig"]` に `reasoning?: OpenRouterReasoningConfig` を追加

### 4.4 `src/config.ts`

- `normalizeOpenRouterReasoning(raw)` を追加
- `loadConfig()` で `openrouterConfig.reasoning` を読み、未指定時は `{ enabled: true, effort: "high" }` を適用
- 無効値はここで吸収する

### 4.5 `src/openrouter-client.ts`

- `OpenRouterConfig` に `reasoning?: { effort?: ..., maxTokens?: number, exclude?: boolean }` を追加
- request body に `reasoning` を条件付き追加
- `maxTokens -> max_tokens` 変換を実施

### 4.6 `src/index.ts`（OpenRouterClient 初期化）

- `new OpenRouterClient({...})` に `reasoning` を渡す

## 5. テスト計画（必須）

専用テストファイルを追加して payload 検証を固定する。  
（`test_phase4_5.ts` の肥大化は避ける）

1. デフォルト設定（未指定）  
- `reasoning.effort=high` が body に入る

2. `reasoning.enabled=false`  
- body に `reasoning` が存在しない

3. `reasoning.effort=medium` 指定  
- body に `reasoning.effort=medium` が入る

4. `reasoning.maxTokens=2048` 指定  
- body に `reasoning.max_tokens=2048` が入る

5. `reasoning.effort` と `reasoning.maxTokens` 同時指定  
- `max_tokens` を優先し、`effort` は送らない

6. `reasoning.exclude=true` 指定  
- body に `reasoning.exclude=true` が入る

7. 既存互換  
- reasoning 以外（`model/max_tokens/temperature`）の既存挙動が変わらない

## 7. リスクとロールバック

### リスク

- デフォルト `high` により、応答遅延と token 消費は増える
- 一部モデルでは `effort` / `max_tokens` の解釈差が出る可能性がある

### ロールバック

- 即時回避: `openrouterConfig.reasoning.enabled=false`
- 低コスト運用: `effort=medium` または `low`
- 緊急時: v0.4.3 相当（reasoning未送信）へ戻す
