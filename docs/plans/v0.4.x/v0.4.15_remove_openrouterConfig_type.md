# 実装プラン: v0.4.15 — `openrouterConfig` 型定義クリーンアップ

---

## 0. このドキュメントの位置づけ

- 目的: `EpisodicPluginConfig.openrouterConfig` が `loadConfig()` で返されない型契約違反を解消する
- 深刻度: **MEDIUM** — 現在の実害なし（0箇所で参照）。将来の開発者が `cfg.openrouterConfig?.maxTokens` を使おうとして常に `undefined` を取得するミスを防ぐ
- 発見元: `docs/incidents/v0.4.14_config_pipeline_regression_audit.md` セクション2
- ソースコード検証: **済**
- **前提**: v0.4.15 の maxTokens リグレッション修正（`openrouterMaxTokens` 追加）と同時適用が前提

---

## 1. 問題定義

`types.ts` L124-132 に `openrouterConfig` ネストインターフェースが存在する:

```typescript
openrouterConfig?: {
  model?: string;
  maxTokens?: number;
  temperature?: number;
  reasoning?: OpenRouterReasoningConfig;
};
```

しかし `loadConfig()` は `openrouterConfig` を返さない。4つのサブフィールドは全て別の平坦化フィールドに分解されている:

| ネストパス | 平坦化フィールド | 状態 |
|-----------|---------------|------|
| `openrouterConfig.model` | `openrouterModel` | v0.4.14 から存在 |
| `openrouterConfig.maxTokens` | `openrouterMaxTokens` | v0.4.15 で追加（前提） |
| `openrouterConfig.temperature` | `narrativeTemperature` | v0.4.14 から存在 |
| `openrouterConfig.reasoning` | `openrouterReasoning` | v0.4.11 から存在 |

`cfg.openrouterConfig` にアクセスするコードは0箇所。ネストオブジェクトは不要な型契約違反。

---

## 2. 修正内容

### 2.1 `types.ts` — `openrouterConfig` ネストを削除

```typescript
// Before (types.ts L124-132):
/** Nested OpenRouter config for narrative generation. */
openrouterConfig?: {
  model?: string;
  maxTokens?: number;
  temperature?: number;
  reasoning?: OpenRouterReasoningConfig;
};

// After: （削除）
```

4つのサブフィールドは全て平坦化フィールドとして存在するため、ネストは不要。

### 2.2 `index.ts` — TypeBox スキーマの `openrouterConfig` を削除

```typescript
// Before: openrouterConfig TypeBox スキーマ定義（ネストオブジェクト）
openrouterConfig: Type.Optional(Type.Object({
  model: Type.Optional(Type.String()),
  maxTokens: Type.Optional(Type.Integer({ minimum: 256, maximum: 32768 })),
  temperature: Type.Optional(Type.Number({ minimum: 0, maximum: 1 })),
  reasoning: Type.Optional(Type.Object({ ... })),
})),

// After: （削除）
```

> **✅ 安全確認済（Pro Engineer Review 追試）**: OpenClaw（DennnouAibou）のランタイムバリデーションは
> `openclaw.plugin.json` の `configSchema` を AJV で検証する（`DennnouAibou/src/plugins/schema-validator.ts`）。
> `episodicClawPlugin.configSchema`（TypeBox）はランタイムバリデーションに使用されないため、
> `additionalProperties: false` による致命的リグレッションは発生しない。
> TypeBox 削除後も `openclaw.plugin.json` の `"openrouterConfig"` スキーマがユーザー設定の入り口として機能する。

`loadConfig()` が `rawConfig?.openrouterConfig?.model` 等で個別に読み取るため、型からの削除は安全。

## 4. リスク評価

- **リスク**: 極低。0箇所で `cfg.openrouterConfig` を参照していないため削除の影響なし
- **注意**: `openclaw.plugin.json` の `"openrouterConfig"` スキーマは残す。ユーザー設定の入り口として必要
- **✅ 安全確認済**: `index.ts` TypeBox スキーマの `openrouterConfig` を削除しても、OpenClaw は `openclaw.plugin.json` の `configSchema`（AJV）でバリデーションするため、ユーザー設定パスに影響なし

- **ロールバック**: 2ファイルの差分で復元可能
