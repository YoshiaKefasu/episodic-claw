# 実装プラン: v0.4.17 — Narrative Output Control Overhaul（物語化出力制御の全面改修）

---

## 0. このドキュメントの位置づけ

- 目的: OpenRouter LLM（特に Gemma 系 free モデル）の物語化出力が「親切なアシスタント」「推論メモ」「箇条書きレポート」に崩れる問題を、プロンプトアーキテクチャ + 品質ゲート + reasoning 設定の3軸で修正する
- 深刻度: **HIGH** — 不良出力が previousEpisode として次回に再注入され、連鎖的に品質が劣化する。v0.4.11 の品質ゲートは圧縮比とオウム返しのみを検査し、assistant-mode 出力を素通りさせる
- 発見元: v0.4.16 リリース後の実運用観測 + GPT 5.4 Extra High 監査（2025-04-19）
- 前提プラン: [v0.4.11](v0.4.11_narrative_quality_gate_overhaul.md)（圧縮比 + オウム返し検出）— 本プランは既存ゲートを拡張する
- ソースコード検証: **済**（narrative-worker.ts, openrouter-client.ts, config.ts, types.ts, openclaw.plugin.json）

---

## 1. 問題定義

### 1.1 発生した事象

v0.4.16 リリース後の実運用で、2つの不良出力が観測された：

**不良出力A — CoT（推論）漏れ** (`conversation-summary-generation.md`, 1161 tokens):

```
Okay, let's start by understanding what the user is asking for. They want a narrative
summary of a conversation between Yosia and Kasou, following specific formatting rules.
The key points are to avoid verbatim repetition, use third-person past tense...
First, I need to parse the provided conversation log...
I should focus on the main topics...
I need to ensure that each part of the conversation is summarized...
Additionally, the user provided examples...
Finally, I must adhere to the character limit...
```

→ モデルの内部推論が `<thinking>` タグなしで本文に混入。末尾に辛うじて物語段落があるが、6段落の推論の後に埋もれている。

**不良出力B — Assistant-mode 崩れ** (`openclaw-update-dennouaibou.md`, 273 tokens):

```
ありがとうございます！このプロジェクトの最後をまとめました。
---
**OpenClawのアップデートと「DennouAibou」の誕生！**
- **開発の背景**
1. **CLIの安定化** –
2. **ベクトル化と最適化** –
3. **並列処理の強化** –
4. **自動デバッグ機能の追加** –
5. **ユーザーフレンドリーなインターフェース** –
- **今後の展望**
✨ *Kasou* ✨ (≧∇≦)b
```

→ 箇条書き・番号リスト・Markdown見出し・絵文字・署名。プロンプトの7項目の禁止事項のうち5項目違反。モデルが「語り手」ではなく「便利なアシスタント」として出力。

### 1.2 既存品質ゲートが検知できない理由

v0.4.11 の品質ゲート（圧縮比 + オウム返し検出）の限界：

| 不良出力 | 圧縮比 | オウム返し | 判定 |
|----------|--------|-----------|------|
| A (CoT漏れ) | 1161/47960 ≈ 2.4% ✅ | 推論文は入力に含まれない ✅ | **通過** ❌ |
| B (Assistant-mode) | 273/47960 ≈ 0.57% ⚠️ | 一部オウム返しあり ⚠️ | 圧縮比で弾ける場合とそうでない場合がある |

→ **出力の「形式」を検査するゲートが存在しない**のが根本欠陥。

### 1.3 根本原因（3主因）

GPT 5.4 Extra High 監査にて特定された3つの主因：

| # | 主因 | 詳細 |
|---|------|------|
| 1 | **重要ルールを `system` に寄せすぎている** | Gemma 系は system ロールの指示追従が弱い。Google 公式ドキュメントでも「system ではなく最初の user に命令を入れろ」と案内している（従来系 Gemma）。Gemma 4 は system 対応をうたうが、free provider での安定性は未検証 |
| 2 | **`reasoning` を強く入れたまま `exclude` していない** | OpenRouter は reasoning トークンをデフォルトでレスポンスに含む。`exclude=true` が抑止策。現在のデフォルトは `effort: high, exclude: false`。推論トークンがタグなしで本文に漏れる |
| 3 | **品質ゲートが「短すぎる / オウム返し」しか見ておらず、assistant-mode を通してしまう** | 箇条書き・見出し・絵文字・CoT前置きを検出するゲートがない |

### 1.4 連鎖劣化の危険性

一度不良出力が通ると、それが `previousEpisode` として次回の LLM 呼び出しに再注入される（[narrative-worker.ts](../../src/narrative-worker.ts) L266, L337）。モデルは前回の「アシスタント風出力」を模倣し、さらにアシスタント風の出力を生成する。**品質ゲートなしでは連鎖的に劣化する**。

### 1.5 証拠

| # | ソース | 内容 |
|---|--------|------|
| 1 | [openrouter-client.ts](../../src/openrouter-client.ts) L83 | system と user は別メッセージで送信 |
| 2 | [config.ts](../../src/config.ts) L110 | reasoning デフォルト: enabled, effort=high |
| 3 | [openclaw.plugin.json](../../openclaw.plugin.json) L205 | reasoning.exclude デフォルト: false |
| 4 | [narrative-worker.ts](../../src/narrative-worker.ts) L97, L107 | 品質ゲートは圧縮比と echo のみ |
| 5 | [test_narrative_quality_gate.ts](../../test_narrative_quality_gate.ts) L20 | テストもその2種類のみ |
| 6 | [narrative-worker.ts](../../src/narrative-worker.ts) L266, L337 | 不良出力が previousEpisode に再注入 |
| 7 | OpenRouter reasoning tokens ドキュメント | https://openrouter.ai/docs/guides/best-practices/reasoning-tokens |
| 8 | Google Gemma prompt structure | https://ai.google.dev/gemma/docs/core/prompt-structure |

| 10 | Gemma 4 model card | https://huggingface.co/google/gemma-4-31B-it |

### 1.6 なぜ Gemma で特に崩れるのか

Gemma が壊れているのではなく、今の使い方が Gemma の苦手に刺さっている：

- 三人称過去形の文学調
- 日本語（Google 自身が「英語以外は prompt だけだと理想通りになりにくい」と書いている）
- 長い会話ログ
- 強い style 制約
- free routing（レイテンシ不安定・モデル実体不明）
- reasoning 有効

この条件では、Gemma は「厳密な narrator」より「説明上手な assistant」に落ちやすい。
しかも今の system prompt 自身が `##`、番号、強調、Good/Bad セクションを含んでいるため、**弱いモデルはその見た目まで真似する**。不良出力Bはまさにこのパターン。

---

## 2. 解決策: 3軸同時修正

```
Axis 1: Prompt Architecture（contract-first 再設計）
  → モデルがそもそも assistant-mode に崩れにくくする

Axis 2: Reasoning Config（exclude=true 既定化）
  → CoT トークンがレスポンスに含まれないようにする

Axis 3: Format Quality Gate（checkNarrativeFormat 追加）
  → Axis 1+2 をすり抜けた不良出力を Retry する
  → 連鎖劣化を断ち切る
```

---

## 3. Axis 1: Prompt Architecture — contract-first 再設計

### 3.1 設計原則

GPT 5.4 の分析に基づく5つの原則：

1. **ルールを user prompt の先頭に置く** — Gemma は user ロールの指示により強く追従する
2. **system prompt は極短にする** — 役割の1行宣言のみ。Gemma free には長い system は重い
3. **悪い例を長く見せない** — Bad example を削除。プロンプト内の Markdown 記号を減らす
4. **出力契約で制御する（人格ではなく形式で固定）** — 「語り手になれ」ではなく「本文だけを書け」
5. **出力プライミング** — ユーザープロンプト末尾で最初のトークンを物語モードに固定

### 3.2 DEFAULT_SYSTEM_PROMPT 変更

**現行** (narrative-worker.ts L17-45): 約30行。few-shot例 + BAD例 + 7ルール。Markdown 記号多数。

**変更後**:

```typescript
const DEFAULT_SYSTEM_PROMPT = `Distill conversation logs into third-person past-tense narrative prose. Output narrative text only.`;
```

**変更点**:
- 30行 → 1行。Gemma が拾いやすい短い命令文
- 役割宣言のみ。ルールは user prompt 側に移動
- 悪い例（Bad output セクション）を削除 — プロンプト内の Markdown 記号がモデルに模倣されるのを防止
- Good example も削除 — user prompt 側に短い書き出し例を移動

### 3.3 DEFAULT_USER_PROMPT_TEMPLATE 変更

**現行** (narrative-worker.ts L48-50):

```typescript
const DEFAULT_USER_PROMPT_TEMPLATE = (previousEpisode: string | undefined, conversationText: string): string =>
  `${previousEpisode ? `Previous episode:\n${previousEpisode}\n---\n` : ""}Please narrativize the following conversation:\n\n${conversationText}`;
```

**変更後**（英語 — 国際ユーザー向け。日本語・中国語・韓国語ユーザーはカスタム .md ファイルで上書き）:

```typescript
const DEFAULT_USER_PROMPT_TEMPLATE = (previousEpisode: string | undefined, conversationText: string): string =>
  `HIGHEST PRIORITY. Violating any rule below makes the output invalid.

Output spec:
Third-person past tense only. One continuous narrative body. Start from the very first character with the story.
Greetings, prefaces, explanations, bullet points, numbered lists, headings, Markdown, emoji, signatures are FORBIDDEN.
"Okay" "Let me" "First" "I need to" "Sure" "Here is" "Thank you" and similar planning notes or assistant-tone phrases are FORBIDDEN.
Never copy-paste from the conversation log. Rephrase all content as natural narrative prose.
Do NOT drop technical details: file names, commands, errors, decisions must be preserved.
Do NOT role-play as a conversation participant. Do NOT explain from outside the story.
Do NOT write reasons you cannot comply. Just write the narrative.

Good opening example:
Late that evening at his desk, he pored over the logs searching for the next move.
${previousEpisode ? `\nPrevious episode:\n${previousEpisode}\n` : ""}
The text below is raw material, NOT your output.
<<<LOG>>>
${conversationText}
<<<END_LOG>>>

Write narrative text only.`;
```

**変更点**:
- ルールを user prompt の先頭に移動（Gemma の指示追従力を活用）
- 「最優先命令」でルールの優先度を明示
- 禁止語のブラックリストを明記（CoT前置き + アシスタント口調）
- 悪い例を長く見せず、短い「良い書き出し例」1行のみ
- `<<<LOG>>>` / `<<<END_LOG>>>` で会話ログを明確に区切る（「これは素材であり出力ではない」の認識を強化）
- 末尾「本文だけを書け。」で出力プライミング（最初のトークンを物語モードに固定）
- Markdown 記号を極限まで削減

### 3.4 選択肢の比較と選定理由

| 選択肢 | 評価 |
|--------|------|
| **contract-first（採用）** | ルール先頭 + 悪い例削除 + 出力プライミング。Gemma の指示追従パターンに合致 |
| 現状維持（system に全ルール） | ❌ Gemma は system ロールの長い指示に弱い。Google 公式も推奨していない |
| 2パス生成パイプライン | ❌ YAGNI 違反。API 呼び出し2倍。システムが過剰に複雑化 |
| 入力前処理（絵文字・口調 strip） | ❌ 会話の「感情の機微」が失われる。物語の質が低下 |

## 4. Axis 2: Reasoning Config — exclude=true 既定化

### 4.1 変更対象

1. `src/config.ts` L111 — `normalizeOpenRouterReasoning()` の呼び出し
2. `openclaw.plugin.json` — `reasoning.exclude` のデフォルト値

### 4.2 変更内容

**config.ts**: `normalizeOpenRouterReasoning` に `exclude: true` をデフォルトとして追加。

現行の `normalizeOpenRouterReasoning()` は `raw.exclude === true` の場合のみ `exclude: true` を返す。デフォルトでは `exclude` は undefined（=false）。

変更: `normalizeOpenRouterReasoning()` の戻り値で、`exclude` が明示的に `false` に設定されていない限り `exclude: true` をデフォルトとする。

```typescript
// 変更前:
if (raw.exclude === true) {
  exclude = true;
}

// 変更後:
// Default exclude=true for narrative path — prevents CoT token leakage into output
// Only disable if user explicitly sets exclude: false
if (raw.exclude !== false) {
  exclude = true;
}
```

**openclaw.plugin.json**: `reasoning` スキーマの `exclude` デフォルト値を `true` に変更。

**理由**: OpenRouter は reasoning トークンをデフォルトでレスポンスに含む（証拠7）。Gemma 系は `<thinking>` タグなしで推論を本文に混ぜる傾向がある。`exclude=true` は API レベルで推論をレスポンスから除外する唯一の確実な手段。

### 4.3 Gemma 系の追加対策（オプション）

Gemma 系モデル（`google/gemma-*`）では `effort` を `low` または `none` に下げることで、推論トークンの消費を減らし、本編のトークン予算を確保できる。ただし、これはモデル名に基づく条件分岐になり KISS に反するため、**v0.4.17 では実装しない**。ユーザーが設定で `effort: low` を指定することで対応可能。

## 5. Axis 3: Format Quality Gate — checkNarrativeFormat()

### 5.1 変更対象

`src/narrative-worker.ts` — 品質ゲート関数の追加と `narrativizeWithRetry()` への統合

### 5.2 checkNarrativeFormat() 仕様

出力が物語形式の要件を満たすかを検査する。**重くしない**。正規表現による5チェックのみ。

```typescript
/**
 * [NEW v0.4.17] Quality gate: check if output conforms to narrative format.
 * Detects assistant-mode outputs, CoT leakage, and prohibited formatting.
 * Returns { pass: boolean, reason: string }.
 */
export function checkNarrativeFormat(text: string): { pass: boolean; reason: string } {
  const lines = text.split(/\r?\n/);
  const firstLine = lines[0] ?? "";
  const firstChars = text.substring(0, 100);

  // Gate 1: Line starts with markdown header / list / numbered list
  for (const line of lines) {
    if (/^\s*#{1,6}\s/.test(line)) {
      return { pass: false, reason: "narrative-format: markdown header detected" };
    }
    if (/^\s*[-*]\s/.test(line)) {
      return { pass: false, reason: "narrative-format: bullet list detected" };
    }
    if (/^\s*\d+\.\s/.test(line)) {
      return { pass: false, reason: "narrative-format: numbered list detected" };
    }
  }

  // Gate 2: English planning phrase in first 100 chars (CoT leakage)
  const cotPatterns = [
    /\bOkay\b.*\b(let me|I need|I should|I'll|first)\b/i,
    /\bLet me\b.*\b(parse|understand|analyze|start|think)\b/i,
    /\bFirst,?\s+I\s+(need|should|will|must)\b/i,
    /\bI need to\b.*\b(parse|understand|focus|ensure)\b/i,
  ];
  for (const pat of cotPatterns) {
    if (pat.test(firstChars)) {
      return { pass: false, reason: "narrative-format: CoT planning phrase detected" };
    }
  }

  // Gate 3: Japanese assistant-mode phrases at the start of the first line
  // [v0.4.17] Scoped to the beginning of firstLine (not full text) to avoid False Positives.
  // When a conversation character legitimately says "ありがとうございます" as role-play,
  // the narrative will embed it mid-sentence (e.g. 彼は「ありがとうございます」と答えた)
  // — which does NOT start the line with the phrase. Only assistant-mode outputs
  // start the output with these phrases (e.g. "ありがとうございます！まとめました。").
  const assistantPhrases = [
    "ありがとうございます",
    "以下の通りです",
    "まとめました",
    "今後の展望",
    "要点をまとめ",
    "お手伝いします",
  ];
  for (const phrase of assistantPhrases) {
    if (firstLine.trimStart().startsWith(phrase)) {
      return { pass: false, reason: `narrative-format: assistant-mode phrase "${phrase}" detected` };
    }
  }

  // Gate 4: Emoji / kaomoji
  const emojiPat = /[\p{Emoji_Presentation}\u{1F600}-\u{1F64F}\u{1F300}-\u{1F5FF}\u{2600}-\u{26FF}\u{2700}-\u{27BF}≧∇≦]/u;
  if (emojiPat.test(text)) {
    return { pass: false, reason: "narrative-format: emoji or kaomoji detected" };
  }

  // Gate 5: First line doesn't look like narrative start
  // Narrative starts with: CJK character, or proper noun, or time expression
  // Assistant starts with: greeting, explanation, or English planning
  const narrativeStartPat = /^[\p{Script=Hiragana}\p{Script=Katakana}\p{Script=Han}A-Z\u00C0-\u017F"「『]/u;
  if (!narrativeStartPat.test(firstLine.trim())) {
    // Allow lowercase Latin starts (some narratives start with "the", "a")
    if (!/^[a-z]/.test(firstLine.trim())) {
      return { pass: false, reason: "narrative-format: first line doesn't look like narrative start" };
    }
  }

  return { pass: true, reason: "" };
}
```

### 5.3 sanitizeNarrativeOutput() の CoT 前置き削除追加

出力プライミングや品質ゲートをすり抜けた場合のセーフティネットとして、サニタイザに CoT 前置きの検出・削除を追加する。

```typescript
// sanitizeNarrativeOutput() に追加（Step 2 の後）
// Step 2.5: Strip CoT planning prefix (untagged reasoning leakage)
// Matches patterns like "Okay, let me..." through to the first narrative-like line
const cotPrefixPat = /^(?:Okay[,.]?\s*(?:let me|I need|I should|I'll|first)[^.]*\.\s*\n?)+/im;
cleaned = cleaned.replace(cotPrefixPat, "");
```

**注意**: これはセーフティネットであり、主力は Axis 1（プロンプト）と Axis 2（exclude=true）。サニタイザでの削除は最後の防衛線。

### 5.4 narrativizeWithRetry() への統合

`checkCompressionRatio()` と `checkEchoDetection()` の後に `checkNarrativeFormat()` を追加する。既存のゲートと同じパターンで Retry を行う。

```typescript
// [v0.4.11] Quality gate 1: Token count & Compression ratio  (既存)
// [v0.4.11] Quality gate 2: Verbatim echo detection            (既存)
// [v0.4.17] Quality gate 3: Narrative format check             (NEW)

const formatCheck = checkNarrativeFormat(text);
if (!formatCheck.pass) {
  console.warn(
    `[NarrativeWorker] Attempt ${attempt + 1}/${MAX_RETRIES}: ${formatCheck.reason} for [${item.id}]. Retrying...`
  );
  await this.sleep(500);
  continue;
}
```

**ゲートの実行順序**:

```
1. MIN_NARRATIVE_TOKENS（最安値・即判定）
2. 圧縮比ゲート（構造的判定）
3. オウム返し検出（文字列比較・やや重い）
4. フォーマット検査（正規表現・やや重い）   ← NEW
```

短い出力は1で弾ける。圧縮比不適は2で弾ける。3と4は2を通ったものだけ実行する。無駄な正規表現実行を避けるため。

## 9. リスク評価

- **リスク**: **低〜中**。プロンプト変更は可逆的。品質ゲートは閾値調整可能。`exclude=true` は API レベルの設定でリスクなし。
- **偽陽性リスク**: `checkNarrativeFormat()` の「最初の行が物語っぽくない」検査で、短い入力の要約が誤弾きされる可能性あり。ただし Gate 5（最初の行の文字種チェック）は CJK / 大文字始まり / 小文字始まりのいずれかなら通過するため、偽陽性は極めて低い。Gate 3（日本語アシスタント口調）は先頭行のみを検査するため、会話中のキャラ演技による正当な「ありがとうございます」等は文中に登場し False Positive にならない。
- **ユーザー影響**: **改善のみ** — 不良出力の保存が防止され、previousEpisode の連鎖劣化が断ち切られる。
- **ロールバック**: 各 Axis は独立してロールバック可能。
  - Axis 1: DEFAULT_SYSTEM_PROMPT / DEFAULT_USER_PROMPT_TEMPLATE を旧版に戻す
  - Axis 2: `normalizeOpenRouterReasoning()` の `exclude` デフォルトを `false` に戻す
  - Axis 3: `checkNarrativeFormat()` 呼び出しを `narrativizeWithRetry()` から削除
- **カスタムプロンプトユーザーへの影響**: Axis 1 の変更は `DEFAULT_*` 定数のみ（英語版）。ユーザーの `narrativeSystemPrompt` / `narrativeUserPromptTemplate` が設定されている場合はそちらが優先される（既存の `resolveSystemPrompt()` / `resolveUserPrompt()` ロジック通り）。日本語ユーザーは日本語版 .md ファイルを config で指定する。
- **多言語プロンプトの配布場所**: `prompts/` ディレクトリに各言語版の system.md / user.md を配置する。ユーザーは自身の言語に合わせて .md パスを config に設定する。
  - 英語: コード内 DEFAULT（.md ファイル不要）
  - 日本語: `prompts/episodic_claw_narrative_system_ja.md` / `prompts/episodic_claw_narrative_user_ja.md`
  - 中国語: `prompts/episodic_claw_narrative_system_zh.md` / `prompts/episodic_claw_narrative_user_zh.md`
  - 韓国語: `prompts/episodic_claw_narrative_system_ko.md` / `prompts/episodic_claw_narrative_user_ko.md`

## 10. ユーザーへのカスタムプロンプト改善推奨

現在のカスタムプロンプト（`episodic_claw_narrative_system.md` / `episodic_claw_narrative_user.md`）は、ユーザー設定で `resolveSystemPrompt()` / `resolveUserPrompt()` から読み込まれるため、コード側の DEFAULT 値変更の影響を受けない。

**推奨**: カスタムプロンプトも contract-first 構造に更新する。具体的には：

**system.md**（日本語版 — 日本人ユーザー向け）:
```markdown
会話記録を三人称・過去形の日本語の地の文へ編纂する。返答は本文のみ。
```

**user.md**（日本語版 — 日本人ユーザー向け）:
```markdown
最優先命令。以下を1つでも破る出力は失敗。

出力仕様:
日本語のみ。三人称・過去形。一つの連続した本文だけを書く。最初の一文字から物語を始める。
挨拶、前置き、説明、箇条書き、番号、見出し、Markdown、絵文字、署名は禁止。
「Okay」「Let me」「First」「I need to」「ありがとうございます」「以下」「まとめました」「今後の展望」などの作業メモや応答者口調は禁止。
会話ログのコピペは禁止。内容は自然な地の文に言い換える。
ファイル名、コマンド、エラー、決定事項などの技術情報は落とさない。
会話参加者として振る舞わない。物語の外から説明しない。
守れない理由も書かない。本文だけを書く。

良い書き出し例:
夜更けの作業机では、彼はログを追いながら次の手を探っていた。

【前回の物語（文脈の接続用）】
{previousEpisode}

以下は記録された会話ログです。これは素材であり、出力ではない。
<<<LOG>>>
{conversationText}
<<<END_LOG>>>

本文だけを書け。
```

> **多言語対応**: 中国語版・韓国語版はそれぞれ別の .md ファイルとして提供する。
> ユーザーは config の `narrativeSystemPrompt` / `narrativeUserPromptTemplate` で該当言語の .md パスを指定する。
