## 1. 問題定義

### 1.0 Bug #0: 設定伝播が壊れている — プラグイン個別設定が loadConfig() に届かない（🔴 BLOCKER）

| 項目 | 内容 |
|------|------|
| 深刻度 | 🔴 **BLOCKER** — Bug #1 よりも先に直さないと、パス解決の修正が一切効果がない |
| 影響 | 全プラットフォーム（Linux/Windows/macOS）でカスタムプロンプトが読み込まれない |
| 発見 | ユーザー指摘「Linuxでも読み込まれない」+ ソースコード追跡（2026-04-21） |

**根本原因**:

index.ts L481-482:
```typescript
const openClawGlobalConfig = api.runtime?.config?.loadConfig?.() || {};
const cfg = loadConfig(openClawGlobalConfig);  // ← グローバル設定全体を渡している！
```

- `api.runtime.config.loadConfig()` は DennouAibou の `config/config.ts` の `loadConfig()` を呼び出し、**openclaw.json 全体**を返す
- `loadConfig(openClawGlobalConfig)` は `rawConfig?.narrativeSystemPrompt`（トップレベル）を探す
- しかし実際の値は `rawConfig?.plugins?.entries?.["episodic-claw"]?.config?.narrativeSystemPrompt` にネストされている
- **結果**: `rawConfig?.narrativeSystemPrompt` は常に `undefined` → `resolvePrompt(undefined)` → `""` → 英語DEFAULT

**データフロー図**:

```
openclaw.json
  └─ plugins.entries.episodic-claw.config
       ├─ narrativeSystemPrompt: "/home/kasou_yoshia/.openclaw/prompts/..."
       └─ narrativeUserPromptTemplate: "/home/kasou_yoshia/.openclaw/prompts/..."

api.runtime.config.loadConfig()  →  グローバル設定全体を返す
                                      ↓
loadConfig(グローバル設定)  →  rawConfig?.narrativeSystemPrompt
                                      ↓
                              undefined（トップレベルにない！）
                                      ↓
                              resolvePrompt(undefined) → ""
                                      ↓
                              英語DEFAULTプロンプト ← 全プラットフォームで発生！
```

**DennouAibou 側の正しいAPI**:

`OpenClawPluginApi` 型（types.ts L2016+）は2つの別々の設定アクセスを提供する:
- `api.config: OpenClawConfig` — グローバル設定（openclaw.json全体）
- `api.pluginConfig?: Record<string, unknown>` — プラグイン個別設定（`plugins.entries.episodic-claw.config` の中身だけ）

episodic-claw の index.ts は `api.pluginConfig` を**一切使用していない**。

**これがLinuxでも読み込まれない理由。プランの前提「Linuxでは動く」は間違いだった。**

（以下 §1.0.1 に詳細説明を記載）

### 1.0.1 Bug #0 詳細説明 — 真の問題の3層データフロー

#### 問題の本質

**ユーザーが設定したカスタムプロンプトが、一切のプラットフォームで読み込まれていない。** これはパス解決の問題（Windows vs Linux）ではなく、もっと前の段階で設定値が消滅している。

#### 3層のデータフロー

設定値は以下の3つのステップを経てプロンプト生成に使われる：

```
Step 1: openclaw.json に設定を書く
Step 2: プラグインの register(api) に設定が渡る
Step 3: loadConfig() が設定値を解釈する
```

**Step 2 → Step 3 の間で設定値が消滅している。**

#### 各ステップの詳細

##### Step 1: ユーザーの openclaw.json

ユーザーはこう設定している：

```json
{
  "plugins": {
    "entries": {
      "episodic-claw": {
        "config": {
          "narrativeSystemPrompt": "/home/kasou_yoshia/.openclaw/prompts/episodic_claw_narrative_system.md",
          "narrativeUserPromptTemplate": "/home/kasou_yoshia/.openclaw/prompts/episodic_claw_narrative_user.md",
          "subagent": { "allowModelOverride": true }
        }
      }
    }
  }
}
```

✅ 設定値は `plugins.entries["episodic-claw"].config` の下に**正しく存在**している。

##### Step 2: DennouAibou のプラグインローダーが `api` オブジェクトを構築

DennouAibou の `loader.ts` は2つの別々の設定アクセスを `api` に設定する：

| API プロパティ | 内容 | 値の構造 |
|---|---|---|
| `api.runtime.config.loadConfig()` | openclaw.json **全体** | `{ agents: {...}, plugins: { entries: {...} }, ... }` |
| `api.pluginConfig` | `plugins.entries["episodic-claw"].config` **だけ** | `{ narrativeSystemPrompt: "/home/...", narrativeUserPromptTemplate: "/home/...", subagent: {...} }` |

`api.pluginConfig` はまさにユーザーが設定した値だけを含む「プラグイン個別設定」。これが正しい入力。

##### Step 3: episodic-claw の index.ts が設定を読み込む（🔴 ここが壊れている）

`index.ts` L481-482：

```typescript
const openClawGlobalConfig = api.runtime?.config?.loadConfig?.() || {};
const cfg = loadConfig(openClawGlobalConfig);  // ← グローバル設定全体を渡している！
```

**`loadConfig(openClawGlobalConfig)` に渡しているのは、openclaw.json 全体。**

そして `config.ts` の `loadConfig()` は：

```typescript
narrativeSystemPrompt: resolvePrompt(rawConfig?.narrativeSystemPrompt),
//                                  ↑ トップレベルを探す！
```

**しかし実際の値はどこにあるか？**

```
openClawGlobalConfig = {
  agents: {...},
  plugins: {
    entries: {
      "episodic-claw": {
        config: {                    ← ここにいる！
          narrativeSystemPrompt: "/home/kasou_yoshia/.openclaw/prompts/..."
        }
      }
    }
  }
}
```

`rawConfig?.narrativeSystemPrompt` → トップレベルを見る → **そんなものはない** → `undefined`

#### 結果の連鎖

```
rawConfig?.narrativeSystemPrompt = undefined
                    ↓
resolvePrompt(undefined) = ""     （config.ts: if (!value) return "";）
                    ↓
cfg.narrativeSystemPrompt = ""    （空文字 = インラインプロンプトなし）
                    ↓
NarrativeWorker は 英語DEFAULTプロンプトを使用
                    ↓
全ての物語（エピソード）が 英語 で生成される ← 全プラットフォームで発生！
```

#### なぜ「Linuxでも読み込まれない」のか

これまでの前提は「Linuxでは動く、Windowsでは動かない（パス解決の問題）」だった。しかし：

| 環境 | `rawConfig?.narrativeSystemPrompt` | 結果 |
|------|-----------------------------------|------|
| **Linux** | `undefined`（ネストされているから） | `""` → 英語DEFAULT |
| **Windows** | `undefined`（同じ理由） | `""` → 英語DEFAULT |
| **macOS** | `undefined`（同じ理由） | `""` → 英語DEFAULT |

**パス解決の問題（Bug #1）は実在するが、Bug #0 が先にブロックしているため、Bug #1 の影響は全く現れていない。** パス解決を直しても、設定値が `resolvePrompt()` に届かなければ意味がない。

#### 正しい修正

`loadConfig()` に渡すべきは **プラグイン個別設定**（`api.pluginConfig`）。修正の完全な仕様は **→ §2.0 Fix 0 参照**。

概要のみ記載：
- `api.pluginConfig`（第一候補）= DennouAibou のローダーが `plugins.entries["episodic-claw"].config` から抽出した値
- フォールバック（`openClawGlobalConfig.plugins.entries["episodic-claw"].config`）= `api.pluginConfig` が提供されない古いバージョンへの防衛策
- `openClawGlobalConfig` 自体は `gateway_start` 内の workspace 解決で引き続き使用するため削除しない

#### 2つのBugの関係

```
Bug #0（設定伝播） ← 🔴 BLOCKER: 全プラットフォームで設定が届かない
    ↓ 修正後
Bug #1（パス解決） ← 🟡 LinuxパスをWindowsで解決できない
    ↓ 修正後
✅ カスタム日本語プロンプトが全プラットフォームで正しく読み込まれる
```

**Bug #0 を直さない限り、Bug #1 の修正は一切効果を持ちません。**

### 1.1 Bug #1: resolvePrompt() がLinux絶対パスをWindows環境で解決できない

| 項目 | 内容 |
|------|------|
| 深刻度 | 🔴 **Bug** |
| 影響 | カスタム日本語プロンプトが読み込まれず、全物語が英語DEFAULTで生成される |
| E2E証拠 | Linux path → `resolved_prompt_len: 0/0`, `fallback_used: true/true` |
| 追加証拠 | 英語エピソード `dennouaibou-system-development.md` の先頭が `DEFAULT_USER_PROMPT_TEMPLATE` の例文 `Late that evening at his desk, he pored over the logs` と一致 |

### 1.2 根本原因

**現状の `resolvePrompt()`**（config.ts L130-138）:

```typescript
function resolvePrompt(value: string | undefined): string {
  if (!value) return "";
  const trimmed = value.trim();
  if (trimmed.endsWith(".md") || trimmed.endsWith(".txt")) {
    try {
      return fs.readFileSync(path.resolve(trimmed), "utf8").trim();
    } catch {
      console.warn(`[Episodic Memory] Failed to read prompt file: ${trimmed}`);
      return "";
    }
  }
  return trimmed;
}
```

**問題**: `path.resolve("/home/kasou_yoshia/.openclaw/prompts/episodic_claw_narrative_system.md")` は Windows では:
- `C:\home\kasou_yoshia\.openclaw\prompts\episodic_claw_narrative_system.md`（cwd ベース解決）
- または `D:\home\kasou_yoshia\...`（ドライブレター依存）
- → ファイルが存在せず `catch` で `""` を返す

**E2E検証の証拠**:

| シナリオ | promptSystemPath | resolved_prompt_len | fallback_used | detected_language |
|---|---|---|---|---|
| Linux path | `/home/kasou_yoshia/.openclaw/prompts/...` | 0/0 | true/true | unknown |
| Windows path | `Y:\kasou_yoshia\.openclaw\prompts\...` | 34/737 | false/false | **ja** |

**伝播経路**: `resolvePrompt()` → `""` → `resolveSystemPrompt()` / `resolveUserPrompt()` が `""` を「カスタムプロンプトなし」と判定 → 英語DEFAULTプロンプトを使用 → 物語が英語で生成される

> **注意**: Bug #0 が修正されるまで、この伝播経路の起点は「パスが解決できない」ではなく「設定値がそもそも届かない」。Bug #0 を先に直さないと、Bug #1 の修正効果は一切確認できない。

### 2.1 Fix 1: resolvePrompt() クロスプラットフォームパス解決

**変更後**:

```typescript
import * as os from "os";  // ← 追加

/**
 * Resolve a prompt value that may be inline text, a file path, or a ~/ path.
 * Cross-platform: handles Linux absolute paths (/home/...), Windows paths (Y:\...),
 * and home-dir shortcuts (~/...) on all platforms.
 *
 * @param value - The prompt value to resolve (inline text, file path, or ~/ path)
 * @param platform - Platform identifier for testing injection (default: process.platform)
 */
function resolvePrompt(value: string | undefined, platform: string = process.platform): string {
  if (!value) return "";
  const trimmed = value.trim();
  if (!trimmed.endsWith(".md") && !trimmed.endsWith(".txt")) {
    return trimmed;  // inline text
  }

  // Build a list of candidate paths to try in order
  const candidates: string[] = [];

  // 1. Resolve ~ / ~/path — works on all platforms
  if (trimmed.startsWith("~/") || trimmed.startsWith("~\\")) {
    const homeDir = process.env.HOME || process.env.USERPROFILE || os.homedir();
    candidates.push(path.join(homeDir, trimmed.slice(2)));
  } else if (trimmed === "~") {
    const homeDir = process.env.HOME || process.env.USERPROFILE || os.homedir();
    candidates.push(homeDir);
  }

  // 2. path.resolve(trimmed) — handles Windows absolute (C:\...) and relative paths
  //    Same behavior as before (backward compatible)
  candidates.push(path.resolve(trimmed));

  // 3. Cross-platform: If original is a POSIX absolute path (/home/user/...) and
  //    we're on Windows, try resolving under the Windows user home directory.
  //    This handles the case where openclaw.json was written on Linux and the
  //    plugin runs on Windows (WSL, dual-boot config share, etc.)
  //    [Deep Review] Uses `platform` parameter instead of process.platform for testability
  if (platform === "win32" && trimmed.startsWith("/home/")) {
    const homeDir = process.env.HOME || process.env.USERPROFILE || os.homedir();
    // /home/kasou_yoshia/.openclaw/... → HOME\.openclaw\...
    // Skip the username segment and join by the path after it
    const relativeFromHome = trimmed.slice("/home/".length);  // kasou_yoshia/.openclaw/...
    const firstSlash = relativeFromHome.indexOf("/");
    if (firstSlash > 0) {
      const afterUsername = relativeFromHome.slice(firstSlash + 1);  // .openclaw/...
      candidates.push(path.join(homeDir, afterUsername));
    }
  }

  // Try each candidate in order; first successful read wins
  for (const candidate of candidates) {
    try {
      const content = fs.readFileSync(candidate, "utf8").trim();
      if (content.length > 0) {
        if (candidate !== path.resolve(trimmed)) {
          console.log(`[Episodic Memory] Prompt file resolved via cross-platform path: ${trimmed} → ${candidate}`);
        }
        return content;
      }
    } catch {
      // Continue to next candidate
    }
  }

  // All candidates failed — log for observability
  console.warn(
    `[Episodic Memory] Failed to read prompt file: ${trimmed}` +
    ` (tried: ${candidates.join(", ")})`
  );
  return "";
}
```

**追加import**: `import * as os from "os";`（config.ts の先頭に追加）

**[Deep Review 調整]**:
- Candidate #4（`candidates.push(trimmed)`）を削除 — Windows で POSIX 絶対パスをそのまま `fs.readFileSync()` に渡すのは確実失敗（Candidate #2 と同義）。無駄な I/O + ログ汚染を除去
- `resolvePrompt()` に `platform` パラメータを追加 — `process.platform` を直接参照せず関数パラメータで注入することで、Linux/macOS CI でも Windows 用ロジックをテスト可能にする。SOLID（依存性注入）に合致
- Candidate #2 のコメントに「Same behavior as before (backward compatible)」を追記 — 後方互換性を明示

### 2.2 解決パスの具体例

| 入力 | Windows での候補リスト | 期待される結果 |
|------|----------------------|---------------|
| `/home/kasou_yoshia/.openclaw/prompts/system.md` | `D:\GitHub\...\home\kasou_yoshia\...`（path.resolve）, `C:\Users\kasou_yoshia\.openclaw\prompts\system.md`（HOME マッピング、HOME=`C:\Users\kasou_yoshia`の場合） | 2番目がヒット |
| `Y:\kasou_yoshia\.openclaw\prompts\system.md` | `Y:\kasou_yoshia\.openclaw\prompts\system.md`（path.resolve直接 — 従来と同じ） | 1番目がヒット |
| `~/.openclaw/prompts/system.md` | `C:\Users\kasou_yoshia\.openclaw\prompts\system.md`（HOME展開） | 1番目がヒット |
| inline text | (ファイルパスとして処理しない) | そのまま返す |

> [Deep Review 調整] 候補リストが4→3に変更。Candidate #4（元パスをそのまま試す）は Windows で Candidate #2 と同義のため削除。

### 2.3 設計判断

- `os.homedir()` / `process.env.HOME` / `process.env.USERPROFILE` の3ソースから home を決定
- `/home/USERNAME/` の USERNAME セグメントをスキップして home 直下にマッピングする（Linux の `/home/kasou_yoshia/` → Windows の `%HOME%\`）。これは最も一般的な設定共有パターン（Linux/WSL と Windows で同じ `~/.openclaw/` ディレクトリを使う）
- 全候補失敗時は、試行したパスのリストをログ出力する（デバッグ容易化）
- 成功時も、クロスプラットフォーム経由で解決された場合にログ出力する（v0.4.19c の観測性要件を先取り）

**動作**: 空の `.md` ファイル（または空白のみ）は「見つからなかった」と同義に扱われ、次の候補にフォールスルーする。最終的に全候補失敗 → `""` 返却 → DEFAULT プロンプト使用。

**推奨**: **変更しない（KISS）。空のプロンプトファイル = プロンプトなし が正しいセマンティクス。** `content.length === 0` のケースを区別するログ追加は複雑化を招くだけで実益がない。ユーザーが空のファイルを意図的に置いたなら、DEFAULT プロンプトにフォールバックするのが最も安全な動作。

#### Issue 4: `HOMEDRIVE`+`HOMEPATH` not in 3-source pattern — 🟢 Acceptable

**現状**: plan は `process.env.HOME || process.env.USERPROFILE || os.homedir()` の3ソースを使用。

**DennouAibou の実装**: `src/infra/fs-safe.ts` も同じ3ソース。`src/infra/home-dir.ts` は `HOMEDRIVE`+`HOMEPATH` も処理するが、`os.homedir()` が内部で `HOMEDRIVE`+`HOMEPATH` を使用するため、`os.homedir()` をフォールバックにしていれば暗黙的にカバーされる。

**評価**: 3ソースで十分。`os.homedir()` が `HOMEDRIVE`+`HOMEPATH` を処理するため、明示的な追加は不要。

### ⚠️ 注意事項 (Deep)

- **🔴 Bug #0 が最優先**: これまでのレビューは「Linuxでは動く、Windowsでは動かない」という前提で行われていたが、ユーザー指摘により「Linuxでも読み込まれない」ことが判明。設定伝播Bugがパス解決Bugより先に直さなければならないBLOCKERである。Deep Review の分析内容は Bug #1 の分析として正しいが、実効性は Bug #0 修正後にのみ現れる
- **Candidate #4 削除の影響**: Git Bash 経由で Node.js を起動した場合、MSYS2 は子プロセスのパスを変換しないため、`fs.readFileSync("/home/...")` は Candidate #2 と同じく失敗する。削除しても実効性は変わらない
- **`process.platform` のテスト注入**: `Object.defineProperty` は Node.js 22+ で non-configurable になる可能性があるため推奨しない。代わりに `resolvePrompt(value, platform?)` のパラメータ化を推奨。SOLID の依存性注入にも合致する
- **v0.4.19b との DRY 統合タイミング**: Option B の `getHomeDir()` 抽出は v0.4.19b で index.ts を変更する際に併せて実施するのが最も効率的。v0.4.19a では config.ts のみの変更に留める
- **macOS `/Users/USERNAME/` パス**: 現在報告されていないため YAGNI でスコープ外。将来 macOS→Windows の設定共有が報告された場合は、`/home/` と同様のマッピングを追加する
- **v0.4.19c クロスリファレンス**: Candidate #4 を削除すると候補リストが4→3になるため、v0.4.19c §2.2 の設定構造ダンプログ（`tried: X candidates`）で候補数が変わる点に留意。v0.4.19c のログは動的に `candidates.length` を使用するため、コードの修正は不要だが、ログ出力の期待値が変わる
- **`api.pluginConfig` の信頼性**: DennouAibou の loader.ts は `pluginConfig: validatedConfig.value` を設定するが、configSchema が提供されない場合は `pluginConfig: {}` になる。episodic-claw は `configSchema: PluginConfigSchema`（TypeBox）を提供しているため、`api.pluginConfig` は常に検証済みの値が入る。フォールバック（`plugins.entries["episodic-claw"].config`）は念のため残す
