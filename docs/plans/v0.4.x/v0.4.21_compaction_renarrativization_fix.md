## 0. 概要

- 目的: 同一コンテンツが繰り返し再物語化される現象を、**compaction経路**と**plugin再起動（warm-start）経路**の両方で止める
- 深刻度: 🔴 HIGH（重複エピソードが継続生成され、ベクトルDBとナラティブ履歴を汚染）
- 発見元: 4/16〜4/19 のエピソード重複 + 4/19ログ（再起動直後に巨大バッファがflushされる時系列）

### 証拠（実データ）

| 種別 | ファイル / ログ | 観測内容 |
|------|------------------|----------|
| 重複エピソード | `episodes/2026/04/16~19/*.md` | 4/16〜4/19 で同系統トピック（Gemini fallback / ep-anchor / Memory search disabled）が反復 |
| 異常ナラティブ | `episodes/2026/04/19/episodic-claw-troubleshooting-report.md` | 先頭にプロンプト解析文（思考漏れ）が混入 |
| 再起動痕跡 | `openclaw-2026-04-19.log` | `Singleton created` / `Starting Go sidecar`（15:17台） |
| 巨大flush | `openclaw-2026-04-19.log` | `Idle timeout ... Auto-finalizing buffer (263 text messages)`（15:34:45） |
| 大量投入 | `openclaw-2026-04-19.log` + `episodic-claw 2026-04-19.log` | 2 chunks / 87155 tokens enqueue → 後続で `cache.ack` + slug refine まで到達 |

**パターン**: 「compaction後の再処理」だけでなく、「再起動直後の初回取り込み」でも既存文脈が再セグメント化され、わずかに文面の異なる同一イベントが増殖する。

## 1. バグ分析

### Bug 1（PRIMARY）: Compaction-induced Re-narrativization

**根本原因**: `EventSegmenter.forceFlush()` が `lastProcessedLength = 0` にリセットするため、compaction 後の `processTurn` が全 compacted メッセージを「新規」として処理する。

**再現フロー**:

```
1. before_compaction → archiver.forceFlush()
   → segmenter.forceFlush()
   → lastProcessedLength = 0 にリセット  ← ここが原因
   → buffer = []

2. OpenClaw compaction 実行
   → メッセージリストが短縮（例: 50msg → 30msg）

3. 次の before_prompt_build → processTurn(compactedList)
   → lastProcessedLength = 0
   → newMessages = compactedList.slice(0) = 全30メッセージ  ← 全て「新規」扱い
   → dedup: buffer空 → 全メッセージ通過
   → "First turn, just absorb" → buffer = 全30メッセージ
   → lastProcessedLength = 30

4. 以降の processTurn で境界検出 → poolAndQueue
   → 全 compacted コンテンツが narrative pipeline に投入
   → 既に物語化済みの内容が再物語化される
```

**現在の context reset 検出の問題**:

```typescript
// segmenter.ts processTurn()
if (this.lastProcessedLength > currentMessages.length) {
  // コンテキストリセット検出 → cursor を 0 に
  this.lastProcessedLength = 0;
  this.buffer = [];
}
```

`lastProcessedLength > currentMessages.length` は「真のコンテキストリセット」（セッション全消去）と「compaction による短縮」を区別しない。compaction 後も `forceFlush` が既に `lastProcessedLength = 0` にしているため、この検出自体が発火しない（`0 > 30` は false）。結果として全メッセージが「初回ターン」として吸収される。

### Bug 2（PRIMARY）: Restart-induced Warm-start Re-absorb

**根本原因**: plugin再起動後に `EventSegmenter` が新規生成され `lastProcessedLength = 0` へ戻る。既存エピソードディレクトリがあると cold-start ingest はスキップされるため、初回 `before_prompt_build` で大きな既存メッセージ列をそのまま吸収し得る。

**根拠（コード）**:

```typescript
// segmenter.ts
private lastProcessedLength = 0;

// index.ts cold-start
if (hasExistingEpisodes) {
  // 既存 episodes がある場合は ingest をスキップ
  continue;
}
```

**根拠（ログ時系列）**:

1. `Singleton created` / `Starting Go sidecar`（15:17台）
2. 約15分後に `Idle timeout ... Auto-finalizing buffer (263 text messages)`（15:34:45）
3. 同秒帯で `Enqueued 2 chunks ... tokens: 87155` → 後続で `cache.ack`

この挙動は、restart直後に過去コンテキストが初回バッファとして再吸収された時系列と整合する。

### Bug 3（SECONDARY）: `handleBatchIngest` 重複ガードなし

`handleBatchIngest` は `md5.Sum([]byte(it.Summary))` で slug を生成するが、既存レコードの存在チェックがない。わずかに異なる叙事文で同一内容のエピソードが異なる slug で保存され、ベクトルDBに重複レコードが蓄積する。

Smart Dedup（`background.go`）は background indexer（ファイルウォッチャー経路）にのみ適用され、`handleBatchIngest` RPC 経路には適用されない。

## 3. 修正詳細

### 3.1 Fix 1: `EventSegmenter.afterCompaction()` — Compaction 後の cursor 跳ばし

**修正** (`segmenter.ts` EventSegmenter クラス):

```typescript
/**
 * [v0.4.21] Compaction 後の再物語化防止。
 * after_compaction handler から呼び出し、次回 processTurn で
 * compacted メッセージを全て「既処理」として扱うようフラグを設定する。
 * 
 * forceFlush() は lastProcessedLength = 0 にリセットするが、
 * compaction 後の processTurn は 0 から全メッセージを再処理してしまう。
 * このメソッドは compaction 特有の「メッセージ短縮」を「既処理」として扱い、
 * 真のコンテキストリセット（セッション全消去）との区別を可能にする。
 */
private pendingCompactionSkip = false;

afterCompaction(): void {
  this.pendingCompactionSkip = true;
  this.buffer = [];
  this.turnSeq = 0;
  this.clearIdleFlushTimer();
  console.log("[Episodic Memory] afterCompaction: set skip flag — next processTurn will advance cursor without reprocessing");
}
```

**修正** (`segmenter.ts` `processTurn()` 先頭):

```typescript

    return false;

    // ...
  }
  // ...

```

### 3.3 Fix 3: Warm-start 初回取り込みガード（再起動経路）

**狙い**: plugin再起動直後に巨大な既存メッセージ列を「新規」と誤認する経路を遮断する。

**修正A** (`segmenter.ts`): cursor を明示同期するユーティリティを追加

```typescript
bootstrapCursor(currentMessagesLength: number, reason: "warm-start" | "manual"): void {
  this.lastProcessedLength = Math.max(0, currentMessagesLength);
  this.buffer = [];
  this.turnSeq = 0;
  this.clearIdleFlushTimer();
  console.log(`[Episodic Memory] bootstrapCursor(${reason}): cursor=${this.lastProcessedLength}`);
}
```

**修正B** (`index.ts` `before_prompt_build`): 初回かつ巨大文脈時のみ bootstrap

```typescript
const WARM_START_SKIP_MIN_MESSAGES = 50;

// 初回 before_prompt_build かつ大量メッセージ時は、再起動復帰とみなして cursor を同期
if (state.lastRecallTurnMessageCount === -1 && msgs.length >= WARM_START_SKIP_MIN_MESSAGES) {
  state.segmenter.bootstrapCursor(msgs.length, "warm-start");
}

state.segmenter.processTurn(msgs, agentWs, agentId).catch(err => {
  console.log("[Episodic Memory] Fallback segmenter error in before_prompt_build:", err);

```

> 補足: `50` は誤検知回避の安全側閾値（通常の短い新規会話は従来挙動のまま）。

## 4. 設計判断

### なぜ `pendingCompactionSkip` フラグなのか

**代替案**: `after_compaction` で `lastProcessedLength = -1`（番兵値）を設定し、`processTurn` で `-1` の場合に cursor を進める。

**却下理由**: 
- 番兵値は型安全性が低い（`-1` は `number` の正当値として扱われる）
- `slice(-1)` が「最後の1要素」を返すという JS の仕様で意図しない挙動のリスク
- boolean フラグは明示的で自己文書化される

### なぜ warm-start ガードを v0.4.21 に含めるのか

4/19の実害は compaction 経路だけでは説明が閉じない。再起動直後に `Idle timeout (263 text messages)` と `Enqueued 2 chunks (87155 tokens)` が連鎖しており、**restart直後の初回取り込み経路**を同時に塞がないと再発余地が残る。

### なぜ `WARM_START_SKIP_MIN_MESSAGES` に閾値を置くのか

初回ターンを常にskipすると新規セッションの取りこぼしが発生し得る。閾値（例: 50）を置くことで、**「再起動復帰でしか出にくい巨大文脈」だけを対象**にして誤検知を抑える。

### なぜ `handleBatchIngest` にも重複ガードを入れるのか

Bug 1/2 を修正しても、再送・再試行・同一batch重複で同一slugが再投入される可能性は残る。defense-in-depth として ingest 層で遮断しておく。

### DRY vs KISS の判断

`afterCompaction()` と `bootstrapCursor()` は cursor 管理に介入するが、既存 `forceFlush()` / `processTurn()` の主処理は維持する。最小限の追加分岐で再発経路を塞ぐ方針（KISS優先）。
