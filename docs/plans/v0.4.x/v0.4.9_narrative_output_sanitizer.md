# 実装プラン: v0.4.9 Narrative Output Sanitizer — LLM出力のタグ汚染修正

---

## あなたへの依頼内容: `episodic-claw` の NarrativeWorker が OpenRouter LLM の出力をサニタイズせずにそのまま episode として保存する問題を修正してください。

---

## 問題の背景

### 現象

エピソードフォルダに以下 2種類の異常ファイルが出力される:

1. **OpenClaw エージェントの生返答**がそのまま保存される (`<final>`, `[[reply_to_current]]`, `[analysis]` タグ付き)
2. **正常な narrative** と **異常な episode** が同じタイムスタンプで並行出力される

### 異常ファイルの例

```
# casual-greeting-exchange.md (28 tokens)
<final>
[[reply_to_current]] おう、バッチリだよ！(≧∇≦)b
今日もなんか面白いと思いついたか？何でも言ってくれ。
</final>

# openclaw-novelization-implementation.md (512 tokens)
[analysis]
この質問は、GPT-5.4以降のOpenClawの開発状況と...

# story-generation-from-logs.md (67 tokens)
[reply_to_current] 物語化で完了よ！(≧∀・)
...
<final>
```

### 根本原因

`narrative-worker.ts` の `narrativizeWithRetry()` が OpenRouter LLM の応答を**一切サニタイズせずに**そのまま返し、`saveNarrative()` でそのまま `batchIngest` に渡している。

OpenRouter free モデルは品質が一定でなく、system prompt の指示に従わずに**会話コンテキスト中のエージェント応答形式をオウム返し**することがある。結果:
- `<final>` / `</final>` タグがそのまま episode 本文に残る
- `[analysis]`, `[[reply_to_current]]`, `[reply_to_current]` が行頭に残る
- エージェントのインストラクション系テキストがそのまま保存される

#### A-2: sanitizeNarrativeOutput() 関数を追加

`NarrativeWorker` クラスの**外側**（ファイル末尾 or クラス定義の直前）に以下のモジュールレベル関数を追加:

```typescript
/**
 * Sanitize OpenRouter LLM output to remove OpenClaw agent response format tags
 * and other non-narrative artifacts that leak through when the model echoes
 * conversation content instead of producing a clean summary.
 *
 * Applied post-LLM, pre-save. Strips:
 * - Reasoning/final/thinking tags via stripReasoningTagsFromText (handles <final>, <thinking>, etc.)
 * - [analysis], [[reply_to_current]], [reply_to_current] OpenClaw response format tags
 * - [output] / [/output] bracket tags (line-start only)
 * - Excessive whitespace from tag removal
 */
export function sanitizeNarrativeOutput(text: string): string {
  // Step 1: Strip <final>, </final>, <thinking>, </thinking> etc.
  let cleaned = stripReasoningTagsFromText(text, { mode: "strict", trim: "both" });

  // Step 2: Strip OpenClaw agent response format tags
  cleaned = cleaned
    .replace(/\[\[reply_to_current\]\]/g, "")
    .replace(/\[reply_to_current\]/g, "")
    .replace(/^\s*\[analysis\]\s*/gim, "")
    .replace(/^\s*\[\/analysis\]\s*/gim, "")
    .replace(/^\s*\[output\]\s*/gim, "")
    .replace(/^\s*\[\/output\]\s*/gim, "");

  // Step 3: Clean up residual whitespace from tag removal
  cleaned = cleaned
    .replace(/\n{3,}/g, "\n\n")
    .replace(/[ \t]+/g, " ")
    .trim();

  return cleaned;
}
```

**設計判断:**
- `stripReasoningTagsFromText` は既に battle-tested 関数。`<final>` と thinking タグはこれで対処
- `[analysis]`, `[[reply_to_current]]` は OpenClaw 固有のレスポンス形式。`retriever.ts` L266 にも同じ `[[reply_to_current]]` パターンが存在するので、一貫性がある
- `[output]` タグ除去はLLMが稀に出す artifacts。行頭のみ除去（本文中の正常な `[output]` は保護）
- ステートレスな pure function — テストしやすい

```typescript
  private async narrativizeWithRetry(item: CacheItem): Promise<NarrativeResult | null> {
    const systemPrompt = this.resolveSystemPrompt();
    const previous = this.lastNarrativeByAgent.get(item.agentId);
    const conversationText = item.rawText;
    const userMessage = this.resolveUserPrompt(previous?.body, conversationText);

    for (let attempt = 0; attempt < MAX_RETRIES; attempt++) {
      try {
        const rawText = await this.client.chatCompletion({ systemPrompt, userMessage });
        const text = sanitizeNarrativeOutput(rawText);

        // Quality gate: skip empty or trivially short outputs
        const tokens = estimateTokens(text);
        if (tokens < MIN_NARRATIVE_TOKENS) {
          console.warn(
            `[NarrativeWorker] Attempt ${attempt + 1}/${MAX_RETRIES}: output too short (${tokens} tokens < ${MIN_NARRATIVE_TOKENS}). Retrying...`
          );
          continue;
        }

        return {
          text,
          tokens,
          model: this.config.openrouterModel ?? "openrouter/free",
        };
      } catch (err) {
        const delayMs = RETRY_BASE_DELAY_MS * Math.pow(2, attempt);
        console.warn(
          `[NarrativeWorker] Attempt ${attempt + 1}/${MAX_RETRIES} failed for [${item.id}]: ${err instanceof Error ? err.message : String(err)}. Retrying in ${delayMs}ms...`
        );
        await this.sleep(delayMs);
      }
    }

    return null;
  }
```

**変更点:**
1. `text` → `rawText` にリネーム。`sanitizeNarrativeOutput(rawText)` で除去してから `text` に代入
2. トークン数チェック追加（Phase B）: `MIN_NARRATIVE_TOKENS` 未満なら `continue` でリトライ

## Phase B: 品質ゲート

### 変更対象: `src/narrative-worker.ts`

#### B-1: MIN_NARRATIVE_TOKENS 定数を追加

既存の定数定義域 (L29-33周辺) の直後に追加:

```typescript
const MIN_NARRATIVE_TOKENS = 10;
```

**この値について:**
- `10 tokens` は、「まともな内容には最低限必要な長さ」の閾値
- 例: `casual-greeting-exchange.md` の sanitize 後は「おう、バッチリだよ！今日もなんか面白いと思いついたか？何でも言ってくれ。」〜 タグ除去後は単なる挨拶（15 tokens）。これは通過するが、品質は低い。ただしこれ以上の品質判定は LLM に委ねるため YAGNI
- `story-generation-from-logs.md` の sanitize 後は「物語化で完了よ！...」〜 タグ除去後 ~50 tokens 〜 通過する。**品質ゲートはあくまで空/超短い出力を弾くため**

#### B-2: narrativizeWithRetry() の変更

Phase A-3 に統合済み。`continue` でリトライループの次の attempt に進む。MAX_RETRIES 回すべて `MIN_NARRATIVE_TOKENS` 未満なら `null` を返し、`cacheRetry` で再キューされる。

## 注意事項

1. **`sanitizeNarrativeOutput()` はモジュールレベルの `export` 関数として定義すること。** テストファイルから直接 import するため。クラスメソッドにはしない。
2. **`reasoning-tags.ts` は変更しないこと。** 既存の `stripReasoningTagsFromText` をそのまま利用する。
3. **`narrativizeWithRetry()` の既存のリトライロジック (catch 内の exponential backoff) は変更しないこと。** 品質ゲートの `continue` は catch の外（try ブロック内）で発動する。
4. **`MIN_NARRATIVE_TOKENS` の値を `10` より大きくしないこと。** 正常な短い narrative（JK の 2-3行で完結な内容になるケース）を弾かないため。
5. **sanitizer のパターンは行頭マッチング (`^\s*`) を使うこと。** 本文中の正常な `[analysis]` 文字列（例: 「narrativeを分析したら」と書かれる場合）を誤除去しない。ただし `[[reply_to_current]]` と `[reply_to_current]` はグローバル除去で問題ない（narrative 本文中に正常に出現することはない）。

## 補足: CacheDB の rawText 保持について

Go sidecar の `cache.ack` (queue.go L202-213) は **deleteAfter=true** でアイテムを PebbleDB から物理削除する。従って、一度 ack されたアイテムの rawText は CacheDB に残らない。失敗した episode の再生成は CacheDB 削除では不可能。

→ この問題は `v0.4.9_persistent_cache_rawtext.md` のプランで別途対応済み（deleteAfter=false + Requeue RPC）。

### 🔍 Detailed Code Review Findings

#### F-1: テストカバレッジの欠如（重要度: HIGH）

プランの Acceptance Criteria #10 は、「`npm test` 全 PASS」だが、現在のテストスイートは:
- `test_phase4_5.ts` — Go sidecar 必要の統合テスト
- `test_reasoning.ts` — `reasoning-tags.ts` のテストのみ

`sanitizeNarrativeOutput()` は**テスト可能な純粋関数**として設計されているのに、テストファイルの追加がプランに含まれていない。これは、「テストしやすさ」を謳いながらテストを書かないという矛盾。

**懸念**: リグレッションの検知が不可能。将来 `stripReasoningTagsFromText` のシグネチャが変わっても気づけない。

#### F-2: 品質ゲートがリトライ回数を消費する（重要度: LOW）

品質ゲートの `continue` は `MAX_RETRIES` (5) の試行回数を消費する。つまり LLM が一貫して短い出力を返す場合は5回すべて「短すぎる」で消費され、エラーログなしに `null` が返る。

ただし、`cacheRetry` で再キューされる前提なので、実用上の問題は小さい。また短い出力の連続はモデル側の異常で稀。**このままでよい。**

#### F-3: `[output]` / `[/output]` の行頭マッチのみ（重要度: LOW）

プランの docstring には「HTML-like tags (e.g., `<antthinking>`, `<output>`)」とあるが、コードは `[output]`（角カッコ）しか処理しない。`<output>`（山カッコ）は `stripReasoningTagsFromText` で除去されない。

ただし、`<output>` はバグ報告に含まれていないし、`<antthinking>` は `stripReasoningTagsFromText` で既に対応済み。docstring の記述がやや不正確だが、実際的には問題なし。YAGNI — 報告されていない `<output>` のためにコードを膨やす必要はない。

#### F-4: 空白正規化の副作用懸念（重要度: LOW）

`.replace(/[ \t]+/g, " ")` は連続スペース/タブを1つに縮める。プロンプトの指定が「800文字以内の内容」なので、コードブロックや表のインデントが壊れる可能性はほぼない。懸念範囲。
