# v0.4.9 Polyglot Query Morphological Upgrade — 言語検出 + 形態素解析による CJK 精度の根本改善

**Parents:** [v0.4.3 Recall Query CJK Attachment Filter Fix](./v0.4.3_recall_query_cjk_attachment_filter_fix.md) / [v0.3.7 Query Improvement Plan](../v0.3.x/v0.3.7_query_improvement_plan.md)
**Date:** 2026-04-13
**Target Version:** v0.4.9
**Status:** Phase 1 + Phase 2 + Phase 4 Implemented

> v0.4.3 で attachment ノイズ除去と script-aware interleaving を入れたが、運用上の recall 精度がまだ良くない。
> 調査の結果、**CJK の regex 塊抽出が根本的に単語を切り出せていない** ことが判明した。
> 本プランは、言語検出 + 形態素解析を導入して CJK クエリ品質を根本から直す改善計画。

## 1. 調査結果 — 現行実装の問題分析

### 1.1 現行 CJK 抽出の致命的欠陥

`src/retriever.ts` L193-232 の `extractPolyglotKeywords()` は、CJK を次の regex で抜き出す：

```typescript
const cjkMatches = text.match(
  /[\p{Script=Han}\p{Script=Katakana}\p{Script=Hiragana}\p{Script=Hangul}]{2,}/gu
) || [];
```

これは **文字の連続をそのまま1塊として拾うだけ**。分かち書きがない CJK では構造的に単語を分離できない。

| 入力 | regex の出力 | 本当に欲しい出力 |
|---|---|---|
| `ツールの盛衰として読むこともできるが` | `ツールの盛衰として読むこともできるが`（1塊） | `ツール 盛衰 読む` |
| `今日は天気がいいね` | `今日は天気がいいね`（1塊） | `今日 天気 いい` |
| `JLPTの申込をした` | `の申込をした`（1塊）+ `JLPT` | `JLPT 申込` |
| `プロダクションエンジニアの視点で` | `プロダクションエンジニアの視点で`（1塊） | `プロダクションエンジニア 視点` |

**stopwords で助詞を弾こうとしても、regex が既に「助詞込みの1塊」を作ってしまっているから手遅れ。** これが「精度が良くない」の正体。

### 1.2 言語検出が文字種比率だけ — JA/ZH/KO を区別できない

`detectDominantScript()` は CJK 文字比率 >= 30% なら `"cjk"` を返すが、日本語・中国語・韓国語を区別しない。各言語で必要な形態素解析が違うのに、一律 `"cjk"` で扱っている。

### 1.3 Go サイドカー側の Bleve も同じ問題

`go/internal/vector/lexical.go` L38：

```go
textFieldMapping.Analyzer = standard.Name
```

standard analyzer は CJK をトークナイズできない。BM25 の CJK 精度も悪い。ただし HNSW（ベクトル検索）が主軸なので、TS 側のクエリ品質改善が先。

### 1.4 Reference 文書との差分

| レイヤ | 現行 (v0.4.8) | Reference 提案 | ギャップ |
|---|---|---|---|
| 言語検出 | `detectDominantScript()` = 文字種比率のみ | `efficient-language-detector-js` = 確率的言語判定 | **致命的** — JA/ZH/KR を区別できない |
| 日本語形態素解析 | なし（regex 塊のみ） | `kuromojin`（MeCab 辞書ベース、POS フィルタ） | **致命的** — クエリ精度の9割を決める |
| 中国語/韓国語 | なし（regex 塊のみ） | `cjk-tokenizer`（スコア付き term 抽出） | **重度** |
| インドネシア語 | stopwords のみ | `sastrawijs`（語幹抽出） | **軽度** — stopwords でそこそこ効く |
| 英語 | regex + stopwords | `natural`（Porter stemmer） | **軽度** — 3文字以上 + stopwords で十分 |

### 1.5 Mixed-Text Handling の問題

現行の script-aware interleaving（CJK 8 + Latin 4）は、**同一文の中に日本語と英語が混在するケース**に弱い。

例：`OpenClawのプラグインとしてepisodic-clawを導入した`

現行の動き：
1. Latin: `OpenClaw`, `episodic`, `claw`（3語）
2. CJK regex: `のプラグインとして`, `を導入した`（2塊、助詞混入）
3. dominant = cjk → CJK 8 + Latin 4 = `のプラグインとして を導入した OpenClaw episodic claw`（助詞ノイズが8枠の主力に）

本当の意図：`OpenClaw プラグイン 導入 episodic claw`

この問題は「CJK 側を形態素解析でちゃんと分離しない限り、interleaving の設計が活きない」という話。Phase 1 が解決すれば interleaving も正しく機能するようになる。

#### 4.1.3 実装詳細

**新規ファイル: `src/lang-detect.ts`**

```typescript
import { Detector } from "efficient-language-detector-js";

let detectorInstance: Detector | null = null;

export type DetectedLanguage = "ja" | "zh" | "ko" | "en" | "id" | "unknown";

export function detectLanguage(text: string): DetectedLanguage {
  if (!detectorInstance) {
    detectorInstance = new Detector();
  }
  const result = detectorInstance.detect(text);
  const lang = result[0]?.language;

  switch (lang) {
    case "ja": return "ja";
    case "zh": return "zh";
    case "ko": return "ko";
    case "en": return "en";
    case "id": return "id";
    default: return "unknown";
  }
}
```

**新規ファイル: `src/cjk-tokenizer.ts`**

```typescript
import { tokenize } from "kuromojin";
import type { DetectedLanguage } from "./lang-detect";

export interface CjkTokenizeResult {
  keywords: string[];
  lang: DetectedLanguage;
}

const JA_CONTENT_POS = new Set(["名詞", "動詞", "形容詞", "副詞"]);

/**
 * 日本語形態素解析。
 * kuromojin は初回呼び出し時に辞書をロードする（async, ~100ms）。
 * 2回目以降は内部キャッシュで同期並みに高速。
 */
async function tokenizeJapanese(text: string): Promise<string[]> {
  const tokens = await tokenize(text);
  return tokens
    .filter(t => {
      const pos = t.pos_detail_1?.split("-")[0] ?? t.pos;
      return JA_CONTENT_POS.has(pos);
    })
    .map(t => t.surface_form);
}

/**
 * CJK テキストを言語に応じてトークナイズ。
 * Phase 1 では日本語のみ形態素解析。ZH/KO は regex フォールバック。
 */
export async function tokenizeCjk(
  text: string,
  lang: DetectedLanguage
): Promise<CjkTokenizeResult> {
  if (lang === "ja") {
    const keywords = await tokenizeJapanese(text);
    return { keywords, lang };
  }

  // ZH/KO フォールバック: 現行 regex（Phase 2 で cjk-tokenizer 導入）
  const cjkMatches = text.match(
    /[\p{Script=Han}\p{Script=Katakana}\p{Script=Hiragana}\p{Script=Hangul}]{2,}/gu
  ) || [];
  return { keywords: cjkMatches, lang };
}
```

#### 4.1.4 Mixed-Text Handling

言語検出はテキスト全体の dominant language を返す。しかし実際の入力は日本語 + 英語混在が普通。

**ルーティング戦略：**

1. **テキスト全体の言語検出** → dominant language を決定
2. **CJK セグメントだけを形態素解析にかける** — Latin 部分は既存の regex + stopwords で処理
3. **両方の結果をマージ** — dominant language 側を primary pool、もう一方を secondary pool にする（現行 interleaving 設計を流用）

具体的なフロー：

```
"OpenClawのプラグインとしてepisodic-clawを導入した"

1. detectLanguage → "ja"
2. CJK セグメント抽出: "のプラグインとして を導入した"
3. kuromojin → ["プラグイン", "導入"]（助詞は POS フィルタで除去）
4. Latin セグメント抽出: "OpenClaw", "episodic", "claw"
5. stopwords フィルタ: "OpenClaw", "episodic", "claw"（全て通る）
6. Interleave: CJK primary 8 + Latin secondary 4
   → "プラグイン 導入 OpenClaw episodic claw"
```

**CJK セグメント分離の実装：**

```typescript
function splitByScript(text: string): { cjk: string; latin: string } {
  // CJK 文字だけを繋げた文字列（形態素解析用）
  const cjkChars = text.match(
    /[\p{Script=Han}\p{Script=Katakana}\p{Script=Hiragana}\p{Script=Hangul}]+/gu
  ) || [];
  // Latin トークンだけを拾う（既存 regex）
  const latinTokens = text.match(/\b[A-Za-z]{3,}\b/g) || [];

  return {
    cjk: cjkChars.join(" "),
    latin: latinTokens.join(" "),
  };
}
```

これにより、**同じ文の中に混在する CJK と Latin を別パスで処理し、最後に統合** できる。

#### 4.1.5 kuromojin 初回ロードの取り扱い

kuromojin は初回呼び出し時だけ辞書ロードで ~100ms かかる。2回目以降はキャッシュで実質同期。

対策：
- プラグイン起動時（`register()` 内）にダミーテキストで `tokenize("初期化")` を呼び、辞書をウォームアップする
- ウォームアップ失敗でもクラッシュさせない（try-catch で包む）
- ウォームアップ未完了の場合は、フォールバックとして現行 regex を使う

```typescript
// src/index.ts の register() 内
let kuromojiReady = false;

try {
  await tokenize("初期化");
  kuromojiReady = true;
  console.log("[episodic-claw] kuromojin dictionary loaded");
} catch (e) {
  console.warn("[episodic-claw] kuromojin warm-up failed, falling back to regex CJK:", e);
}
```

## Post-Phase 追加実装: insufficient_keywords ガード（2026-04-13）

### 背景

Phase 1-4 でクエリ品質は大幅に改善したが、短文（1-2語のキーワードしか生成できない入力）では
ノイズの多い低品質な recall 結果が注入されるリスクがあった。

インドネシア語で `recall_empty` が多発するか検証した結果、
短文でも最低1語は拾えるため `recall_empty` 自体は発生しなかったが、
1-2語のクエリで recall を実行してもセマンティック検索の精度が低く、
ノイズ注入になることが判明した。

### 実装内容

`src/retriever.ts` L444-449 に insufficient_keywords ガードを追加：

```typescript
// Skip recall when query has too few keywords — insufficient signal for meaningful retrieval
const queryKeywords = query.split(/\s+/).filter(Boolean);
if (queryKeywords.length <= 2) {
  return { ..., reason: "insufficient_keywords", ... };
}
```

- **閾値: `<= 2`**（1-2語のクエリは即スキップ、3語以上は通す）
- **reason 型**に `"insufficient_keywords"` を追加（`RecallInjectionOutcome.reason`）
- **テスト正規表現**も更新済み

### 判定根拠

| クエリ例 | キーワード数 | 判定 |
|---|---|---|
| `remember exam practice` | 3 | OK（通す） |
| `Recall exam memory` | 3 | OK（通す） |
| `selesai`（インドネシア語「完了」） | 1 | スキップ |
| `terima kasih`（インドネシア語「ありがとう」） | 2 | スキップ |
| `おはよう`（日本語「おはよう」） | 1 | スキップ |

### Phase 3（インドネシア語 stemming）の YAGNI 確定根拠

インドネシア語で `recall_empty` が多発するかテストした結果：
- 長文: 3-4語抽出 → OK
- 短文: 1-2語抽出 → `insufficient_keywords` でスキップ
- **stemming 不要** — 現行 regex + stopwords で十分。短文の問題は stemming ではなくキーワード不足であり、`insufficient_keywords` ガードで対処済み
