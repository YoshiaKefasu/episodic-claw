# v0.4.3 Cache Idle Poll Log Storm Fix

**Parents:** [v0.4.2 Cold Start Buffer Architecture](./v0.4.2_cold_start_buffer_architecture.md) / [v0.4.1 Post-Release Hardening](./v0.4.1_post_release_hardening.md)  
**Date:** 2026-04-10  
**Target Version:** v0.4.3  
**Status:** 収束完了（Release Gate Closed / v0.4.3 released / operational verification）

> **Source-of-truth:** この文書は fix 単体の記録です。v0.4.3 全体の release verdict / sign-off は parent の [v0.4.3_release_readiness_assurance_plan.md](./v0.4.3_release_readiness_assurance_plan.md) を正本とします。

> 本プランは、v0.4.2 の Cache-and-Drain 基盤そのものを否定するものではありません。  
> 実運用で顕在化した「空キュー時の `cache.leaseNext` ポーリングがログを埋める」「info ログが host UI では warn に見える」という観測ノイズを、v0.4.3 で静かに閉じるための修正計画です。

## 1. 何が起きているか

今回のログ `X:\tmp\episodic-claw\2026-04-10.log` を**本ドキュメント執筆時点のスナップショット**で集計すると、次の状態でした。

- 総ログ行数: `478`
- `Method: cache.leaseNext`: `467` 行
- 観測区間: 約 `474` 秒
- ほぼ `59.11 calls/min` の空ポーリング
- 同区間で `cache.enqueueBatch` / `cache.ack` / `cache.retry` は観測されず

つまり、**キュー故障ではなく、アイドル時の正常空振りがそのまま高頻度ログ化されている** のが本件です。

## 2. Current Reality (Log vs. Source Code)

### 2.1 `NarrativeWorker` は空でも 1 秒ごとに lease を叩く

`src/narrative-worker.ts`

- `POLL_INTERVAL_MS = 1000`
- `start()` 後は常時 `pollNext()`
- `processNextFromCache()` は item がなくても毎回 RPC を投げる

該当箇所:

- `src/narrative-worker.ts:31`
- `src/narrative-worker.ts:106`
- `src/narrative-worker.ts:121`
- `src/narrative-worker.ts:126`

### 2.2 Go 側は全 RPC method を毎回 info ログに出す

`go/main.go`

- `handleConnection()` で `EmitLog("Method: %s", req.Method)` を無条件実行
- `cache.leaseNext` も hot path なのに例外扱いされていない

該当箇所:

- `go/main.go:2915`
- `go/main.go:2960`

### 2.3 Go logger は全レベルを stderr に流している

`go/internal/logger/logger.go`

- `logger.Emit()` は `debug/info/warn/error` 全部を `os.Stderr` へ出力

該当箇所:

- `go/internal/logger/logger.go:124`
- `go/internal/logger/logger.go:128`

### 2.4 TypeScript 側は sidecar stderr をすべて `console.warn()` に投げている

`src/rpc-client.ts`

- child stderr の内容をそのまま `console.warn(...)`
- そのため Go 側 JSONL では `info` でも、host UI では `warn` バッジに見える

該当箇所:

- `src/rpc-client.ts:230`
- `src/rpc-client.ts:231`

### 2.5 これは queue ロジック破綻ではない

`go/internal/cache/queue.go`

- `LeaseNext()` は queued item がなければ `nil` を返すだけ
- dead-letter 暴走や retry ループは今回のログには出ていない

該当箇所:

- `go/internal/cache/queue.go:135`
- `go/internal/cache/queue.go:199`

## 3. Core Problem

> **空キュー時の正常ポーリングが、固定 1 秒間隔 + 無条件 method logging + stderr→warn bridge の組み合わせで、運用上は“異常に見えるログ嵐”へ増幅されている。**

| Check | Result | Note |

## 5. Solution Options

### Option A — Adaptive Idle Poll + Hot-RPC Log Suppression + Severity Remap *(推奨)*

**Approach**:  
queue プロトコルは壊さず、v0.4.2 の pull-based worker をそのまま活かす。  
その代わり、空振り時だけポーリング間隔を指数的に広げ、`cache.leaseNext` の method trace を hot path から外し、host UI では `info` を `warn` に誤表示しないようにする。

**Implementation cost**: Medium  
**Risk**: Low  
**Why recommended**:  
YAGNI/KISS に最も合う。queue 設計を全面的に変えずに、CPU・ログ量・UI 誤警告の 3 つを同時に下げられる。

**Concrete steps**:

1. `src/narrative-worker.ts` に idle backoff state を追加する
   - `consecutiveEmptyPolls`
   - `nextPollDelayMs`
   - 推奨スケジュール: `1s -> 2s -> 4s -> 8s -> 15s` cap
   - item を lease できたら即 `1s` にリセット

2. `NarrativeWorker` に `wake()` を追加する
   - `setTimeout` を clear
   - 次 poll を即時実行
   - 二重 wake でも壊れない idempotent 実装にする

3. enqueue 成功時に worker を起こす
   - live ingest: `src/segmenter.ts`
   - force flush: `src/segmenter.ts`
   - gap archive (>50): `src/archiver.ts`
   - cold-start enqueue: `src/rpc-client.ts` helper か `src/index.ts` 呼び出し側
   - 実装は callback 注入で十分。新しい global singleton は作らない

4. `go/main.go` の `EmitLog("Method: %s", req.Method)` を hot-path aware にする
   - `cache.leaseNext`
   - `ping`
   - 将来必要なら `watcher.start` 以外の noisy method も skip list 化
   - ルール: **“every request” ではなく “state change / error / sampled debug” を記録**

5. `src/rpc-client.ts` の sidecar stderr bridge を severity-aware にする
   - `[core] [info] ...` → `console.log` or `console.info`
   - `[core] [warn] ...` → `console.warn`
   - `[core] [error] ...` → `console.error`
   - 将来のために `stdout` listener も追加し、片側依存を減らす

6. host に見せるべきログを残し、見せなくてよい空振りを落とす
   - 残す: queue 初期化、lease 成功、ack、retry、dead-letter、continuity load、fatal error
   - 落とす: 空キュー時の `Method: cache.leaseNext`

7. テストを追加する
   - `NarrativeWorker` empty polls で delay が増える
   - item lease 後に delay が reset される
   - enqueue success 後に `wake()` で即 poll へ戻る
   - `cache.leaseNext` empty path が per-call info log を吐かない
   - stderr bridge が `[info]` を warn 扱いしない

### Option B — Event-Driven Queue Wakeup / Long Poll

**Approach**:  
`cache.leaseNext` の定期 polling をやめ、Go 側に `waitForWork` か wake channel を持たせる。enqueue 時に waiter を起こし、worker はブロッキング or 長めタイムアウトで待つ。

**Implementation cost**: High  
**Risk**: Medium  
**When to choose this instead**:  
v0.4.3 ではなく、その先で “アイドル時 RPC をほぼゼロにしたい” 要件が明確な場合。

**Concrete steps**:

1. Go 側に agent ごとの wake channel を追加
2. `enqueueBatch` 成功時に broadcast
3. `cache.leaseNext` を long-poll 化するか `cache.waitForWork` を追加
4. worker を `wake -> lease -> process` の 2 段へ変更
5. reconnect / shutdown / multi-agent を再設計

### Option C — ログだけ消す

**Approach**:  
`Method: cache.leaseNext` のログだけ止める。

**Implementation cost**: Low  
**Risk**: Medium  
**Why not recommended**:  
UI 上の警告は減るが、1 秒ポーリングそのものは残る。症状の半分しか閉じない。

## 6. Pro Recommendation

> **Choose Option A because** これが一番小さな変更で、運用上の痛みをちゃんと止められるからです。  
> queue 自体は壊れていないので、v0.4.3 で protocol redesign まで行くのはやりすぎです。  
> まずは idle path を静かにし、host UI の severity 誤表示を直し、それでも不足するなら v0.4.4+ で event-driven 化を検討するのが現場的です。

**Estimated implementation**: 0.5 - 1.5 day  
**Rollback plan**:

1. `NarrativeWorker` の adaptive backoff を fixed 1s に戻す
2. method log suppression を元に戻す
3. stderr bridge parser を単純 `console.warn` に戻す
