# v0.4.3 Narrative Surprise Metadata Fix

**Parents:** [v0.4.2 Cold Start Buffer Architecture](./v0.4.2_cold_start_buffer_architecture.md)  
**Date:** 2026-04-10  
**Target Version:** v0.4.3  
**Status:** 実装完了（core fix landed / operational sign check tracked in parent）

> **Source-of-truth:** この文書は Gate C の局所記録です。v0.4.3 全体の release verdict / sign-off は parent の [v0.4.3_release_readiness_assurance_plan.md](./v0.4.3_release_readiness_assurance_plan.md) を正本とします。

> 本プランは、「保存された narrative episode の footer に `\"surprise\": 0` が入るのは正常か？」に対する修正計画です。  
> 結論は **“いまは正常に見えるが、実装としては metadata loss”** です。source chunk 側の `surprise` を narrative 保存時に落としているため、結果が常にゼロへ潰れています。

---

## 1. Incident Summary

観測された episode footer には次の状態がありました。

- `saved_by: "main"`
- `tags: ["narrative", "auto-segmented"]`
- `surprise: 0`

一方で、v0.4.2 の cache queue 設計では chunk metadata に `surprise` が存在します。  
つまり、「source では持っていた値が narrative save 時に落ちている」可能性が高いです。

---

## 2. Current Reality (Source Code vs. Metadata)

### 2.1 chunk queue item には `surprise` がある

`src/narrative-queue.ts`

- `CacheQueueItem.surprise` が定義されている
- 単一 chunk ならそのまま保持
- 複数 chunk なら first chunk のみ `surprise` を保持し、後続 chunk は `0`

該当箇所:

- `src/narrative-queue.ts:20`
- `src/narrative-queue.ts:63`
- `src/narrative-queue.ts:94`

### 2.2 `NarrativeWorker` のローカル型が `surprise` を持っていない

`src/narrative-worker.ts`

- `interface CacheItem` に `surprise` がない
- `cacheLeaseNext()` も `any | null` 返しで型が弱い

該当箇所:

- `src/narrative-worker.ts:34`
- `src/rpc-client.ts:614`

### 2.3 保存時に `surprise: 0` を固定で書いている

`src/narrative-worker.ts`

- `saveNarrative()` が `batchIngest()` に渡す item で `surprise: 0` をハードコードしている

該当箇所:

- `src/narrative-worker.ts:183`
- `src/narrative-worker.ts:193`

### 2.4 `batchIngest` / Go 側は `surprise` を保存できる

`src/rpc-client.ts` / `go/main.go`

- TS 側 `batchIngest` params には `surprise` がある
- Go 側も `Surprise` を受けて episode metadata に書ける

該当箇所:

- `src/rpc-client.ts:566`
- `go/main.go:1472`
- `go/main.go:1544`
- `go/main.go:1614`

---

## 3. Core Problem

> **queue chunk に存在する `surprise` が、narrative 保存経路で型ごと落ち、最後に `surprise: 0` が固定で書き戻されている。**

---

## 4. Principle Filter

| Check | Result | Note |
|---|---|---|
| YAGNI — 今すぐ必要か | ✅ Yes | 既に保持している metadata を失っているだけで、追加機能ではない |
| KISS — もっと単純に直せるか | ✅ Yes | `item.surprise` を通すだけでまず十分 |
| DRY — 重複はあるか | ⚠️ Yes | queue item / worker item / batchIngest item で metadata 契約が分断されている |
| SOLID — 実害ある違反か | ✅ Yes | source metadata と saved metadata の責務が切れてしまっている |

---

## 5. Solution Options

### Option A — Preserve `item.surprise` End-to-End *(推奨)*

**Approach**:  
queue item の `surprise` を worker 型にも明示し、`saveNarrative()` でそのまま `batchIngest` に渡す。

**Implementation cost**: Low  
**Risk**: Low  
**Why recommended**:  
最小変更で metadata loss を止められる。いま欲しいのは分析軸の維持であって、新しい scoring ではない。

**Concrete steps**:

1. `src/narrative-worker.ts` の `CacheItem` に `surprise: number` を追加する
2. `src/rpc-client.ts` の `cacheLeaseNext()` を `any` ではなく queue item 型へ寄せる
3. `saveNarrative()` の `surprise: 0` を `surprise: item.surprise ?? 0` に変更する
4. multi-chunk case は現状仕様を維持する
   - first chunk: 元 surprise を保存
   - later chunks: `0`
5. テストを追加する
   - single chunk ingest で narrative footer の `surprise` が source と一致
   - split chunk で first のみ保持、後続は 0

### Option B — `sourceSurprise` を別フィールドで追加する

**Approach**:  
`surprise` は narrative 自体の値として残しつつ、元 chunk の値を `sourceSurprise` へ別保存する。

**Implementation cost**: Medium  
**Risk**: Medium  
**When to choose this instead**:  
将来「narrative quality surprise」と「segment boundary surprise」を分けて扱いたい場合。

**Why not recommended now**:  
v0.4.3 では設計を増やしすぎ。今はまず loss を止めればよい。

### Option C — `surprise: 0` を仕様化

**Approach**:  
narrative episode の surprise は常にゼロと定義する。

**Implementation cost**: Low  
**Risk**: High  
**Why not recommended**:  
既存 queue metadata を捨てる説明がつかない。ユーザー観測ともズレる。

---

## 6. Pro Recommendation

> **Choose Option A because** これは不具合というより契約落ちです。  
> source で持っている値を保存時に通すだけで閉じるので、v0.4.3 ではそれが正解です。

**Estimated implementation**: 1 - 2 hours  
**Rollback plan**:

1. `saveNarrative()` を元の固定 0 に戻す
2. worker 型の `surprise` を optional 化または削除

---

## 7. Acceptance Criteria

1. surprise-boundary 由来の single-chunk narrative episode で `surprise` が 0 固定にならない
2. size-limit / force-flush / cold-start でも source chunk metadata がそのまま保存される
3. multi-chunk case の first-only preserve ルールが壊れない
4. episode markdown footer と DB metadata が一致する

### 📍 Current Reality (Source Code vs. Document)

- ✅ `NarrativeWorker` の leased item 型は queue 本体型へ寄せられました。
  - `src/narrative-worker.ts` では `type CacheItem = CacheQueueItem` になっており、型二重化は解消されています。
- ✅ `saveNarrative()` は固定 `0` ではなく、`item.surprise ?? 0` を保存します。
  - これで worker save path での metadata loss は止まっています。
- ✅ `cacheLeaseNext()` は `any` ではなく `CacheQueueItem | null` を返します。
  - `src/rpc-client.ts` は `CacheQueueItem` を import して型契約を寄せています。
- ✅ queue 側の first-only preserve ルールは維持されています。
  - `src/narrative-queue.ts` では single chunk はそのまま、multi-chunk は first chunk のみ `surprise` を保持します。
- ✅ Go 側の ingest 保存経路は `params.Surprise` を footer / vector record へ書けます。
  - `go/main.go` の `handleIngest` で `frontmatter.EpisodeMetadata.Surprise` と vector record の `Surprise` に流れます。
- ✅ 回帰テストは 3 層に増えています。
  - `runSurpriseMetadataRegression()` は type alias / rpc 型 / save path / first-only preserve を固定しています。
  - `runSurpriseMetadataRoundTrip()` は chunking と save path 契約を helper-based に往復確認しています。
  - `runReleaseGateC()` も追加されています。
- ⚠️ ただし Gate C はまだ release-grade proof ではありません。
  - `runReleaseGateC()` は queue -> worker -> RPC -> Go -> frontmatter の chain 契約確認です。
  - まだ「保存された markdown を実読して footer metadata の `surprise` を確認する」persisted-footer integration ではありません。

### 🎯 Core Problem (1 sentence)

> `surprise: 0` 固定化バグはほぼ解消済みだが、release 前に必要な最後の論点は persisted markdown footer まで実読して `surprise` 一致を固定することです。
