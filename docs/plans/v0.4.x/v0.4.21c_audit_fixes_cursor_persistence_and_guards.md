## 2. 修正 #1: 通常ターンの cursor 永続化（debounce 付き）

### 2.1 問題の詳細

`persistCursor()` は以下の2箇所からしか呼ばれない：
- `processTurn` の compaction-skip 分岐（`pendingCompactionSkip = true` の時）
- `bootstrapCursor()` （`reason === "manual" | "warm-start"` の時）

通常の `processTurn` で `lastProcessedLength` が進んでも **state.db には保存されない**。
結果：cursor=10 から再開 → 数ターン進んで cursor=15 → 再起動 → DB には 10（または bootstrap 時の値）のまま →
11〜15 が再処理される → **重複ナラティブ生成の原因**。

### 2.2 修正仕様

`segmenter.ts` に debounce 付き cursor 永続化を追加する。
ただし **compaction-skip 分岐は即時 persist のまま維持**し、通常ターン経路だけを `maybePersistCursor` の対象にする。

```typescript
// --- 追加フィールド ---
private cursorPersistCounter = 0;
private readonly CURSOR_PERSIST_INTERVAL = 3; // every 3rd processTurn that advances cursor

// --- processTurn 内、通常経路の lastProcessedLength 更新後に追加 ---
// 例: 通常ターン終了時
this.lastProcessedLength = currentMessages.length;
this.maybePersistCursor(agentWs, agentId);  // ← 追加

// --- 新規メソッド ---
/** [v0.4.21c] Debounced cursor persistence for normal turns.
 *  Writes every CURSOR_PERSIST_INTERVAL advances to reduce DB I/O.
 *  In-memory cursor is always up-to-date; DB is a restart-recovery backup.
 *  Max staleness: CURSOR_PERSIST_INTERVAL - 1 turns (acceptable on crash).
 */
private maybePersistCursor(agentWs: string, agentId: string): void {
  if (!agentWs || !agentId) return;
  this.cursorPersistCounter++;
  if (this.cursorPersistCounter % this.CURSOR_PERSIST_INTERVAL !== 0) return;
  this.persistCursor(agentWs, agentId);
}
```

### 2.3 配置箇所

`processTurn` 内の `this.lastProcessedLength = currentMessages.length;` は **7箇所** にある：
1. L192: compaction-skip 分岐（`persistCursor` 即時実行を維持、`maybePersistCursor` には置換しない）
2. L255: duplicates-or-empty 分岐（**新規追加**）
3. L262: first-turn absorb 分岐（**新規追加**）
4. L287: no-text slice 分岐（**新規追加**）
5. L299: time-gap boundary 分岐（**新規追加**）
6. L358: 正常終了分岐（**新規追加**）
7. L364: error fallback 分岐（**新規追加**）

※ `bootstrapCursor` 内の `persistCursor` も即時実行のまま（restart 直後の重要値）。

### 2.4 影響範囲

- `segmenter.ts` のみ
- RPC / Go 側は変更不要（既存の `setSegmenterCursor` を利用）
- `CURSOR_PERSIST_INTERVAL = 3` により、DB I/O は通常ターンでも3回に1回 →
  1時間10ターン程度なら 1時間あたり3〜4回の書き込み（`pebble.Sync` でも許容範囲）

### 2.5 テスト観点

- [ ] 通常ターンが3回進んだ後に `state.get` で cursor が更新されていること
- [ ] crash をシミュレートして、最後の永続化値から再開できること
- [ ] compaction-skip / bootstrapCursor の即時 persist がデバウンスされていないこと

## 3. 修正 #2: `recentSaveHashes` の agent/workspace 分離

### 3.1 問題の詳細

`narrative-worker.ts` の `recentSaveHashes` は `Map<string, number>` の単一インスタンス。
key は rawText の hash のみで、agent/workspace の識別子を含まない。

persistence は `agent:{agentId}:ws:{agentWs}:narrative-save-hashes` と agent 単位で保存・復元するが、
`loadSaveHashes()` で全て同じ Map にマージされるため：
- Agent A と Agent B が同じ rawText を持つ → A の hash が B の保存を止める
- persistence 導入で混線が再起動後も固定化される

### 3.2 修正仕様

Map の key に `agentWs:agentId:` prefix を追加し、agent/workspace 単位で分離する。

```typescript
// --- 変更: hash key の生成 ---
// 旧: this.recentSaveHashes.has(rawHash)
// 新: this.recentSaveHashes.has(`${agentWs}:${agentId}:${rawHash}`)

// 具体箇所:
// 1. saveNarrative() 内の duplicate check
private async saveNarrative(result: NarrativeResult, item: CacheItem): Promise<void> {
  const rawHash = this.hashRawText(item.rawText);
  const scopedKey = `${item.agentWs}:${item.agentId}:${rawHash}`;  // ← scoped
  if (this.recentSaveHashes.has(scopedKey)) {
    const savedAt = this.recentSaveHashes.get(scopedKey)!;
    // ... 既存の TTL check ...
    return;  // duplicate
  }
  // ... 保存処理 ...
  this.recentSaveHashes.set(scopedKey, Date.now());  // ← scoped
}

// 2. loadSaveHashes() 内の復元
private async loadSaveHashes(agentWs: string, agentId: string): Promise<void> {
  const hashes = await this.rpcClient.getNarrativeSaveHashes(agentWs, agentId) // DB は agent-scoped
    .catch(() => []);
  for (const h of hashes) {
    // DB から読み出した hash key に agentWs:agentId: prefix を付与して in-memory Map に格納
    const scopedKey = `${agentWs}:${agentId}:${h.key}`;
    this.recentSaveHashes.set(scopedKey, h.timestamp);  // ← scoped で格納
  }
}

// 3. persistSaveHashes() 内の保存
private persistSaveHashes(agentWs: string, agentId: string): void {
  if (!agentWs || !agentId) return;
  this.saveHashPersistCounter++;
  if (this.saveHashPersistCounter % this.SAVE_HASH_PERSIST_INTERVAL !== 0) return;
  // agentWs:agentId: で始まる entry だけをフィルタして保存
  const prefix = `${agentWs}:${agentId}:`;
  const entries = Array.from(this.recentSaveHashes.entries())
    .filter(([key]) => key.startsWith(prefix))
    .map(([key, timestamp]) => ({ key: key.slice(prefix.length), timestamp })); // DB には rawHash のみ保存
  this.rpcClient.setNarrativeSaveHashes(agentWs, agentId, entries)
    .catch((err) => console.warn("[NarrativeWorker] Failed to persist save hashes:", err));
}
```

### 3.3 注意点

- `saveNarrative` の呼び出し箇所で `item.agentWs` / `item.agentId` が常に利用可能であることを確認
- `persistSaveHashes` は現在 `saveNarrative` の最後で `item.agentWs/item.agentId` を渡して呼ばれている（L590）
- TTL eviction loop (L509) は key が変わっても同様に動作する（Date.now() 比較は変わらず）
- **DB 格納形式は変更しない**（rawHash のみ保存。prefix は in-memory のみ）→ 既存データとの互換性維持

### 3.4 テスト観点

- [ ] Agent A と Agent B が同じ rawText を持つ場合、互いに影響しないこと
- [ ] loadSaveHashes 後に正しい scoped key で格納されていること
- [ ] persistSaveHashes が該当 agent の entry のみを保存すること

## 4. 修正 #3: `handleIngest` getStore 失敗時の best-effort 化

### 4.1 問題の詳細

v0.4.21b で `getStore()` を slug 生成直後に前倒しした結果、
**store 初期化失敗 = ingest 全体中止** になった。
旧実装（v0.4.21b 以前）は「vector 保存だけスキップ、file save は継続」の best-effort だった。

現在のコード:
```go
vstore, storeErr := getStore(params.AgentWs)
if storeErr != nil {
    EmitLog("ai.ingest: Store init failed: %v", storeErr)
    sendResponse(conn, req.ID, RPCError{Code: -32000, Message: "Store init failed"})
    return  // ← 全体中止！
}
```

一時的な Pebble ロック・I/O 問題で、本来 markdown として残せた内容まで落とす回帰。

### 4.2 修正仕様

`getStore()` 失敗時は **dedup ガードをスキップ** し、file save は継続する。
vector store add だけでなく、`vstore` を使う全処理（`Get` / `Add` / `SetMeta`）を `vstore != nil` で保護する。

```go
vstore, storeErr := getStore(params.AgentWs)
if storeErr != nil {
    // [v0.4.21c] Best-effort: dedup guard skipped, file save continues
    EmitLog("ai.ingest: Store init failed (best-effort mode): %v", storeErr)
    // vstore は nil のまま → 後続の dedup check と vstore.Add をスキップ
}

// ... slug generation ...

// [v0.4.21c] Duplicate guard: only when vstore is available
if vstore != nil {
    if _, getErr := vstore.Get(slug); getErr == nil {
        ingestDbDuplicateSkips.Add(1)
        EmitLog("Ingest: duplicate slug already exists in DB, skipping: %s (total_skips=%d)", slug, ingestDbDuplicateSkips.Load())
        sendResponse(conn, req.ID, map[string]interface{}{"path": "", "slug": slug})  // ← #6 で修正
        return
    }
}

// ... serialize (file save) — always proceeds ...

// [v0.4.21c] Vector store add: only when vstore is available
if vstore != nil && embedErr == nil && emb != nil {
    // ... vstore.Add ...
}

// [v0.4.21c] last_activity meta update: only when vstore is available
if vstore != nil {
    _ = vstore.SetMeta("last_activity", []byte(fmt.Sprintf("%d", now.Unix())))
}
```

### 4.3 影響範囲

- `main.go` handleIngest 関数内のみ（`vstore` 使用箇所すべてを nil ガード）
- handleBatchIngest は既に best-effort に近い構造（個別 item の失敗が他に波及しない）なので変更不要

### 4.4 テスト観点

- [ ] getStore 失敗時に file save が完了すること
- [ ] getStore 失敗時に vector store に保存されないこと
- [ ] getStore 成功時の dedup / 保存が従来通り動作すること

## 5. 修正 #4: 日本語 CoT regex `[ましょうて]` の文字クラスバグ修正

### 5.1 問題の詳細

```typescript
/^.*を(整理|分析|まとめ|考察)し[ましょうて]/
```

`[ましょうて]` は正規表現の**文字クラス**であり、「ま・し・ょ・う・て」の**どれか1文字**にマッチする。
意図は「しましょう」「して」等の代替グループだった。

実害例: `資料を整理していた彼は...` → `整理し` + `て` でマッチ → **false positive**
→ 品質 gate が不必要にリトライ・破棄

### 5.2 修正仕様

文字クラスを非キャプチャグループに置換し、さらに語尾境界を追加して false positive を抑制する:

```typescript
// 旧:
/^.*を(整理|分析|まとめ|考察)し[ましょうて]/,

// 新:
/^.*を(整理|分析|まとめ|考察)(?:しましょう|して)(?:[、。]|$)/,
```

`(?:しましょう|して)(?:[、。]|$)` は:
- `整理しましょう。` / `分析して` などの CoT 起点文を検出
- `整理していた` / `分析してから` のような叙事文中フレーズは語尾境界がないため除外

### 5.3 Gate 3 への影響

Gate 3 (`assistantPhrases`) は `startsWith` で判定しており、regex ではないため影響なし。

### 5.4 テスト観点

- [ ] `資料を整理していた彼は...` が Gate 2 を**通過**すること（false positive 解消）
- [ ] `資料を整理しましょう。` が Gate 2 で**検出**されること
- [ ] `データを分析して`（文末 or 句点終端）が Gate 2 で**検出**されること
- [ ] `情報をまとめてみる` が Gate 2 を**通過**すること（叙事文寄り表現の誤検知を避ける）

## 6. 修正 #5: duplicate save 時の `cacheAck` 二重実行の解消

### 6.1 問題の詳細

現在のコード:

```typescript
// saveNarrative() 内 (L560-565):
if (this.recentSaveHashes.has(scopedKey)) {
  // ... duplicate detected ...
  await this.rpcClient.cacheAck(item.id, "narrative-worker");  // ← Ack 1
  return;
}

// processNextFromCache() 内 (L349):
await this.saveNarrative(result, item);
await this.rpcClient.cacheAck(item.id, "narrative-worker");    // ← Ack 2
```

現状は duplicate 分岐で `saveNarrative` 内で Ack した後、呼び出し元でも Ack するため、
**実際に二重 Ack が発生し得る**。また Ack の責任が二箇所に分散していて保守性が低い。

### 6.2 修正仕様

saveNarrative 内の duplicate 分岐から `cacheAck` を削除し、**呼び出し元で統一的に Ack** する。
戻り値の型変更（`Promise<boolean>`）は行わず、`Promise<void>` のまま最小修正で対応する。

```typescript
// saveNarrative は Promise<void> のまま
private async saveNarrative(result: NarrativeResult, item: CacheItem): Promise<void> {
  const rawHash = this.hashRawText(item.rawText);
  const scopedKey = `${item.agentWs}:${item.agentId}:${rawHash}`;
  if (this.recentSaveHashes.has(scopedKey)) {
    const savedAt = this.recentSaveHashes.get(scopedKey)!;
    if (Date.now() - savedAt < this.DUPLICATE_WINDOW_MS) {
      console.log(JSON.stringify({ source: "episodic-claw", event: "narrative-duplicate-skip",
        hash: rawHash, ageMs: Date.now() - savedAt }));
      return;  // duplicate。Ack は呼び出し元に任せる
    }
    this.recentSaveHashes.delete(scopedKey);
  }
  // ... 保存処理 ...
}

// processNextFromCache 側:
await this.saveNarrative(result, item);  // duplicate でも save でも
await this.rpcClient.cacheAck(item.id, "narrative-worker");  // 常に呼び出し元で Ack
```

### 6.3 テスト観点

- [ ] duplicate 検出時に cacheAck が1回だけ呼ばれること
- [ ] 正常保存時に cacheAck が1回だけ呼ばれること

## 7. 修正 #6: duplicate `ai.ingest` 戻り値の `path` 改善

### 7.1 問題の詳細

現在、duplicate 検出時の `ai.ingest` 戻り値が:
```json
{ "path": "", "slug": "episode-..." }
```
`index.ts` L1354 で `Got it — filed that away at ${slugRes.path}` と表示されるため、空 path がユーザに露出する。

### 7.2 修正仕様

duplicate 検出時に `vstore.Get(slug)` で取得した既存レコードの `SourcePath` を返す。

```go
if _, getErr := vstore.Get(slug); getErr == nil {
    ingestDbDuplicateSkips.Add(1)
    EmitLog("Ingest: duplicate slug already exists in DB, skipping: %s (total_skips=%d)", slug, ingestDbDuplicateSkips.Load())
    // [v0.4.21c] Return existing record's path instead of empty string
    existingRec, _ := vstore.Get(slug)  // already know it exists
    existingPath := ""
    if existingRec != nil {
        existingPath = existingRec.SourcePath
    }
    sendResponse(conn, req.ID, map[string]interface{}{"path": existingPath, "slug": slug})
    return
}
```

※ `vstore.Get(slug)` は既に一度呼んで成功しているので、2回目の呼び出しは必ず成功する。
パフォーマンスへの影響は微小（同一 slug の Get はキャッシュされる可能性も高い）。

代替案として1回の Get で済ませるリファクタリングも検討:

```go
existingRec, getErr := vstore.Get(slug)
if getErr == nil {
    // duplicate
    ingestDbDuplicateSkips.Add(1)
    existingPath := ""
    if existingRec != nil {
        existingPath = existingRec.SourcePath
    }
    sendResponse(conn, req.ID, map[string]interface{}{"path": existingPath, "slug": slug})
    return
}
```

**→ 1回の Get で済ませる方を採用。**

### 7.3 スコープ境界（handleBatchIngest は対象外）

`handleBatchIngest` の RPC 契約は `[]string`（slug配列）であり `path` を返さない。
したがって本修正は **`handleIngest` のみ** を対象とし、batch 側は契約変更しない。

### 7.4 テスト観点

- [ ] `handleIngest` duplicate 検出時に空でない path（`existingRec.SourcePath`）が返ること
- [ ] existingRec が nil の場合に path: "" が返ること（フォールバック）
- [ ] 通常の ingest は変更なく動作すること
- [ ] `handleBatchIngest` の戻り値契約（`[]string`）が変わらないこと

## 8. 修正 #7: state.db の Close() を shutdown で呼ぶ

### 8.1 問題の詳細

`globalStateStore` (PebbleDB) が Go sidecar の shutdown パスで Close() されていない。
`pebble.Sync` しているので durability 的には致命ではないが、file lock 解放と clean shutdown の観点で問題。

### 8.2 修正仕様

Go 側の終了パス（watchdog EOF 経路）に `globalStateStore.Close()` を追加する。
現在 `startWatchdog()` は EOF 検知後に `os.Exit(0)` を直接呼ぶため、defer では close されない。

修正後（概略）:
```go
func closeGlobalStateStore() {
    stateStoreMu.Lock()
    defer stateStoreMu.Unlock()
    if globalStateStore != nil {
        if err := globalStateStore.Close(); err != nil {
            EmitLog("state.db: close error: %v", err)
        }
        globalStateStore = nil
    }
}

func startWatchdog() {
    go func() {
        buf := make([]byte, 1)
        _, err := os.Stdin.Read(buf)
        EmitLog("Stdin closed (parent death detected): %v. Terminating.", err)
        closeGlobalStateStore()
        os.Exit(0)
    }()
}
```

補足: 将来 graceful shutdown 経路（signal handler）を入れる場合も `closeGlobalStateStore()` を共通利用する。

### 8.3 テスト観点

- [ ] sidecar 終了後に state.db の file lock が解放されていること
- [ ] 再起動後に state.db が正常に open できること

## 9. 修正 #8: `state.get` の read error と not-found の区別

### 9.1 問題の詳細

`handleStateGet()` は PebbleDB の一時的 I/O エラー・破損と not-found を同じ `result: ""` として返す。
TS 側では区別不可 → cursor/save-hash restore が silently 無効化される。

現在のコード:
```go
val, getErr := stateStore.Get(params.Key)
if getErr != nil {
    EmitLog("state.get: failed to read key %s: %v", params.Key, getErr)
    // not-found も real error も同じ空文字を返す
    sendResponse(conn, req.ID, "")
    return
}
sendResponse(conn, req.ID, val)
```

### 9.2 修正仕様

`store.Get()` は既に not-found の場合 `("", nil)` を返すように実装済み（v0.4.21b）。
したがって `getErr != nil` は **real error** のみを意味する。
real error の場合は空文字ではなく RPC error を返す。

```go
val, getErr := stateStore.Get(params.Key)
if getErr != nil {
    // [v0.4.21c] Distinguish real DB errors from not-found.
    // store.Get() returns ("", nil) for not-found, so getErr != nil means actual failure.
    EmitLog("state.get: failed to read key %s: %v", params.Key, getErr)
    sendResponse(conn, req.ID, RPCError{Code: -32000, Message: fmt.Sprintf("state.get read error: %v", getErr)})
    return
}
sendResponse(conn, req.ID, val)
```

### 9.3 TS 側の影響

`rpc-client.ts` の `stateGet()` は現在:
```typescript
async stateGet(key: string, agentWs?: string, agentId?: string): Promise<string> {
    return this.request<string>("state.get", { key, agentWs, agentId });
}
```

RPC error が返ると `this.request` が例外を投げる。
各呼び出し側は既に `.catch(() => "")` でフォールバックしているので、
**既存のフォールバック動作は変わらない**。
ただし、エラーの観測性が向上する（`.catch` のログが出力される可能性）。
