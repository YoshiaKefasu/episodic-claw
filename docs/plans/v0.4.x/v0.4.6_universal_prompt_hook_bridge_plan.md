## 3. 現実的な制約（正直な断言）

> **OpenClaw本体を触らない前提では、CLI経路で `before_prompt_build` 同等の強制注入を100%保証することはできない。**

なので v0.4.6 は「Plugin-onlyで実現可能な最善解」を採用する。

## 4. 採用方針（Option A: Plugin-only）

### A-1. Embedded経路（非CLI）

- **Tool-first + 条件付き実行フィルター** を主経路にする
- Tool-first が有効なターンでは、注入経路を事前に閉じる
  - `assemble()` は `retrieveRelevantContext` を呼ばない
  - `before_prompt_build` は常に no-op return (`{}`)
- 理由: Tool-first の `ep-recall` 結果は tool result として同ターン文脈に乗るため、追加注入は不要

### A-1.1 重複イベント抑制（Embedded限定）

- `before_prompt_build` は OpenClaw API 上で `handled/cancel` を返せないため、明示キャンセルは不可
- 代わりにプラグイン側で **no-op return (`{}`)** を常用し、重複イベント経路を遮断する
- `assemble` と `before_prompt_build` の二重 `retrieveRelevantContext` 実行を禁止する

### A-1.2 例外フォールバック（ハード失敗時のみ）

- 以下の **事前に判定できるハード失敗** のときだけ、注入フォールバックを許可する（任意）
  - `ep-recall` ツール非公開 / 利用不可
  - memory runtime 未起動（gateway未起動等）
  - `toolFirstRecall.enabled=false`
- それ以外は Tool-first 経路を優先し、`retrieveRelevantContext` を走らせない

### A-2. CLI経路（本件の救済）

- **Tool-first Recall** をプラグイン側で明示
- システムプロンプト（memory section）に、次の契約を追加する
  - 各ユーザーターンで **条件付き実行フィルター** を先に通し、条件一致時のみ `ep-recall` を1回呼ぶ
  - `query` は v0.3.7 からの既存 Query 生成系（messages Parse/Rewrite）で構築する
  - 「最新ユーザーメッセージ生文をそのまま query に使う」方式は採用しない
  - `recallQueryRecentMessageCount` は Query 生成に渡す直近メッセージ窓サイズを決める設定であり、生文直投げ設定ではない
  - `k=3`（初期値）
  - 関連があれば同ターン回答に統合
  - ヒットなしなら捏造せず通常回答

### A-2.1 条件付き実行フィルター（Fast Path）

- 目的: 「毎ターン強制 `ep-recall`」をやめ、ノイズと無駄なツール呼び出しを削減する
- 判定順（短絡評価）:
  1. `novelty gate`: 低情報メッセージ（OK/了解/thanks等）なら即skip
  2. `intent gate`: 記憶参照意図（過去参照・継続タスク・比較/再確認）がなければskip
  3. `fingerprint gate`: Parse/Rewrite後のQuery指紋が直近と同一ならskip
  4. `negative cache gate`: 同一指紋で no-hit バックオフ中ならskip
  5. ここまで通過した場合のみ `ep-recall` 実行
- no-hit対策:
  - no-hit連続時は同一指紋の再実行を `3 -> 6 -> 12` ターンで指数バックオフ
  - no-hit時はユーザー向け文面を増やさず通常回答へ戻る（ノイズ抑止）

### A-2.2 2010年級CPU向けレイテンシ目標（MS単位）

- 前提: 判定処理はネットワークI/Oなし、メモリ内データのみ
- 目標（1ターンあたり、フィルター判定パス）:
  - p50 <= 1.0ms
  - p95 <= 3.0ms
  - p99 <= 5.0ms
- ステージ別目安:
  - 正規化/軽量トークン化: <= 0.4ms
  - intent/novelty判定: <= 0.8ms
  - 指紋生成（SHA1/xxhash等）: <= 0.4ms
  - negative cache照会/更新: <= 0.2ms
  - 合計（skip判定完了まで）: <= 2.0ms 目標
- 注記: `ep-recall` 実行後のRPC/検索時間は別枠（この目標は「打つか打たないか判定」部分）

### A-3. 期待値

- CLIでも「ユーザー発話 -> Query実行 -> 同ターン反映」の実行確率を大きく上げる
- OpenClaw本体非改修でも即リリース可能

## 5. 実装スコープ（Episodic-Clawのみ）

1. `src/index.ts`
- memory prompt section/supplement へ Tool-first Recall契約を追加
- Tool-first 有効判定（例: `shouldUseToolFirstForTurn`）を追加
- Tool-first 有効ターンでは `assemble` の `retrieveRelevantContext` を実行しない（論理無効化）
- Tool-first 有効ターンでは `before_prompt_build` は常に no-op return (`{}`)
- ハード失敗時のみ `retrieveRelevantContext` フォールバックを許可（任意）
- CLI向けに条件付き実行フィルターを追加（intent/novelty/fingerprint/negative-cache）
- no-hit指数バックオフ（3/6/12 turns）を追加
- フィルターは O(window) + O(1) lookup で完結し、全判定をメモリ内で処理する
- 判定/分岐ログを追加（例: `source=tool_first|fallback`, `skip_reason=*`）

2. `openclaw.plugin.json`
- CLI fallback方針の説明を追記
- 必要なら toggle 用設定を追加（例: `toolFirstRecall.enabled`, `toolFirstRecall.k`）
- `recallQueryRecentMessageCount` の説明に「Parse/Rewrite窓サイズ」であることを明記

3. `README.md` / `README.ja.md`
- 経路差分を明文化
- CLIは「hook注入」ではなく「tool-first」で等価目的を達成することを明記
- Query は「最新1件の生文」ではなく、既存の messages Parse/Rewrite で作ることを明記

4. `RELEASE_NOTES_v0.4.6.md`
- 本リリースの目的と制約を明記

## 6. テスト計画（Plugin-only）

1. 非CLI provider で Tool-first 有効ターンは `assemble`/`before_prompt_build` とも注入no-opになり、`retrieveRelevantContext` が呼ばれない
2. CLI provider で「フィルター判定 -> 条件一致時のみ `ep-recall` 実行」が動作する
3. Query 構築が「最新1件生文直投げ」ではなく、既存 Parse/Rewrite 系であることをログで確認できる
4. `recallQueryRecentMessageCount` 変更時に Query 窓サイズだけが変わり、生成方式自体は不変である
5. no-hit時に同一Query指紋の再実行がバックオフされ、毎ターン連打されない
6. `ep-recall` ヒット時、回答本文に記憶参照が反映される
7. `ep-recall` ノーヒット時、捏造なしで通常回答
8. Tool-first無効化設定時に既存挙動へ戻る（toggle導入時）
9. Embeddedで Tool-first 有効時、重複Query/重複注入イベントが発生しない
10. `ep-recall` ハード失敗時のみ、許可設定に応じて `retrieveRelevantContext` フォールバックが動作する
11. フィルター判定レイテンシが 2010年級CPU相当プロファイルで p95 <= 3.0ms を満たす

## 7. リスクと対応

### リスク

- CLIモデルが命令に従わず、`ep-recall` をスキップする可能性
- ツール呼び出し増加による遅延・コスト増
- Tool-first 判定が厳しすぎて必要ターンを skip する可能性
- フィルター実装が重くなると低スペックCPUで体感遅延が出る可能性

### 対応

- 命令文を短く強制力の高い表現に固定
- ログで `ep-recall` 実行有無を観測し、スキップ率を追跡
- 失敗時フォールバックとして手動 `ep-recall` 運用も残す
- 注入フォールバックはハード失敗時のみ許可し、通常ターンの二重経路を禁止する
- `skip_reason` ログを必須化して誤判定を追跡可能にする
- フィルター計算をメモリ内・固定窓・短絡評価に限定し、重い正規表現/全文走査を禁止
- `filter_eval_ms` を常時ログ化し、しきい値超過時に警告を出す

---

## 8. リリース判定（Go/No-Go）

1. OpenClaw本体への差分がゼロであること
2. 非CLIで Tool-first 有効時に注入経路が論理無効化され、重複イベントがゼロであること
3. CLIで条件付きフィルターが機能し、不要ターンでは `ep-recall` を実行しないこと
4. docs / release notes が実装と一致していること
5. フィルター判定の p95 が 3.0ms 以下であること（低スペック環境プロファイル）

---

> OpenClaw本体に依存しない即応策として、CLIは Tool-first Recall で運用ギャップを埋める。  
> これが今の制約下で最短・最小リスクの現実解。

### 🛤️ Solution Options

#### Option A — Plugin-only Bridge Hardening *(推奨)*
**Approach**: OpenClaw本体を触らず、Episodic-Claw側で「契約一致 + 単一路化 + CLI用Tool-first」を成立させる。  
**Implementation cost**: Medium  
**Risk**: Low-Medium  
**Why recommended**: 既存方針（Plugin-only）を維持しつつ、今の失敗点を最小差分で解消できる。  
**Concrete steps**:
1. `assemble()` 返却を `prependSystemContext` ではなく `systemPromptAddition` に合わせる（契約一致）。
2. Tool-first 有効ターンでは `assemble` の `retrieveRelevantContext` を実行しない（論理無効化）。
3. Tool-first 有効ターンでは `before_prompt_build` を常時 no-op return (`{}`) にする。
4. CLI向け Tool-first ガイダンスは `api.registerMemoryCapability({ promptBuilder })` か `registerMemoryPromptSection` で注入する。
5. Tool-first の query は既存 Parse/Rewrite を共通関数化して再利用し、`recallQueryRecentMessageCount` は user窓として明示する。
6. ノイズ抑止は `novelty -> intent -> fingerprint -> negative cache` の短絡評価 + `3/6/12` バックオフで固定。
7. 診断ログを `source=tool_first|fallback` で統一し、重複有無を追えるようにする。

#### Option B — 最小保守（重複抑止のみ）
**Approach**: Embedded 側の契約修正と重複抑止だけを先に実施し、CLI Tool-first 自動化は見送る。  
**Implementation cost**: Low  
**Risk**: Low  
**When to choose this instead**: v0.4.6 を緊急で切る必要があり、CLI側の挙動差を短期的に許容できる場合。  
**Concrete steps**:
1. `assemble` の返却契約を修正。
2. `before_prompt_build` の重複 no-op を先行実装。
3. CLI は暫定で手動 `ep-recall` 運用 + ドキュメント警告を強化。

### ✅ Pro Recommendation
> **Choose Option A because**: Plugin-only制約を守りながら、CLI/Embedded の両経路で「Query→同ターン反映」を現実的に揃えられる。  
> Estimated implementation: 0.5〜1.0日  
> Rollback plan: `toolFirstRecall.enabled=false` と no-op ガード無効化で v0.4.5 相当挙動へ即時復帰。
