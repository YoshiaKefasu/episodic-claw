### Phase 1: TS側 recall 結果キャッシュ + 二重実行排除

> **最大のインパクト: キャッシュヒット時に200〜3000ms削減。実装1〜2時間。**

#### 5.1.1 TS側 recall 結果 LRU キャッシュ

**問題:**
Go側に `recallCache`（sync.Map、query→embedding vector、15分TTL）があるが、TS→Go のRPC往復は**常に発生**する。
同一会話内で連続ターンを送ると、毎回同じ（または類似の）クエリでRPC + Embedding APIを叩く。

**設計:**

```typescript
// src/retriever.ts — モジュールレベルの recall 結果キャッシュ
type RecallResultCache = {
  queryHash: string;
  result: RecallInjectionOutcome;
  cachedAt: number;
};

const RECALL_CACHE_TTL_MS = 60_000; // 1分（会話テンポに合わせた短命TTL）
let recallResultCache: RecallResultCache | null = null;
```

**キャッシュ無効化の統合:**
- `invalidateRecallCacheForWorkspace()` が呼ばれたとき（新エピソード追加後）にTS側キャッシュも `null` にリセット
- `lastInjectedResultHash` のリセットと同じタイミングで無効化

**フロー:**

```
retrieveRelevantContext() {
  queryHash = SHA1(query)
  
  // キャッシュヒット判定
  if (recallResultCache
      && recallResultCache.queryHash === queryHash
      && Date.now() - recallResultCache.cachedAt < RECALL_CACHE_TTL_MS) {
    return recallResultCache.result;  // RPC往復を完全スキップ
  }
  
  // キャッシュミス → 通常のRPC実行
  const result = await rpcClient.recall(...);
  recallResultCache = { queryHash, result, cachedAt: Date.now() };
  return result;
}
```

**リスクと緩和策:**

| リスク | 影響度 | 緩和策 |
|---|---|---|
| stale なキャッシュで新エピソードが検索されない | Medium | `invalidateRecallCacheForWorkspace()` で無効化。TTL 1分で自然失効 |
| メモリ使用量増加 | Low | キャッシュは直近1件のみ保持。RecallInjectionOutcome は数KB |
| re-injection guard との干渉 | Low | キャッシュヒットでも re-injection guard は別途チェック済み |

**Acceptance Criteria:**
1. 同一クエリハッシュの連続ターンでRPCが発行されない（ログで確認）
2. 新エピソード保存後にキャッシュが無効化される
3. TTL経過後にキャッシュが自然失効する
4. re-injection guard が正常に機能する
5. `npm run build:ts` PASS
6. `npm test` 全 PASS

#### 5.1.2 before_prompt_build と assemble() の二重 recall 排除

**問題:**
`src/index.ts` の構造:
- `before_prompt_build`（L863-1014）: セグメンテーション + recall + 注入
- `assemble()`（L1049-1125）: 「保険として」再度セグメンテーション + recall + 注入

contextEngine slot が `"episodic-claw"` の場合、**both** が呼ばれるため同一のrecallが2回実行される。

**設計:**
`assemble()` 内で「before_prompt_build が既にこのターンで recall を実行済みか」を判定し、済んでいればrecallをスキップする。

```typescript
// AgentRuntimeState に追加
type AgentRuntimeState = {
  // ... existing fields ...
  /** before_prompt_build が直近のターンで recall を実行していれば true */
  lastRecallTurnMessageCount: number;
};
```

```
[before_prompt_build]
  → recall 実行 → state.lastRecallTurnMessageCount = msgs.length
  → return { prependSystemContext }

[assemble()]
  → if (msgs.length === state.lastRecallTurnMessageCount) {
       // before_prompt_build で既に recall 済み → recall スキップ
       // prependSystemContext は OpenClaw が before_prompt_build の結果をマージ済み
       return { messages: msgs, prependSystemContext: "", estimatedTokens: 0 };
     }
  → else { 通常の recall 実行（before_prompt_build がブロックされた場合の保険） }
```

**リスクと緩和策:**

| リスク | 影響度 | 緩和策 |
|---|---|---|
| before_prompt_build がブロックされた場合に recall されない | Low | `msgs.length !== lastRecallTurnMessageCount` で保険パスが動く |
| OpenClaw が before_prompt_build の prependSystemContext を無視した場合 | Low | contextEngine slot が episodic-claw なら OpenClaw は prependSystemContext をマージする |

**Acceptance Criteria:**
1. contextEngine slot が `"episodic-claw"` の場合、recall が1ターンにつき1回だけ実行される
2. before_prompt_build がブロックされた場合、assemble() の保険recallが正常に動く
3. ログに `assemble() called` と `before_prompt_build` の両方が出るが、`ai.recall payload` は1回のみ
4. `npm run build:ts` PASS
5. `npm test` 全 PASS

#### 5.2.1 isAttachmentDominant() と stripAttachmentNoise() の統合

**問題:**
`src/retriever.ts` の `instantDeterministicRewrite()` 内で:
1. `isAttachmentDominant(text)` — ATTACHMENT_BOILERPLATE 6本の正規表現でマッチ判定（L59-98）
2. `stripAttachmentNoise(text)` — **同じ** ATTACHMENT_BOILERPLATE 6本の正規表現で除去（L104-112）

同一テキストに対して同じ正規表現を2回走らせている。

**修正方針:**
1パスで「判定 + 除去」を同時に行う統合関数を作る:

```typescript
/**
 * Classify and clean attachment noise in a single pass.
 * Returns { isDominant, cleanedText }.
 */
export function classifyAndStripAttachment(text: string): {
  isDominant: boolean;
  cleanedText: string;
} {
  let cleaned = text;
  let markerCount = 0;

  for (const pattern of ATTACHMENT_BOILERPLATE) {
    const matches = cleaned.match(pattern);
    if (matches) markerCount += matches.length;
    cleaned = cleaned.replace(pattern, "");
  }

  // Clean up leftover whitespace
  cleaned = cleaned.replace(/\n{3,}/g, "\n\n").replace(/[ \t]+/g, " ").trim();

  // Dominance check on cleaned text
  const cjkChars = (cleaned.match(/[\p{Script=Han}\p{Script=Katakana}\p{Script=Hiragana}\p{Script=Hangul}]/gu) || []).length;
  const isDominant = cleaned.length < 2
    || (cjkChars === 0 && cleaned.length < 5)
    || MEDIA_ONLY_SENTINEL.test(text.trim());

  return { isDominant, cleanedText: cleaned };
}
```

```diff

     const rawText = extractText(m.content);

-    if (isAttachmentDominant(text)) {
-      const caption = stripAttachmentNoise(text);
-      if (caption.length >= 3) {
-        eligibleTexts.push(caption);
```

**Acceptance Criteria:**

## 7. Solution Options

### Option A — 3フェーズ段階実装（推奨）

**Approach:**
Phase 0 → Phase 1 → Phase 2 の順に段階的に実装。各フェーズでビルド・テスト・動作確認を行う。

**Implementation cost:** Low → Medium
**Risk:** Low
**Why recommended:**
各フェーズが独立しており、Phase 0 だけでも即効性がある。ロールバックも容易。

### Option B — 全フェーズ一括実装

**Approach:**
Phase 0〜2 を一気に実装する。

**Implementation cost:** Medium
**Risk:** Medium
**When to choose this instead:**
開発時間が限られており、1回のビルド・デプロイで全てを入れたい場合。

### Option C — Phase 0 のみ実装、残りは見送り

**Approach:**
デバッグ残骸の除去と recallFeedback 非同期化だけ行い、Phase 1-2 は YAGNI として見送る。

**Implementation cost:** Low
**Risk:** Low
**When to choose this instead:**
2コアCPU環境での運用が当面の予定になく、Phase 1-2 の恩恵が薄い場合。

### ⚠️ 非ブロッキング懸念（3件）

**1. `classifyAndStripAttachment` と旧関数の併存（DRY境界）**

`isAttachmentDominant()` と `stripAttachmentNoise()` が関数として残っているのに、ホットパスでは `classifyAndStripAttachment()` を使う。テストが旧関数を直接テストしている場合、ロジックが乖離するリスクがある。

→ **現時点では問題なし。** 旧関数がテストのアサーション対象であり続ける限り、統合関数との挙動差異はテストが検出する。将来的に旧関数を deprecated 扱いにするかは YAGNI で判断。

**2. キャッシュの agentWs 単位 vs グローバル単一変数**

`_recallResultCache` はモジュールレベルの単一変数。マルチエージェント環境でエージェントAのキャッシュがエージェントBの recall で上書きされる。ただし agentWs チェックがあるためキャッシュ**ヒット**は正しいエージェントのみ。**ミス時に前のキャッシュが捨てられるだけ** — 機能的には正しい。

→ **問題なし。** LRU Map にするほどの並行エージェント数は想定されていない。KISS 適合。

**3. assemble() のアンカー注入は duplicate guard でスキップされない**

`src/index.ts` L1112-1119: ガードは anchorPrependText 計算の**後**に挿入されている。anchorPrependText が存在する場合、`prependSystemContext = anchorPrependText.trim()` として返される。

→ **設計通り。** アンカー注入と recall は独立した関心事。recall だけスキップし、アンカーは通す。正しい。
