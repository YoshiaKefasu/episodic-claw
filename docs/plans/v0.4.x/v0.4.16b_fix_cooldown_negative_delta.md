## 1. 問題定義

### 1.1 根本原因

| 変数 | 定義場所 | 再起動後 | 永続化 |
|------|----------|----------|--------|
| `turnSeq` | `segmenter.ts` L33 | **0 にリセット** | ❌ メモリのみ |
| `LastBoundaryTurn` | `segstate.go` L21 → Pebble DB | **前回値を維持** | ✅ Pebble |

TS 側の `turnSeq` はプロセス再起動で 0 に戻る一方、Go 側の `st.LastBoundaryTurn` は Pebble DB から前回の値（例: 100）を復元する。この不一致により、再起動後の cooldown 判定で負値の差分が発生する。

### 1.2 再現シナリオ

```
再起動後の最初のセッション:

  TS: turnSeq = 1 (0から再開)
  Go: st.LastBoundaryTurn = 100 (Pebbleから復元)

  Go cooldown判定:
    params.Turn(1) - st.LastBoundaryTurn(100) = -99
    -99 <= cooldown(2) → true → isBoundary = false (強制)

  結果: 再起動後、負値の差分でも cooldown 条件が成立し、
        境界判定が過剰に抑制される
```

### 1.3 影響範囲

- **v0.4.16 の評価への影響**: v0.4.16 導入後に「分割が減った」と観測された場合、その変化の一部は λ=1.5→2.0 によるものではなく、この cooldown 過剰抑制によるものの可能性がある
- **長期的な影響**: 再起動のたびに cooldown が過剰抑制されるが、最初の境界検出後に `st.LastBoundaryTurn` が新しいターン番号で更新されるため、不一致は自然解消する。実害は軽微だが、毎回の再起動で再発する

|---|----------|-----|------|
| 1 | `segmenter.ts` | L33 | `private turnSeq = 0;` — メモリのみ、再起動で0にリセット |

## 2. 修正内容

### 2.1 Go (`main.go`) — cooldown 負値デルタガード（explicit `delta` 変数）

**変更箇所**: `handleSegmentScore()` 内の cooldown 判定（L1390-1394）

```go
// Before:
if isBoundary && cooldown > 0 && st.LastBoundaryTurn > 0 && params.Turn > 0 {
    if params.Turn-st.LastBoundaryTurn <= cooldown {
        isBoundary = false
        reason = "cooldown"
    }
}

// After:
if isBoundary && cooldown > 0 && st.LastBoundaryTurn > 0 && params.Turn > 0 {
    delta := params.Turn - st.LastBoundaryTurn
    if delta >= 0 && delta <= cooldown {
        isBoundary = false
        reason = "cooldown"
    }
}
```

**変更点**: 差分を明示的な `delta` 変数に抽出し、`delta >= 0` で負値を除外。

GPT 5.3 は `delta > 0`（同一ターン除外）を推奨したが、Gemini 深掘り検証にて **`delta >= 0` が正しい**ことを確認:
- `delta == 0`（同一ターンでの連続境界検出）は cooldown で抑制すべき → `>= 0` が必要
- `delta < 0`（再起動後の負値ケース）→ cooldown 不成立 → 境界判定が正しく行われる
- `delta > 0`（通常ケース）→ 従来通りの cooldown 動作

**`delta` 変数の利点**（GPT 5.3 推奨理由）:
- 意図が明示的（「デルタが非負かつ閾値以内」が一行で表現）
- Go の慣用句に合致（`if x-y <= z` より `delta := x-y; if delta >= 0 && delta <= z` が可読性良好）
- デバッグ時のログ出力に `delta` 変数を直接利用可能

### 2.2 TypeScript — 変更なし

`turnSeq` の永続化は不要。Go 側のガードで十分対応できる。

## 4. `delta >= 0` vs `delta > 0` の選定理由

GPT 5.3 は `delta > 0` を推奨したが、Gemini 深掘り検証にて **`delta >= 0` が正しい**ことを確認。

| 条件 | `delta > 0` の場合 | `delta >= 0` の場合 | 正しい動作 |
|------|-------------------|-------------------|------------|
| `delta == 0`（同一ターン再検出） | cooldown **不成立**（境界が通る） | cooldown **成立**（境界が抑制される） | **成立** — 同じターンでの連続境界は抑制すべき |
| `delta < 0`（再起動後の負値） | cooldown 不成立 ✅ | cooldown 不成立 ✅ | 不成立（再起動ケース） |
| `delta > 0` 且つ `delta <= cooldown` | cooldown 成立 ✅ | cooldown 成立 ✅ | 成立（通常ケース） |

**`delta > 0` が危険な理由**: TS 側の `processTurn()` には `before_prompt_build` とフォールバック `assemble` の2つのトリガーパスがあり、並行/ファイヤーアンドフォーゲットの呼び出し経路が存在する。同一ターンでの重複呼び出しが発生し得るため、`delta == 0` のケースを cooldown で抑制することは安全側に倒した正しい設計である。

**稀なエッジケース**: 再起動後の最初のターンが偶然前回の `LastBoundaryTurn` と同じ値になる可能性があるが、これは確率的に極めて低く（ターン番号は1から再開）、仮に発生しても「1ターンだけ cooldown が余分に効く」だけで有害ではない。

## 9. 同一関数内の関連技術債務（スコープ外）

> GPT 5.3 Extra High 監査にて指摘された3つの追加バグ。v0.4.16b では修正しないが、将来の技術債務解消に向けて文書化する。

### 9.1 `GetSegmentationState` エラー無視（LOW）

```go
// main.go L1334
st, _ := vstore.GetSegmentationState(params.AgentId)
```

- **影響**: Pebble 読み取り失敗 or JSON デコード失敗時、ゼロ値構造体が返る。数学は安全に `inWarmup = true` にフォールバックするためクラッシュなし。ただし適応閾値のチューニング状態が一時的に失われる。
- **推奨修正**: エラーログ出力（`log.Printf`）を追加。リトライは不要（次回リクエストで自然回復）。

### 9.2 `PutSegmentationState` エラー無視（LOW）

```go
// main.go L1433
_ = vstore.PutSegmentationState(params.AgentId, st)
```

- **影響**: 書き込み失敗時、DB が更新されず次回リクエストで古い状態を読む。EWMA 計算が遅延するがクラッシュやパニックなし。
- **推奨修正**: エラーログ出力を追加。重要な状態遷移（境界検出時の `LastBoundaryTurn` 更新等）の喪失を検知可能にする。

### 9.3 ゴルーチン内の状態競合（LOW）

```go
// main.go L2956
go handleSegmentScore(conn, req)
```

- **影響**: `handleSegmentScore` がゴルーチンで実行されるため、同一 `agentId` に対する並行リクエストで読み書きの競合が発生し得る（check-then-act race）。ただし、TS 側の `processTurn()` は順次処理（`await`）であり、同一 `agentId` での真の並行実行はほぼ不可能。最悪ケースでも浮動小数点の EWMA 更新1回分の損失に留まる。
- **推奨修正**: 将来的に `sync.Mutex` を `agentId` 単位で導入。ただし現状の実害は皆無で、ロック導入によるレイテンシ増加のリスクが上回るため見送り。
