## Goal

既存の Segmenter の「即座にチャンク分割 → batchIngest」フローを、「Pool バッファに蓄積 → 物語化キューに投入」フローに置き換える。

```
v0.3.x:  processTurn → chunkAndIngest → batchIngest(生テキスト)
v0.4.0:  processTurn → poolAndQueue → fire-and-forget(.raw.md) + NarrativeWorker.enqueue()
```

### Core Concern: chunkAndIngest の置き換えスコープ

`chunkAndIngest` は 83 行で以下の責務を持つ:
1. バッファからテキスト抽出 (`extractText`)
2. チャンクサイズ制限 (`maxCharsPerChunk`)
3. 各チャンクに `buildSummaryForLevel` で要約タイトル付与
4. `rpc.batchIngest()` で Go sidecar に送信
5. バッファクリア

この全部を一気に置き換えるのはリスキー。段階的に:

**Step A:** `NarrativePool` を作成し、`chunkAndIngest` からの呼び出しを追加
**Step B:** `chunkAndIngest` 内で「Pool モード」と「Legacy モード」を分岐
**Step C:** Pool モード安定後に Legacy コードパスを削除

### Step 1: `src/narrative-pool.ts` 新規作成

**前提: Step 0 完了後。`PoolFlushItem` は `types.ts` から import する。**

```typescript
import { Message, extractText } from "./segmenter";
import { PoolFlushItem } from "./types";  // Step 0 で types.ts に移動済み

export class NarrativePool {
  private buffer: Message[] = [];
  private charCount: number = 0;
  private maxPoolChars: number;

  constructor(maxPoolChars: number);

  /** メッセージをプールに追加。flush 条件を満たしたら PoolFlushItem を返す */
  add(messages: Message[], surprise: number, agentWs: string, agentId: string): PoolFlushItem | null;

  /** 強制 flush (compaction 前、session_end) */
  forceFlush(agentWs: string, agentId: string): PoolFlushItem | null;

  /** プール内の文字数 */
  get currentChars(): number;
}
```

**実装の要点:**
- `add()` は `charCount` を追跡し、`maxPoolChars` を超過したら flush

- `PoolFlushItem.rawText` は `extractText()` を呼んでプールの全メッセージを連結したテキスト

### Step 3: `poolAndQueue` の実装

```typescript
private async poolAndQueue(agentWs: string, agentId: string, surprise: number): Promise<void> {
  if (!this.pool || !this.narrativeWorker) return;

  const item = this.pool.add(this.buffer.slice(), surprise, agentWs, agentId);
  this.buffer = [];
  this.lastProcessedLength = 0;

  if (item) {
    // Fire-and-forget: 生ログを .raw.md として即座保存
    this.saveRawLog(item, agentWs, agentId).catch(err =>
      console.error("[Episodic Memory] Raw log save failed:", err)
    );

    // 物語化キューに投入
    this.narrativeWorker.enqueue(item).catch(err =>
      console.error("[Episodic Memory] Narrative enqueue failed:", err)
    );
  }
}
```

### Step 4: `saveRawLog` の実装

```typescript
private async saveRawLog(item: PoolFlushItem, agentWs: string, agentId: string): Promise<void> {
  // .raw.md として保存。物語化が全失敗した場合のフォールバックデータ。
  const timestamp = new Date().toISOString().replace(/[:.]/g, "-");
  const rawPath = path.join(agentWs, `raw-${timestamp}.raw.md`);
  const rawContent = item.rawText;
  await fs.promises.writeFile(rawPath, rawContent, "utf8");
}
```

**Note:** `.raw.md` は Go sidecar の watcher が拾ってインデックスするが、拡張子が `.raw.md` なので通常のエピソードとは区別できる。Go 側の watcher に `.raw.md` を無視するフィルタを追加するかどうかは Phase 3 で判断。

### Findings

#### [P2-F1] `poolAndQueue` で `pool.add()` が null を返した場合の処理 — 情報

**場所:** `segmenter.ts` L279

**観察:** `pool.add()` は `maxPoolChars` 未達なら `null` を返す。その場合 L284 の `if (item)` で `saveRawLog` も `enqueue` もスキップされるが、L280 の `pool.clear()` と L281-282 の `buffer = [], lastProcessedLength = 0` は実行される。

**影響:** バッファが破棄された上で、Pool 内のメッセージもクリアされる。`add()` が `null` を返した = まだ Pool に蓄積している途中なのに、その蓄積分が消える。

**根本原因:** `pool.add()` はメッセージを内部バッファに追加した上で flush 判定をする。`null` = まだ溜めている、ということ。しかし `pool.clear()` が **無条件に呼ばれている** ため、蓄積途中のメッセージが消失する。

**修正案:**
```typescript
private async poolAndQueue(agentWs: string, agentId: string, surprise: number, reason: string): Promise<void> {
  if (!this.pool || !this.narrativeWorker) return;

  const item = this.pool.add(this.buffer.slice(), surprise, agentWs, agentId);
  // pool.clear() はここでは呼ばない — add() が pool 内部でバッファ管理しているため
  this.buffer = [];
  this.lastProcessedLength = 0;

  if (item) {
    this.pool.clear(); // flush 後のみクリア

    this.saveRawLog(item, agentWs).catch(err =>
      console.error("[Episodic Memory] Raw log save failed:", err)
    );
    this.narrativeWorker.enqueue(item).catch(err =>
      console.error("[Episodic Memory] Narrative enqueue failed:", err)
    );
  }
}
```

**Severity:** **高** — データロスト。Pool に蓄積中のメッセージが毎回消される。

#### [P2-F2] `forceFlush` 内の `pool.forceFlush()` が segmenter の buffer を渡していない — 中

**場所:** `segmenter.ts` L319

**問題:** `pool.forceFlush()` は Pool 内部のバッファを flush するが、segmenter の `this.buffer` を先に Pool に `add()` していない。つまり segmenter に溜まっているが Pool にまだ渡されていないメッセージは **flush されない**。

**分析:** `processTurn` では `handleSegmentBoundary` → `poolAndQueue` → `pool.add(this.buffer.slice())` で buffer を Pool に渡すが、`forceFlush` は直接 `pool.forceFlush()` を呼ぶだけで、segmenter の buffer を Pool に入れるステップがない。

**修正案:**
```typescript
async forceFlush(agentWs: string, agentId: string = ""): Promise<void> {
  if (this.buffer.length === 0) return;
  try {
    if (this.pool && this.narrativeWorker) {
      // segmenter の buffer を先に Pool に追加してから flush
      this.pool.add(this.buffer.slice(), 0, agentWs, agentId);
      const item = this.pool.forceFlush(agentWs, agentId);
      // ...
```

**Severity:** **高** — `session_end` / `before_compaction` 時に直近のバッファ内容がロストする。

#### [P2-F3] `NarrativePool.add()` が segmenter と重複してメッセージを持つ — 設計注意

**場所:** `narrative-pool.ts` L25-29

**観察:** `pool.add()` はメッセージを Pool の内部バッファにコピーする。しかし `segmenter.ts` も `this.buffer` にメッセージを持っている。つまり surprise 境界で `poolAndQueue` が呼ばれる前は、メッセージは **segmenter にのみ** 存在し、Pool は空。`poolAndQueue` 呼び出し時に segmenter から Pool に一括転送される。

**現状の問題:** Pool の `maxPoolChars` は「Pool 内部のバッファサイズ」で判定するが、`poolAndQueue` は毎回 segmenter の全バッファを一括で `add()` する。つまり **Pool の蓄積機能が実際には使われていない** — 毎回 segmenter から一括投入 → 即 flush (または null) → clear() で消すパターン。

Pool 設計の本来の意図は「複数回の surprise 境界を跨いで蓄積 → maxPoolChars 超過で flush」のはずだが、現実装では「1 回の surprise 境界 = 1 回の pool.add() = 即 flush か即消去」になっている。

**影響:** Pool のバッファリング効果が発揮されない。segmenter の `maxBufferChars` (default: 7200) が先に効くため、Pool に届く前にチャンクが分割される。

**判断:** Phase 2 時点では **そのままで OK**（KISS — まず動くことが大事）。Phase 3 / Phase 4 で Pool の蓄積効果を実際に検証して、必要なら segmenter → pool のデータフローを見直す。

#### [P2-F4] `.raw.md` のファイルパスが `agentWs` 直下 — 低
