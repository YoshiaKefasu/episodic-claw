# 実装プラン: v0.4.10 Media Attachment Query Leak Fix — 画像添付メッセージの Recall クエリ汚染修正

---

## あなたへの依頼内容: `episodic-claw` の Recall クエリ生成が、Gateway WebUI から送信した複数画像添付メッセージのシステムメタデータ（ファイルパス・auto-reply boilerplate）をキーワードとして混入させてしまう問題を修正してください。

---

## 問題の背景

### 現象

Gateway WebUI から複数枚の画像を送信すると、その後の Recall クエリに以下のようなノイズキーワードが混入する:

```
query="わかる 固定 今後 方針 決め バック エンド マジ send image prefer message"
skippedImageLikeMessages=0
reason=recall_empty
```

`send image prefer message` は画像添付時に Gateway が自動付加する auto-reply boilerplate 由来のテキスト。Recall に無関係なキーワードが混入し、ノイズクエリが無駄に発火する。

### 問題のメッセージ形式

Gateway WebUI が複数画像添付時に `user` ロールで渡すメッセージ:

```
[media attached: 10 files]
[media attached 1/10: /home/kasou_yoshia/.openclaw/media/inbound/file_384---af970a7f-a543-4a36-b894-794c9c8a88a0.jpg (image/jpeg) |
/home/kasou_yoshia/.openclaw/media/inbound/file_384---af970a7f-a543-4a36-b894-794c9c8a88a0.jpg]
[media attached 2/10: /home/kasou_yoshia/.openclaw/media/inbound/file_385---d8ee6358-a9ec-42a8-a42b-6b829a8c9738.jpg (image/jpeg) |
/home/kasou_yoshia/.openclaw/media/inbound/file_385---d8ee6358-a9ec-42a8-a42b-6b829a8c9738.jpg]
...（10件分）
To send an image back, prefer the message tool (media/path/filePath). If you must inline, use MEDIA:https://example.com/image.jpg (spaces ok, quote if needed) or a safe relative path like MEDIA:./image.jpg. Avoid absolute paths (MEDIA:/...) and ~ paths - they are blocked for security. Keep caption in the text body.

media:image
```

### 根本原因

`src/retriever.ts` の `ATTACHMENT_BOILERPLATE` 正規表現が、このメッセージ形式を完全に除去できていない。理由は3つ:

1. **マルチライン `[media attached N/M: ...]` ブロックのパイプ2行目が漏れる**
   - 既存パターン: `\[media attached(?:\s+\d+\/\d+)?:[^\]]*\]`
   - `[^\]]*` は改行を含まないため、パイプ `|` の後ろの2行目のパス文字列がマッチから除外される
   - 結果: `/home/kasou_yoshia/.openclaw/media/inbound/file_384---...jpg` がそのまま残る

2. **auto-reply boilerplate が複数行にまたがる**
   - 既存パターン: `To send an image back[^\n]*?(?:prefer|caption|Keep caption|URL)\.[ \t]*`
   - 実際のメッセージは `To send an image back...` から `Keep caption in the text body.` まで1段落分あり、複数行を跨ぐ
   - 結果: `prefer` `message` `Keep` `caption` `MEDIA` 等がテキストに残留する

3. **`media:image`（山括弧なし）が未対応**
   - 既存パターン: `<media:image>` (山括弧あり)
   - Gateway が末尾に付加する `media:image` は山括弧なしの素のテキストで、既存パターンにマッチしない
   - 結果: `image` がキーワードとして残る

`MEDIA_ONLY_SENTINEL` も同様に上記の残骸テキストを見落とすため `isDominant = false` と判定し、`instantDeterministicRewrite()` に汚染テキストが流れ込む。

### DRY 違反の補足（評価で発見）

`isAttachmentDominant()` (deprecated, L80-119) が `ATTACHMENT_BOILERPLATE` 定数を使わず、L92-97 で `.replace()` をハードコードしている。`ATTACHMENT_BOILERPLATE` を更新してもこの関数には反映されないため、同じバグが再発する。Phase 3 で委譲に置き換える。

## Phase 1: `ATTACHMENT_BOILERPLATE` の修正

**変更ファイル:** `src/retriever.ts` (L44-58 付近)

```diff
  const ATTACHMENT_BOILERPLATE: RegExp[] = [
    // [media attached: ...] or [media attached 1/2: ...] — indexed multi-attachment support
    // Covers: [media attached: /path], [media attached 1/2: media://inbound/...], etc.
-   /\[media attached(?:\s+\d+\/\d+)?:[^\]]*\]/gi,
+   /\[media attached(?:\s+\d+\/\d+)?:[^\]]*\](?:\s*\|\s*[^\n]*)*/gi,
    // <media:image>, <media:document>, <media:audio>, <media:video> with optional (N ...) suffix
    /<media:(image|document|audio|video)>(\s*\([^)]*\))?/gi,
    // [User sent media without caption]
    /\[User sent media without caption\]/gi,
    // To send an image back, prefer ... (match up to period, preserve text after)
-   /To send an image back[^\n]*?(?:prefer|caption|Keep caption|URL)\.[ \t]*/gi,
+   /To send an image back[\s\S]*?Keep caption in the text body\./gi,
    // standalone "attached files" / "attachment" lines without meaningful text
    /^\s*attached files\s*$/gi,
    // media://inbound/<id> standalone
    /media:\/\/inbound\/[^\s]+/gi,
+   // [media attached: N files] — summary header line (e.g., "[media attached: 10 files]")
+   /\[media attached:\s*\d+\s*files?\]/gi,
+   // Standalone "(image/jpeg)" or "(image/png)" MIME type annotations leaked from multi-line markers
+   /\(image\/\w+\)/gi,
+   // Bare "media:image" or "media:document" without angle brackets (Gateway WebUI format)
+   /^media:(image|document|audio|video)\s*$/gim,
+   // MEDIA: inline URL/path hints from auto-reply boilerplate
+   /MEDIA:[^\s]+/gi,
  ];
```

**変更詳細:**

| 変更 | 理由 |
|---|---|
| `[^\]]*\]` → `[^\]]*\](?:\s*\|\s*[^\n]*)*` | パイプ `\|` で続く2行目のパスも巻き込んで除去する |
| `To send an image back` パターンを `[\s\S]*?Keep caption in the text body.` に拡張 | 複数行にまたがる auto-reply boilerplate を末尾の確定文字列まで一括除去する |
| `\[media attached:\s*\d+\s*files?\]` 追加 | `[media attached: 10 files]` というサマリーヘッダー行を除去する |
| `\(image\/\w+\)` 追加 | パイプ後ろの MIME type 注釈 `(image/jpeg)` の残骸を除去する |
| `^media:(image|document|audio|video)\s*$` 追加 | 山括弧なしの `media:image` 行（Gateway WebUI 末尾追記形式）を除去する |
| `MEDIA:[^\s]+` 追加 | `MEDIA:https://...` や `MEDIA:./image.jpg` 形式のインライン URL ヒントを除去する |

## Phase 2: `MEDIA_ONLY_SENTINEL` の動的構築化

**変更ファイル:** `src/retriever.ts` (L71 付近)

1行に12個の選択肢を詰め込んだ正規表現は可読性が極めて低く、`ATTACHMENT_BOILERPLATE` との整合性も確認しづらい。パターン配列から `new RegExp()` で動的に組み立てる方式に変更する。

**現在のコード (L71):**

```typescript
const MEDIA_ONLY_SENTINEL = /^\s*(?:\[media attached[^\]]*\]|<media:[^>]+>(?:\s*\([^)]*\))?|\[User sent media without caption\]|attached files|media:\/\/inbound\/[^\s]+|\s)*$/i;
```

**変更後:**

```typescript
// Media-only sentinel: a message consisting almost entirely of attachment markers.
// Built from a string array so each pattern is on its own line with a comment,
// making it easy to keep in sync with ATTACHMENT_BOILERPLATE.
const MEDIA_ONLY_SENTINEL_PARTS = [
  "\\[media attached[^\\]]*\\](?:\\s*\\|[^\\n]*)*",  // [media attached: ...] with pipe continuation
  "<media:[^>]+>(?:\\s*\\([^)]*\\))?",                // <media:image> etc.
  "\\[User sent media without caption\\]",            // LINE/Discord no-caption marker
  "attached files",                                    // standalone "attached files" line
  "media://inbound/[^\\s]+",                          // Gateway media URI
  "To send an image back[^\\n]*",                     // auto-reply template first line
  "\\(image/\\w+\\)",                                 // MIME type (image/jpeg)
  "Keep caption[^\\n]*",                               // auto-reply tail line
  "media:(?:image|document|audio|video)",             // bare media:type (no angle brackets)
  "MEDIA:[^\\s]+",                                     // MEDIA: URL hints
  "\\s",                                               // whitespace
];
const MEDIA_ONLY_SENTINEL = new RegExp(
  `^\\s*(?:${MEDIA_ONLY_SENTINEL_PARTS.join("|")})*$`, "i"
);
```

**設計判断:**
- 各パターンが1行に1つ並び、コメント付き。将来のパターン追加は1行足すだけ
- `ATTACHMENT_BOILERPLATE` と `MEDIA_ONLY_SENTINEL_PARTS` の対応関係が一目で分かる
- `new RegExp()` はリテラル正規表現より初期化コストがかかるが、モジュールロード時の1回のみ。実行時のホットパスではないので問題なし

## Phase 3: `isAttachmentDominant()` のハードコード排除

**変更ファイル:** `src/retriever.ts` (L80-119)

`isAttachmentDominant()` (deprecated) の L92-97 が `ATTACHMENT_BOILERPLATE` を使わず `.replace()` をハードコードしている。この関数が `export` されている以上、外部コードから呼ばれる可能性があり、ハードコードが残ると同じバグが再発する。

**現在のコード (L80-98):**

```typescript
export function isAttachmentDominant(text: string): boolean {
  // Check for media-only sentinel pattern
  if (MEDIA_ONLY_SENTINEL.test(text.trim())) return true;

  // Count attachment markers vs actual text
  let markerCount = 0;
  for (const pattern of ATTACHMENT_BOILERPLATE) {
    const matches = text.match(pattern);
    if (matches) markerCount += matches.length;
  }

  // If attachment markers dominate the text, it's attachment-dominant
  const nonMarkerText = text.replace(/\[media attached(?:\s+\d+\/\d+)?:[^\]]*\]/gi, "")
    .replace(/<media:(image|document|audio|video)>(\s*\([^)]*\))?/gi, "")
    .replace(/\[User sent media without caption\]/gi, "")
    .replace(/To send an image back[^\n]*?(?:prefer|caption|Keep caption|URL)\.[ \t]*/gi, "")
    .replace(/media:\/\/inbound\/[^\s]+/g, "")
    .replace(/\s+/g, " ")
    .trim();
```

**変更後:**

```typescript
export function isAttachmentDominant(text: string): boolean {
  // Check for media-only sentinel pattern
  if (MEDIA_ONLY_SENTINEL.test(text.trim())) return true;

  // Count attachment markers vs actual text
  let markerCount = 0;
  for (const pattern of ATTACHMENT_BOILERPLATE) {
    const matches = text.match(pattern);
    if (matches) markerCount += matches.length;
  }

  // Delegate to stripAttachmentNoise() instead of hardcoded .replace() chain.
  // This ensures isAttachmentDominant() always stays in sync with ATTACHMENT_BOILERPLATE.
  const nonMarkerText = stripAttachmentNoise(text);
```

**変更点:**
- L92-97 の6行の `.replace()` チェーン → `stripAttachmentNoise(text)` の1行に置き換え
- 以降の dominance 判定ロジック (L100-118) は変更なし
- `stripAttachmentNoise()` は `ATTACHMENT_BOILERPLATE` をイテレートするため、Phase 1 の変更が自動反映される

**これにより3関数すべてが `ATTACHMENT_BOILERPLATE` を経由する:**

| 関数 | パターン参照 | Phase 1 変更の反映 |
|---|---|---|
| `classifyAndStripAttachment()` | `ATTACHMENT_BOILERPLATE` をイテレート | ✅ 自動 |
| `stripAttachmentNoise()` | `ATTACHMENT_BOILERPLATE` をイテレート | ✅ 自動 |
| `isAttachmentDominant()` (deprecated) | **変更前**: ハードコード → **変更後**: `stripAttachmentNoise()` 経由 | ✅ 自動 |

## 注意事項

1. **`ATTACHMENT_BOILERPLATE` のパターンを追加・変更したら、必ず `MEDIA_ONLY_SENTINEL_PARTS` も同期すること。** Phase 2 の配列方式なら1行足すだけだが、忘れると sentinel が古いままになり `isDominant` 判定が正しく動かない。
2. **`isAttachmentDominant()` 内に `.replace()` のハードコードを戻さないこと。** Phase 3 の委譲により、今後は `stripAttachmentNoise()` 経由で `ATTACHMENT_BOILERPLATE` を参照する。
3. **`test_phase4_5.ts` のスタンドアロンコピーは本番コードをコピーしたもの。** 本番を変更したら必ずこちらも同期すること。同期を忘れると `npm test` が fail する。
4. **`test_phase4_5.ts` L2207-2219 の同期チェックパターンも更新すること。** このチェックは本番ソースに旧パターン文字列が含まれているかを検証する。パターンを変更したら検証パターンも新しいものに更新しないと assert fail する。
5. **`test_classify_attachment.ts` は新規ファイル。** `package.json` の `test` スクリプトにも追加すること。

## 品質改善（プラン外の追加修正）

classifyAndStripAttachment() L189-193 で ATTACHMENT_INDICATORS の適用対象を text (original) から cleaned (サニタイズ後) に変更している。これは正しい改善。元のパスや拡張子が original text に残っていると、画像+キャプション混在時に indicator count が不当に増えて isDominant: true と誤判定されるバグがあった。修正後はサニタイズ済みテキストに対してのみ indicator をカウントするため、キャプション付きメッセージが正しく isDominant: false になる。
