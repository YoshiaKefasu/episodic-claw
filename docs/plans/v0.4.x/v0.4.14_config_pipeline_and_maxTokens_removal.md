## 2. 修正B: `narrativeMaxTokens` 削除 — OpenRouterのデフォルトに任せる

### 2.1 問題定義

`narrativeMaxTokens` は3つの矛盾した状態にある:

| 場所 | 状態 |
|------|------|
| `openclaw.plugin.json` スキーマ | トップレベルに `"narrativeMaxTokens"` が存在（L412-416）。**設定可能** |
| `types.ts` L123 | `narrativeMaxTokens?: number`。Deprecatedコメント付きだが型定義あり |
| `config.ts` L100 | `narrativeMaxTokens: rawConfig?.openrouterConfig?.maxTokens ?? rawConfig?.narrativeMaxTokens`。**読み取っている** |
| `index.ts` L507 | `maxTokens: cfg.narrativeMaxTokens`。**OpenRouterClient に渡している** |
| `openrouter-client.ts` L89-91 | `if (this.config.maxTokens !== undefined) { body.max_tokens = this.config.maxTokens; }`。**undefined なら max_tokens を送らない** |

**矛盾**: スキーマが設定可能にしているのに、型定義は「Deprecated」、コードフローは `openrouterConfig.maxTokens` を第一優先としつつフラットも読む、という状態。

### 2.2 削除の理由

OpenRouter API で `max_tokens` を送らない（`undefined`）場合、**OpenRouterがモデルのデフォルト最大トークン数を使う**。これは推奨される挙動である:

- OpenRouter ドキュメント: "Omit to let OpenRouter/model decide (recommended)"
- `openclaw.plugin.json` の `openrouterConfig.maxTokens` 説明（L193-194）: "Omit to let OpenRouter/model decide (recommended)"
- `openrouter-client.ts` L9 のコメント: `// undefined = let OpenRouter/model decide (recommended)`

つまり、ユーザーが明示的に `maxTokens` を設定しない限り、**OpenRouterのデフォルトに任せるのが最適**。`narrativeMaxTokens` の Deprecated エイリアスを維持する理由はない。

### 2.3 影響分析

**`narrativeMaxTokens` のデータフロー**:

```
openclaw.json: narrativeMaxTokens: 2048
  → OpenClaw スキーマ検証 → rawConfig.narrativeMaxTokens = 2048  ✅ (スキーマに存在)
    → loadConfig() L100: rawConfig?.openrouterConfig?.maxTokens ?? rawConfig?.narrativeMaxTokens
    → ネストパス優先: openrouterConfig.maxTokens がなければ narrativeMaxTokens を使う
      → cfg.narrativeMaxTokens = 2048
        → index.ts L507: maxTokens: cfg.narrativeMaxTokens  (= 2048)
          → OpenRouterClient.config.maxTokens = 2048
            → openrouter-client.ts L90: body.max_tokens = 2048
```

**`openrouterConfig.maxTokens` のデータフロー**:

```
openclaw.json: openrouterConfig: { maxTokens: 2048 }
  → OpenClaw スキーマ検証 → rawConfig.openrouterConfig.maxTokens = 2048  ✅
    → loadConfig() L100: rawConfig?.openrouterConfig?.maxTokens (= 2048) ?? rawConfig?.narrativeMaxTokens (undefined)
    → ネストパスが第一優先
      → cfg.narrativeMaxTokens = 2048
        → 後は同じ
```

両方とも最終的に `body.max_tokens = 2048` に到達する。違いは **設定方法** だけ:
- フラット（`narrativeMaxTokens`）: Deprecated エイリアス
- ネスト（`openrouterConfig.maxTokens`）: 推奨パス

## 3. `openrouterModel` と `narrativeTemperature` の扱い

### 3.1 `openrouterModel` — 修正不要（有効なレガシー互換）

**データフロー**:
```
openclaw.json: openrouterConfig: { model: "qwen/qwen3-235b-a22b:free" }
  → rawConfig.openrouterConfig.model = "qwen/qwen3-235b-a22b:free"
    → loadConfig() L99: rawConfig?.openrouterConfig?.model ?? rawConfig?.openrouterModel ?? "openrouter/free"
    → cfg.openrouterModel = "qwen/qwen3-235b-a22b:free"
      → index.ts L506: model: cfg.openrouterModel
        → OpenRouterClient.config.model
```

`openrouterModel` フラットフィールドはスキーマにないため `rawConfig?.openrouterModel` は常に `undefined`。しかし、**ネストパス `openrouterConfig.model` が第一優先で機能している**。`loadConfig()` はこれを `cfg.openrouterModel` に平坦化して渡し、`index.ts` がそれを `OpenRouterClient` に渡す。フローは正しい。

`openrouterModel` を型や `loadConfig()` から削除すると、`cfg.openrouterModel` を参照している `narrative-worker.ts:302` と `index.ts:506` を両方修正する必要がある。変更箇所が増える割に利点は薄い。**v0.4.14では修正不要。**

### 3.2 `narrativeTemperature` — 修正不要（有効なレガシー互換）

`narrativeMaxTokens` と同じ構造だが、**ユーザーが温度設定を変更できる正当なユースケースがある**。`narrativeTemperature` は `loadConfig()` でクランプ処理（0-1範囲）を行っており、削除するとクランプ保護が失われる。**修正不要。**

## 6. リスク評価

- **リスク**: 低
  - `narrativeMaxTokens` 削除: ユーザーが `narrativeMaxTokens: 2048` を設定している場合、**その設定が無視される**ようになる。しかしスキーマ削除と同時に行うため、OpenClaw がバリデーションエラーで弾くため実害なし
  - `recallQueryRecentMessageCount` 追加: デフォルト値 `4` は既存動作と同一
  - `queryExcludedKeywords` 追加: デフォルト値 `[]` は既存動作と同一
  - `toolFirstRecall` / `runtimeBridgeMode` スキーマ削除: v0.4.13 でコード削除済み。ユーザーの古い設定ファイルにこれらがあっても、`additionalProperties: false` により OpenClaw が既に弾いているはず
- **破壊的変更**: `narrativeMaxTokens` のTypeBoxスキーマ削除（`index.ts`）により、このフィールドを使っていたユーザーの `openclaw.json` で OpenClaw のランタイムバリデーションエラーが発生する可能性がある。ただし `openclaw.plugin.json` には元から `narrativeMaxTokens` プロパティが存在しないため（`additionalProperties: false` により既に弾かれている）、実害なし
- **ロールバック**: 約11箇所の差分で復元可能
