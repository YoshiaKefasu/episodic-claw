## 0-1. v0.4.11で見つかった設計欠陥

### Fallback要約の3つの汚染問題

v0.4.11の`buildDeterministicFallback()`は、全Retry失敗時にrawTextの最初20行をそのままMDファイルとして保存する。これは3つの汚染を引き起こす：

**汚染1: コンテキスト汚染（最も深刻）**

`saveNarrative()` L326:
```typescript
this.lastNarrativeByAgent.set(item.agentId, { episodeId: `narrative-${Date.now()}`, body: result.text });
```

Fallback要約の生テキストが次の物語化の「Previous episode」としてLLMに渡される。LLMに「前回の物語はこれです」と**生の会話ログを食わせる**。オウム返しの原因が次のコンテキストにも伝播する。

**汚染2: ベクトルストア汚染**

Fallback要約は`batchIngest`→Go sidecar→ベクトル埋め込みの通常パスを通る。`fallback-summary`タグは付くが、リコールパイプラインにこのタグでフィルタリングする仕組みがない。結果: 生テキストの埋め込みがベクトルストアに混入し、ユーザーのクエリに対してオウム返しの埋め込みがマッチする。

**汚染3: UX汚染**

`Y:\kasou_yoshia\.openclaw\workspace\episodes\`に生テキストのMDが保存される。タグは`fallback-summary`だが、ファイル名も内容も正常な物語と区別がつかない。HealingWorkerがリネームしたらさらに区別不可能。

### 結論

> **「失敗時は何も保存しない」方が「生テキストを保存する」よりマシ。**
> rawTextはCacheDBに残っている（`Ack(deleteAfter=false)`）。再処理可能。

## 1. 問題定義

### 1.1 v0.4.11のFallback要約が引き起こす3つの汚染

前述の通り。Fallback要約をそのまま保存するとコンテキスト・ベクトル・UXの3方向に汚染が拡散する。

### 1.2 低遅延・効率化の課題

現状のアーキテクチャには以下の遅延・効率課題がある：

| 課題 | 現状 | 影響 |
|---|---|---|
| **ポーリング間隔** | 1秒（アイドル時は最大15秒） | アイドル時の無駄なRPC呼び出し |
| **チャンク分割** | 行単位の分割。会話の文脈境界を無視 | 1つの会話が2チャンクに分割され、物語化品質が低下 |
| **連続性参照** | `lastNarrativeByAgent`のメモリキャッシュのみ | プロセス再起動でリセット。冷スタート後の最初の物語化は「前回の物語」を参照できない |
| **再処理手段** | Retry（20回上限）とRequeue（手動RPC）のみ | dead-letterになったチャンクの手動再処理方法が不明 |
| **Fallback保存** | rawTextの最初20行をそのままMD保存 | 3方向汚染 |

---

## 2. 解決策: 6層パイプライン再設計

### Layer 0: Fallback要約の撤廃 → Dead-Letter + アラート

**最優先。v0.4.11のFallback要約を撤廃し、代わりにcacheRetry + アラートログに変更する。**

```
現状 (v0.4.11):
  5リトライ失敗 → buildDeterministicFallback() → saveNarrative(fallback) → Ack
  ❌ コンテキスト汚染
  ❌ ベクトル汚染
  ❌ UX汚染

修正後:
  5リトライ失敗 → cacheRetry("Quality gate exhausted", MAX_CACHE_ATTEMPTS=20)
  ✅ データはCacheDBに残る（再処理可能）
  ✅ コンテキスト汚染なし（lastNarrativeByAgentに生テキストが入らない）
  ✅ ベクトル汚染なし（MDファイルが作られない）
  ✅ UX汚染なし（ゴミMDが保存されない）
```

`cacheRetry`は指数バックオフで再キューする。`MAX_CACHE_ATTEMPTS=20`に達するとdead-letterになる。dead-letterになったチャンクのrawTextはPebbleDBに残っており、手動でRequeue RPCを呼ぶことで再処理可能。

### Layer 1: Dead-Letter監視ログ

品質ゲートで弾かれた場合のログを充実する：

```typescript
console.warn(
  `[NarrativeWorker] Quality gate exhausted for [${item.id}] after ${MAX_RETRIES} attempts. ` +
  `Tokens: ${item.estimatedTokens}→attempted, Compression ratios: [logged per attempt]. ` +
  `Item re-queued with backoff. Attempts remaining: ${MAX_CACHE_ATTEMPTS - item.attempts}/${MAX_CACHE_ATTEMPTS}.`
);
```

Dead-letterに到達した場合：

```typescript
console.error(
  `[NarrativeWorker] DEAD LETTER: [${item.id}] exhausted all ${MAX_CACHE_ATTEMPTS} cache attempts. ` +
  `rawText preserved in PebbleDB. Use 'cache.requeue' RPC to retry manually.`
);
```

### Layer 2: 連続性参照の永続化

`lastNarrativeByAgent`のメモリキャッシュはプロセス再起動でリセットされる。`initContinuity()`でGo sidecarから最新の物語MDを読み込むが、これはファイルスキャンなので遅い。

修正: `cacheGetLatestNarrative()`の結果をキャッシュするだけでなく、新しく作成された物語もキャッシュに反映するようになっている（L326）。問題は**Fallback要約を保存した場合にキャッシュが汚染される**こと。Layer 0でFallbackを撤廃すればこの問題も解決する。

### Layer 3: チャンク分割の文脈境界改善

現状の`splitIntoChunks()`は行単位の分割のみ。会話の文脈境界（user→assistantのペアなど）を無視する。

将来的な改善（v0.4.13以降）:
- `role: "user"` から `role: "assistant"` の応答までを1つのチャンクにまとめる
- チャンクの先頭に直前の数行をコンテキストとして含める

**当面は変更しない。** 現状の分割でも品質ゲートが弾けば再キューされるので、致命的ではない。

### Layer 4: ポーリング最適化

現状の1秒ポーリングはアイドル時に15秒までバックオフするが、最初のポーリングサイクルで複数エージェントを順次スキャンするため、1サイクルあたりのレイテンシが高い。

将来的な改善（v0.4.13以降）:
- カスタムRPC `cache.subscribe` でPush型に変更
- PebbleDBのウォッチャーでキュー追加イベントを通知

**当面は変更しない。** アイドルバックオフはv0.4.3で実装済み。

### Layer 5: 明示的なRequeue CLIコマンド

dead-letterになったチャンプを再処理するための手動RPCコマンドを追加する。Go sidecarに既に`Requeue()` RPCがある（queue.go L248-267）が、TS側から呼ぶ手段がない。

追加: `rpc-client.ts`に`cacheRequeue()`メソッドを追加:

```typescript
async cacheRequeue(id: string): Promise<boolean> {
  return this.request<boolean>("cache.requeue", { id });
}
```

## 4. Phase A: Fallback要約撤廃

### 4.1 変更対象

`src/narrative-worker.ts` — `processNextFromCache()` の null ハンドリング

### 4.2 現行（v0.4.11）

```typescript
} else {
  // [NEW v0.4.11] All retries exhausted → generate deterministic fallback summary
  const fallbackText = this.buildDeterministicFallback(item);
  const fallbackResult: NarrativeResult = {
    text: fallbackText,
    tokens: estimateTokens(fallbackText),
    model: "fallback-summary",
  };
  // Pass isFallback=true to add specific tag
  await this.saveNarrative(fallbackResult, item, true);
  await this.rpcClient.cacheAck(item.id, "narrative-worker");
  console.warn(
    `[NarrativeWorker] LLM narrativization failed for [${item.id}] after ${MAX_RETRIES} attempts. Saved deterministic fallback summary (${fallbackResult.tokens} tokens).`
  );
}
```

### 4.3 変更後

```typescript
} else {
  // [v0.4.12] Quality gate exhausted → re-queue with backoff instead of saving fallback
  // Fallback summary would pollute: context (lastNarrativeByAgent), vector store, and UX
  // rawText is preserved in PebbleDB (Ack deleteAfter=false) for manual requeue later
  await this.rpcClient.cacheRetry(item.id, "narrative-worker", "Quality gate exhausted: all retries failed", MAX_CACHE_ATTEMPTS);
  console.warn(
    `[NarrativeWorker] Quality gate: all ${MAX_RETRIES} LLM attempts exhausted for [${item.id}]. Re-queued for later retry (${item.attempts + 1}/${MAX_CACHE_ATTEMPTS}).`
  );
}
```

### 4.4 削除対象

1. `buildDeterministicFallback()` メソッド（L336-347）
2. `saveNarrative()` の `isFallback` パラメータ（L304, L307-309）
3. `processNextFromCache()` 内の Fallback分岐（L226-238）

### 4.5 設計判断

- **cacheRetryを選ぶ理由**: rawTextはPebbleDBに残っている。`cacheRetry`は`backoffUntil`を設定して遅延再キューする。LLMモデルが一時的に品質ゲートを通らない場合でも、時間が経てば通過する可能性がある（free モデルはモデルが変わる）
- **Ackしない理由**: AckするとrawTextのステータスが`done`になり、再処理にRequeue RPCが必要になる。cacheRetryなら自動的に再キューされる
- **MAX_CACHE_ATTEMPTS=20**: 品質ゲートは5回のLLMリトライ×4回のキャッシュリトライ=最大20回のLLM呼び出し。それでも通らない場合はdead-letterになる

## 7.5 Phase G: 品質ゲート失敗時にsleep追加

### 7.5.1 問題

現行の `narrativizeWithRetry()` で品質ゲート失敗時は `continue` で即リトライする。APIエラー時は指数バックオフ（1s→2s→4s→8s→16s）があるが、品質ゲート失敗時には**待機なし**で5回連続でOpenRouterを叩く。

free モデルは同じ入力に対して同じような出力を返す傾向がある。sleepなしで5回叩いても同じ結果になる可能性が高い。

### 7.5.2 変更対象

`src/narrative-worker.ts` — `narrativizeWithRetry()` 内の品質ゲート分岐

### 7.5.3 変更内容

品質ゲート失敗時（圧縮比不足・オウム返し検出）に500msのsleepを追加する：

```typescript
// [v0.4.11] Quality gate 1: Token count & Compression ratio
if (!checkCompressionRatio(tokens, item.estimatedTokens)) {
  console.warn(
    `[NarrativeWorker] Attempt ${attempt + 1}/${MAX_RETRIES}: compression ratio too low (${tokens}/${item.estimatedTokens} = ${(tokens / item.estimatedTokens * 100).toFixed(2)}% < ${MIN_COMPRESSION_RATIO * 100}%). Retrying...`
  );
  await this.sleep(500); // [NEW v0.4.12] Brief pause before retry — free models tend to produce similar output without pause
  continue;
}

// [v0.4.11] Quality gate 2: Verbatim echo detection
if (!checkEchoDetection(text, conversationText)) {
  console.warn(
    `[NarrativeWorker] Attempt ${attempt + 1}/${MAX_RETRIES}: output appears to be verbatim echo of input. Retrying...`
  );
  await this.sleep(500); // [NEW v0.4.12] Brief pause before retry
  continue;
}
```

### 7.5.4 設計判断

- **500ms**: OpenRouter APIの利用制限を回避しつつ、モデルの出力に変化をもたらす最低限の待機時間。1秒未満ならユーザー体験に影響なし
- **指数バックオフではない理由**: 品質ゲート失敗は「APIは成功しているが出力が不良」なので、指数バックオフは過剰。500msの固定遅延で十分
- **APIエラー時**: 既存の指数バックオフ（1s→2s→4s→8s→16s）は変更なし。品質ゲート失敗とAPIエラーは別のルート

## 13. 評価メモ（実装には不要・背景理解用）

### なぜ「何も保存しない」が最善なのか

Fallback要約のコンテキスト汚染を図示:

```
チャンク#1 (物語化失敗)
  ↓ Fallback要約を保存
  ↓ lastNarrativeByAgent["main"] = "おう、バッチリ聞こえてるぜ！..."
  ↓
チャンク#2 (次のチャンク)
  ↓ Previous episode: "おう、バッチリ聞こえてるぜ！..."
  ↓ LLM: 「前回の物語がオウム返しなら、今回もオウム返しでいいんだな」
  ↓ 物語化品質が連鎖的に劣化
```

cacheRetryなら:

```
チャンク#1 (物語化失敗)
  ↓ cacheRetry → キューに戻す
  ↓ lastNarrativeByAgent["main"] = （更新なし、前回の正常な物語のまま）
  ↓
チャンク#2 (次のチャンク)
  ↓ Previous episode: （前回の正常な物語）
  ↓ LLM: 正常なコンテキストで物語化
  ↓ 品質劣化なし
```

### なぜRequeue RPCが必要なのか

`Ack(deleteAfter=false)` でrawTextはPebbleDBに残るが、ステータスが`dead-letter`になるとNarrativeWorkerのポーリングで取得されなくなる。手動で再処理するには:

1. Go sidecarの`Requeue()` RPCを呼ぶ
2. アイテムのステータスが`done` → `queued`に戻る
3. NarrativeWorkerが次のポーリングサイクルで取得し、再物語化を試みる

現状はこのRPCのTS側ラッパーがない。Phase Cで追加する。

### 将来的な改善（v0.4.13以降）

| 改善 | 内容 | 優先度 |
|---|---|---|
| チャンク分割の文脈境界 | role境界で分割、コンテキストオーバーラップ付与 | 中 |
| Push型ポーリング | Go側のウォッチャーでキュー追加時にTS側に通知 | 低 |
| dead-letterダッシュボード | CLIでdead-letter一覧取得・再処理コマンド | 中 |
| モデルフォールバック | freeモデル失敗時に有料モデルにフォールバック | 低 |

### 14.3 Phase E: checkEchoDetection のメモリ最適化

#### 現状の問題

```typescript
// 現状 (v0.4.11)
export function checkEchoDetection(output: string, input: string): boolean {
  const collapsedOutput = output.replace(/\s+/g, "").trim();  // ← 192KB のフルコピー
  if (collapsedOutput.length < MIN_ECHO_LENGTH) return true;
  const echoSample = collapsedOutput.substring(0, ECHO_SAMPLE_LENGTH);
  const collapsedInput = input.replace(/\s+/g, "");           // ← さらに大きなコピー
  return !collapsedInput.includes(echoSample);
}
```

48KトークンのrawText（約192KB）に対して`replace(/\s+/g, "")`を呼ぶと:
1. **192KBの文字列コピーを2回生成**（output + input）
2. V8のGCが若い世代ヒープを圧迫する
3. 2010年CPUではこのコピーだけで0.5〜2msかかる

#### 修正: input先頭5K文字のみスキャン

エージェントの生返答は入力の先頭付近にある（47Kトークン中の最初の200トークン程度）。input全体（約192KB）の空白削除を避け、先頭5K文字だけをスキャンする：

```typescript
const MAX_ECHO_SCAN_CHARS = 5000; // Only scan first 5000 chars of input

export function checkEchoDetection(output: string, input: string): boolean {
  // Collapse whitespaces in output (small, typically <500 chars)
  const collapsedOutput = output.replace(/\s+/g, "").trim();
  if (collapsedOutput.length < MIN_ECHO_LENGTH) return true;

  const echoSample = collapsedOutput.substring(0, ECHO_SAMPLE_LENGTH);

  // Only collapse the first MAX_ECHO_SCAN_CHARS of input (avoid 192KB full copy)
  // Echoes are always near the beginning of the input
  const inputPrefix = input.substring(0, MAX_ECHO_SCAN_CHARS).replace(/\s+/g, "");
  return !inputPrefix.includes(echoSample);
}
```

**効果:**
- input（192KB）のフルコピー生成を廃止 → 5K文字（約20KB）のみ空白削除
- 2010年CPUで2ms→0.3ms（**85%削減**）
- 実装が3行で済む（KISS原則に合致）
- スライディングウィンドウは複雑すぎるため採用しない（Pro Engineer Reviewの指摘）

### 14.4 Phase F: estimateTokens の for...of → for...let i 最適化

#### 現状

```typescript
export function estimateTokens(text: string): number {
    let count = 0;
    for (const char of text) {
        count += char.charCodeAt(0) > 0x2E80 ? 1.5 : 0.25;
    }
    return Math.ceil(count);
}
```

`for...of` は文字列のイテレータを使う。2010年CPUでは文字列イテレータのオーバーヘッドが無視できない。

#### 修正: for...let i に変更（振る舞い不変）

```typescript
export function estimateTokens(text: string): number {
    let count = 0;
    for (let i = 0; i < text.length; i++) {
        count += text.charCodeAt(i) > 0x2E80 ? 1.5 : 0.25;
    }
    return Math.ceil(count);
}
```

**設計判断（Pro Engineer Review）:**
- **長さベース近似（`text.length * 0.5`）はYAGNIリスクあり**: `splitIntoChunks()` のチャンク分割閾値判定に±20%の誤差を入れると、会話が分断されるリスクがある
- **`for...let i` は振る舞い不変**: イテレータオーバーヘッドの排除のみで、推定精度は変わらない
- **KISS原則に合致**: 最小限の変更で最大限の効果

3. OpenRouter freeモデルはバックエンドがラウンドロビンで複数モデルに分散する場合がある。少し間を置くことで別モデルに割り当てられ、異なる出力が得られる可能性がある

**注意点:**
- 同じ入力→同じ出力の問題は500ms待っても根本解決しない。確率は上がるが、5回連続同じモデルにルーティングされれば同じ結果が返る
- より確実な対策はv0.4.13の「モデルフォールバック」（free → 有料モデル切り替え）
- しかし**害はない**。最悪でも2.5秒の遅延が増えるだけ。現在の「即リトライで同じ結果5回」より明確にマシ
