## 1. 問題定義

### データフロー

```
openclaw.json: "recallQueryRecentMessageCount": 2
  → OpenClaw が rawConfig に渡す ✅
    → loadConfig() が rawConfig.recallQueryRecentMessageCount を読み取らない ❌
      → this.config.recallQueryRecentMessageCount = undefined
        → ?? 4 でデフォルト値にフォールバック ❌
```

`loadConfig()`（`config.ts:67-110`）は約30個のフィールドを読み取って `EpisodicPluginConfig` オブジェクトを構築しているが、**スキーマに定義された全38フィールドのうち8個が欠落している**。

これは v0.4.13 で修正した `narrativePreviousEpisodeRef` と**全く同じパターンのバグ**（スキーマ定義済・型定義済・`loadConfig()` で読み取り漏れ）。

## 2. 修正内容

### 2.1 `loadConfig()` に欠落フィールドを追加

**対象ファイル**: `src/config.ts`

```typescript
// 追加する2行（recallQueryRecentMessageCount と queryExcludedKeywords）

// Recall query config
recallQueryRecentMessageCount: Math.max(1, Math.min(12, rawConfig?.recallQueryRecentMessageCount ?? 4)),
queryExcludedKeywords: rawConfig?.queryExcludedKeywords ?? [],
```

#### 値の設計判断

- **`recallQueryRecentMessageCount`**: `Math.max(1, Math.min(12, ...))` でクランプ。0はクエリ生成不能（empty query）、13以上はToken浪費（ユーザー設定の意味がない）。`openclaw.plugin.json` のスキーマ定義 `minimum: 1, maximum: 12` と一致させる。
- **`queryExcludedKeywords`**: デフォルト `[]`。`rawConfig?.queryExcludedKeywords` が `undefined` の場合は空配列。TypeScript の `string[]` 型に合致。

### 2.2 `openrouterConfig` の型定義調整

`openrouterConfig` 自体は `loadConfig()` で分解して読み取っているため、オブジェクトとしては `EpisodicPluginConfig` に存在しない。しかし、`openclaw.plugin.json` のスキーマには `"openrouterConfig"` がネストしたオブジェクトとして定義されており、`rawConfig?.openrouterConfig?.model` のように部分的にアクセスされている。

これは現在の実装で動作しているため**変更不要**。型定義の `openrouterConfig` インターフェースも L127-132 に既に存在する。

### 2.3 変更しないもの

- **`enabled`**: OpenClaw 本体が制御する。`loadConfig()` で読み取る必要なし。
- **`runtimeBridgeMode`**: v0.4.7 deprecated。v0.4.13 で `resolveRuntimeBridgeMode()` を削除済み。
- **`toolFirstRecall`**: v0.4.6 deprecated。v0.4.13 で `tool-first-gate.ts` を削除済み。
- **`allowCrossAgentRecall`**: スキーマに「Disabled and ignored」と明記済み。
- **`sharedEpisodesDir`**: スキーマに「Disabled and ignored」と明記済み。

## 3. `retriever.ts` の参照方法

現在の `retriever.ts` は `this.config?.recallQueryRecentMessageCount ?? 4` と `this.config?.queryExcludedKeywords` でアクセスしている。`loadConfig()` にフィールドを追加すれば、`this.config` に正しい値が渡るため、`retriever.ts` 側の変更は**不要**。

ただし、デフォルト値の二重定義を防ぐため、`retriever.ts` の `?? 4` を削除して `loadConfig()` 側に一元化する。

**変更箇所**: `src/retriever.ts:519`

```typescript
// Before:
const recentMessageCount = this.config?.recallQueryRecentMessageCount ?? 4;

// After:
const recentMessageCount = this.config?.recallQueryRecentMessageCount ?? 4;
// NOTE: Default is now defined in loadConfig() (Math.max(1, Math.min(12, rawConfig?.recallQueryRecentMessageCount ?? 4)))
// The ?? 4 here is a defensive fallback for when config is undefined (e.g., unit tests)
```

実質的に変更不要。`this.config` が `loadConfig()` の戻り値であれば、`?? 4` は到達しない安全ネットとして機能する。ユニットテスト等で `config` が `undefined` の場合のフォールバックとして残す。

## 5. 根本原因分析と再発防止

### なぜこのバグが繰り返し発生したか

v0.4.13 で `narrativePreviousEpisodeRef` が同じパターンで無視されていた。根本原因は**スキーマ定義・型定義・`loadConfig()` パースの3層が手動で同期されていない**こと。

### 再発防止策

`loadConfig()` の戻り値を `EpisodicPluginConfig` 型にキャストするのではなく、TypeScript の `satisfies` 演算子を使って型の完全性をコンパイル時に検証する。

```typescript
// config.ts の loadConfig() の最後で:
return {
  tombstoneRetentionDays: rawConfig?.tombstoneRetentionDays ?? 14,
  // ... 全フィールド ...
} satisfies EpisodicPluginConfig;
```

`satisfies` を使うと、`EpisodicPluginConfig` に定義されているフィールドが1つでも欠けると**コンパイルエラー**になる。これにより、スキーマにフィールドを追加しても `loadConfig()` の読み取りを忘れるとビルドが壊れるため、検出が確実になる。

**注意**: `satisfies` の導入は型安全性の改善だが、`EpisodicPluginConfig` の全フィールドがオプショナル（`?:`）であるため、単に `satisfies EpisodicPluginConfig` を付けるだけでは「空オブジェクト `{}` も型を満たす」問題がある。これを防ぐには、`Required<EpisodicPluginConfig>` に対して `satisfies` を使うか、コンパイル時テストを追加する必要がある。

**本次の対象外とする**: `satisfies` の導入は別コミット（v0.4.15 以降）で実施する。v0.4.14 では欠落フィールドの追加のみを行う。
