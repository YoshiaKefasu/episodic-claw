## 2. 修正 D1: `handleBatchIngest` の getStore 失敗時 best-effort 化

### 2.1 問題の詳細

v0.4.21c で `handleIngest` の getStore 失敗時 early return を best-effort に修正したが、
**`handleBatchIngest` が同一の回帰を抱えたまま** だった。

現在のコード（L1801付近）:
```go
vstore, err := getStore(params.AgentWs)
if err != nil {
    sendResponse(conn, RPCResponse{JSONRPC: "2.0", Error: &RPCError{Code: -32000, Message: fmt.Sprintf("failed to get store: %v", err)}, ID: req.ID})
    return  // ← バッチ全件が失敗！
}
```

一時的な Pebble ロック・I/O 問題で、本来 markdown として残せたバッチ全件が落とされる。
handleIngest と同じ回帰パターン。

### 2.2 修正仕様

handleIngest と同じ best-effort パターンを適用する。

```go
vstore, storeErr := getStore(params.AgentWs)
if storeErr != nil {
    // [v0.4.21d] Best-effort: dedup guard skipped, file saves continue
    EmitLog("ai.batchIngest: Store init failed (best-effort mode, dedup skipped): %v", storeErr)
    // vstore is nil — duplicate guard and vector store ops will be skipped below
}

// ... item processing loop ...

// Duplicate guard 2 (DB-level): only when vstore is available
if vstore != nil {
    existingRec, getErr := vstore.Get(slug)
    if getErr == nil {
        // duplicate — skip this item but continue processing other items in batch
        // [v0.4.21d] Return existing record's path for consistency with handleIngest
        existingPath := ""
        if existingRec != nil {
            existingPath = existingRec.SourcePath
        }
        mu.Lock()
        slugs = append(slugs, slug)  // still append slug so caller doesn't retry
        mu.Unlock()
        continue  // skip this item, process others
    }
}

// ... serialize (file save) — always proceeds per item ...

// Vector store add: only when vstore is available
if vstore != nil && embedErr == nil && emb != nil {
    vstore.Add(ctx, vector.EpisodeRecord{...})
}
```

### 2.3 handleBatchIngest 固有の考慮

- バッチ処理では個別 item の失敗が他 item に波及しない構造（goroutine per item + WaitGroup）
- getStore は バッチ全体で1回だけ呼ばれる → 失敗時は全 item が best-effort モードになる
- dedup guard が無効になっても、file save が継続するため実害は小さい
- 重複 insert は slug の一意性で事後に healing で修復可能

### 2.4 他の getStore 呼び出しの分類

main.go 内の全 getStore 呼び出しを分析した結果：

| Line | Handler | getStore失敗時 | Best-effort対象？ |
|------|---------|----------------|-------------------|
| 464 | (init path) | warn only | ✅ 既に best-effort |
| 602 | (watchdog) | `_, _ = getStore(ws)` | ✅ 既に無視 |
| 1016 | handleIndexerRebuild | early return | ❌ rebuild は store が必須（best-effort 非対象） |
| 1369 | handleSurprise / handleSegmentScore | early return | ❌ 評価系は store 必須 |
| 1575 | handleIngest | ✅ v0.4.21c で best-effort 化済み | — |
| 1801 | handleBatchIngest | **early return** | ✅ **D1 対象** |
| 2370 | handleRecall | early return | ⚠️ 検討（recall は store 必須だが、fallback 返答もあり） |
| 2481 | handleRecallFeedback | early return | ❌ feedback 記録に store 必須 |
| 2533 | handleGetWatermark | early return | ⚠️ 検討（watermark 読めない → 空返答で耐えるか？） |
| 2558 | handleSetWatermark | early return | ❌ watermark 設定に store 必須 |
| 2650 | (background indexing) | early return | ❌ indexing は store 必須 |
| 2681 | (background task) | early return | ❌ store 必須 |

**結論**: D1 は `handleBatchIngest` のみを対象とする。
他の handler は store が本質的に必須（recall / feedback / watermark / indexing）であり、
best-effort 化の恩恵が小さいか、意味がない。

## 3. 修正 D2: `saveHashPersistCounter` の per-agent 分離

### 3.1 問題の詳細

v0.4.21c で追加した debounce counter のうち、
**実害があるのは `NarrativeWorker` 側の `saveHashPersistCounter` のみ**。

`EventSegmenter` は agent ごとに個別インスタンスで生成されるため、
`cursorPersistCounter` はプロセス全体共有ではなく agent 単位で隔離される。

```typescript
// narrative-worker.ts
private saveHashPersistCounter = 0;  // ← 全 agent 共有
private readonly SAVE_HASH_PERSIST_INTERVAL = 5;
```

**実害**: 
- Agent A が高頻度（1分に5保存）→ Agent B が低頻度（10分に1保存）の場合
- A の保存で counter が進み、B の保存タイミングが本来より早く/遅く揺れる
- **persist タイミングが agent 間で不均一になり、save-hash 永続化の予測可能性が下がる**

### 3.2 修正仕様

`saveHashPersistCounter` を `Map<string, number>` に変更し、per-agent で追跡する。

```typescript
// narrative-worker.ts
private saveHashPersistCounters = new Map<string, number>();
private readonly SAVE_HASH_PERSIST_INTERVAL = 5;

private persistSaveHashes(agentWs: string, agentId: string): void {
  if (!agentWs || !agentId) return;
  const key = `${agentWs}:${agentId}`;
  const count = (this.saveHashPersistCounters.get(key) ?? 0) + 1;
  this.saveHashPersistCounters.set(key, count);
  if (count % this.SAVE_HASH_PERSIST_INTERVAL !== 0) return;
  // ... persist logic unchanged ...
}
```

### 3.3 メモリ影響

- Map のサイズは agent 数に比例（通常 1〜5 agent）
- 各 entry は string key + number value → 数バイト
- agent 終了時の明示的なクリーンアップは不要（TTL eviction なしでも実害なし）

### 3.4 スコープ明確化

- `segmenter.ts` の `cursorPersistCounter` は **今回の修正対象外**。
- 理由: `index.ts` で `EventSegmenter` が agent 単位で生成され、
  インスタンス境界で既に分離されているため（実害なし）。

### 3.5 テスト観点

- [ ] 高頻度 agent の save が低頻度 agent の save-hash persist タイミングに影響しないこと
- [ ] 各 agent が独立して SAVE_HASH_PERSIST_INTERVAL で persist すること
- [ ] 新規 agent の counter が 0 から開始すること

## 4. 修正 D3: state key の agentWs canonicalization

### 4.1 問題の詳細

`rpc-client.ts` で state DB の key を構築する際、raw `agentWs` path を直に入れている。

```typescript
// rpc-client.ts
const key = `agent:${agentId}:ws:${agentWs}:segmenter-cursor`;
const key = `agent:${agentId}:ws:${agentWs}:narrative-save-hashes`;
```

`agentWs` はファイルシステムパス（例: `C:\Users\foo\project`）であり：
- Windows と POSIX でスラッシュ方向が異なる → 同一 workspace が別 key になる
- 大文字小文字の違い（Windows は case-insensitive）→ 同一フォルダが別 key になる
- symlink / junction → 実体は同じだがパスが異なる → 別 key になる
- 末尾スラッシュの有無 → 別 key になる

**実害**: 同一 workspace を異なるパス表現で開いた場合、cursor / save-hash が別エントリに保存され、
再起動後の restore が期待値と異なる（v0.4.21b の persistence の信頼性が低下）。

### 4.2 修正仕様

`agentWs` を key に組み込む前に **hash 化** する。

```typescript
// rpc-client.ts — 新規 helper
private agentWsHash(agentWs: string): string {
  // Normalize path separators and case (Windows compat)
  // Then hash to produce a stable, collision-resistant key component.
  const normalized = agentWs.replace(/\\/g, '/').toLowerCase();
  // Use a fast, non-crypto hash (FNV-1a or simple DJB2)
  let hash = 5381;
  for (let i = 0; i < normalized.length; i++) {
    hash = ((hash << 5) + hash + normalized.charCodeAt(i)) & 0x7FFFFFFF;
  }
  return hash.toString(36);
}
```

key 構築の変更:
```typescript
// 旧:
const key = `agent:${agentId}:ws:${agentWs}:segmenter-cursor`;
// 新:
const key = `agent:${agentId}:ws:${this.agentWsHash(agentWs)}:segmenter-cursor`;
```

### 4.3 移行戦略

**既存データとの互換性**が最大の懸念。

**Option A — hash 化 + マイグレーション**

1. 新しい key 形式で読み書きする
2. 初回 read 時、新しい key が存在しなければ古い key (`raw agentWs`) で探す
3. 古い key にデータがあれば新しい key にコピーし、古い key は削除
4. マイグレーション完了後は新しい key のみ使用

```typescript
async getSegmenterCursor(agentWs: string, agentId: string): Promise<{ lastProcessedLength: number }> {
  const newKey = `agent:${agentId}:ws:${this.agentWsHash(agentWs)}:segmenter-cursor`;
  const raw = await this.stateGet(newKey, agentWs, agentId).catch(() => "");
  
  // [v0.4.21d] Migration: if new key is empty, try legacy key
  if (!raw) {
    const legacyKey = `agent:${agentId}:ws:${agentWs}:segmenter-cursor`;
    const legacyRaw = await this.stateGet(legacyKey, agentWs, agentId).catch(() => "");
    if (legacyRaw) {
      // Migrate: write to new key, delete old key
      await this.stateSet(newKey, legacyRaw, agentWs, agentId).catch(() => {});
      await this.stateSet(legacyKey, "", agentWs, agentId).catch(() => {});  // delete by empty
      return JSON.parse(legacyRaw);
    }
    return { lastProcessedLength: 0 };
  }
  return JSON.parse(raw);
}
```

**Option B — hash 化のみ（マイグレーションなし）** *(推奨)*

- 実装は最もシンプル
- 既存の cursor / save-hash データは読み出せなくなる → 再起動時に bootstrap からやり直し
- ユーザへの影響は「再起動後に少し重複ナラティブが出る可能性」程度

### 4.4 影響範囲

- `rpc-client.ts`: `getSegmenterCursor`, `setSegmenterCursor`, `getNarrativeSaveHashes`, `setNarrativeSaveHashes` の4メソッド
- `main.go`: 変更不要（Go 側は key 文字列を透過的に扱う）
- `narrative-worker.ts`: `agentWs` hash は `rpc-client.ts` 内部で処理されるので変更不要
- `segmenter.ts`: 同上

### 4.5 テスト観点

- [ ] Windows パス `C:\Users\foo\project` と `c:/users/foo/project` が同じ hash になること
- [ ] 末尾スラッシュあり・なしで同じ hash になること
- [ ] Windows パス canonicalization 後に key が安定すること（大文字小文字/区切り差分）
- [ ] 既存 state がない前提で cursor/save-hash が再生成されること（Option B）
- [ ] 既存 state が残っていてもクラッシュせず新 key で継続動作すること

## 5. 修正 D4: `restoreCursor` → `bootstrapCursor` 同値書き戻しの防止

### 5.1 問題の詳細

現在のカーソル復元フロー（実コード）:

```typescript
// index.ts (caller)
const savedCursor = await segmenter.restoreCursor(agentWs, agentId);
segmenter.bootstrapCursor(savedCursor, "manual", agentWs, agentId);
```

`restoreCursor` → DB から値を読む
`bootstrapCursor` → `this.lastProcessedLength = currentMessagesLength` + `this.persistCursor()` を**無条件**に呼ぶ

問題: `restoreCursor` が返した値と `lastProcessedLength` の現在地が同じ場合、
**DB に同じ値を書き戻す**無駄な write + Sync が発生する。

v0.4.21c で cursor debounce を導入した後、通常ターンでは3回に1回しか persist しないのに、
restart 直後の bootstrap では**毎回** 同値書き戻しが発生するのは非効率。

### 5.2 修正仕様

`bootstrapCursor` に「値が変わった時だけ persist」の条件を追加し、
restore 呼び出し側の reason も合わせて整理する。

```typescript
bootstrapCursor(currentMessagesLength: number, reason: "warm-start" | "manual", agentWs: string = "", agentId: string = ""): void {
  const next = Math.max(0, currentMessagesLength);
  const changed = this.lastProcessedLength !== next;
  this.lastProcessedLength = next;
  this.buffer = [];
  this.clearIdleFlushTimer();
  // [v0.4.21d] Only persist when value changed or user-explicit manual operation
  if (changed || reason === "manual") {
    this.persistCursor(agentWs, agentId);
  }
}
```

合わせて `index.ts` の restore 直後呼び出しは以下のどちらかに統一する。

- Option A（推奨）: restore 経路を `"warm-start"` に変更し、同値書き戻しを確実に抑止
- Option B: restore 経路を `"manual"` のまま維持し、`changed` 判定のみで write 抑制（挙動が曖昧になりやすい）

**推奨は Option A**。`manual` はユーザ明示操作だけに意味を限定し、reason の責務を明確にする。

### 5.3 エッジケース

- `lastProcessedLength` が 0（初期状態）→ restore も 0 → 書き戻しスキップ ✅
- `lastProcessedLength` が 0 → restore が 15 → persist 実行 ✅
- `lastProcessedLength` が 15 → manual で 15 → persist 実行（manual は常に persist） ✅
- compaction 後 → `pendingCompactionSkip` で進んだ値 → restore より大きい → persist 実行 ✅

### 5.4 テスト観点

- [ ] `bootstrapCursor(15, "warm-start")` で `lastProcessedLength === 15` の時に persistCursor が呼ばれないこと
- [ ] `bootstrapCursor(15, "warm-start")` で `lastProcessedLength === 10` の時に persistCursor が呼ばれること
- [ ] `bootstrapCursor(15, "manual")` で `lastProcessedLength === 15` の時に persistCursor が呼ばれること（ユーザ明示操作）
- [ ] `index.ts` restore 経路が `"warm-start"` に統一され、同値時に write-back しないこと
