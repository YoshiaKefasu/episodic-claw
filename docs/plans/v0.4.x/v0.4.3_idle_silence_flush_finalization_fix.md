> 本プランは、「15分以上放置したのに新しい episode が作られない」症状を対象にした修正計画です。  
> 結論から言うと、現状の実装は idle 中に自動 finalize しません。`segmentationTimeGapMinutes` は「次の新規 user message が来た時だけ」効いており、静かに放置しただけでは flush されないのが現状です。

## 1. Incident Summary

今回の観測では、次のズレがありました。

- `X:\tmp\episodic-claw\2026-04-10.log` では `04:06:36` に `cache.enqueueBatch` が走り、`04:06:57` に episode file create が出ている
- その後 `04:11:27` に `ai.segmentScore` と `ai.recall` は走っている
- しかし `04:11:27` 以降は `cache.enqueueBatch` / `ai.batchIngest` / `Watcher event: CREATE ...` が続かない
- 一方で `Y:\kasou_yoshia\.openclaw\workspace\episodes\2026\04\10` には新規 narrative file が増えていない

つまり、**worker や queue が止まったのではなく、segmenter 側で「まだ finalize する条件に達していない」と判断されたまま buffer が開きっぱなし** になっています。

## 3. Core Problem

> **現行設計では「しばらく沈黙したのでこの buffer を締める」という idle-finalize 経路が存在せず、time-gap 設定も“次の user message 到着時の比較”にしか使われていない。**

## 5. Solution Options

### Option A — Segmenter-Level Idle Flush Timer *(推奨)*

**Approach**:  
`EventSegmenter` に「buffer 非空のまま一定時間 message が来なければ強制 flush」タイマーを追加する。  
新しい user-facing config は増やさず、まずは `segmentationTimeGapMinutes` を idle flush にも流用する。

**Implementation cost**: Medium  
**Risk**: Low  
**Why recommended**:  
一番小さい変更で、今のユーザー期待と実装を一致させられる。queue も narrative worker も触りすぎずに済む。

**Concrete steps**:

1. `src/segmenter.ts` に idle flush timer state を追加する
   - `idleFlushTimer: NodeJS.Timeout | null`
   - `lastBufferActivityAt`
   - `clearIdleFlushTimer()`
   - `scheduleIdleFlush(agentWs, agentId)`

2. buffer が非空になった地点で timer を張る
   - first absorb (`src/segmenter.ts:203-207`)
   - non-boundary append (`src/segmenter.ts:290-293`)

3. 新規 message 到着時は timer を張り直す
   - 「最後の activity から N 分無入力」で発火するようにする

4. timer 発火時は `forceFlush()` 相当を呼ぶ
   - ただし reason を `idle-timeout` としてログに残す
   - narrative cache path では `splitIntoChunks(..., "force-flush", 0)` ではなく、`idle-timeout` を新設して区別した方が後追いしやすい

5. flush 後は segmenter buffer / timer / `lastProcessedLength` を整合的に reset する
   - 二重 flush 防止
   - context reset と race しないようにする

6. 最低限の guard を入れる
   - image-only / empty-only buffer では idle flush しない
   - text length が極端に小さい場合は flush せず継続でもよい
   - ただし v0.4.3 初手では「buffer に text が1件でもあれば flush」で十分

7. テストを追加する
   - fake timer で `segmentationTimeGapMinutes` 経過時に enqueue が起きる
   - 途中で新 message が来ると timer が延長される
   - forceFlush / context reset と二重実行しない

### Option B — Host / Session Hook で Idle Finalize

**Approach**:  
segmenter ではなく、gateway / session status 側から「しばらく会話が止まったので flush しろ」を送る。

**Implementation cost**: Medium  
**Risk**: Medium  
**When to choose this instead**:  
将来「chat idle」「gateway idle」「agent suspended」を区別して制御したい場合。

**Why not recommended now**:  
v0.4.3 では広げすぎ。責務が分散し、まず直したい不具合に対して大回り。

### Option C — 現状を仕様として明文化

**Approach**:  
「次の message が来るまで finalize されない」と仕様化する。

**Implementation cost**: Low  
**Risk**: High  
**Why not recommended**:  
ユーザー期待とズレたまま残る。今回の incident を“仕様です”で押し切るのは現場的に悪手。

---

## 6. Pro Recommendation

> **Choose Option A because** 今の設計の穴は segmenter の idle finalize 不在だからです。  
> queue や worker をいじる前に、buffer の閉じ方を正すのが筋です。

**Estimated implementation**: 0.5 - 1 day  
**Rollback plan**:

1. idle flush timer を無効化
2. `segmentationTimeGapMinutes` の idle 流用を外す
3. 既存の surprise / size / forceFlush のみへ戻す

---

## 7. Acceptance Criteria

1. 最後の textful user message から `segmentationTimeGapMinutes` 超の無入力で、新 episode が自動 finalize される
2. その flush は `cache.enqueueBatch` まで進み、worker が通常通り narrativize する
3. 二重 flush / 重複 episode が出ない
4. 既存の surprise boundary / size limit / forceFlush は壊れない
5. image-only ノイズで空 episode が量産されない
