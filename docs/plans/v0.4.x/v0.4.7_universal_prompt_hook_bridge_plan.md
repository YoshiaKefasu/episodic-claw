## 1. 非交渉条件（固定）

1. OpenClaw本体ソースは変更しない（参照のみ）
2. 変更は Episodic-Claw プラグインのみ
3. 目的は「CLI/Embeddedどちらでも、1ターン内で Recall と episodic ingest を壊さない」こと

## 2. 事実確認（原因と証拠）

## 2.1 根本原因

- OpenClaw は `isCliProvider(...)` のとき `runCliAgent(...)` に直分岐する。
- この経路は Embedded 側の `before_prompt_build` / context-engine `assemble|ingest` パイプラインを通らない。
- Episodic-Claw の live ingest と注入が `before_prompt_build` / `assemble` 依存のため、CLIターンで入口が消失する。

参照:
- `openclaw-source/src/agents/command/attempt-execution.ts`
- `openclaw-source/src/agents/pi-embedded-runner/run/attempt.ts`
- `episodic-claw/src/index.ts`

## 2.2 ログ一致（再現）

- OpenClaw 側は対象時刻に `agent/cli-backend cli exec` が連続。
- Episodic 側は `SleepTimer Idle xxh` だけ進み、`processTurn` / `Idle timeout ... Auto-finalizing buffer` が出ない。
- 結果として当日エピソードが増えず、`v0.4.3_idle_silence_flush_finalization_fix` の実効果が出ない。

参照:

- `X:\tmp\episodic-claw\2026-04-11.log`

## 3. CLI分岐で破綻する機能マップ（v0.4.7で回収対象）

1. 自動 ingest 入口  
- 破綻経路: `before_prompt_build` / `assemble` 不達  
- 影響: `segmenter.processTurn` 未実行、bufferが育たない

2. Idle自動確定（v0.4.3修正対象）  
- 破綻経路: `processTurn` が走らず idle timer 自体が再スケジュールされない  
- 影響: `Auto-finalizing buffer` が発火しない

3. Narrative queue / OpenRouter narrativization  
- 破綻経路: segment boundary 不成立  
- 影響: queue enqueue が増えず、当日物語化が進まない

4. 自動 Recall注入（旧注入経路）  
- 破綻経路: `retrieveRelevantContext` 呼び出し入口消失  
- 影響: Query改善が生きていても同ターン注入されない

5. Compactionアンカー系  
- 破綻経路: compaction hooks は Embedded パイプライン前提  
- 影響: CLI連続運用ではアンカー注入機会が実質減る

6. 観測性  
- 破綻経路: 上記入口不達  
- 影響: Query/Injection/Finalize系ログが見えず運用診断不能

## 4. v0.4.7採用方針（One-Shot Reconnect）

## 4.1 Primary ingress を provider 非依存フックへ移す

`before_dispatch` を **主入口** にする。  
理由: provider分岐（CLI/Embedded）より手前で動き、`sessionKey` が取れるため agent/workspace 解決が安定する。

実施:
- `api.on("before_dispatch", ...)` を追加
- その場で `prepareWorkspaces(...)`
- `sessionKey` 起点で agentId 解決し、ingest対象を確定
- ユーザー入力を turn payload 化して segmenter 側へ渡す
- **重要:** `before_dispatch` の payload は単発メッセージなので、既存 `processTurn(currentMessages[])` に直接渡さない
- `processTurn` の `lastProcessedLength` 前提を壊さないため、incremental ingest 専用入口を追加する（詳細は 5.2）

## 4.2 Assistant側の片肺化を防ぐ

`message_sent` を **補助入口** として追加し、送信成功テキストを ingest 側へ渡す。  
これで「ユーザーだけ保存される」劣化を防ぐ。

実施:
- `api.on("message_sent", ...)` を追加
- `before_dispatch` で作った session mapping（conversation/sessionKey/agentId）を参照
- assistant payload を同一ターンへ合流
- mapping は短命TTLで管理し、解決不能時は no-op + 診断ログのみ（誤workspace書き込みを防止）

## 4.3 Tool-first方針は維持、注入経路は明示的に閉じる

- `toolFirstRecall.enabled=true` のとき:
  - `before_prompt_build` は常時 no-op (`{}`)
  - `assemble()` は recall注入を行わない
- 例外は「ツール呼び出し不能などのハード失敗時」のみ（任意フォールバック）

## 4.4 Query生成の契約（v0.3.7系維持）

- `query` は「最新1件の生文直投げ」ではない
- 既存 Parse/Rewrite 系で作る
- `recallQueryRecentMessageCount` は Parse/Rewrite の user message 窓サイズとして扱う

## 4.5 非対象（明示）

- `openclaw agent` の command path（`dispatch-from-config` 非経由）は v0.4.7 の主対象外
- ただし、この経路で壊さないこと（no regression）はテストで担保する

## 5. 実装スコープ（Episodic-Clawのみ）

1. `src/index.ts`
- `before_dispatch` フックを追加（primary ingest）
- `message_sent` フックを追加（assistant ingest補完）
- `sessionKey + fingerprint + run/turn` ベースの idempotency marker 追加
- `before_prompt_build` no-op の明示ログ統一
- `assemble()` の recall注入停止条件を v0.4.7 仕様に合わせて固定

2. `src/segmenter.ts`（必要最小限）
- 既存 `processTurn` 前提だけで足りない場合、incremental append 用の薄い入口を追加
- 目的は「フル履歴必須」依存を外し、hook payload 単位でも安全に処理すること
- 例: `processIncrementalTurn(deltaMessages, agentWs, agentId, cursorKey)` を追加し、`lastProcessedLength` 非依存で buffer を進める
- 既存 `processTurn` は互換維持（context-engine 経路で継続利用）

3. `src/tool-first-gate.ts`
- `before_dispatch` 主経路と `ep-recall` 実行時で turn drift しないようカウンタ規約を明文化
- `skip_reason` / `filter_eval_ms` / `source` ログを統一

4. `src/index.ts`（ingress state 管理）
- `conversationId/channelId -> sessionKey/agentId` の短命マップを追加（TTL + サイズ上限 + LRU）
- `message_sent` 側で session 解決不能時は保存処理をしない（誤注入回避）
- idempotency marker は `sessionKey + role + normalizedTextHash + minuteBucket` を推奨（turn未提供経路でも安定）

5. `openclaw.plugin.json` / README系 / RELEASE_NOTES
- v0.4.7 で ingest入口を provider非依存化したことを明記
- Known limitation と fallback 条件を明記

## 6. パフォーマンス要件（2010年級CPU）

対象は「Recall 条件付きフィルター判定」部分のみ（I/O除外）。

- p50 <= 1.0ms
- p95 <= 3.0ms
- p99 <= 5.0ms

制約:
- 短絡評価順を固定: novelty -> intent -> fingerprint -> negative cache
- 重い正規表現連鎖・全履歴走査を禁止
- 判定はメモリ内完結

## 7. テスト計画（v0.4.7必須）

1. CLI provider で `before_dispatch` ingest が毎ターン動く  
2. CLI provider で `segmenter` が進み、idle auto-finalize ログが復帰する  
3. `message_sent` 補助入口で assistant 側も episodic 化される  
4. Tool-first有効時、`before_prompt_build` は常時 no-op で重複しない  
5. Embedded で二重イベントにならない（primary ingress 成功時は旧注入経路が静止）  
6. `recallQueryRecentMessageCount` が query窓サイズとしてのみ効く  
7. no-hitバックオフ `3 -> 6 -> 12` が維持される  
8. v0.4.3 idle silence flush 回帰が再現しない（当日エピソードが生成される）
9. `before_dispatch` 単発payloadでも segmenter が継続動作し、`lastProcessedLength` 破綻が起きない
10. `message_sent` で session mapping 解決不能時に no-op で安全に落ちる
11. `openclaw agent` command path でクラッシュ・誤保存がない（非対象だがno regression）

## 8. Go / No-Go

1. OpenClaw本体差分ゼロ  
2. CLIターンで episodic ingest が実測で復帰  
3. 同一ターン重複注入ゼロ  
4. Query生成契約（Parse/Rewrite）維持  
5. docs/release notes が実装と一致

---

## 9. リスクと対処

1. `before_dispatch` で sessionKey が欠落する経路  
- 対処: no-op + 明示ログ + `message_received` 低信頼フォールバック（保存のみ）

2. フック増加による重複 ingest  
- 対処: idempotency marker を必須化（sessionKey+fingerprint+turn）

3. message_sent だけで session 解決できないケース  
- 対処: `before_dispatch` 時点で conversation→session の短命マップを構築

4. OpenClaw公式の memory 方針変更  
- 対処: v0.4.x では plugin-only維持。将来、公式が Tool-first一本化したら `retrieveRelevantContext` 完全停止を再評価。

5. `openclaw agent` 直実行のように `before_dispatch` を経由しない経路  
- 対処: 本版の対象外として明記し、次版で command-path専用入口を別検討。

---

## 10. Pro Recommendation

`v0.4.7` は **primary ingest を provider分岐前フックへ移す** ことで、  
CLI分岐起点の連鎖破綻（ingest/idle finalize/narrative/注入観測）を一括で再接続する。  
この方針が、OpenClaw本体非改修での最短・最小リスクな一発復旧策。

> Perspective: Google / IBM Production Engineering  
> Principles applied: YAGNI · KISS · DRY · SOLID  
> Source code verified: ✅ (as of 2026-04-12)

### 📍 Current Reality (Source Code vs. Document)
- ✅ Document matches code: CLI provider は `runCliAgent` 直分岐で Embedded の `before_prompt_build` / `assemble|ingest` を通らない。
- ✅ Document matches logs: 対象時刻は `agent/cli-backend cli exec` が継続し、episodic 側は `SleepTimer Idle` のみで live ingest 痕跡がない。
- ⚠️ Discrepancy found: `before_dispatch` は単発 payload であり、既存 `segmenter.processTurn(currentMessages[])` の前提（累積メッセージ長）と一致しない。
- ⚠️ Discrepancy found: `message_sent` の context には `sessionKey` が含まれないため、session mapping 設計を明示しないと誤保存リスクがある。
- ⚠️ Scope caveat: `openclaw agent` command path は `dispatch-from-config` 非経由のため、`before_dispatch` bridge の対象外。

### 🎯 Core Problem (1 sentence)
> 入口を `before_dispatch` に移す方針は正しいが、segmenter のデータ契約を合わせないと ingest が再び空振りする。

### 🔍 Principle Filter
| Check | Result | Note |
|-------|--------|------|
| YAGNI — Is this actually needed now? | ✅ Yes | 入口再接続は必須。過剰機能ではない。 |
| KISS — Is there a simpler solution? | ⚠️ Simpler exists | まず incremental ingest 入口を1本追加するのが最短。 |
| DRY — Any duplication to eliminate? | ✅ Manageable | ingest 入口は増えるが、segmenter本体ロジックは共通化可能。 |
| SOLID — Any violation causing real problems? | ⚠️ Found | `processTurn` へ単発payload直投入は責務前提を崩す。 |

### 🛤️ Solution Options

#### Option A — Incremental Ingest Adapter *(推奨)*
**Approach**: `before_dispatch/message_sent` からは incremental 専用 API に渡し、segmenterの既存境界判定を内部再利用する。  
**Implementation cost**: Medium  
**Risk**: Low  
**Why recommended**: `processTurn` の前提を壊さず、CLI/Embedded の両経路を最小差分で統合できる。  
**Concrete steps**:
1. `segmenter.ts` に incremental 入口を追加（cursorは sessionKey 単位）。
2. `index.ts` に `before_dispatch` と `message_sent` を追加し、同入口へ集約。
3. session mapping は TTL/LRU 付き短命マップで運用。
4. tool-first有効時の注入経路 no-op を維持。

#### Option B — Session JSONL Reconstruct Path
**Approach**: `before_dispatch` ごとに session JSONL を読み直して `processTurn(fullMessages)` を維持する。  
**Implementation cost**: Medium-High  
**Risk**: Medium  
**When to choose this instead**: incremental API を触りたくない短期対応時。  
**Concrete steps**:
1. `sessionKey` から session file を解決。
2. 最新 JSONL を parse して `processTurn` へ渡す。
3. I/Oコスト増を受け入れて安定優先にする。

### ✅ Pro Recommendation
> **Choose Option A because**: plugin-only制約下で最小の運用リスクで「一発復旧」を実現できる。  
> Estimated implementation: 0.5〜1.0日  
> Rollback plan: `before_dispatch/message_sent` 登録を feature toggle で無効化し、v0.4.6 挙動へ即時復帰。
