# v0.4.3 Recall Query CJK and Attachment Filter Fix

**Parents:** [v0.4.1 Post-Release Hardening](./v0.4.1_post_release_hardening.md) / [v0.4.2 Roadmap Hardening Gap Closure](./v0.4.2_roadmap_hardening_gap_closure.md)  
**Date:** 2026-04-10  
**Target Version:** v0.4.3  
**Status:** 収束完了（core fix + runtime regression test in default npm test harness）

> 本プランは、「`recallQueryRecentMessageCount: 2` にしているのに query が古い英語ノイズになる」「日本語 query がほぼ出ない」「画像送信 message を recall query から外したい」という incident を対象にした修正計画です。  
> 結論は、**設定値は読めているが、eligible message 選別と polyglot keyword 抽出が弱く、画像添付/OCR/attachment 文言が query を汚染している** です。
>
> なお本件は **Telegram 専用の正規表現で直してはいけません**。`openclaw-source` 側では Telegram / LINE / Discord で添付 message の text 化パターンが少しずつ違うため、v0.4.3 の修正は **channel-agnostic** に組む必要があります。

### 2.2.1 あるべき意味は「最後の2 raw user message を strict に守り、その窓内で eligible text だけ抽出する」

今回の incident で問題になったのは、画像送信 message が次のような混合 payload になっていた点です。

- `[media attached: ...]`
- `[media attached: media://inbound/<id>]`
- `<media:image>` / `<media:document>`
- `[User sent media without caption]`
- 絶対パス / ファイル名
- `To send an image back, prefer ...`
- gateway / system boilerplate
- 末尾に短い caption 本文

この形だと、raw では確かに「最後の user message」ですが、**message 全体を丸ごと recall query 材料にするのは不適格** です。  
v0.4.3 では `recallQueryRecentMessageCount` の意味を次のように再定義するのが妥当です。

> `recallQueryRecentMessageCount = 2`  
> = **最後の 2 件の raw user messages を strict に対象化し、その 2 件の中から recall eligible な本文だけを抽出して query を組む**

ここでの recall eligible ルールは、少なくとも次を含むべきです。

- image / file attachment boilerplate だけの message は除外
- `media attached`, `media://inbound`, `<media:image>`, `<media:document>`, `attached files`, `jpg/jpeg/png/webp`, 絶対パス列は query 材料から除外
- `[User sent media without caption]` のような media-only sentinel は query 材料から除外
- `To send an image back...` のような transport/gateway boilerplate は除外
- 画像 message に caption がある場合は **caption 本文だけ残す**
- caption が実質空なら、その user message 自体を recall query 対象から外す
- **その結果 query 材料が足りない / 空になるなら、古い message へは遡らず recall 自体を skip する**

短い日常文でも、この strict window の中で意味語を拾えれば query は作れます。  
例:

- raw message: `おはよう、朝は早いね…`
- query: `おはよう 朝 早い`

つまり、**数え方のバグではなく、strict recent window の中で本文だけを救い出す sanitize が足りない** のが今回の本質です。

### 2.2.2 OpenClaw source との相互性は「Telegram 専用」ではなく「共通添付表現」ベースで合わせるべき

`openclaw-source` を見ると、添付 message の text 化はチャネルごとに少し違います。

- 共通 auto-reply / gateway 側
  - `src/auto-reply/media-note.ts`
  - `src/gateway/chat-attachments.ts`
  - `[media attached: ...]`
  - `[media attached: media://inbound/<id>]`
  - `To send an image back, prefer ... Keep caption in the text body.`
- LINE 側
  - `extensions/line/src/bot-message-context.ts`
  - 画像やファイルを `<media:image>` / `<media:document>` に落とす
  - text が無い media-only message でも placeholder 本文を作る
- Discord 側
  - `extensions/discord/src/monitor/message-utils.ts`
  - 添付を `<media:image> (N images)` / `<media:document> (N files)` に落とす
  - content が空なら attachment placeholder がそのまま message body になる

つまり v0.4.3 では、**Telegram の絶対パスつき添付文だけを見て正規表現を書くのは不十分** です。  
実装は provider 名で分岐するのではなく、OpenClaw 共通で現れる transport / attachment marker を落とし、caption / user-authored text だけを救い出す形にするべきです。

## 3. Core Problem

> **設定された recent message 数は読めているが、recall query を作る前段で「どの user message を query 材料にしてよいか」の選別がなく、さらに keyword extractor が Latin-first なので、画像添付/OCR/filename 由来の英語ノイズが CJK 意図を押し流している。**

## 5. Solution Options

### Option A — Recall-Only Eligibility Filter + Script-Aware Query Builder *(推奨)*

**Approach**:  
recall query 用にだけ、直近 user messages をさらに選別する。  
image-only / attachment-dominant / filename-heavy message を落とし、残った text から script-aware に query を組み立てる。  
このとき判定は Telegram 専用ではなく、OpenClaw 共通の attachment marker と LINE / Discord placeholder を含む **channel-agnostic sanitize** として実装する。

**Implementation cost**: Medium  
**Risk**: Low  
**Why recommended**:  
今の incident を一番素直に閉じられる。`recallQueryRecentMessageCount` の契約はそのまま保てる。

**Concrete steps**:

1. `src/retriever.ts` に recall 専用 eligibility filter を追加する
   - `isRecallEligibleMessage(message)`
   - 条件例:
     - empty / whitespace only を除外
     - `media attached`, `media://inbound`, `<media:image>`, `<media:document>`, `attached files`, `image jpeg`, `jpg`, `png`, `webp` だけで大半を占める文を除外
     - `[User sent media without caption]` のような media-only sentinel を除外
     - filename token 比率が高すぎる文を除外
     - text length が短すぎる attachment ack 文も除外
     - image/file message に caption がある場合は caption だけ抽出し、boilerplate 側は切り落とす
     - provider 名では分岐せず、OpenClaw 共通の transport marker と placeholder で判定する

2. `recentMessageCount` は strict に守る
   - まず `.filter(role=user).slice(-recentMessageCount)` で raw window を固定する
   - その window の中でだけ eligibility / caption salvage / boilerplate strip を行う
   - **window 外の古い user messages には遡らない**
   - その結果 query が空なら `empty_query` 扱いで recall しない
   - つまり仕様としては `last 2 raw user messages` を strict window とし、`last 2 recall-eligible user texts` に置き換えない

3. `large-payload.ts` に recall query 向け sanitize helper を追加する
   - 汎用 normalize を壊さず、recall query 専用に使う
   - `stripAttachmentNoise(...)`
   - `stripFilenameLikeTokens(...)`
   - `stripMediaMetaPhrases(...)`
   - `extractRecallEligibleText(...)` のように caption salvage を含む helper 名でもよい
   - strip 対象には少なくとも次を含める
     - `[media attached: ...]`
     - `[media attached: media://inbound/<id>]`
     - `<media:image>` / `<media:document>` / `<media:audio>` / `<media:video>`
     - `[User sent media without caption]`
     - `To send an image back, prefer ... Keep caption in the text body.`
   - ただし caption 本文は落とさず、行単位で boilerplate 行だけ剥がす

4. `extractPolyglotKeywords()` を script-aware に直す
   - dominant script を判定
   - dominant が CJK なら CJK を先に採用
   - Latin と CJK を別 bucket 化し、片方だけで 12 件を埋めない
   - 例: `CJK 8 + Latin 4` か interleave

5. observability を追加する
   - `eligibleRecentMessages`
   - `skippedImageLikeMessages`
   - `dominantScript`
   - `finalQuery`
   - `queryHash`
   - こうして「設定無視」と「sanitize failure」を見分けられるようにする

6. テストを追加する
   - Telegram 形式の `[media attached: /abs/path ...]` でも caption 本文だけ残る
   - gateway offload 形式の `[media attached: media://inbound/<id>]` でも query ノイズ化しない
   - LINE 形式の `<media:image>` / `<media:document>` でも caption 本文だけ残る
   - Discord 形式の `<media:image> (N images)` / `<media:document> (N files)` でも query ノイズ化しない
   - 日本語2件 + 画像添付1件で日本語 query が優先される
   - strict recent 2件が両方 media-only なら、前の message へ遡らず recall を skip する
   - `media attached files ... jpg` 系の query がそのまま出ない
   - Latin-heavy mixed text でも CJK dominant 時は CJK query が残る

### Option B — Full LLM Rewrite Query Builder

**Approach**:  
deterministic rewrite をやめて、短い query 生成を LLM に委ねる。

**Implementation cost**: Medium - High  
**Risk**: Medium  
**When to choose this instead**:  
将来、多言語 mix・画像説明・tool trace まで含めて柔軟に query 生成したい時。

**Why not recommended now**:  
コストもブレも増える。v0.4.3 は deterministic path を強くする方が安全。

### Option C — `queryExcludedKeywords` に個別ノイズ語を追加し続ける

**Approach**:  
`jpg`, `image`, `attached`, `files` などを stopword に足して逃げる。

**Implementation cost**: Low  
**Risk**: Medium  
**Why not recommended**:  
対症療法にしかならない。filename / OCR / media phrase はまた別名で出る。

## 7. Acceptance Criteria

1. `recallQueryRecentMessageCount: 2` は strict raw window として守られ、window 外へ勝手に backfill しない
2. 直近2件の raw user messages が両方とも query 材料ゼロなら、古い message へ遡らず recall を skip する
3. `ai.recall payload` に `media attached files ... jpg` のような query がそのまま出ない
4. 日本語 conversation では CJK query が優先される
5. assistant message を混ぜずに、user intent を保った query になる
6. 既存の English-only conversation では今まで通り English query が作れる
7. 画像付き message では boilerplate ではなく caption 本文が recall query 候補として扱われる
8. Telegram / LINE / Discord のいずれでも、attachment placeholder 自体は query に残らない
9. provider ごとの if 文ではなく、共通 sanitize helper で通る
