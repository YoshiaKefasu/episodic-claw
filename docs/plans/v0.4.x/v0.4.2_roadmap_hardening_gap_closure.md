## 1. 先に結論

コード上では、v0.4.0 / v0.4.1 の中核だった以下はすでに入っています。

1. D1 consolidation の停止 (`go/main.go`)
2. SleepTimer の `pebble:not found` ノイズ抑制 (`go/main.go`)
3. Embedding 429 回復の TTL + probe と capped backoff (`go/main.go`, `go/internal/ai/provider.go`)
4. v0.4.2 の cache-and-drain 基盤 (`src/narrative-queue.ts`, `src/narrative-worker.ts`, `go/internal/cache/queue.go`)

一方で、**まだ v0.4.2 で閉じる価値が高い欠落** は次の4系統です。

1. **README / manifest / TypeBox schema が旧D0/D1世界のまま**
2. **設定項目の意味がコード実態とズレている**
3. **互換層が「legacy / compatibility」として整理されず現役に見えている**
4. **v0.4.0 / v0.4.1 から後ろ倒しされた運用確認が未着手**

この4つは、放置すると「コードは正しいが公開契約が古い」状態を固定してしまう。  
逆に言うと、ここを閉じれば v0.4.2 はかなり締まる。

## 3. 実コードで確認できたこと

### 3.1 v0.4.1 の hardening 本体はコード上かなり入っている

**D1 排除**
- `go/main.go` の `checkSleepThreshold()` は consolidation を呼ばず、idle ログだけ出す
- `go/main.go` の `handleConsolidate()` は noop レスポンス
- `ai.consolidate` RPC ディスパッチは残っているが、互換性維持のための入口に留まる

**H-2 ノイズ抑制**
- `go/main.go` で `pebble:not found` は正常系として握り潰している

**H-3 Embedding 429 対策**
- `go/main.go` で `heal_429` の TTL 30min + probe 復帰が入っている
- `go/internal/ai/provider.go` で `Retry-After` は 5 分 cap

つまり、**v0.4.1 本体のコード欠落は小さい**。  
いまの問題は、実装そのものより **周辺契約が古い** ことです。

### 3.2 README と公開契約 — 同期完了

README 3言語と公開面の旧アーキ説明は、別エージェントによって同期済み。

1. D0 / D1 二層構造 → narrative + cache-and-drain 主軸に更新
2. Background Workers → Healing / rebuild / replay に説明更新
3. `ep-expand` → episode lookup / expand に更新
4. `maxBufferChars` / `maxPoolChars` → advanced guard として明記
5. `enableBackgroundWorkers` → maintenance workers に更新

加えて、以下も同期済み。

1. `openclaw.plugin.json` の設定説明
2. `src/index.ts` TypeBox schema / plugin description
3. `src/types.ts` の型コメント
4. `sharedEpisodesDir` / `allowCrossAgentRecall` の disabled compatibility 化
5. flat narrative alias の legacy 扱い明記

したがって、**公開契約のズレは回収済み**。

### 3.3 `v0.4.2_redundant_config_cleanup.md` — 再スコープ完了

このプランは当初「64K cache-and-drain が入ったから、旧サイズ設定は全部不要」という前提だった。
しかし、現行コードを確認した結果、以下のように再分類された。

**`maxBufferChars`**
- `src/segmenter.ts` の boundary 判定にまだ使用 → **advanced flush guard** として再分類

**`maxPoolChars`**
- `src/narrative-pool.ts` の flush 条件に使用 → **advanced pool flush guard** として再分類

**`maxCharsPerChunk`**
- `src/segmenter.ts` の legacy `chunkAndIngest()` で使用 → **deprecated / legacy-only** に格下げ

つまり、**v0.4.2 では staged deprecation を採用**。実削除は v0.4.3 以降へ分離。

### 3.4 旧設計の互換層 — 格下げ完了

runtime 側の互換層は、以下のように格下げ済み。

**compatibility / noop として明記済み**
- `ai.consolidate` → deprecated no-op compatibility RPC (`@deprecated` JSDoc)
- `sharedEpisodesDir` / `allowCrossAgentRecall` → disabled compatibility key
- flat narrative alias → legacy alias for `openrouterConfig.*`

**fallback / legacy として明記済み**
- `triggerBackgroundIndex` → fallback/legacy path
- `chunkAndIngest()` → legacy path
- `summary-escalation` → fallback path

これらは残すこと自体より、「現役機能に見えてしまうこと」が問題だったが、v0.4.2 では適切な肩書きを付与済み。

## 7. Solution Options

### Option A — Contract Sync + Release Readiness Bundle *(推奨)*

**Approach**  
v0.4.2 ではランタイムの大手術を避け、**コード実態に合わせて公開契約と運用準備を締める**。  
`v0.4.2_redundant_config_cleanup.md` は「全削除」ではなく「Phase 1: deprecate / reword / narrow scope」へ縮小する。

**Implementation cost**  
Low to Medium

**Risk**  
Low

**Why recommended**  
- すでに動いている `v0.4.1-hotfix` / `v0.4.2` の基盤を壊さない
- README / manifest / schema のズレを先に閉じられる
- 24h テストや品質レビューを release checklist として正式化できる
- YAGNI / KISS に合う

**Concrete steps**

1. **README reality sync**
   - `README.md`
   - `README.ja.md`
   - `README.zh.md`
   - D0/D1 の主役表現をやめ、v0.4.x の narrative + cache を主軸に書き換える
   - `enableBackgroundWorkers` の説明から D1 整理の文言を外す
   - `ep-expand` の説明を D1 中心から episode lookup / expand の実態へ寄せる
   - 「3設定は完全削除済み」の false claim を消す

2. **設定契約の同期**
   - `openclaw.plugin.json`
   - `src/index.ts` TypeBox schema
   - `src/types.ts`
   - `src/config.ts`
   - ここでは **削除ではなく明文化** を優先する
     - `maxBufferChars`: segmenter boundary guard
     - `maxPoolChars`: narrative pool flush guard
     - `maxCharsPerChunk`: legacy path only
     - `enableBackgroundWorkers`: healing / rebuild / replay worker の on/off として説明を更新
     - `sharedEpisodesDir` / `allowCrossAgentRecall`: disabled compatibility key
     - flat narrative fields: legacy alias for `openrouterConfig.*`

3. **互換層の格下げ明記**
   - `ai.consolidate`: deprecated no-op compatibility RPC
   - `triggerBackgroundIndex` / `chunkAndIngest()` / `summary-escalation`: fallback / legacy path
   - 現役機能に見える表現は外す

4. **`v0.4.2_redundant_config_cleanup.md` の再スコープ**
   - v0.4.2 で安全に触れるのは「契約整理」まで
   - 実削除は v0.4.3 以降へ分離
   - 少なくとも `maxCharsPerChunk` は legacy path 廃止前に消さない

5. **運用確認のドキュメント化**
   - 24h soak test checklist
   - narrative 品質レビュー観点
   - recall 比較の acceptance criteria
   - これはコード変更なしでも v0.4.2 に入れる価値が高い

### Option B — Runtime Cleanup Bundle

**Approach**  
v0.4.2 の中で、設定削除・legacy path 整理・README 更新・release test まで全部まとめてやる。

**Implementation cost**  
Medium to High

**Risk**  
Medium

**When to choose this instead**  
v0.4.2 を「最後の v0.4 系整理リリース」にしたくて、多少の実装リスクを受け入れる場合。

**Why not recommended now**
- `maxBufferChars` / `maxPoolChars` はまだ live path で使っている
- `maxCharsPerChunk` も legacy path では生きている
- README 修正と runtime cleanup を同時にやると、切り分けが難しくなる

## 8. Pro Recommendation

> **Choose Option A because**:  
> いま閉じるべきは「コードそのもの」より「コードと外向き契約のズレ」です。  
> v0.4.2 では runtime を無理に削るより、README / manifest / schema / release checklist をコード実態に合わせるほうが、はるかに安全で効果が大きい。

**Estimated implementation**
- ドキュメント同期: 2-3h
- manifest / schema 説明更新: 1-2h
- release checklist 整備: 1h
- 合計: 半日以内

**Rollback plan**
- ドキュメント更新中心なので rollback は容易
- config 説明の更新だけなら runtime 影響は最小
