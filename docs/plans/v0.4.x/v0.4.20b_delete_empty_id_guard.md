## 0. 概要

- 目的: v0.4.20の空ID防御に`Delete()`/`deleteLocked()`/`CleanOrphans()`が含まれていないギャップを修正
- 背景: v0.4.20のFix 3は`Add()`/`BatchAdd()`に空ID拒否を追加したが、`Delete()`系は見落とし
- 深刻度: 🟡 MED（低確率だが実在する防御ギャップ）
- 発見元: v0.4.20実装の横断的検証（全呼び出し経路精査）

### 問題の詳細

`deleteLocked("")` が実行されると:

1. `epKey = append(prefixEp, []byte("")...)` → `"ep:"`（プレフィクスそのもの）
2. `s2iKey = append(prefixS2I, []byte("")...)` → `"s2i:"`（同上）
3. PebbleDBの`Get("ep:")`/`Get("s2i:")` → 通常は`ErrNotFound`（実害なし）
4. `batch.Delete("ep:", nil)` / `batch.Delete("s2i:", nil)` → プレフィクスキーそのものを削除（不正操作）
5. `enqueueSysLexq(batch, "DELETE", "")` → **Fix 1で安全にスキップされる** ✅

**唯一の発火経路**: `CleanOrphans()` が `rec.ID == ""` かつ `rec.SourcePath != ""` かつ対象ファイルが存在しないレコードを検出した場合。

```go
// store.go CleanOrphans()
for iter.First(); iter.Valid(); iter.Next() {
    var rec EpisodeRecord
    if err := msgpack.Unmarshal(iter.Value(), &rec); err == nil && rec.SourcePath != "" {
        if _, statErr := os.Stat(rec.SourcePath); os.IsNotExist(statErr) {
            toDelete = append(toDelete, string(rec.ID))  // ← rec.ID == "" の場合あり
        }
    }
}
// ...
for _, id := range toDelete {
    s.Delete(id)  // ← Delete("") が呼ばれる
}
```

## 2. 修正詳細

### 2.1 Fix 1: `deleteLocked()` 空IDガード

**修正** (`store.go` `deleteLocked()` 冒頭):

```go
func (s *Store) deleteLocked(id string) error {
    // [v0.4.20b] Guard: skip deletion for empty ID — cannot form valid Pebble keys
    if id == "" {
        logger.Warn(logger.CatStore, "deleteLocked: skipped deletion for empty ID")
        return nil
    }
    epKey := append(append([]byte(nil), prefixEp...), []byte(id)...)
    // ... existing code ...
}
```

**設計判断**:
- `return nil`（エラーにしない）理由: `Delete("")`の呼び出し元は`CleanOrphans()`のみで、空IDレコードの削除失敗は致命的ではない。v0.5.0のDB移行で自然に消滅する。
- `nil`を返すことで呼び出し元のエラーハンドリングに影響しない。

### 2.2 Fix 2: `Delete()` 空IDガード

**修正** (`store.go` `Delete()` 冒頭):

```go
func (s *Store) Delete(id string) error {
    // [v0.4.20b] Guard: skip deletion for empty ID
    if id == "" {
        return nil
    }
    s.mutex.Lock()
    defer s.mutex.Unlock()
    return s.deleteLocked(id)
}
```

**設計判断**:
- `Delete()`の公開APIレイヤーでもガードすることで、将来の呼び出し元も保護。
- 二重ガード（Delete + deleteLocked）はDRY原則に反するが、KISS（各関数が自己完結的に安全）とSOLID（公開APIの事前条件チェック）を優先。

### 2.3 Fix 3: `CleanOrphans()` 空IDスキップ

**修正** (`store.go` `CleanOrphans()` ループ内):

```go
for iter.First(); iter.Valid(); iter.Next() {
    var rec EpisodeRecord
    if err := msgpack.Unmarshal(iter.Value(), &rec); err == nil && rec.SourcePath != "" {
        if strings.TrimSpace(rec.ID) == "" {
            logger.Warn(logger.CatStore, "CleanOrphans: skipping record with empty ID (SourcePath=%s)", rec.SourcePath)
            continue
        }
        if _, statErr := os.Stat(rec.SourcePath); os.IsNotExist(statErr) {
            toDelete = append(toDelete, rec.ID)
        }
    }
}
```

**設計判断**:
- `Delete("")` 呼び出し自体を発生させないことで、同一空IDレコードへの不要な再試行を抑える。
- `Delete`/`deleteLocked` の防御と責務が重複するが、入力正規化（CleanOrphans）と防御（Delete層）は目的が異なるため許容。

**Approach**: 現プランの Fix 1-2 に加えて、`CleanOrphans()` でも `rec.ID==""` を早期スキップしてログ化する。  
**Implementation cost**: Low  
**Risk**: Low  
**Why recommended**: 呼び出し元・公開API・内部実装の3層で防御でき、同じ空IDを毎回Deleteしにいく無駄を減らせる。
