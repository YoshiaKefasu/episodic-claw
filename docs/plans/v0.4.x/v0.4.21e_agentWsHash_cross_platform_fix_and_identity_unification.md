## 2. E1: agentWsHash Cross-Platform Fix

### 問題

現在の `agentWsHash()` (rpc-client.ts L608-614):
```typescript
private agentWsHash(agentWs: string): string {
  const normalized = agentWs.replace(/\\/g, '/').toLowerCase().replace(/\/+$/, '');
  // DJB2 hash...
}
```

`.toLowerCase()` が **全プラットフォーム** で適用される:
- **Windows**: case-insensitive → lowerCase 化は正しい（`C:\Users` と `c:\users` は同一）
- **macOS (default)**: case-insensitive HFS+/APFS → lowerCase 化は概ね正しい
- **Linux / case-sensitive macOS**: `/Work/Foo` と `/work/foo` は**異なるディレクトリ**だが同一ハッシュになる → **cursor/save-hash が別 workspace 間で混入** = 実バグ

### 修正方針

**platform 判定を入れて win32 のみ lowerCase 化**。非Windowsは slash 正規化のみ。

```typescript
private agentWsHash(agentWs: string): string {
  let normalized = agentWs.replace(/\\/g, '/').replace(/\/+$/, '');
  // Case normalization only on Windows (case-insensitive by default)
  if (process.platform === 'win32') {
    normalized = normalized.toLowerCase();
  }
  let hash = 5381;
  for (let i = 0; i < normalized.length; i++) {
    hash = ((hash << 5) + hash + normalized.charCodeAt(i)) & 0x7FFFFFFF;
  }
  return hash.toString(36);
}
```

### 注意事項

- macOS は case-insensitive volume が多いが、case-sensitive volume も現実に存在するため、`darwin` 一律 lowerCase は誤同一化リスクを残す
- 本プランでは **誤同一化回避を優先** して `win32` のみ lowerCase とする（KISS + 安全側）
- macOS の同一化精度（`/Work` と `/work`）は将来課題。必要になった時点で fs 依存の実判定を検討（YAGNI）
- この修正により、**Windows で既存 DB のキーは変わらない**（lowerCase 化は従来通り）→ migration 不要
- **Linux/macOS ではキーが変わる可能性がある**（従来 lowerCase 化されていたのが止まる）が、v0.4.21d の D3 自体がまだリリース前のため、実質的な regression なし

## 3. E3: agentWsHash の Shared Utility 化

### 問題

`agentWsHash` は `RpcClient` の `private` メソッド。`NarrativeWorker` から E2 のために呼び出せない。

### 修正方針

**`utils.ts` に export して共有化**。`utils.ts` は既に `estimateTokens` を export しているため、同じパターンで追加。

```typescript
// utils.ts に追加
/** [v0.4.21e] Canonicalize agentWs path for stable state key construction.
 *  Platform-aware: lowercases on case-insensitive filesystems (win32/darwin) only.
 *  Normalizes path separators and trailing slashes on all platforms.
 *  Returns DJB2 hash (base36) of the normalized path.
 */
export function agentWsHash(agentWs: string): string {
  let normalized = agentWs.replace(/\\/g, '/').replace(/\/+$/, '');
  if (process.platform === 'win32') {
    normalized = normalized.toLowerCase();
  }
  let hash = 5381;
  for (let i = 0; i < normalized.length; i++) {
    hash = ((hash << 5) + hash + normalized.charCodeAt(i)) & 0x7FFFFFFF;
  }
  return hash.toString(36);
}
```

`rpc-client.ts` 側:
- `private agentWsHash` メソッドを削除
- `import { agentWsHash } from './utils'` を追加
- 4メソッドの `this.agentWsHash(agentWs)` → `agentWsHash(agentWs)` に変更

## 4. E2: In-Memory Key も agentWsHash 経由に統一

### 問題

| レイヤー | key 構築 | 正規化 |
|----------|----------|--------|
| **DB (永続化)** | `agent:${agentId}:ws:${agentWsHash(agentWs)}:...` | hash(normalize(path)) |
| **in-memory recentSaveHashes** | `${agentWs}:${agentId}:${rawHash}` | raw path |
| **in-memory saveHashPersistCounters** | `${agentWs}:${agentId}` | raw path |

同一 workspace が異なる path 表現で参照された場合（例: symlink、大文字小文字違い）、
永続化側は統一されるがインメモリ側は統一されない → **同一 process 内でも dedup が回避される**。

### 修正方針

インメモリ key の `agentWs` 部分も `agentWsHash(agentWs)` 経由に統一。

**変更箇所**:

1. **`recentSaveHashes` key**: `${agentWs}:${agentId}:${rawHash}` → `${agentWsHash(agentWs)}:${agentId}:${rawHash}`
   - `saveNarrative()` の `scopedKey` 構築 (L572)
   - `persistSaveHashes()` の `prefix` 構築 (L535)
   - `loadSaveHashes()` の `prefix` 再構築 (L554-557)

2. **`saveHashPersistCounters` key**: `${agentWs}:${agentId}` → `${agentWsHash(agentWs)}:${agentId}`
   - `persistSaveHashes()` の `agentKey` 構築 (L528)

3. **import 追加**: `import { agentWsHash } from './utils'`

### 注意事項

- この変更により、**プロセス内の in-memory dedup と DB dedup が同じ identity で動作**する
- `loadSaveHashes()` は DB から読み込んだ後、in-memory map に `prefix + h.key` 形式で格納する。
  DB 側の key は `agentWsHash` 経由で統一済みなので、`loadSaveHashes` の prefix も `agentWsHash` 経由にするだけで整合する
- **プロセス再起動時**: 既存 DB エントリの key は `agentWsHash` 経由（v0.4.21d で変更済み）なので、load 時の整合性は保たれる

## 5. E4: Best-Effort Recovery-Gap Observability

### 問題

`vstore == nil` の時:
- ✅ file save は継続
- ❌ dedup guard は無効
- ❌ `triggerHealing()` はスキップ
- ❌ vector/index DB への追加もスキップ

**現在のログ**: `Store init failed (best-effort mode, dedup skipped)` のみ。  
**欠落**: 「この episode は disk に保存されたが vector DB に未索引である」ことの記録がない。

もし watcher/healing も同時に degrade していれば、**markdown は disk に存在するが永久に索引されない**状態が発生し得る。

### 修正方針

`sync/atomic` カウンタを追加し、best-effort mode で保存された未索引 episode 数を追跡。
定期的（または healing worker 起動時）にログ出力。

**handleIngest**（`main.go` の vstore nil 分岐）:
```go
// 既存の vstore nil ブロックの末尾に追加
if vstore == nil {
    unindexedSaves.Add(1)
    EmitLog("Ingest: Episode saved to disk but NOT indexed (vstore unavailable). unindexed_total=%d. Run rebuild when store recovers.", unindexedSaves.Load())
}
```

**handleBatchIngest**（`main.go` の getStore 失敗ログ + goroutine 内 vstore nil 分岐）:
```go
// getStore 失敗時のログに追記
EmitLog("ai.batchIngest: Store init failed (best-effort mode, dedup skipped). Episodes will be saved but NOT indexed until store recovers.")
```

**カウンタ定義**（`main.go` ファイル先頭の var ブロック）:
```go
var unindexedSaves atomic.Uint64 // [v0.4.21e] Episodes saved to disk but not indexed (vstore nil)
```

**healing worker での活用**:
既存の `RunAsyncHealingWorker` が unindexedSaves > 0 の場合に追加ログを出力するよう修正。
（healing worker 自体のロジック変更は不要 — 追加ログのみ）

### 注意事項

- カウンタは**プロセス再起動でリセット**される。永続化は不要（disk 上の episode は healing worker が自動検出するため）
- `triggerHealing()` を vstore nil 時に呼ばないのは意図的（healing も vstore を必要とするため）。ただし vstore が復旧した後に healing が走れば回復可能
