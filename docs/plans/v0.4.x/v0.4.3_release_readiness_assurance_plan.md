## 4. Gate A — Idle Poll Wake Latency Guarantee

### 4.1 Goal

`wake()` が単に callback として存在するだけでなく、**15s cap に入った idle worker を enqueue 後すぐ再開できる**ことを実時間で証明する。

### 4.2 What must be true

1. worker を `15s` cap backoff 状態まで進められる
2. その状態で enqueue が発生すると、既存 timer を破棄して即 `pollNext()` が走る
3. live ingest / force flush / large gap archive / cold-start の4経路すべてで同じ契約になる
4. Go 側で `Method: cache.leaseNext` が hot-path method log に戻っていない
5. host 側で `[info]` が `warn` 表示されない

### 4.3 Required test shape

- benchmark or runtime integration test を 1 本追加
- 測る対象:
  - `enqueue callback fired at`
  - `next poll started at`
- 合格条件:
  - cap 中 enqueue から次 poll 開始まで `<= 1500ms`
  - 4経路とも同一上限を満たす

### 4.4 Evidence to capture

- `npm test` の該当 test 名と PASS
- bench/test の出力ログ
- 5分 idle + 1回 enqueue の観測ログ
  - `cache.leaseNext` method spam が出ていないこと
  - enqueue 後に cap 待ちせず再開したこと

### 4.5 Blocker condition

以下のどれかが起きたら release blocker:

- enqueue 後に `> 1500ms` かかる
- 経路ごとに wake 契約がバラつく
- `cache.leaseNext` method spam が回帰する
- host UI で info が warn に戻る

## 5. Gate B — Idle Silence Flush Integration Guarantee

### 5.1 Goal

`segmentationTimeGapMinutes` 超過後に新規 message が来なくても、`EventSegmenter` 本体が **1回だけ** idle flush / finalize することを統合テストで証明する。

### 5.2 What must be true

1. non-empty text buffer は silence 経過で flush される
2. image-only / tool-only buffer は flush されない
3. `idle-timeout` reason が queue までそのまま伝わる
4. flush 後も `lastProcessedLength` が壊れない
5. timer は boundary 後の新 buffer でも張り直される

### 5.3 Required test shape

- fake timer を使った integration test を追加
- standalone helper ではなく、`EventSegmenter` 本体の timer scheduling を直接踏む
- 最低ケース:
  1. text buffer flushes after silence
  2. image-only does not flush
  3. tool-only does not flush
  4. reason propagation = `idle-timeout`
  5. cursor preserved after flush

### 5.4 Evidence to capture

- `npm test` の fake-timer integration test 名と PASS
- 生成された queue item の reason assertion
- flush 後に次 `processTurn()` が履歴全体を再読しないことの assertion

### 5.5 Blocker condition

以下のどれかが起きたら release blocker:

- silence 経過で flush されない
- image-only / tool-only が誤 flush される
- `idle-timeout` が潰れる
- `lastProcessedLength` が reset する

## 6. Gate C — Surprise Footer Persistence Guarantee

### 6.1 Goal

queue chunk が持つ `surprise` が、worker save path を通って **保存済み narrative markdown footer** まで失われないことを証明する。

### 6.2 What must be true

1. leased item の `surprise` が `saveNarrative()` へ渡る
2. `batchIngest` params に同じ値が乗る
3. Go 側 ingest 保存後の footer / parsed metadata に同じ値が残る
4. multi-chunk は first-only preserve の現仕様を維持する

### 6.3 Required test shape

- persisted-footer integration test を 1 本追加
- 最低ケース:
  1. single chunk with non-zero surprise
  2. multi-chunk first-only preserve
- 期待値:
  - single chunk は saved footer の `surprise` が source と一致
  - multi-chunk は first chunk だけ preserve され、後続は `0`

### 6.4 Evidence to capture

- `npm test` の persisted-footer test 名と PASS
- 生成 markdown の footer snapshot
- parsed metadata assertion

### 6.5 Blocker condition

以下のどれかが起きたら release blocker:

- saved footer が `0` に潰れる
- source と saved metadata が不一致
- first-only preserve が崩れる

## 12. 徹底的にやること

この節は、v0.4.3 を本当に release してよい状態まで持っていくための**実行ルール**です。  
単に「テストを増やす」ではなく、**どこまで証明したら Gate を閉じてよいか**を明文化します。

### 12.1 Gate A で徹底的にやること

1. `wake()` を source code で確認するだけで終わらせない  
   実 `NarrativeWorker` を立ち上げ、`15s cap` 相当の idle 状態へ進めてから `enqueue -> wake -> poll resume` を測る。
2. 計測は wall-clock で取る  
   `Date.now()` でも `performance.now()` でもよいが、**変数代入だけの疑似計測は禁止**。
3. 4経路を全部通す  
   live ingest / force flush / large gap archive / cold-start の4本を同じ test suite で確認する。
4. 「timer が clear された」だけでなく「次 poll が実際に始まった」を取る  
   `clearTimeout()` 呼び出し確認だけでは Gate A は閉じない。
5. `cache.leaseNext` の quietness を一緒に見る  
   Go skip list 回帰と stderr severity remap 回帰がないことを、最低1ケースで同時確認する。

### 12.2 Gate B で徹底的にやること

1. `EventSegmenter` 本体を直接使う  
   helper 関数の再実装や source grep だけではなく、実体インスタンスで timer scheduling を踏む。
2. fake timer か controllable timer を差し込む  
   `segmentationTimeGapMinutes` 超過を deterministic に進める。
3. flush するケースと、しないケースを同じ suite で固定する
   - text buffer は flush
   - image-only は no flush
   - tool-only は no flush
4. `idle-timeout` reason を queue item まで見る  
   「発火したっぽい」ではなく、enqueue payload の reason を assertion する。
5. cursor 保持まで確認する  
   flush 後の次 `processTurn()` で履歴全体を再読しないことを assertion する。

### 12.3 Gate C で徹底的にやること

1. synthetic markdown 自前生成だけで済ませない  
   `saveNarrative -> batchIngest -> Go ingest -> saved markdown footer` の実経路を通す。
2. 実ファイルを保存して、実際に読み返す  
   `<!-- episodic-meta {JSON} -->` 形式の footer を parse し、`surprise` 値一致を取る。
3. single-chunk と multi-chunk を分けて固定する
   - single-chunk: source surprise と saved surprise が一致
   - multi-chunk: first-only preserve が維持される
4. body extraction も一緒に確認する  
   footer 除去後に本文だけが残ることを見て、保存形式の regressions を同時に捕まえる。
5. Go 側 frontmatter path まで source と runtime の両面で確認する  
   TS だけ green で終わらせない。

### 12.4 `runGatewayStartSmoke()` で徹底的にやること

1. flaky failure を「環境差」で片付けない  
   log order assertion 失敗と sidecar `ETIMEDOUT` は、どちらも default `npm test` の release blocker として扱う。
2. ログ期待値を現実に合わせて固定する  
   `Auto-Rebuild skipped: GEMINI_API_KEY not set` が並び順で揺れるなら、順序を厳密に取るべき箇所と、存在確認で十分な箇所を分ける。
3. sidecar 起動待ちを deterministic にする  
   port open / socket ready / watcher.start 完了のどれを readiness と見なすかを test 側で固定する。
4. 1回 PASS で終わらせない  
   最低でも連続複数回 green を取って、suite stability を確認する。

### 12.5 ドキュメント更新で徹底的にやること

1. 親文書を先に更新する  
   release verdict は必ず [v0.4.3_release_readiness_assurance_plan.md](/d:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/docs/plans/v0.4.x/v0.4.3_release_readiness_assurance_plan.md) を正本にする。
2. 子3本は親に従わせる

3. `Status` と `Release sign-off` を分ける  
   実装完了と release 可否は別物として書く。
4. test 結果は build / targeted proof / suite stability を分けて書く

## 13. やらないべきこと

### 13.1 テストでやらないこと

- source code に文字列があるだけで Gate PASS 扱いにしない
- helper の再実装を「integration test」と呼ばない
- `setTimeout(50)` を置いただけで real timer proof と言い切らない
- synthetic markdown を自前で書いただけで end-to-end persistence proof と言い切らない
- 1回だけ通った flaky test を fixed とみなさない

### 13.2 実装でやらないこと

- Gate を閉じるためだけに queue protocol 全面改修へ広げない
- 新しい config を増やして論点を拡散しない
- release gate の不足を、ログ抑制や assertion 緩和でごまかさない
- `gateway_start` の不安定さを別 issue に逃がして v0.4.3 を先に出さない
- `currentMessages` / `lastProcessedLength` / `surprise` の既存契約をテストなしで再設計しない

### 13.3 ドキュメントでやらないこと

- 親が `NO-GO` のまま子だけ `Release sign-off: OK` にしない
- `helper proof green` を `integration proof complete` と言い換えない
- `build green` と `release ready` を同義にしない
- 「ほぼ大丈夫」「実質完了」で曖昧に締めない

### 13.4 リリース判断でやらないこと

- Gate A/B/C のうち1つでも弱いまま、他が通ったから出す
- default `npm test` が不安定なまま、manual spot-check だけで release を決める
- 再現しない失敗を「たまたま」と扱って記録を残さない
