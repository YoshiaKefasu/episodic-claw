## 1. 問題定義

### 1.1 挙動 #4: 同一チャンクのダブル生成に見える挙動

| 項目 | 内容 |
|------|------|
| 深刻度 | 🟢 **挙動**（Bug ではない） |
| 影響 | 品質ゲート5回失敗→cache.retryの再キューが「ダブル生成」に見える |
| E2E証拠 | log:3324 `Enqueued 2 chunks`, log:3349-3351 `5 attempts exhausted` |

**GPT 5.3分析**:
- log:3324 — `Enqueued 2 chunks ... tokens: 85673`。2つのチャンクが別IDでleaseされている（同一チャンク重複ではない）
- log:3349-3351 — chunk 0001 が品質ゲート5回失敗→`cache.retry`で再キュー
- これが「ダブル生成」に見える主因

**判定**: 期待通りの動作。品質ゲートが正しく機能している証拠。ただし、Bug #1（英語DEFAULT）のためモデルが日本語会話を英語で物語化しようとして品質ゲートに連続失敗しやすくなっている。v0.4.19a 修正後は再試行回数も減るはず。

### 1.2 改善 #4: resolvePrompt() の観測性向上

| 項目 | 内容 |
|------|------|
| 深刻度 | 🟢 **改善** |
| 影響 | プロンプト解決の失敗原因がログから分かりにくい |

**現状**: `resolvePrompt()` 失敗時に `[Episodic Memory] Failed to read prompt file: /home/kasou_yoshia/...` とだけ出力される。どのパスを試したか、なぜ失敗したかが分からない。

**v0.4.19a で先取り済み**: v0.4.19a の Fix 1 に観測性向上が含まれている（候補リスト全失敗時に試行パス一覧を出力、クロスプラットフォーム解決成功時にログ出力）。本プランでは追加の観測性改善を検討する。

- 結果として「ダブル生成に見える挙動」も自然に減少する

### 2.2 改善: 設定読み込み時の構造ダンプログ + 設定値ログ

v0.4.19a Fix 0 で `api.pluginConfig` を `loadConfig()` に渡すようになったが、設定伝播の成否をログから確認できるようにする。

**構造ダンプログ**（起動時1回だけ出力）:

```typescript
// index.ts — v0.4.19a Fix 0 の直後に追加（v0.4.19a のコードは既に実装済み）
// const openClawGlobalConfig = ...  ← v0.4.19a で既に追加済み
// const pluginSpecificConfig = api.pluginConfig || ...  ← v0.4.19a で既に追加済み
// const cfg = loadConfig(pluginSpecificConfig);  ← v0.4.19a で既に追加済み

// [v0.4.19c] Config structure diagnostic — helps debug config propagation issues
// [v0.4.19c rev.2] Fixed: hasEpisodicClawInGlobal path was missing .entries layer (always returned false).
// Removed hasRecallQueryDirect (plugin-specific key never exists at global config root → always false = YAGNI violation).
// Added hasConfigOverrides: detects if user has set any plugin config values.
const configStructure = {
  hasPluginConfig: !!api.pluginConfig,
  pluginConfigKeys: api.pluginConfig ? Object.keys(api.pluginConfig) : [],
  hasPluginsInGlobal: !!openClawGlobalConfig?.plugins,
  hasEpisodicClawInGlobal: !!openClawGlobalConfig?.plugins?.entries?.["episodic-claw"]?.config,
  hasConfigOverrides: Object.keys(pluginSpecificConfig).length > 0,
  topLevelKeys: typeof openClawGlobalConfig === "object" ? Object.keys(openClawGlobalConfig).slice(0, 20) : [],
};
console.log(
  `[Episodic Memory] Config structure: ${JSON.stringify(configStructure)}`
);
```

**設定値ログ**（構造ダンプの直後）:

```typescript
console.log(
  `[Episodic Memory] Config loaded: recallQueryRecentMessageCount=${cfg.recallQueryRecentMessageCount}, ` +
  `narrativeSystemPrompt=${cfg.narrativeSystemPrompt ? `(custom, ${cfg.narrativeSystemPrompt.length} chars)` : "(default)"}, ` +
  `model=${cfg.openrouterModel}`
);
```

**設計判断**:
- **トップレベルキーは最初の20個まで出力**（巨大な設定オブジェクトのログ溢れを防止）
- **構造ダンプでは値を出力しない**（APIキー等の機密情報漏洩を防止）
- **設定値ログでは主要設定値のみを出力**（`recallQueryRecentMessageCount`、`narrativeSystemPrompt` の長さ、`openrouterModel`）。APIキー等は出力しない
- 起動時に1回だけ出力（毎ターン出力はノイズになる）
- `hasPluginConfig` / `pluginConfigKeys` を追加 — v0.4.19a Fix 0 の `api.pluginConfig` が提供されているかを確認可能にする
- **[rev.2] `hasEpisodicClawInGlobal` のパス修正**: `plugins["episodic-claw"]` → `plugins.entries["episodic-claw"].config`（実際のコード index.ts L490-491 と同一パスに。旧パスは1段ネストが欠落し常に `false` を返していた）
- **[rev.2] `hasRecallQueryDirect` 削除 → `hasConfigOverrides` に置換**: `recallQueryRecentMessageCount` はプラグイン固有キーであり OpenClaw グローバル設定直下には存在しないため、常に `false` を返し診断価値がなかった。代わりに `Object.keys(pluginSpecificConfig).length > 0` で「ユーザーが何か設定したか」を確認（YAGNI + 実用性の両立）

### 2.3 改善: narrative worker の chunk 処理サマリログ

narrative-queue.ts の `splitIntoChunks()` で、分割結果のサマリをログ出力する。

> **⚠️ ソースコード照合に基づく修正**: `splitIntoChunks()` には2つのリターンパスがある:
> - 単一チャンクパス (L71-93): `totalTokens <= HARD_TOKEN_CAP` の場合、インラインで `return [{ ... }]` する。`chunks` 変数は未定義。
> - 複数チャンクパス (L96-192): `const chunks: CacheQueueItem[] = []` で配列を構築し `return chunks` する。
>
> 単一チャンク時は境界分割が発生していないため、サマリログの情報価値がない。
> **複数チャンクパスの `return chunks;` 直前のみ**にサマリログを追加する（YAGNI + KISS）。

```typescript
// narrative-queue.ts — L191 の return chunks; 直前に追加（複数チャンクパスのみ）
console.log(
  `[Episodic Cache] Split summary: ${chunks.length} chunk(s), ` +
  `totalTokens=${totalTokens}, ` +
  `roleBoundarySplits=${chunks.filter(c => {
    const firstLine = c.rawText.split("\n")[0];
    return detectRoleLine(firstLine) !== null;
  }).length}/${chunks.length} start at role boundary`
);
```

**設計判断**:
- 各チャンクの最初の行が role ラベル行かどうかを集計する
- role境界開始率が 100% でない場合、会話境界ミスアラインの残存を示唆する
- v0.4.19b の Fix 5b と組み合わせて効果を観測する
- **単一チャンク時はサマリログを出力しない**: 1チャンクの場合は境界分割が発生していないため、「1/1 role boundary」は観測ノイズになる

### 🛤️ Solution Options

#### Option A — 複数チャンクパスのみにサマリログ追加 *(推奨)*

**Approach**: `splitIntoChunks()` の複数チャンクパス（L96 以降）の `return chunks;` 直前にのみサマリログを追加。単一チャンクはロギング不要（1チャンク=境界分割の概念が無意味）。  
**Implementation cost**: Low（3行追加）  
**Risk**: Low  
**Why recommended**: 単一チャンク時のサマリログ（「1 chunk, X tokens, 1/1 role boundary」）は情報価値がほぼゼロ。複数チャンク時の境界アライン率だけが観測すべき指標。YAGNI + KISS に適合。  
**Concrete steps**:
1. `narrative-queue.ts:191`（`return chunks;` の直前）にサマリログを追加
2. 単一チャンクパス（L71-93）には追加しない
3. プランの§2.3に「単一チャンク時はサマリログを出力しない」旨を追記

#### Option B — 両パスにサマリログ追加

**Approach**: 単一チャンクパスでも `return` の前にサマリログを追加。変数名 `chunks` の代わりにローカル変数を使う。  
**Implementation cost**: Low（5行追加、早期リターン前の2箇所）  
**Risk**: Low  
**When to choose this instead**: 全チャンク分割イベントを網羅的にログ記録したい場合。ただし単一チャンクの「1/1 role boundary」は観測ノイズになる。

### ✅ Pro Recommendation

> **Choose Option A**: 単一チャンク時は境界分割が発生していないため、サマリログの情報価値がない。複数チャンク時のみロギングすることで、YAGNI に適合しログノイズを最小化する。推定実装時間: 10分。
