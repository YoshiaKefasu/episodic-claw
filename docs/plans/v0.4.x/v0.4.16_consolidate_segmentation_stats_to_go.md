# 実装プラン: v0.4.16 — セグメンテーション統計の Go 一元化（Welford 削除）

---

## 0. このドキュメントの位置づけ

- 目的: TypeScript 側の Welford オンライン統計（`segCount`/`segMean`/`segM2`）を削除し、セグメンテーション統計を Go サイドカーの EMA（永続化済み）に一元化する
- 深刻度: **HIGH** — λ が **永久に 1.5 に固定**される実行中バグの修正。計算上の意図閾値 `mean + 2.0×std` に対し、実際は `mean + 1.5×std` で過剰分割状態。既存ユーザーの行動変化（エピソードが長くなる）を伴うため、リリースノートでの告知が必要
- 発見元: v0.4.15 レビュー中の `segmentationWarmupCount` 永続化分析 → 深掘りで `updateSegStats()` 未呼び出しを発見
- ソースコード検証: **済**（2025-07-18, 再検証 2025-07-19）
- **⚠️ 2025-07-19 慎重検証による大幅改訂**: 初版では「再起動後の10ターンだけλ=1.5」と記載していたが、実際は `updateSegStats()` に呼び出し元が存在せず `segCount` が永久に0のまま → λは**常に1.5**。また Go側のウォームアップλキャップは無意味であることが判明（Goはウォームアップ中λを無視するため）。プランを全面的に修正

## 1. 問題定義

### 1.1 二重統計の現状

| レイヤー | 統計量 | アルゴリズム | 永続化 | 用途 | ウォームアップ判定 |
|----------|--------|-------------|--------|------|-------------------|
| **TypeScript** (`segmenter.ts`) | `segCount`, `segMean`, `segM2` | Welford（正確なオンライン統計） | ❌ メモリのみ | `getEffectiveLambda()` のみ | `segCount < 10` |
| **Go sidecar** (`main.go`) | `st.Count`, `st.Mean`, `st.Variance` | EMA（指数移動平均、ウィンドウ=50） | ✅ Pebble DB | 境界判定の閾値計算（`mean + lambda × std`） | `st.Count < warmup` |

### 1.2 実行中バグ：λ が永久に 1.5 に固定（再起動後「だけ」ではない）

**初版の誤り**: 元の分析では「再起動後の10ターンだけλ=1.5」と記載していたが、
これは**誤り**だった。実際のバグはもっと深刻である。

```
根本原因: updateSegStats() に呼び出し元が存在しない

  segCount = 0  ← 永久に変化しない（updateSegStats のみインクリメントするが、呼び出し元なし）
  getEffectiveLambda() → if (segCount < 10) return Math.min(1.5, 2.0) → 常に 1.5

  TS → Go: lambda = 1.5 (全ターンで常に)
  Go:     threshold = mean + (1.5 × std)   ← 本来は mean + 2.0 × std のはず

  影響: ウォームアップ終了後も閾値が低く、過剰分割状態が続く
  （エピソードが意図より短く細かく分割される）
```

**証明**: `segmenter.ts` 全文検索で `this.updateSegStats` の呼び出しは0件。
定義(L157)はあるが、`processTurn()` 等のどこからも呼ばれない。
結果として `segCount` は初期値0のまま不変。

### 1.4 影響範囲の調査結果

- `updateSegStats()` の呼び出し元: **なし**（定義はあるが `processTurn()` 等のどこからも呼ばれない）
- `getSegStd()` の呼び出し元: **なし**
- `shrinkSegStats()` の呼び出し元: **なし**
- `segCount`/`segMean`/`segM2` の読み取り元: **`getEffectiveLambda()` のみ**（segCountは常に0のため1.5を返すのみ）

### 1.5 Go側ウォームアップ中のλ取扱い（重要）

Go側の `handleSegmentScore()` において、ウォームアップ中は **λ を閾値計算に使用しない**:

```go
// L1374-1382
reason := "warmup"
inWarmup := st.Count < warmup
if !inWarmup {              // ← ウォームアップ中はこのブロック全体をスキップ
    threshold = mean + (lambda * std)
    ...
    reason = "surprise-z"
}
```

ウォームアップ中:
- `threshold = fallback` (0.2 固定)
- `isBoundary = false` (強制)
- λ は閾値計算に一切影響しない

**結論**: 初版プランで提案した「Go側にウォームアップλキャップ (`if inWarmup && lambda > 1.5`)** を追加する変更は **無意味** — Go は既にウォームアップ中にλを無視しているため。

## 2. アルゴリズム比較と選定理由

### 2.1 Welford vs EMA

| 観点 | Welford (TS) | EMA (Go) | 優位性 |
|------|-------------|----------|--------|
| 統計モデル | 定常分布を仮定（全履歴に等重み） | 非定常対応（指数減衰：最近のデータが重要） | **EMA** — 会話は非定常プロセス |
| 減衰機構 | `shrinkSegStats()` — 50%一括カット（粗いハック） | `alpha = 2/(eff+1)` — 自然な指数減衰 | **EMA** — 滑らかな適応 |
| 永続化 | ❌ メモリのみ | ✅ Pebble DB | **EMA** — 一貫性あり |
| 分散推定精度 | 正確（Welford は数値的に安定） | 近似（`alpha*delta²` は真のEMA分散より微小ズレ） | Welford — ただし境界閾値用途では実差ゼロ |
| 境界判定 | ❌ 行わない | ✅ 実際の境界判定を行う | **EMA** — 全ロジックを集約 |

**結論: Go EMA が優位。Welford は削除対象。**

### 2.2 EMA 分散近似の妥当性

Go側の EMA分散更新:
```go
prev.Variance = (1 - alpha) * (prev.Variance + (alpha * delta * delta))
```

理論上の正確なEMA分散は `(x - new_mean)` との交差項を含むが、
`alpha << 1` の場合 `old_mean ≈ new_mean` なので差は無視可能。
境界閾値 `mean + lambda * std` の用途では、`lambda=2.0` 時の閾値誤差は <<0.1%。
**実用上の問題なし。**

## 5. リスク評価

- **リスク**: **中**。TS側の削除は安全（デッドコード）だが、λ=1.5→2.0 への変更は **既存ユーザーのセグメンテーション行動を変化させる**
- **ユーザー影響**:
  - 新規ユーザー: **変更なし** — ウォームアップ中はGo側がλを無視するため（fallback閾値使用）
  - 既存ユーザー: **行動変化あり** — ウォームアップ終了後の閾値が `mean + 1.5×std` → `mean + 2.0×std` に上昇 → **エピソードが長くなる（分割が減る）**
  - この変化は「バグ修正による意図通りの動作への復帰」だが、ユーザー体感としては「以前より分割されにくくなった」と感じる可能性あり
- **ロールバック（第一優先: 設定変更）**: ユーザー設定 `segmentationLambda: 1.5` を指定すれば、コード変更なしで旧体感に即座に復元可能。Go側は送られたλをそのまま閾値計算に使用するため、1.5を送れば `mean + 1.5×std` に戻る
- **ロールバック（第二優先: コード復元）**: TS側3フィールド+4メソッドの復元（`this.getEffectiveLambda()` に戻す）で λ=1.5 に復元可能
- **リリース告知必須**: CHANGELOG またはリリースノートで「セグメンテーション閾値の計算バグを修正。ウォームアップ終了後のエピソードが以前より長くなる場合があります」と明記すること

### 6.3 行動変化の詳細分析

**旧動作（バグ）**: λ=1.5 固定 → `threshold = mean + 1.5×std`
**新動作（修正後）**: λ=2.0（デフォルト） → `threshold = mean + 2.0×std`

閾値の差: `0.5×std` — Surprise スコアの標準偏差が典型的に 0.1〜0.3 の範囲であるため、
閾値の上昇は 0.05〜0.15 程度。これは「少し分割されにくくなる」程度の変化であり、
極端な動作変化ではないが、ユーザーが明確に体感する可能性がある。

## 8. 既知の関連バグ：`turnSeq` / `LastBoundaryTurn` 再起動不整合

> **スコープ外**（v0.4.16 では修正しない）だが、v0.4.16 の挙動評価を歪める可能性があるため文書化する。

### 8.1 問題の概要

| 変数 | 定義場所 | 再起動後 | 永続化 |
|------|----------|----------|--------|
| `turnSeq` | `segmenter.ts` L33 | **0にリセット** | ❌ メモリのみ |
| `LastBoundaryTurn` | `segstate.go` L21 → Pebble DB | **前回値を維持** | ✅ Pebble |

### 8.2 再現シナリオ

```
再起動後の最初のセッション:

  TS: turnSeq = 1 (0から再開)
  Go: st.LastBoundaryTurn = 100 (Pebbleから復元)

  Go cooldown判定:
    params.Turn(1) - st.LastBoundaryTurn(100) = -99
    -99 <= cooldown(2) → true → isBoundary = false (強制)

  結果: 再起動後、cooldown が負値でも成立してしまい、
        境界判定が過剰に抑制される
```

### 8.3 Go側のコード（`main.go` L1390-1394）

```go
if isBoundary && cooldown > 0 && st.LastBoundaryTurn > 0 && params.Turn > 0 {
    if params.Turn-st.LastBoundaryTurn <= cooldown {
        isBoundary = false
        reason = "cooldown"
    }
}
```

負値の差分（`-99 <= 2`）が条件を満たしてしまうため、再起動直後は境界が過剰に抑制される。

### 8.4 影響範囲

- **v0.4.16 の評価への影響**: v0.4.16 導入後に「分割が減った」と観測された場合、その変化の一部は λ=1.5→2.0 によるものではなく、この cooldown 過剰抑制によるものの可能性がある。評価を正確に行うには、再起動直後の数ターン（cooldown が自然解除されるまで）を除外する必要がある
- **長期的な影響**: 再起動のたびにcooldownが過剰抑制されるが、`st.Count` が増加すると `LastBoundaryTurn` が新しいターン番号で更新され、不一致は自然解消する。実害は軽微

### 8.5 推奨される修正方針（v0.4.17 以降）

```go
// 修正案: 負値の差分は cooldown 対象外とする
if params.Turn-st.LastBoundaryTurn <= cooldown && params.Turn > st.LastBoundaryTurn {
    isBoundary = false
    reason = "cooldown"
}
```

または、TS側の `turnSeq` をファイル等に永続化して再起動後も継続させる。
