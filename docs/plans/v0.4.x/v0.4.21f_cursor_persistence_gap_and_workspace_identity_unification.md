## 2. F1: handleIdleFlush の cursor 復元未永続化

### 2.1 問題の詳細

`handleIdleFlush()` (segmenter.ts L132-157) の制御フロー:

```
1. savedLastProcessedLength = this.lastProcessedLength   // 保存
2. this.lastProcessedLength = currentMessages.length     // 一時進行
3. handleSegmentBoundary()                               // セグメント処理
4. this.lastProcessedLength = savedLastProcessedLength   // ← 復元
5. this.buffer = []                                      // クリア
   // ❌ persistCursor() / maybePersistCursor() なし
```

**失敗シナリオ**:
1. DB cursor = 117、in-memory cursor は 120 に進んでいる
2. idle flush が発火 → step 2 で cursor が currentLength に進む → step 4 で 120 に復元
3. **この時点で in-memory = 120、DB = 117**
4. プロセス再起動 → `restoreCursor()` → DB から 117 を読む → messages 118-120 が再処理される → **重複ナラティブ**

### 2.2 なぜ今まで気付かなかったか

v0.4.21c は `maybePersistCursor` を通常ターン（`processTurn`）の全出口に追加したが、**idle flush は非同期の別カーソルパス**であり、通常ターンの出口網羅から漏れていた。

idle flush は cursor を一時的に進めてから元に戻すという特殊な制御フローを持ち、`processTurn` と異なり「cursor が変わったら persist」という単純なルールが適用できないと誤って想定されていた。しかし実際は step 4 の復元後の cursor 値も DB と異なり得るため、永続化が必要。

### 2.3 修正仕様

`handleIdleFlush` の `lastProcessedLength` 復元後に `persistCursor()` を追加する。

```typescript
// segmenter.ts handleIdleFlush() 内
this.lastProcessedLength = savedLastProcessedLength;
this.buffer = [];
// [v0.4.21f] Persist cursor after idle flush restoration — the restored value
// may differ from the DB value if normal turns had advanced the cursor since last persist.
this.persistCursor(agentWs, agentId);
```

**スコープ確認**: `handleIdleFlush(agentWs: string, agentId: string)` は既に両パラメータを
メソッド引数として受け取っている（L132）。idle timer のコールバック（L122-124）は
closure でこれらを capture しているため、1行追加で対応可能。

**`persistCursor` を選択する理由**:
- F1 は「idle flush 後に再起動すると cursor が退行し得る」穴なので、**復元直後に確定書き込み**する方が意図に一致する
- idle flush は通常ターンほど高頻度ではなく、1回の同期 write コストは運用上許容できる
- `maybePersistCursor` の debounce は「書き込み遅延」を意図的に残すため、F1 の確実な穴閉じとしては不十分

### 2.4 エッジケース

| ケース | 挙動 | 正しいか |
|--------|------|----------|
| idle flush 後、cursor = DB 値と同じ | `persistCursor` は1回実行（冪等な上書き） | ✅ |
| idle flush 後、cursor ≠ DB 値 | `persistCursor` で即時同期 | ✅ |
| idle flush 後、即座に processTurn が発火 | 追加の debounce persist が走る可能性はあるが機能上無害 | ✅ |
| `agentWs` / `agentId` が空文字 | `persistCursor` が早期 return | ✅（既存ガード） |

### 2.5 テスト観点

- [ ] idle flush 発火後、DB cursor が in-memory 値と一致すること（再起動で退行しないこと）
- [ ] idle flush 後に `state.set` が1回発行されること（空ID時は除く）
- [ ] `agentWs` 空文字時の早期 return が機能すること

## 3. F2: lastNarrativeByAgent の workspace 非分離

### 3.1 問題の詳細

```typescript
// narrative-worker.ts L238, L240
private lastNarrativeByAgent = new Map<string, { episodeId: string; body: string }>();
private knownAgentIds = new Set<string>();
```

`lastNarrativeByAgent` の key が `agentId` のみで、`agentWs` を含まない。

**失敗シナリオ**:
1. Workspace A (agentId=`main`) でナラティブ生成 → `lastNarrativeByAgent.set("main", episodeA)`
2. Workspace B (agentId=`main`) でナラティブ生成 → `lastNarrativeByAgent.set("main", episodeB)` → **episodeA を上書き**
3. Workspace A で次回ナラティブ生成 → `narrativizeWithModel()` が episodeB を前回ナラティブとして読む → **cross-workspace continuity bleed**

**補足（knownAgentIds）**:
- `knownAgentIds` は `cacheLeaseNext(workerId, agentId, ...)` の入力集合として使われるため、
  現行契約では `agentId` 単位のまま維持する方が安全。
- workspace 分離が必要な本体は `lastNarrativeByAgent` 側。

### 3.2 影響範囲の全特定

`lastNarrativeByAgent` の参照箇所（現コード）:
- L307: `initContinuity()` — `this.lastNarrativeByAgent.set(agentId, ...)`
- L432: `narrativizeWithModel()` — `this.lastNarrativeByAgent.get(item.agentId)`
- L611: `saveNarrative()` — `this.lastNarrativeByAgent.set(item.agentId, ...)`

`knownAgentIds` の参照箇所（契約確認）:
- L303: `initContinuity()` — `this.knownAgentIds.add(agentId)`
- L335: `processNextFromCache()` — `Array.from(this.knownAgentIds)` を `cacheLeaseNext(..., agentId, ...)` へ渡す

### 3.3 修正仕様

`lastNarrativeByAgent` の key のみ `agentWsHash(agentWs):agentId` 形式に統一する。
（`knownAgentIds` は `cacheLeaseNext` 契約に合わせて `agentId` のまま維持）

```typescript
// 変更前
private lastNarrativeByAgent = new Map<string, { episodeId: string; body: string }>();
private knownAgentIds = new Set<string>();

// 参照箇所の変更
// initContinuity: workspace 分離キーで continuity を格納
const identityKey = `${agentWsHash(agentWs)}:${agentId}`;
this.lastNarrativeByAgent.set(identityKey, { episodeId: result.episodeId, body: result.body });

// narrativizeWithModel: 前回ナラティブ参照も同じ identityKey を使う
const prevKey = `${agentWsHash(item.agentWs)}:${item.agentId}`;
const previous = this.lastNarrativeByAgent.get(prevKey);

// saveNarrative: 更新時も同じ identityKey
const saveKey = `${agentWsHash(item.agentWs)}:${item.agentId}`;
this.lastNarrativeByAgent.set(saveKey, { episodeId: `narrative-${now}`, body: result.text });

// knownAgentIds は agentId のまま（cacheLeaseNext 契約維持）
this.knownAgentIds.add(agentId);
```

### 3.4 メモリ影響

- key が `agentId`（数文字）→ `agentWsHash(agentWs):agentId`（hash 値 + 区切り + agentId）に延長
- 通常 1〜5 workspace × 1〜5 agent = 最大 25 エントリ
- 各 key の延長は高々 20 バイト程度 → **無視できる**

### 3.5 整合性確認

v0.4.21e で統一した identity model との整合:

| レイヤー | key 形式 | v0.4.21f で変更 |
|----------|----------|-----------------|
| state DB (cursor/save-hash) | `agent:${agentId}:ws:${agentWsHash(agentWs)}:...` | なし（既に統一済み） |
| in-memory save-hash (scopedKey/prefix/agentKey) | `${agentWsHash(agentWs)}:${agentId}:...` | なし（v0.4.21e で統一済み） |
| in-memory lastNarrativeByAgent | `${agentWsHash(agentWs)}:${agentId}` | **F2 で統一** |
| in-memory knownAgentIds | `${agentId}`（lease 契約キー） | 変更なし（契約維持） |

### 3.6 テスト観点

- [ ] 複数 workspace で同じ agentId を使用した際、continuity が混入しないこと
- [ ] initContinuity が workspace A の前回ナラティブを workspace B に渡さないこと
- [ ] 単一 workspace の従来動作が変わらないこと

## 4. F3: TS 側 stateGet の error 潰し

### 4.1 問題の詳細

```typescript
// rpc-client.ts L606, L626
const raw = await this.stateGet(key, agentWs, agentId).catch(() => "");
```

v0.4.21c で Go 側 `state.get` は DB read エラーを RPC error として返すよう修正したが、
TS 側の `.catch(() => "")` は**実エラーも not-found も同じ空文字**として処理する。

**失敗シナリオ**:
1. state DB が一時的に破損/IO エラー
2. Go 側 → RPC error response（ログにもエラー出力）
3. TS 側 → `.catch(() => "")` で空文字に変換 → 「cursor なし」として扱う
4. `bootstrapCursor(0, "warm-start")` → 全メッセージ再処理 → 重複ナラティブ

**なぜ完全修正が難しいか**:
- stateGet の戻り値は `string`（JSON 文字列 or 空文字）という既存契約
- エラーを上位に伝播すると、呼び出し側全てのエラーハンドリングが必要になる → 変更範囲が大きい
- 現状の「空文字 = not-found 扱い」でも自己修復は動く（重複ナラティブは出るがクラッシュはしない）

### 4.2 修正仕様（段階的改善）

**最小対応**: `.catch(() => "")` に console.warn を追加して、DB エラーと not-found を区別してログ出力する。動作は変えない（空文字扱いは維持）が、**観測性を向上**させる。

```typescript
// rpc-client.ts getSegmenterCursor
const raw = await this.stateGet(key, agentWs, agentId).catch((err) => {
  // [v0.4.21f] Distinguish DB errors from not-found for observability.
  // Behavior is unchanged (return empty string), but now errors are logged
  // instead of silently swallowed.
  // [v0.4.21f] Defensive normalization: catch variable may be unknown.
  const msg = err instanceof Error
    ? (err.stack || err.message)
    : String(err);
  if (msg && msg !== "undefined") {
    console.warn(`[episodic-claw] stateGet failed for key "${key}": ${msg}. Falling back to empty state.`);
  }
  return "";
});
```

```typescript
// rpc-client.ts getNarrativeSaveHashes
const raw = await this.stateGet(key, agentWs, agentId).catch((err) => {
  const msg = err instanceof Error
    ? (err.stack || err.message)
    : String(err);
  if (msg && msg !== "undefined") {
    console.warn(`[episodic-claw] stateGet failed for key "${key}": ${msg}. Falling back to empty state.`);
  }
  return "";
});
```

### 4.3 将来改善案

- **v0.4.22**: stateGet の戻り値を `{ value: string; error?: Error }` に変更し、呼び出し側でエラーに応じた分岐を可能にする
- 現状は KISS 原則に従い、ログ出力のみの最小対応とする

### 4.4 テスト観点

- [ ] DB read エラー時に console.warn が出力されること
- [ ] not-found 時に console.warn が出力されないこと — not-found は stateGet が空文字を返すだけで throw しないため .catch 自体が発火しない。DB エラー時のみ throw → .catch 発火 → console.warn 出力
- [ ] 動作が従来と変わらないこと（空文字 fallback）

## 6. リスク分析

| リスク | 確率 | 影響 | 軽減策 |
|--------|------|------|--------|
| F1: idle flush 後に毎回1回の永続化が走る | 低 | Sync I/O 微増 | idle flush は低頻度。退行バグ解消効果が上回る |
| F1: idle flush 後の cursor 値が DB 値と同じ場合の冪等 write | 中 | 意味上は変化なし | 冪等な上書きとして許容。必要なら将来 `lastPersistedCursor` キャッシュで最適化 |
| F2: agentWsHash 計算の CPU コスト増 | 極低 | hash 計算が3箇所追加 | DJB2 は O(path長)、数バイト〜数十バイトの文字列。無視できる |
| F2: workspace 分離後に既存の lastNarrativeByAgent エントリが読めなくなる | 中 | 初回再起動で continuity がリセット | 現行動作と同じ（プロセス再起動で in-memory は消える）ため実質影響なし |
| F3: catch 引数が非 Error でログ欠落 | 低 | ログ欠落 | `err instanceof Error ? ... : String(err)` で文字列化して出力 |

   - 根拠: `src/narrative-worker.ts:303,335`（knownAgentIds の用途） / `:432,611`（continuity参照・更新の実体）。

3. **F3 のコード例を `unknown` 安全な catch へ補強**
   - `err instanceof Error ? ... : String(err)` で型安全にログ化。
