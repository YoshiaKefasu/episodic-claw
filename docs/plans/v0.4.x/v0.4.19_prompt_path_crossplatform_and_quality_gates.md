### 1.2 Bug #1 の詳細 — 英語出力の根本原因

**現状の `resolvePrompt()`**（config.ts L130-138）:

```typescript
function resolvePrompt(value: string | undefined): string {
  if (!value) return "";
  const trimmed = value.trim();
  if (trimmed.endsWith(".md") || trimmed.endsWith(".txt")) {
    try {
      return fs.readFileSync(path.resolve(trimmed), "utf8").trim();
    } catch {
      console.warn(`[Episodic Memory] Failed to read prompt file: ${trimmed}`);
      return "";
    }
  }
  return trimmed;
}
```

**問題**: `path.resolve("/home/kasou_yoshia/.openclaw/prompts/episodic_claw_narrative_system.md")` は Windows では:
- `C:\home\kasou_yoshia\.openclaw\prompts\episodic_claw_narrative_system.md`（cwd ベース解決）
- または `D:\home\kasou_yoshia\...`（ドライブレター依存）
- → ファイルが存在せず `catch` で `""` を返す

**E2E検証の証拠**:

| シナリオ | promptSystemPath | resolved_prompt_len | fallback_used | detected_language |
|---|---|---|---|---|
| Linux path | `/home/kasou_yoshia/.openclaw/prompts/...` | 0/0 | true/true | unknown |
| Windows path | `Y:\kasou_yoshia\.openclaw\prompts\...` | 34/737 | false/false | **ja** |

**追加証拠**: 英語エピソード `dennouaibou-system-development.md` の先頭が、`DEFAULT_USER_PROMPT_TEMPLATE` の例文 `Late that evening at his desk, he pored over the logs` と一致。これは英語DEFAULTプロンプトが使われた決定的証拠。

### 1.3 Bug #2 の詳細 — 段落区切り消失

`episode-3f29971b1081bec875cdf3d910eb8280.md`:
- 1238トークンにも関わらず1行・1段落
- 句読点の後にスペースだけが続き、改行が全くない
- 品質ゲート `checkNarrativeFormat()` は段落数を検査していない
- カスタム日本語プロンプトにも「一つの連続した本文」と書いてある（§1.4 も参照）

**E2E対照**: Windows path シナリオで生成されたエピソードは `paragraph_count=4`, `line_count=7`。段落区切りは維持されている。これはカスタムプロンプトが正しく読み込まれた場合の正常な挙動。

**推定原因**: 英語DEFAULTプロンプトでは `One continuous narrative body` と指示しているため、モデルが段落区切りを全く入れない。日本語カスタムプロンプトでは「一つの連続した本文」だが、日本語モデルは文脈に応じて自然に段落区切りを入れる傾向があるため、日本語プロンプトが効いていれば段落は維持されるはず。

### 1.4 設定ズレ #3 の詳細

`openclaw.json` L840 で `recallQueryRecentMessageCount: 2` が設定されているが、ログ L3315 で `recentMsgs=4` が出力されている。

**想定される原因**:

1. **A. index.ts の `loadConfig()` 呼び出し時の引数**: `api.runtime?.config?.loadConfig?.()` が返すグローバル設定全体を `loadConfig()` に渡している。`EpisodicPluginConfig` の `recallQueryRecentMessageCount` は **プラグイン設定** の中にあるはずだが、`loadConfig()` に渡される `rawConfig` がグローバル設定全体の場合、`rawConfig?.recallQueryRecentMessageCount` が `undefined` になり、デフォルト値 `4` が使われる
2. **B. 別の呼び出しパス**: `recallQueryRecentMessageCount` は `retriever.ts` でも参照され、そちらは `cfg` からではなく直接値を渡している可能性

**確認が必要**: `api.runtime?.config?.loadConfig?.()` が返すオブジェクトの構造。`openclaw.json` の `plugins["episodic-claw"].config` なのか、グローバル直下なのか。

### 1.5 Bug #5 の詳細 — チャンク分割の会話論理境界ミスアライン

**問題**: 2層の原因が重なっている。

**層1: `buildFlushItem()` が role 情報を消失**

```typescript
// narrative-pool.ts — buildFlushItem()
const rawText = this.buffer
  .map((m) => extractText(m.content))  // ← m.role が捨てられる
  .filter(Boolean)
  .join("\n");
```

結果: `rawText` は純粋なテキストの羅列になり、どこがユーザーの発話でどこがアシスタントの応答か区別できない。

**層2: `splitIntoChunks()` が会話構造を認識できない**

```typescript
// narrative-queue.ts — splitIntoChunks()
const lines = rawText.split("\n");
for (const line of lines) {
  if (currentTokens + lineTokens > SOFT_TOKEN_TARGET && currentLines.length > 0) {
    pushChunk(currentLines, chunkIndex);  // ← 任意の行境界で切断
    // role 境界（user→assistant 遷移）を認識できない
  }
}
```

**具体例** — ユーザーの質問とアシスタントの応答が別チャンクに分断:

```
チャンク1:
  ...会話の続き...
  ユーザー: 次のステップは何？   ← ユーザーの質問だけ残る

チャンク2:
  アシスタント: 次は...          ← アシスタントの応答が別チャンクに
  ...新しい会話...
```

モデルはチャンク1の「次のステップは何？」という質問に対する応答を持たず、チャンク2は「次は...」という応答の文脈を欠く。結果として**物語の論理構造が崩壊**する。

**既存の `PoolFlushItem` には `messages: Message[]` が存在**するが、`splitIntoChunks()` は `rawText: string` しか受け取らないため、会話構造情報が使われていない。

### 2.2 Fix 1: resolvePrompt() クロスプラットフォームパス解決

**現行** (config.ts L130-138):

```typescript
function resolvePrompt(value: string | undefined): string {
  if (!value) return "";
  const trimmed = value.trim();
  if (trimmed.endsWith(".md") || trimmed.endsWith(".txt")) {
    try {
      return fs.readFileSync(path.resolve(trimmed), "utf8").trim();
    } catch {
      console.warn(`[Episodic Memory] Failed to read prompt file: ${trimmed}`);
      return "";
    }
  }
  return trimmed;
}
```

**問題**: `path.resolve("/home/kasou_yoshia/...")` は Windows では `/home/` を相対パスとして扱い、cwd ベースで解決する。結果として存在しないパスになる。

**変更後**:

```typescript
/**
 * Resolve a prompt value that may be inline text, a file path, or a ~/ path.
 * Cross-platform: handles Linux absolute paths (/home/...), Windows paths (Y:\...),
 * and home-dir shortcuts (~/...) on all platforms.
 */
function resolvePrompt(value: string | undefined): string {
  if (!value) return "";
  const trimmed = value.trim();
  if (!trimmed.endsWith(".md") && !trimmed.endsWith(".txt")) {
    return trimmed;  // inline text
  }

  // Build a list of candidate paths to try in order
  const candidates: string[] = [];

  // 1. Resolve ~ / ~/path — works on all platforms
  if (trimmed.startsWith("~/") || trimmed.startsWith("~\\")) {
    const homeDir = process.env.HOME || process.env.USERPROFILE || os.homedir();
    candidates.push(path.join(homeDir, trimmed.slice(2)));
  } else if (trimmed === "~") {
    const homeDir = process.env.HOME || process.env.USERPROFILE || os.homedir();
    candidates.push(homeDir);
  }

  // 2. path.resolve(trimmed) — handles Windows absolute (C:\...) and relative paths
  candidates.push(path.resolve(trimmed));

  // 3. Cross-platform: If original is a POSIX absolute path (/home/user/...) and
  //    we're on Windows, try resolving under the Windows user home directory.
  //    This handles the case where openclaw.json was written on Linux and the
  //    plugin runs on Windows (WSL, dual-boot config share, etc.)
  if (process.platform === "win32" && trimmed.startsWith("/home/")) {
    const homeDir = process.env.HOME || process.env.USERPROFILE || os.homedir();
    // /home/kasou_yoshia/.openclaw/... → C:\Users\kasou_yoshia\.openclaw\... or
    // the actual HOME-based path
    const relativeFromHome = trimmed.slice("/home/".length);  // kasou_yoshia/.openclaw/...
    const firstSlash = relativeFromHome.indexOf("/");
    if (firstSlash > 0) {
      // Skip the username segment and join from the path after it
      const afterUsername = relativeFromHome.slice(firstSlash + 1);  // .openclaw/...
      candidates.push(path.join(homeDir, afterUsername));
    }
    // Also try: if HOME itself is like Y:\kasou_yoshia, the path might be
    // Y:\kasou_yoshia\.openclaw\... — already covered by the above if
    // HOME = Y:\kasou_yoshia and afterUsername = .openclaw/prompts/...
    // → Y:\kasou_yoshia\.openclaw\prompts\...
  }

  // 4. Try the original path as-is (some Windows Git Bash setups can read /home/... paths)
  candidates.push(trimmed);

  // Try each candidate in order; first successful read wins
  for (const candidate of candidates) {
    try {
      const content = fs.readFileSync(candidate, "utf8").trim();
      if (content.length > 0) {
        return content;
      }
    } catch {
      // Continue to next candidate
    }
  }

  // All candidates failed — log for observability
  console.warn(
    `[Episodic Memory] Failed to read prompt file: ${trimmed}` +
    ` (tried: ${candidates.join(", ")})`
  );
  return "";
}
```

**追加import**: `import * as os from "os";`（config.ts の先頭に追加）

**解決パスの具体例**:

| 入力 | Windows での候補リスト | 期待される結果 |
|------|----------------------|---------------|
| `/home/kasou_yoshia/.openclaw/prompts/system.md` | `C:\Users\kasou_yoshia\.openclaw\prompts\system.md`, `D:\GitHub\...\home\kasou_yoshia\...`, `Y:\kasou_yoshia\.openclaw\prompts\system.md`（HOME=`Y:\kasou_yoshia`の場合）, 元パス | 3番目がヒット |
| `Y:\kasou_yoshia\.openclaw\prompts\system.md` | `Y:\kasou_yoshia\.openclaw\prompts\system.md`（path.resolve直接） | 1番目がヒット |
| `~/.openclaw/prompts/system.md` | `Y:\kasou_yoshia\.openclaw\prompts\system.md`（HOME展開） | 1番目がヒット |
| inline text | (ファイルパスとして処理しない) | そのまま返す |

**重要な設計判断**:
- `os.homedir()` / `process.env.HOME` / `process.env.USERPROFILE` の3ソースから home を決定
- `/home/USERNAME/` の USERNAME セグメントをスキップして home 直下にマッピングする（Linux の `/home/kasou_yoshia/` → Windows の `%HOME%\`）。これは最も一般的な設定共有パターン（Linux/WSL と Windows で同じ `~/.openclaw/` ディレクトリを使う）
- 全候補失敗時は、試行したパスのリストをログ出力する（デバッグ容易化）

### 2.3 Fix 2: checkNarrativeFormat() Gate 6 — 段落構造ゲート

**現行**: `checkNarrativeFormat()` は5つのGateを持つが、段落数・行数の検査がない。1238トークンの1ブロック平文が品質ゲートを通過する。

**追加**: Gate 6 — 最小段落構造要件

```typescript
// Gate 6: Minimum paragraph structure
// Narrative output with 500+ characters should have at least 2 paragraphs
// (separated by \n\n). Single-paragraph wall-of-text indicates model failure
// to structure output — common with free models producing "stream of consciousness".
const MIN_CHARS_FOR_PARAGRAPH_CHECK = 500;
const MIN_PARAGRAPH_COUNT = 2;

if (text.length >= MIN_CHARS_FOR_PARAGRAPH_CHECK) {
  // Count paragraph breaks: sequences of \n\n (possibly with whitespace)
  const paragraphBreaks = text.split(/\n\s*\n/);
  if (paragraphBreaks.length < MIN_PARAGRAPH_COUNT) {
    return { pass: false, reason: `narrative-format: single-paragraph wall-of-text (${text.length} chars, ${paragraphBreaks.length} paragraph(s)). Minimum ${MIN_PARAGRAPH_COUNT} paragraphs required for texts over ${MIN_CHARS_FOR_PARAGRAPH_CHECK} chars.` };
  }
}
```

**設計判断**:
- **500文字未満は除外**: 短い物語（1段落でも完結する内容）を誤弾しない
- **段落区切り定義**: `\n\s*\n` — 空行（空白のみの行を含む）で区切られたテキストブロック
- **MIN_PARAGRAPH_COUNT = 2**: 最低2段落。これは非常に緩い要件。日本語の自然な物語では3〜5段落が普通
- **偽陽性リスク**: 500文字以上の1段落物語が弾かれる。しかし、500文字以上で段落区切りが全くないのは、モデルが「stream of consciousness」で出力した兆候であり、品質ゲートで弾くのが適切

### 2.4 Fix 3: 設定伝播ズレの修正

**現状のデータフロー** (index.ts L481-482):

```typescript
const openClawGlobalConfig = api.runtime?.config?.loadConfig?.() || {};
const cfg = loadConfig(openClawGlobalConfig);
```

`api.runtime?.config?.loadConfig?.()` が返すのは **グローバル設定全体**（`openclaw.json` のルートオブジェクト）。`loadConfig()` は `rawConfig?.recallQueryRecentMessageCount` を見るが、実際の設定値は `rawConfig?.plugins?.["episodic-claw"]?.config?.recallQueryRecentMessageCount` にある可能性がある。

**ただし**: `openclaw.plugin.json` の `configSchema` を通じて、OpenClaw本体がプラグイン設定を抽出して `loadConfig()` に渡す可能性もある。この場合、`rawConfig` は既にプラグイン設定に絞り込まれている。

**修正方針**: `loadConfig()` に渡す前に、グローバル設定からプラグイン設定を抽出する。既にプラグイン設定が抽出済みの場合はそのまま通す。

```typescript
/**
 * Extract the episodic-claw plugin config from the global OpenClaw config.
 * OpenClaw may pass either the full config or just the plugin config,
 * depending on the version. This function handles both cases.
 */
function extractPluginConfig(globalConfig: any): any {
  if (!globalConfig || typeof globalConfig !== "object") return {};

  // If the object already looks like a plugin config (has known field names),
  // it's likely already extracted by OpenClaw
  const hasPluginFields = (
    typeof globalConfig.recallQueryRecentMessageCount === "number" ||
    typeof globalConfig.openrouterApiKey === "string" ||
    typeof globalConfig.narrativeSystemPrompt === "string" ||
    typeof globalConfig.dedupWindow === "number" ||
    typeof globalConfig.maxPoolChars === "number"
  );
  if (hasPluginFields) return globalConfig;

  // Try to extract from plugins["episodic-claw"].config
  const plugins = globalConfig.plugins || globalConfig.openclawPlugins;
  if (plugins && typeof plugins === "object") {
    const pluginEntry = plugins["episodic-claw"];
    if (pluginEntry && typeof pluginEntry === "object") {
      return pluginEntry.config || pluginEntry;
    }
  }

  // Fallback: return as-is (might work if OpenClaw already extracts)
  return globalConfig;
}
```

`index.ts` の呼び出し箇所を変更:

```typescript
const openClawGlobalConfig = api.runtime?.config?.loadConfig?.() || {};
const pluginRawConfig = extractPluginConfig(openClawGlobalConfig);
const cfg = loadConfig(pluginRawConfig);
```

**観測性向上**: 設定の反映状況をログ出力:

```typescript
console.log(
  `[Episodic Memory] Config loaded: recallQueryRecentMessageCount=${cfg.recallQueryRecentMessageCount}, ` +
  `narrativeSystemPrompt=${cfg.narrativeSystemPrompt ? `(custom, ${cfg.narrativeSystemPrompt.length} chars)` : "(default)"}, ` +
  `model=${cfg.openrouterModel}`
);
```

### 2.6 Fix 5a: buildFlushItem() に role ラベル付与

**現行** (narrative-pool.ts — `buildFlushItem()`):

```typescript
const rawText = this.buffer
  .map((m) => extractText(m.content))  // ← m.role が捨てられる
  .filter(Boolean)
  .join("\n");
```

**問題**: ユーザーの質問とアシスタントの応答が区別できないテキストの羅列になる。

**変更後**:

```typescript
private buildFlushItem(reason: PoolFlushItem["reason"], surprise: number, agentWs: string, agentId: string): PoolFlushItem {
  const rawText = this.buffer
    .map((m) => {
      const text = extractText(m.content);
      if (!text) return "";
      // [v0.4.19] Role labels for conversation-boundary-aware chunking + narrative model context
      // Format: "role: text" — only for primary conversation roles
      const role = m.role === "user" || m.role === "assistant" ? m.role : null;
      return role ? `${role}: ${text}` : text;
    })
    .filter(Boolean)
    .join("\n");

  return {
    messages: [...this.buffer],
    rawText,
    surprise,
    reason,
    agentWs,
    agentId,
  };
}
```

**設計判断**:
- **ラベル対象**: `user` と `assistant` のみ。`tool_use` は既に `[Tool Used: ...]` 形式でラベル付き、`toolResult`/`tool_result` は EXCLUDED_ROLES で除外済み
- **フォーマット**: `role: text` — シンプルでLLMが自然に理解できる形式。`[user]` や `<user>` 等のタグ形式はトークン効率が悪い
- **後方互換**: role ラベルのない既存キャッシュアイテムは `splitIntoChunks()` で「role ラベルなし = 1行1メッセージ」として従来通り処理される（Fix 5b 参照）
- **charCount への影響**: role ラベルの文字数（`user: ` = 6文字、`assistant: ` = 11文字）は charCount に含まれない。charCount はプールフラッシュ判定用であり、rawText の長さとは独立しているため、修正不要

**rawText の具体例**:

```
user: テスト送るね
assistant: バッチリ聞こえてるぜ
user: 次はv0.4.9のリリースノート書いて
assistant: 了解、Cherry-Pickアップデートの内容も含めるよ
[Tool Used: code_searcher]
assistant: リリースノートの草案ができた
```

### 2.7 Fix 5b: splitIntoChunks() 会話境界対応分割

**現行** (narrative-queue.ts — `splitIntoChunks()`):

```typescript
const lines = rawText.split("\n");
for (const line of lines) {
  if (currentTokens + lineTokens > SOFT_TOKEN_TARGET && currentLines.length > 0) {
    pushChunk(currentLines, chunkIndex);  // ← 任意の行境界で切断
  }
}
```

**問題**: `\n` 行単位でしか分割ポイントを探せないため、user/assistant の会話ターンの途中で切断される。

**変更後** — 会話境界を優先分割ポイントとして認識:

```typescript
/**
 * [v0.4.19] Detect if a line is a role-labeled conversation boundary.
 * Returns the role prefix ("user" or "assistant") or null.
 */
function detectRoleLine(line: string): "user" | "assistant" | null {
  if (line.startsWith("user: ")) return "user";
  if (line.startsWith("assistant: ")) return "assistant";
  return null;
}

/**
 * Split raw text into chunks that fit within the 64K token limit.
 * [v0.4.19] Conversation-boundary-aware: prefers splitting at role transitions
 * (user→assistant boundaries) over arbitrary line boundaries.
 */
export function splitIntoChunks(
  rawText: string,
  agentWs: string,
  agentId: string,
  source: CacheQueueItem["source"],
  reason: CacheQueueItem["reason"],
  surprise: number,
): CacheQueueItem[] {
  const parentIngestId = `ingest-${Date.now().toString(36)}-${Math.random().toString(36).slice(2, 8)}`;
  const totalTokens = estimateTokens(rawText);

  // Single chunk — no split needed
  if (totalTokens <= HARD_TOKEN_CAP) {
    _chunkCounter++;
    const now = new Date().toISOString().replace(/[:.]/g, "-");
    const orderKey = `${now}-${String(_chunkCounter).padStart(4, "0")}`;
    return [{ /* same as before */ }];
  }

  // Split into multiple chunks — conversation-boundary-aware
  const chunks: CacheQueueItem[] = [];
  const lines = rawText.split("\n");
  let currentLines: string[] = [];
  let currentTokens = 0;
  let lastRoleInChunk: string | null = null;  // [v0.4.19] Track last role in current chunk

  const pushChunk = (lines: string[], chunkIndex: number) => { /* same as before */ };

  let chunkIndex = 0;
  for (const line of lines) {
    const lineTokens = estimateTokens(line);
    const lineRole = detectRoleLine(line);  // [v0.4.19]

    // [v0.4.19] Conversation-boundary-aware split:
    // When we need to split (soft target exceeded), prefer splitting at role transitions.
    // Specifically: prefer splitting BEFORE a "user:" line (i.e., keep Q&A pairs together).
    // This prevents a user question from being separated from its assistant response.
    if (currentTokens + lineTokens > SOFT_TOKEN_TARGET && currentLines.length > 0) {
      // If the next line is a role transition (user: or assistant:),
      // this is an ideal split point — split now.
      // Otherwise, if the next line is NOT a role line, we still split at the line boundary
      // (hard fallback) but log a warning about misaligned boundary.
      if (!lineRole) {
        // Non-role line: we're splitting mid-turn. Log for observability.
        console.log(
          `[Episodic Cache] Chunk split at non-conversation boundary ` +
          `(chunkIndex=${chunkIndex}, lastRole=${lastRoleInChunk}). ` +
          `Consider reducing maxPoolChars for smaller chunks.`
        );
      }
      pushChunk(currentLines, chunkIndex);
      chunkIndex++;
      currentLines = [];
      currentTokens = 0;
      lastRoleInChunk = null;
    }

    // Hard cap: force split even mid-line if needed (same as before)
    if (lineTokens > HARD_TOKEN_CAP) { /* same as before */ }

    currentLines.push(line);
    currentTokens += lineTokens;
    if (lineRole) lastRoleInChunk = lineRole;  // [v0.4.19] Track role
  }

  // Push remaining (same as before)
  if (currentLines.length > 0) {
    pushChunk(currentLines, chunkIndex);
  }

  return chunks;
}
```

**設計判断**:
- **分割ポイントの優先順位**: role ラベル行 (`user:` / `assistant:`) が来たタイミングで既に SOFT_TARGET を超えている場合 → role 境界で分割。role ラベル行でない場合 → 従来通り行境界で分割（フォールバック）
- **`user:` の前で分割する理由**: Q&Aペアを同じチャンクに保つため。`user: 質問` → `assistant: 応答` のセットが同一チャンクに入る確率が最も高い
- **後方互換**: role ラベルなしの既存キャッシュアイテムでは `detectRoleLine()` が常に `null` を返すため、全ての行が「非role行」として扱われ、従来と全く同じ動作になる
- **ログ出力**: 非role境界での分割時に警告ログを出力。これは `maxPoolChars` の調整指標になる

### 2.8 Fix 5 の影響範囲

**Go sidecar への影響**: なし。Go sidecar は `rawText` を opaque string として格納するのみ。`batchIngest` は `summary` フィールド（物語化結果）を保存し、`rawText` はキャッシュキューのメタデータとしてのみ存在する。

**既存キャッシュアイテムへの影響**: なし。Fix 5b の `detectRoleLine()` は role ラベルなしの行に対して `null` を返すため、既存のラベルなしキャッシュアイテムは従来通り処理される。

**narrative model への影響**: ポジティブ。role ラベルにより、モデルは会話の構造（誰が何を言ったか）を正確に把握できる。これは物語化の品質向上に直結する。特に日本語カスタムプロンプトで「ユーザーの発話とアシスタントの応答を区別して物語化する」ような指示が可能になる。

**DEFAULT プロンプトへの影響**: 修正不要。`DEFAULT_USER_PROMPT_TEMPLATE` の `The text below is raw material` は role ラベル付きテキストにも適用される。モデルは自然に `user:` / `assistant:` を解釈する。ただし、カスタム日本語プロンプトでは role ラベルを活用した指示を追加できる（per_agent_narrative_customization.md で対応）。

## 7. リスク評価

- **リスク**: **低〜中** — Fix 1〜3 は低リスク。Fix 5 は rawText フォーマットの変更を伴うため中リスクだが、後方互換性を維持する
- **後方互換性**: ✅
  - Fix 1: `resolvePrompt()` の候補リストの最初は `path.resolve(trimmed)` なので、Windows絶対パス・相対パスはこれまでと同じ挙動
  - Fix 2: 500文字未満の物語はGate 6の対象外。既存の短いエピソードには影響しない
  - Fix 3: `extractPluginConfig()` は「既に抽出済み」の場合はそのまま通すので、OpenClawが正しく設定を渡している環境では何も変わらない
  - Fix 5a: role ラベルは `buildFlushItem()` でのみ追加。`messages[]` は変更なし。Go sidecar は rawText を opaque string として扱う
  - Fix 5b: `detectRoleLine()` は role ラベルなしの行に `null` を返すため、既存キャッシュアイテムは従来通り処理される
- **Fix 5 の追加リスク**: 
  - rawText フォーマット変更により、Go sidecar が rawText を解析している場合に影響する可能性。ただし、Go sidecar は rawText を opaque string として PebbleDB に格納するのみであり、解析は行わない
  - `echo detection` (`checkEchoDetection`) は rawText の先頭 5000 文字と照合する。role ラベル (`user: ` / `assistant: `) が付与されることで、echo 検出のホワイトスペース collapsing に影響する可能性がある。ただし、role ラベルは固定プレフィックス（6〜11文字）であり、echo 検出は whitespace collapse 後の比較なので、実質的な影響はない（`user:` と `assistant:` は whitespace collapse でそのまま残る）
  - 非常長の assistant メッセージ（HARD_TOKEN_CAP 超過）の分割は、role ラベルが付与されていても中間強制分割が発生する。これは既存の動作と同じ
- **偽陽性リスク**: Gate 6 により、500文字以上の1段落物語が弾かれる。これは意図的 — 500文字超の1ブロック平文は品質問題の兆候
- **偽陰性リスク**: `/home/` パスマッピングが `HOME` に依存する。`HOME` が設定されていない場合はクロスプラットフォーム解決が働かない。ただし `os.homedir()` がフォールバックとして機能するため、実質的に問題ない
- **ロールバック**: 各Fixは独立してリバート可能。Fix 5a のみリバートした場合、rawText は role ラベルなしに戻り、Fix 5b の `detectRoleLine()` は全ての行に `null` を返すため、従来の分割動作に自動的にフォールバックする

### ⚠️ 注意事項

- **`/home/USERNAME/` → HOME マッピングの精度**: Linux の `/home/kasou_yoshia/` を Windows の `%HOME%` にマッピングする際、`/home/` の次のセグメント（ユーザー名）をスキップする。これは Linux と Windows でユーザー名が異なる場合でも機能する（例: `/home/kasou_yoshia/` → `Y:\kasou_yoshia\` でも `C:\Users\Kasou\` でも、HOME 配下の `.openclaw/` が同じなら解決する）。ただし、HOME 配下に `.openclaw/` が存在しない場合は解決できない — この場合は候補リストの全てが失敗し、フォールバックとして `""`（DEFAULT英語プロンプト）が使われる。これは現状より悪くはない
- **Gate 6 の閾値**: `MIN_CHARS_FOR_PARAGRAPH_CHECK = 500` と `MIN_PARAGRAPH_COUNT = 2` は暫定値。実運用で偽陽性が多ければ調整する。特に日本語の短い物語（300〜600文字）では1段落でも完結する内容があり得るため、600文字以上に引き上げる可能性もある
- **Fix 5 の role ラベルフォーマット**: `role: text` 形式を選択したが、もし会話内容自体に `user: ` や `assistant: ` で始まる行が含まれる場合、`detectRoleLine()` が誤検出する可能性がある。ただし、これは非常に稀なケース（ユーザーが意図的に `user: ` で始まるメッセージを送る場合）。軽減策: `detectRoleLine()` は行の先頭のみをチェックする（`startsWith`）ため、文中の `user:` は誤検出しない
- **Fix 5b の分割ロジックの限界**: 現在の実装は「次の行が role 行なら role 境界で分割」のみ。より高度な「Q&Aペアを同じチャンクに保つ」ロジック（ユーザーの質問 + アシスタントの応答を1セットとして扱う）は実装していない。これは KISS 原則に基づく — ペアリングロジックは複雑で、SOFT_TARGET の制約下では必ずしもペアを維持できるとは限らない（assistant の応答が非常に長い場合など）。将来的に per_agent_narrative_customization.md で高度な分割戦略を検討する
