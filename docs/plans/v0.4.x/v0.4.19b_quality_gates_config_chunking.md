### 1.3 Bug #5: チャンク分割の会話論理境界ミスアライン

| 項目 | 内容 |
|------|------|
| 深刻度 | 🟠 **Bug** |
| 影響 | ユーザーの質問とアシスタントの応答が別チャンクに分断され、モデルが文脈を失う |
| E2E証拠 | `buildFlushItem()` が `extractText()` のみ → role消失; `splitIntoChunks()` が `\n` 行単位でしか分割ポイントを探せない |

**2層の原因**:

**層1: `buildFlushItem()` が role 情報を消失**

```typescript
// narrative-pool.ts — buildFlushItem()
const rawText = this.buffer
  .map((m) => extractText(m.content))  // ← m.role が捨てられる
  .filter(Boolean)
  .join("\n");
```

**層2: `splitIntoChunks()` が会話構造を認識できない**

```typescript
// narrative-queue.ts — splitIntoChunks()
const lines = rawText.split("\n");
for (const line of lines) {
  if (currentTokens + lineTokens > SOFT_TOKEN_TARGET && currentLines.length > 0) {
    pushChunk(currentLines, chunkIndex);  // ← 任意の行境界で切断
  }
}
```

**具体例** — ユーザーの質問とアシスタントの応答が別チャンクに分断:

```
チャンク1:
  ...会話の続き...
  ユーザー: 次のステップは何？   ← ユーザーの質問だけ残る

チャンク2:
  アシスタント: 次は...          ← アシスタントの応答が別チャンクに
  ...新しい会話...
```

**既存の `PoolFlushItem` には `messages: Message[]` が存在**するが、`splitIntoChunks()` は `rawText: string` しか受け取らないため、会話構造情報が使われていない。

### 2.3 Fix 2b: checkNarrativeFormat() Gate 6 — 段落構造ゲート

**追加**: Gate 6 — 最小段落構造要件

```typescript
// Gate 6: Minimum paragraph structure
// Narrative output with 500+ characters should have at least 2 paragraphs
// (separated by \n\n). Single-paragraph wall-of-text indicates model failure
// to structure output — common with free models producing "stream of consciousness".
const MIN_CHARS_FOR_PARAGRAPH_CHECK = 500;
const MIN_PARAGRAPH_COUNT = 2;

if (text.length >= MIN_CHARS_FOR_PARAGRAPH_CHECK) {
  // Count paragraph breaks: sequences of \n\n (possibly with whitespace)
  const paragraphBreaks = text.split(/\n\s*\n/);
  if (paragraphBreaks.length < MIN_PARAGRAPH_COUNT) {
    return { pass: false, reason: `narrative-format: single-paragraph wall-of-text (${text.length} chars, ${paragraphBreaks.length} paragraph(s)). Minimum ${MIN_PARAGRAPH_COUNT} paragraphs required for texts over ${MIN_CHARS_FOR_PARAGRAPH_CHECK} chars.` };
  }
}
```

**設計判断**:
- **500文字未満は除外**: 短い物語（1段落でも完結する内容）を誤弾しない
- **段落区切り定義**: `\n\s*\n` — 空行（空白のみの行を含む）で区切られたテキストブロック
- **MIN_PARAGRAPH_COUNT = 2**: 最低2段落。日本語の自然な物語では3〜5段落が普通
- **偽陽性リスク**: 500文字以上の1段落物語が弾かれる。しかし、500文字以上で段落区切りが全くないのは、モデルが「stream of consciousness」で出力した兆候

### 2.5 Fix 5a: buildFlushItem() に role ラベル付与

**現行** (narrative-pool.ts — `buildFlushItem()`):

```typescript
const rawText = this.buffer
  .map((m) => extractText(m.content))  // ← m.role が捨てられる
  .filter(Boolean)
  .join("\n");
```

**変更後**:

```typescript
private buildFlushItem(reason: PoolFlushItem["reason"], surprise: number, agentWs: string, agentId: string): PoolFlushItem {
  const rawText = this.buffer
    .map((m) => {
      const text = extractText(m.content);
      if (!text) return "";
      // [v0.4.19] Role labels for conversation-boundary-aware chunking + narrative model context
      // Format: "role: text" — only for primary conversation roles
      const role = m.role === "user" || m.role === "assistant" ? m.role : null;
      return role ? `${role}: ${text}` : text;
    })
    .filter(Boolean)
    .join("\n");

  return {
    messages: [...this.buffer],
    rawText,
    surprise,
    reason,
    agentWs,
    agentId,
  };
}
```

**設計判断**:
- **ラベル対象**: `user` と `assistant` のみ。`tool_use` は既に `[Tool Used: ...]` 形式でラベル付き、`toolResult`/`tool_result` は EXCLUDED_ROLES で除外済み
- **フォーマット**: `role: text` — シンプルでLLMが自然に理解できる形式。`[user]` や `<user>` 等のタグ形式はトークン効率が悪い
- **後方互換**: role ラベルのない既存キャッシュアイテムは `splitIntoChunks()` で「role ラベルなし = 1行1メッセージ」として従来通り処理される（Fix 5b 参照）
- **charCount への影響**: role ラベルの文字数（`user: ` = 6文字、`assistant: ` = 11文字）は charCount に含まれない。charCount はプールフラッシュ判定用であり、rawText の長さとは独立しているため、修正不要

**rawText の具体例**:

```
user: テスト送るね
assistant: バッチリ聞こえてるぜ
user: 次はv0.4.9のリリースノート書いて
assistant: 了解、Cherry-Pickアップデートの内容も含めるよ
[Tool Used: code_searcher]
assistant: リリースノートの草案ができた
```

### 2.6 Fix 5b: splitIntoChunks() 構造化チャンク分割（Option A）

**現行** (narrative-queue.ts — `splitIntoChunks()`):

```typescript
export function splitIntoChunks(
  rawText: string, agentWs: string, agentId: string,
  source: string, reason: string, surprise: number
): NarrativeChunk[] {
  const lines = rawText.split("\n");
  for (const line of lines) {
    if (currentTokens + lineTokens > SOFT_TOKEN_TARGET && currentLines.length > 0) {
      pushChunk(currentLines, chunkIndex);  // ← 任意の行境界で切断
    }
  }
}
```

**変更後** — `messages?` オプションを追加し、構造化データがあれば role 境界を直接判定、なければ `detectRoleLine()` フォールバック:

```typescript
export function splitIntoChunks(
  rawText: string, agentWs: string, agentId: string,
  source: string, reason: string, surprise: number,
  messages?: Message[]  // [v0.4.19] Option A: structured conversation-boundary-aware chunking
): NarrativeChunk[] {
```

**detectRoleLine()** — rawText フォールバック用（cold-start/gap-archive パス）:

```typescript
/**
 * [v0.4.19] Detect if a line is a role-labeled conversation boundary.
 * Returns the role prefix ("user" or "assistant") or null.
 * Used as fallback when messages[] is not available (cold-start, gap-archive).
 */
function detectRoleLine(line: string): "user" | "assistant" | null {
  if (line.startsWith("user: ")) return "user";
  if (line.startsWith("assistant: ")) return "assistant";
  return null;
}
```

**構造化パス** — `messages` が提供された場合:

```typescript
if (messages && messages.length > 0) {
  // Structured path: use messages[].role directly for boundary detection.
  // Build line→role mapping from messages, then use it in the split loop.
  const lineToRole = new Map<string, "user" | "assistant" | null>();
  for (const m of messages) {
    const text = extractText(m.content);
    if (!text) continue;
    const role = m.role === "user" || m.role === "assistant" ? m.role : null;
    // Match the rawText line format: "role: text" or just "text"
    const lineKey = role ? `${role}: ${text}` : text;
    lineToRole.set(lineKey, role);
  }

  // In the split loop, use lineToRole instead of detectRoleLine()
  for (const line of lines) {
    const lineTokens = estimateTokens(line);
    const lineRole = lineToRole.get(line) ?? detectRoleLine(line);  // structured first, string fallback

    if (currentTokens + lineTokens > SOFT_TOKEN_TARGET && currentLines.length > 0) {
      if (!lineRole) {
        console.log(
          `[Episodic Cache] Chunk split at non-conversation boundary ` +
          `(chunkIndex=${chunkIndex}, lastRole=${lastRoleInChunk}). ` +
          `Consider reducing maxPoolChars for smaller chunks.`
        );
      }
      pushChunk(currentLines, chunkIndex);
      chunkIndex++;
      currentLines = [];
      currentTokens = 0;
      lastRoleInChunk = null;
    }

    if (lineTokens > HARD_TOKEN_CAP) { /* same as before */ }

    currentLines.push(line);
    currentTokens += lineTokens;
    if (lineRole) lastRoleInChunk = lineRole;
  }
} else {
  // Fallback path: rawText-only (cold-start, gap-archive, legacy)
  // Same logic as structured path but using detectRoleLine() for role detection
  for (const line of lines) {
    const lineTokens = estimateTokens(line);
    const lineRole = detectRoleLine(line);

    if (currentTokens + lineTokens > SOFT_TOKEN_TARGET && currentLines.length > 0) {
      if (!lineRole) {
        console.log(/* same warning */);
      }
      pushChunk(currentLines, chunkIndex);
      chunkIndex++;
      currentLines = [];
      currentTokens = 0;
      lastRoleInChunk = null;
    }

    if (lineTokens > HARD_TOKEN_CAP) { /* same as before */ }

    currentLines.push(line);
    currentTokens += lineTokens;
    if (lineRole) lastRoleInChunk = lineRole;
  }
}
```

**設計判断**:
- **2つのパス**: `messages` あり = 構造化パス（live-turn）、`messages` なし = フォールバックパス（cold-start/gap-archive）。ロジックは同一だが、role 判定方法が異なる
- **構造化パスの利点**: `lineToRole` Map は `messages[].role` から直接構築するため、会話内容自体が `user: ` で始まる場合でも**誤検出しない**（Discrepancy 3 の解消）
- **フォールバックパスの利点**: `Message[]` が存在しない cold-start/gap-archive でも role ラベル付き rawText から会話境界を推定できる。`detectRoleLine()` は「最善の推測」だが、完全ではない
- **`user:` の前で分割する理由**: Q&Aペアを同じチャンクに保つため。`user: 質問` → `assistant: 応答` のセットが同一チャンクに入る確率が最も高い
- **後方互換**: `messages` なしの呼び出し（archiver.ts, rpc-client.ts）はフォールバックパスに自動的に入る。role ラベルなしの既存キャッシュアイテムでは `detectRoleLine()` が常に `null` を返すため、従来と全く同じ動作になる
- **ログ出力**: 非role境界での分割時に警告ログを出力。これは `maxPoolChars` の調整指標になる
- **DRY 解消**: `PoolFlushItem.messages[].role` が single source of truth。構造化パスでは rawText への再エンコード→再パースを経ずに直接 role を参照する

## 7. リスク評価

- **リスク**: **中** — Fix 2a は DEFAULT プロンプトの変更であり、カスタムプロンプトなしで運用している全ユーザーの出力形式が変化する（中リスク）。Fix 2b は低リスク。Fix 5 は rawText フォーマットの変更を伴うため中リスクだが、後方互換性を維持する。Fix 3 は v0.4.19a に移行済み
- **後方互換性**: ✅
  - Fix 2: 500文字未満の物語はGate 6の対象外。既存の短いエピソードには影響しない
  - ~~Fix 3~~: v0.4.19a Fix 0 に移行（`api.pluginConfig` 渡し）
  - Fix 5a: role ラベルは `buildFlushItem()` でのみ追加。`messages[]` は変更なし。Go sidecar は rawText を opaque string として扱う
  - Fix 5b: `detectRoleLine()` は role ラベルなしの行に `null` を返すため、既存キャッシュアイテムは従来通り処理される
- **Fix 5 の追加リスク**:
  - `echo detection` (`checkEchoDetection`) は rawText の先頭 5000 文字と照合する。role ラベルが付与されることで影響する可能性がある。ただし、role ラベルは固定プレフィックス（6〜11文字）であり、whitespace collapse 後の比較なので実質的な影響はない
  - 非常に長い assistant メッセージ（HARD_TOKEN_CAP 超過）の分割は、role ラベルが付与されていても中間強制分割が発生する。これは既存の動作と同じ
- **偽陽性リスク**: Gate 6 により、500文字以上の1段落物語が弾かれる。これは意図的。ただし日本語の短い物語（300〜600文字）では1段落でも完結する内容があり得るため、閾値は運用で調整する可能性あり
- **ロールバック**: 各Fixは独立してリバート可能。Fix 5a のみリバートした場合、rawText は role ラベルなしに戻り、Fix 5b の `detectRoleLine()` は全ての行に `null` を返すため、従来の分割動作に自動的にフォールバックする
