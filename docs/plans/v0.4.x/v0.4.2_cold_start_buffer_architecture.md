# v0.4.2 Cold-Start Buffer Architecture

**Parent:** [v0.4.0 Narrative Architecture Roadmap](../v0.4.0_narrative_architecture_roadmap.md)
**Date:** 2026-04-09
**Base Version:** v0.4.1-hotfix
**Target Version:** v0.4.2
**Status:** 収束完了（コア基盤完了 / small gap は意図的例外 / parsing/contract smoke PASS）

> 本プランは、既存の長大な会話セッション（500kトークン超など）へ本プラグインを途中導入した際に発生する「Cold-Start Context Exceeded」問題、および「再起動時の NarrativeWorker の State揮発」問題を根本的に解決するためのアーキテクチャ刷新プランです。

---

## 1. 解決する課題

1. **巨大メッセージの一括流入 (H-5)**: プラグイン導入直後に膨大な履歴が `processTurn` へ流れ込むと、`NarrativePool` が一塊として吸収し、200k枠等のOpenRouterに送信してエラー (400 Bad Request) になる。
2. **State の揮発性と文脈断絶**: 現在の `NarrativeWorker` はキューと直前の物語 (`lastFullEpisode`) をメモリ (RAM) に持っているため、再起動時に処理待ちログが揮発し、物語の「前回からの続き」という接続も断たれてしまう。
3. **Ingest 経路の分裂**: 現在は live ingest、cold-start、pre-compaction gap archive が別ルートで動いており、どれか1本だけを直しても system 全体の整合性が取れない。

---

## 2. 新アーキテクチャ設計 (Sequential Cache-and-Drain Model)

> ユーザーから提示されたホワイトボード設計図（Ingest Messages → 新規Pebble DB Cache → Sequential 物語化 → 本番DB）に完全準拠した設計。

### 処理フロー

1. **Ingest Messages (入力と分割)**
   `processTurn` (live)、cold-start ingest、pre-compaction gap archive (large gap >50 msgs) の producer は、**同一の enqueue helper** `splitIntoChunks` + `enqueueNarrativeChunks` を呼ぶ。small gap (≤50 msgs) は既存の `batchIngestWithEscalation` を維持する（将来的に cache 化検討）。入力が規定サイズ（64K token class）を超えている場合は、**安全な単位で複数チャンクに分割**する。

2. **Sequential タイムスタンプ付与**
   分割された各チャンクには `YYYY-MM-DD HH:MM:ss-001` 等の厳格な連番タイムスタンプを付与し、本来の会話の順番を失わないようにする。順序は `agentId` ごとに保証し、agent 間の混線を防ぐ。

3. **New Pebble DB for Cache (先行保存レイヤー ＝ 全件経由)**
   64K以下の単一チャンクであっても、512Kの複数チャンクであっても、**LLMへ送る前に必ず「キャッシュ用Pebble DB」へ一旦先行保存（Enqueue）**します。
   この経路の一本化により、突然のプロセス終了やPCの電源オフでも、あらゆる処理待ちのキューが完全に永続化・保護されます。

4. **Sequential 的な物語化 (Drain ループ)**
   `NarrativeWorker` は、この Cache DB をポーリング（またはイベント駆動）し、一番古い `YYYY-MM-DD HH:MM` スタンプのチャンクから1つずつ順番に lease 取得する。
   OpenRouter で物語化を実行し、失敗時は Exponential Backoff で再試行する。一定回数を超えたジョブは削除せず `dead-letter` 状態へ遷移し、後続ジョブを永久ブロックしない。

5. **物語化 MD ファイルの最終保存**
   物語化が成功したら、生成された MD ファイルを **v0.4.2 では現行の本番DB（Pebble + go-hnsw）へ保存する**。成功確認後にだけ Cache DB 上のジョブを Ack / Pop する。Comet DB 移行は v0.5.0 の別タスクとし、本プランの commit target には含めない。

### このプランで強制的に一本化する入口

1. **Live ingest**: `processTurn -> NarrativePool -> enqueueNarrativeChunk()`
2. **Cold-start**: `gateway_start -> ingestColdStartSession -> enqueueNarrativeChunk()`
3. **Pre-compaction gap archive (large gap >50 msgs)**: `archiveUnprocessed -> enqueueNarrativeChunk()`

> **注:** small gap (≤50 msgs) は既存の `batchIngestWithEscalation` を維持。v0.4.2 では large gap からの cache 化を開始し、small gap は将来的に cache 化検討。

この主要3経路は、v0.4.2 以降 **直接 `NarrativeWorker.enqueue()` や `triggerBackgroundIndex()` を叩いてはいけない**。必ず Cache DB を経由する。

### 64K の定義

図では「64K Token」としているが、実装では tokenizer の厳密計測を各分割点で毎回走らせると重い。そこで v0.4.2 では以下の2段ガードにする。

- **Soft target**: `estimatedTokens <= 48_000`
- **Hard cap**: `estimatedTokens <= 64_000`

`estimateTokens()` は CJK を含む近似値なので、soft target を低めに置いて provider 差異のブレを吸収する。split の目的は「理論上ぴったり64Kに合わせること」ではなく、「危険な oversized payload を cache へ入れる前に確実に分割すること」に置く。

### Queue State Machine

| State | 意味 | 次状態 |
|---|---|---|
| `queued` | 未処理。lease 取得待ち | `leased`, `dead-letter` |
| `leased` | Worker が処理中。`lease_until` を持つ | `queued`, `done`, `dead-letter` |
| `done` | 本番DB保存と Ack 完了 | 削除対象 |
| `dead-letter` | retry 上限超過または poison pill | 手動回復 / 再投入 |

`Peek -> Pop` の2段だけでは crash 時に二重処理や永久ロックが起きやすいため、実装では `lease` を持つ `queued/leased/done/dead-letter` の4状態を正式仕様とする。

## 3. 実装のステップバックダウン

### Step 0: Producer の一本化
最初にやるべきは「分割ロジック」ではなく「入口の統一」。以下3箇所の producer は、共通 helper `enqueueNarrativeChunk()` 以外の queue 追加手段を持たないようにする。（small gap ≤50 msgs は既存 `batchIngestWithEscalation` を維持する。）

1. `EventSegmenter.poolAndQueue()` の live 経路
2. `ingestColdStartSession()` の既存セッション取り込み
3. `archiveUnprocessed()` の gap archive（large gap >50 msgs のみ。small gap は対象外）

この helper は以下を担当する。

1. 生ログ正規化
2. 64K class への分割
3. `CacheQueueItem[]` の構築
4. `CacheEnqueueBatch` RPC の発行
5. Enqueue 成功ログの出力

### Step 1: TypeScript 側の 64K 分割ロジック
`NarrativePool` の拡張ではなく、**共通 producer helper の内部責務** として分割する。こうすることで live / cold-start / gap archive が同じ規則で chunking される。

**📝 分割アルゴリズムの仕様:**
1. 単純な固定文字数での分割ではなく、直近の `role: "user"`、time-gap 境界、既存の surprise boundary を優先候補にする。
2. 優先境界が見つからない場合だけ、hard cap 未満の最も近い位置で強制切断する。
3. 各 chunk は `estimatedTokens` を記録して enqueue し、worker 側が再計算しなくて済むようにする。
4. 分割後も順序が壊れないよう、1つの論理入力から生成された chunk には共通 `parentIngestId` を持たせる。
5. キューに積まれるアイテムの型定義：
```typescript
export interface CacheQueueItem {
  id: string;                 // "agentId:YYYYMMDDHHMMSS-0001"
  agentWs: string;
  agentId: string;
  source: "live-turn" | "cold-start" | "gap-archive";
  parentIngestId: string;     // 同一入力から分割された chunk 群を束ねる
  orderKey: string;           // "YYYYMMDDHHMMSS-0001"
  surprise: number;
  reason: "size-limit" | "surprise-boundary" | "cold-start-import" | "gap-archive" | "force-flush";
  rawText: string;
  estimatedTokens: number;
  status: "queued" | "leased" | "dead-letter";
  attempts: number;
  createdAt: string;
  updatedAt: string;
  leaseOwner?: string;
  leaseUntil?: string;
  lastError?: string;
}
```

### Step 2: Go Sidecar の Cache DB 実装 (New Pebble DB)
既存の `episodes.db` (Pebble) とは別に、一時キュー用途に特化した軽量な `episodes/cache.db` を新設する。WAL を有効にし、value は JSON で保持する。key は順序ソートしやすい `agentId:orderKey` を使う。

**📝 RPC Schema (go/internal/rpc):**
```go
// 1. EnqueueBatch (TS -> Go)
type CacheEnqueueBatchReq struct {
    Items []CacheQueueItem `json:"items"`
}

// 2. LeaseNext (TS -> Go)
// 最古の queued job を leased 状態に更新して返す。
type CacheLeaseNextReq struct {
    WorkerID     string `json:"workerId"`
    AgentID      string `json:"agentId"`
    LeaseSeconds int    `json:"leaseSeconds"`
}

type CacheLeaseNextResp struct {
    Item *CacheQueueItem `json:"item"` // 無ければ null
}

// 3. Ack (TS -> Go)
// 本番DB保存成功後に done として削除。
type CacheAckReq struct {
    ID       string `json:"id"`
    WorkerID string `json:"workerId"`
}

// 4. Retry / Release (TS -> Go)
// 失敗時は attempts を増やし queued に戻す。上限超過時は dead-letter。
type CacheRetryReq struct {
    ID          string `json:"id"`
    WorkerID    string `json:"workerId"`
    Error       string `json:"error"`
    BackoffSec  int    `json:"backoffSec"`
    MaxAttempts int    `json:"maxAttempts"`
}

// 5. LatestNarrative (TS -> Go)
// agent 単位で直近の narrative エピソードを返す。
type GetLatestNarrativeReq struct {
    AgentWs string `json:"agentWs"`
    AgentID string `json:"agentId"`
}

type GetLatestNarrativeResp struct {
    EpisodeID string `json:"episodeId"`
    Body      string `json:"body"`
    Found     bool   `json:"found"`
}
```

**📝 DB value の最小フィールド**

1. `status`
2. `attempts`
3. `leaseOwner`
4. `leaseUntil`
5. `backoffUntil`
6. `lastError`

`leaseUntil` を過ぎた `leased` job は sidecar 起動時、および `LeaseNext` 実行時に自動回収して `queued` へ戻す。

### Step 3: NarrativeWorker の永続キューポーリング移行
`src/narrative-worker.ts` はインメモリ配列（`private queue = []`）を完全に捨て、定期的に Go 側へ `CacheLeaseNext` を投げる「Pull型ワーカー」へ改修される。queue orchestration は Cache DB に寄せ、worker は「1件取り出す -> 物語化する -> Ack/Retry」を責務に限定する。

**📝 処理サイクル:**
1. 起動直後に `GetLatestNarrative(agentWs, agentId)` を呼び、agent ごとの continuity state をロードする。
2. Worker がアイドリング中に 1秒間隔で `CacheLeaseNext` をポーリングする。
3. lease 取得後、OpenRouter へプロンプト送信する。
4. OpenRouter 単体の短期 retry は worker 内で実行する。
5. 物語化成功後、現行の `batchIngest` を呼び出し、本番DB保存が成功した場合のみ `CacheAck` を呼ぶ。
6. OpenRouter 失敗や本番DB保存失敗時は `CacheRetry` を呼び、job を再試行対象へ戻す。
7. `attempts` が上限を超えた job は `dead-letter` に移す。削除はしない。

**📝 Cold-Start 文脈復元機能:**
プロセス起動直後、`this.lastFullEpisode = ""` にして妥協（単発エピソード化）していた処理を廃止する。
ただし state は単一文字列ではなく、**agent 単位の Map** とする。

```typescript
private lastNarrativeByAgent = new Map<string, {
  episodeId: string;
  body: string;
}>();
```

起動時に既存DBから「agent ごとの最新 narrative episode」を1件取得する RPC（`GetLatestNarrative`）を呼び出し、`lastNarrativeByAgent` へロードする。`genesis-archive`、`fallback`、`gap-compacted` は continuity source から除外する。

### Step 4: Retry / Backoff / Poison Pill 方針

**短期 retry** と **ジョブ全体の retry** を分ける。

1. **短期 retry**: OpenRouter 呼び出し単体に対する 5 回 retry。worker 内で処理。
2. **ジョブ全体 retry**: `CacheRetry` で queue に戻す retry。DB で `attempts` を持つ。

**Backoff ルール**

```text
backoff = min(300s, 2^attempt)  // 指数 backoff、最大 300 秒
```

> **注:** 実装は jitter なしの純粋な指数 backoff。計画書では jitter を記載していたが、実装はシンプルに `1 << min(attempts, 8)` で最大 256s、上限 300s キャップ。

**Dead-letter 条件**

1. `attempts >= 20`
2. 4xx のうち明らかな fatal error が連続する
3. 本文が空、または正規化後に unusable と判定された

Dead-letter は「削除」ではなく「隔離」であり、管理者が後で再投入できるよう残す。

### Step 5: 既存コードとの接続点

v0.4.2 では以下を明示的に置き換える。

1. `EventSegmenter.poolAndQueue()` は `NarrativeWorker.enqueue()` を直接呼ばず `CacheEnqueueBatch` を呼ぶ。
2. `ingestColdStartSession()` は temp JSON を `triggerBackgroundIndex()` へ渡す経路を廃止し、queue item を直接 enqueue する。
3. `archiveUnprocessed()` は large gap 時の `legacy_backlog_*.json + triggerBackgroundIndex()` を廃止し、queue item を直接 enqueue する。

`triggerBackgroundIndex()` は v0.4.2 以降も残してよいが、用途は「既存 `.md` の index rebuild」など non-narrative path に限定する。

### Step 6: 起動時の crash recovery

sidecar 起動時に以下を行う。

1. `cache.db` を開く
2. `leased` だが `leaseUntil < now` の job を `queued` に戻す
3. `dead-letter` 件数を集計してログ出力
4. `GetLatestNarrative` 用に最新 narrative episode の cursor / meta を warmup

Node 側 worker は `gateway_stop` 時に 15 秒 drain を試みるが、失敗しても queue の durability に頼ってよい。ここが v0.4.0 と v0.4.2 の大きな差分になる。

## 4. 完全な可視化とトレーサビリティ (No Black-box)

本アーキテクチャでは、データの流れがバックグラウンドに移動するため、ユーザーや開発者が状態を完全に把握できるよう、強固なログ出力を義務付ける。各ステップで以下の形式のログ（`EmitLog` etc.）を必須とする：

1. **[Cache Enqueue]** `[Episodic Cache] Enqueued 3 chunks for agentWs. source=cold-start parent=cs-abc123 (tokens: 41k, 39k, 12k. queue=3)`
2. **[Cache Lease]** `[NarrativeWorker] Leased chunk [agent-1:20260409...] attempt=3 lease=60s tokens=39k`
3. **[OpenRouter Status]** `[NarrativeWorker] Attempt 1/5 failed (429). Retrying in 2000ms...`
4. **[Narration Success]** `[NarrativeWorker] Successfully narrativized chunk [YYYY-MM-DD...]. (Output: 450 tokens)`
5. **[Final Commit]** `[Episodic DB] Saved episode [ID] to main Pebble DB. Acked cache job. (remaining=2)`
6. **[Cache Retry]** `[NarrativeWorker] Returned chunk [YYYY-MM-DD...] to queue. attempts=4 backoff=32s error=429`
7. **[Dead Letter]** `[Episodic Cache] Moved chunk [YYYY-MM-DD...] to dead-letter after 20 attempts`

これによりシステムが「裏でフリーズしているのか」「APIの回復待ち（Retry中）なのか」が一切ブラックボックスにならず、常にターミナルで確認可能となる。

### 運用観測値

v0.4.2 ではログだけでなく、最低限以下の値を sidecar 側で集計できる状態にする。

1. `queued_count`
2. `leased_count`
3. `dead_letter_count`
4. `oldest_queued_age_sec`
5. `latest_success_at`
6. `latest_failure_at`
7. `latest_narrative_episode_id_by_agent`

1. cold-start 500k 相当の session が direct path ではなく cache enqueue されること

3. worker 再起動後に `leased` job が自動回収されること
4. multi-agent 環境で `lastNarrativeByAgent` が混線しないこと
5. dead-letter が生成されても後続 job が止まらないこと

### 未収束項目の対応方針（詳細）

ここから先は「何が残っているか」の列挙ではなく、**どう閉じるか** を現場判断で固定する。

#### 1. Gap archive ≤50 msgs の扱い

**現状**
- `archiveUnprocessed()` は `unprocessed.length > 50` のときだけ cache enqueue に流す。
- `<=50` は現行の `batchIngestWithEscalation()` を通る。
- つまり、設計思想は「全面 cache 化」に寄っているが、実装は「large gap のみ cache 化」で止まっている。

**Option A — v0.4.2 では small gap を現行 batchIngest のまま維持する（推奨）**

理由:
- `small gap` は payload が小さく、現行経路でも OpenRouter / batchIngest の失敗率が低い。
- ここまで cache 化すると、v0.4.2 の主目的である cold-start / large gap / restart safety から一歩外れ、検証面積だけが増える。
- YAGNI / KISS 的には、`large gap` と `cold-start` を先に閉じた今、`small gap` まで無理に一本化しなくても実害は少ない。

**v0.4.2 での正式判断**
- `small gap (<=50 msgs)` は **例外として batchIngestWithEscalation を維持** する。
- 文書上も「主要3経路は cache 化済み。small gap は intentional exception」と明記する。
- これを「未実装」ではなく「スコープ上の意図的除外」として扱う。

**この判断で必要な文書整備**
1. 「すべての producer が同一 enqueue helper」を言い切る箇所には、必ず `small gap` 例外を併記する。
2. Quick Wins の「archiveUnprocessed の producer を共通 helper に寄せる」は、`large gap のみ完了` と書き換える。
3. 成功条件から「small gap も cache 経由」を暗黙に期待させる文言を外す。

**Option B — small gap も v0.4.2 で cache 化する**

理由:
- アーキテクチャの説明はより美しくなる。
- `archiveUnprocessed()` が完全に 1 本化され、説明と実装のズレが消える。

代償:
- `batchIngestWithEscalation()` の安定経路を捨てるため、pre-compaction 周りの退避経路を新たに焼き直す必要がある。
- リリース前の検証面積が増え、得られる実益は large gap 対応ほど大きくない。

**プロの推奨**
> v0.4.2 では **Option A** を採る。  
> `small gap` は intentional exception として固定し、cache への完全統一は v0.4.3 以降の整理タスクに回す。

### Functional

1. 10K 相当の通常 live ingest が 1 chunk で enqueue -> narrativize -> batchIngest される
2. 500K 相当の cold-start session が複数 chunk に分割され、順序どおりに処理される

### Failure / Recovery

1. OpenRouter 429 を返し続けると `CacheRetry` が backoff 付きで増える
2. 20回超過で dead-letter へ移る
3. worker 処理中に gateway を kill しても、次回起動で lease 失効 job が回収される
4. `batchIngest` 失敗時に Ack されず、job が再試行される

### Continuity

1. agent A と agent B が同時に会話しても previous episode が混線しない
2. 再起動後、最新 narrative episode が continuity source として復元される
3. `genesis-archive` と `fallback` が continuity source に使われない
