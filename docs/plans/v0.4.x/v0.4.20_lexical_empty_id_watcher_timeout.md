## 2. 修正詳細

### 2.1 Fix 1: `enqueueSysLexq()` 空IDガード

**問題**: `enqueueSysLexq()` は `recordID` が空文字列でもキー `sys_lexq:{ts}:` を生成する。`lexicalWorker` は `SplitN(":", 3)` で解析するため `parts[2] = ""` となり、Bleve が "document ID cannot be empty" で拒否する。

**修正** (`store.go` L337-343):

```go
// enqueueSysLexq writes a lexical queue task directly into PebbleDB to guarantee processing.
func (s *Store) enqueueSysLexq(batch *pebble.Batch, action string, recordID string) {
	// [v0.4.20] Guard: skip empty record IDs — Bleve rejects document ID ""
	if recordID == "" {
		logger.Warn(logger.CatLexical, "enqueueSysLexq: skipped empty recordID for action=%s", action)
		return
	}
	key := []byte(fmt.Sprintf("sys_lexq:%d:%s", time.Now().UnixNano(), recordID))
	val := []byte(action)
	if batch != nil {
		_ = batch.Set(key, val, nil)
	} else {
		_ = s.db.Set(key, val, pebble.NoSync)
	}
}
```

**設計判断**:
- 空IDを黙ってスキップする理由: v0.5.0で統合インデックス化されるまでの暫定対応であり、ログで検知可能にすれば十分
- エラーを返さない理由: `Add()` / `BatchAdd()` / `deleteLocked()` / `UpdateRecord()` の全呼び出し元が `enqueueSysLexq` の戻り値を無視しているため、戻り値変更の恩恵がない

### 2.2 Fix 2: `RebuildLexicalIndex()` 空IDスキップ

**問題**: `RebuildLexicalIndex()` は `rec.SourcePath == ""` と `os.IsNotExist` はチェックするが、`rec.ID == ""` はチェックしない。空IDレコードがPebbleDBに存在すると、Fix 1 のガードがなく `enqueueSysLexq(batch, "UPDATE", "")` が実行される。

**修正** (`store.go` `RebuildLexicalIndex()` L1833付近):

```go
for iter.First(); iter.Valid(); iter.Next() {
    var rec EpisodeRecord
    if err := msgpack.Unmarshal(iter.Value(), &rec); err != nil {
        continue
    }
    // [v0.4.20] Skip records with empty ID — cannot be indexed by Bleve
    if rec.ID == "" {
        logger.Warn(logger.CatStore, "RebuildLexicalIndex: skipping record with empty ID (SourcePath=%s)", rec.SourcePath)
        continue
    }
    if rec.SourcePath == "" {
        continue
    }
    if _, err := os.Stat(rec.SourcePath); os.IsNotExist(err) {
        continue
    }
    s.enqueueSysLexq(batch, "UPDATE", rec.ID)
    count++
}
```

### 2.3 Fix 3: `Add()` / `BatchAdd()` 空ID拒否

**問題**: `Store.Add()` と `Store.BatchAdd()` は `rec.ID` が空文字列でも受け入れる。空IDレコードは:
- `ep:` キーで保存される → 他の空IDレコードと衝突
- `s2i:` キーで保存される → 同上
- `enqueueSysLexq(batch, "ADD", "")` が呼ばれる → Fix 1 で防がれるが、そもそもDBに入れるべきではない

**修正** (`store.go` `Add()` L363直後):

```go
func (s *Store) Add(ctx context.Context, rec EpisodeRecord) error {
    // [v0.4.20] Guard: reject records with empty ID
    if rec.ID == "" {
        return fmt.Errorf("episode record ID must not be empty")
    }
    // ... existing code ...
}
```

**修正** (`store.go` `BatchAdd()` L438直後):

```go
func (s *Store) BatchAdd(ctx context.Context, records []EpisodeRecord) error {
    if len(records) == 0 {
        return nil
    }
    // [v0.4.20] Guard: reject batch with any empty ID
    for i, rec := range records {
        if rec.ID == "" {
            return fmt.Errorf("episode record ID must not be empty (index %d)", i)
        }
    }
    // ... existing code ...
}
```

**影響分析**:
- `handleIngest` は slug が空になるケースで既に `Summary is empty` チェックで早期リターンしている
- `handleBatchIngest` も同様
- `runAutoRebuild` は `item.doc.Metadata.ID` を使用 — frontmatterパース後にIDが空になることはフロントマター破搈のみ
- `RunAsyncHealingWorker` は `slug` を使用 — slugは常に非空（MD5ハッシュ由来）
- **結論**: 既存の正常経路では空IDは発生しない。このガードはDB破損・マイグレーション不整合に対する防衛

### 2.4 Fix 4: `lexicalWorker` 空IDタスク安全処理

**問題**: Fix 1 が導入されれば今後空IDタスクは `sys_lexq` キューに積まれない。しかし、既存のDBに既に蓄積されている `sys_lexq:{ts}:` キーが残存する可能性がある。`lexicalWorker` は `parts[2] = ""` の場合に `bleveBatch.Index("", data)` を発行してエラーになる。

**修正** (`lexical.go` タスク解析直後、L143付近):

```go
for _, task := range tasks {
    batchKeys = append(batchKeys, task.key)
    action := task.action

    // [v0.4.20] Skip tasks with empty ID — Bleve rejects document ID ""
    if task.id == "" {
        logger.Warn(logger.CatLexical, "Skipping lexical task with empty document ID (action=%s, key=%s)", action, string(task.key))
        continue
    }

    if action == "ADD" || action == "UPDATE" {
        // ... existing code ...
```

**設計判断**:
- `continue` でスキップ → `batchKeys` には追加済みなので、コミット後にこのキーはDBから削除される
- つまり、残存する空IDタスクは次回のポーリングで消費され、二度と現れない

### 2.5 Fix 5: `ensureWatcher()` タイムアウト延長 + リトライ

**問題**: `src/index.ts` の `ensureWatcher()` は `watcher.start` RPC に5秒の `Promise.race` タイムアウトを設定している。しかし `handleWatcherStart`（`main.go`）は以下を実行する:

1. `watcher.New(1500, ...)` — ウォッチャー生成
2. `quarantineLegacyNestedEpisodeTree(path)` — レガシーツリーの隔離（ファイルI/O）
3. `w.AddRecursive(path)` — 大規模ディレクトリの再帰スキャン
4. `go getStore(ws)` — 非同期だがストア初期化を含む
5. `w.Start()` — FS ウォッチャー起動

大規模ワークスペース（数千エピソードファイル）では、ステップ2〜3だけで5秒を超過する可能性がある。

**修正** (`src/index.ts` `ensureWatcher()`):

```typescript
// [v0.4.20] Increased timeout from 5s to 15s + 1 retry
const WATCHER_START_TIMEOUT_MS = 15_000;  // was 5_000
const WATCHER_START_MAX_RETRIES = 1;

let watcherStarted = false;
for (let attempt = 0; attempt <= WATCHER_START_MAX_RETRIES; attempt++) {
  try {
    await Promise.race([
      rpcClient.startWatcher(agentWs),
      new Promise((_, reject) =>
        setTimeout(() => reject(new Error("Watcher Start Timeout")), WATCHER_START_TIMEOUT_MS)
      ),
    ]);
    watcherStarted = true;
    break;
  } catch (err) {
    if (attempt < WATCHER_START_MAX_RETRIES) {
      console.warn(`[Episodic Memory] Watcher start attempt ${attempt + 1} failed, retrying...`, err);
    } else {
      console.warn(`[Episodic Memory] Watcher start failed after ${attempt + 1} attempts.`, err);
    }
  }
}

if (!watcherStarted) {
  // Rebuild fallback path (unchanged)
  try {
    await rpcClient.rebuildIndex(agentWs, agentWs);
  } catch (rebuildErr) {
    console.error("[Episodic Memory] Rebuild fallback failed.", rebuildErr);
    console.warn("[Episodic Memory] One or more watcher workspaces are degraded; rebuild fallback is active.");
  }
}
```

**設計判断**:
- 5s → 15s: 大規模ワークスペースの `AddRecursive` + `quarantineLegacyNestedEpisodeTree` に余裕を持たせる
- リトライ1回: 一時的なリソース枯渇（inotify等）に対する保険
- 合計最大遅延: 15s × 2 = 30s（rebuildフォールバックより遥かに速い）

**⚠️ リトライ時の留意点**: `handleWatcherStart` は goroutine で実行される（`main.go` L2954）。最初の試行がTS側で15sタイムアウトしてもGo側では処理が継続し、ウォッチャーが起動完了する可能性がある。2回目のリトライで `startWatcher` RPC を送ると "Watcher already active on path" の成功レスポンスが返る。これは良性だが、実装時に「already active」ログを見ても驚かないように留意すること。

### 2.6 Fix 6: `ensureWatcher()` rebuild fallback 非同期化（起動非ブロッキング）

**問題**: 現在の `ensureWatcher()` は watcher起動失敗時に `await rpcClient.rebuildIndex(...)` を実行するため、起動処理 (`gateway_start`) が rebuild 完了まで同期ブロックされる。

`indexer.rebuild` は内部で `Clear()` + `runAutoRebuild()`（ファイル走査 + frontmatter parse + embedding API + index再構築）を行うため、負荷時に数十秒〜数分かかり得る。これを起動同期パスで待つ設計が、体感遅延と timeout 連鎖の温床になる。

**修正** (`src/index.ts` `ensureWatcher()` fallback path):

```typescript
if (!watcherStarted) {
  _singleton!.watcherWorkspaces.delete(agentWs);
  _singleton!.watcherDegradedWorkspaces.add(agentWs);
  console.warn("[Episodic Memory] Failed to start watcher; scheduling async rebuild fallback.");

  // [v0.4.20] Non-blocking fallback: do not block gateway_start on rebuild
  void (async () => {
    try {
      await rpcClient.rebuildIndex(agentWs, agentWs); // timeout remains default (120000ms)
      console.log(`[Episodic Memory] Async rebuild fallback completed for ${agentWs}.`);
      invalidateRecallCacheForWorkspace(agentWs);
    } catch (rebuildErr) {
      console.error("[Episodic Memory] Async rebuild fallback failed.", rebuildErr);
      console.warn("[Episodic Memory] One or more watcher workspaces are degraded; rebuild fallback is active.");
    }
  })();
}
```

**設計判断**:
- KISS: timeout値を大きくするより、重い処理を起動同期パスから外す方が副作用が小さい
- 既存の `rpc-client.ts` 120s デフォルトは維持（YAGNI: まずは非同期化で効果確認）
- rebuild失敗時は degraded フラグ維持 + ログ通知で運用可視性を確保

### 2.7 Fix 7: `ensureWatcher()` degraded ワークスペース重複 rebuild 回帰防止

**問題**: Fix 6 の非同期 rebuild fallback により、以下の回帰が発生した:

1. `ensureWatcher()` が watcher 開始に失敗
2. `watcherWorkspaces.delete(agentWs)` + `watcherDegradedWorkspaces.add(agentWs)` を実行
3. 非同期 rebuild IIFE を起動して即座に return
4. 次の `before_prompt_build` で再度 `ensureWatcher()` が呼ばれる
5. `watcherWorkspaces.has(agentWs)` → `false`（delete 済み）
6. **再び watcher 開始を試行 → 失敗 → 再度非同期 rebuild 発行**
7. 4-6 が毎ターン無限に繰り返される

Go側の `rebuildInProgress` atomic guard によりDB破損は起きないが、不要なRPC通信とログスパムが発生する。

**修正** (`src/index.ts` `ensureWatcher()` 先頭):

```typescript
const ensureWatcher = async (agentWs: string): Promise<void> => {
  if (!agentWs) return;
  if (_singleton!.watcherWorkspaces.has(agentWs)) return;
  // [v0.4.20] Early return for already-degraded workspaces — prevents repeated
  // async rebuild RPCs on every before_prompt_build call when watcher is down.
  if (_singleton!.watcherDegradedWorkspaces.has(agentWs)) return;
  _singleton!.watcherWorkspaces.add(agentWs);
  // ... rest of function ...
};
```

**設計判断**:
- degraded 状態はセッション中持続する（v0.5.0でウォッチャーアーキテクチャ見直し）
- rebuild IIFE は初回の degradation 時に1回だけ発行される
- rebuild 成功時は `watcherDegradedWorkspaces` から削除しない（rebuild は全インデックス再構築であり、FSウォッチャー不在のまま機能する設計）

**発見経路**: code-reviewer-lite レビューで 🔴 Critical 回帰として発見。非同期fallback化により `ensureWatcher()` が即座に返るようになった結果、`watcherWorkspaces.delete()` 後の再呼び出しでガードが効かなくなっていた。

## 6. v0.5.0 移行に関する注記

v0.5.0で Pebble → Comet/LanceDB 置換が実施されるため、以下は本プランでは対応しない（YAGNI）:

- **PebbleDB内の既存空IDレコードのクリーンアップ** — Fix 1-4 で新規発生は防止される。既存の空IDレコードは `CleanOrphans()` に見えない（同関数は `SourcePath` が存在しない場合のみ削除し、`rec.ID == ""` はチェックしない）。`RunGarbageCollector()` も同様。v0.5.0 のDB移行で自然に消滅するため、対応不要
- **HNSW と Bleve の統合** — v0.5.0 で LanceDB の統合インデックスに置換
- **`handleWatcherStart` の非同期レスポンス化** — 現在の同期レスポンス（5s〜15sで完了する前提）は暫定として許容。v0.5.0 でウォッチャーアーキテクチャ自体が見直される可能性
- **`rpc-client.ts` の rebuild timeout 恒久延長** — 現時点では採用しない。まずは非同期fallback化で起動ブロック解消を優先
