## 主張: `cacheRetry()` Called with `backoffSec=0` — Hot-Loop Risk

### 監査文書の記載

> `cacheRetry` の `backoffSec` パラメータを省略するとデフォルト `0` になり、Go サイドカーが即座にリース可能な状態で再エンキューする。同じ失敗アイテムが即座に再リースされ、ホットループが発生する可能性がある。

**重大度**: HIGH と分類

### ソースコード検証結果: **偽**

Go サイドカーの `queue.go` L237-242 を確認:

```go
// Retry() 内のバックオフ計算
backoff := backoffSec
if backoff <= 0 {
    backoff = 1 << min(item.Attempts, 8) // 2^attempt, cap at 2^8=256s
    if backoff > 300 {
        backoff = 300
    }
}
item.BackoffUntil = time.Now().UTC().Add(time.Duration(backoff) * time.Second).Format(time.RFC3339)
```

**`backoffSec=0`（デフォルト）は「バックオフなし」ではなく「自動指数バックオフ」を意味する。**

- `backoff <= 0` の場合、Go 側は自動的に `2^attempts`（最大 256秒、キャップ 300秒）の指数バックオフを適用する
- `backoffSec=0` はホットループを引き起こさない
- むしろ非ゼロの `backoffSec` を渡すと、**自動指数バックオフが上書きされて固定遅延になる**ため、現状の `backoffSec=0`（デフォルト）の方が優れた挙動

### 結論

**この主張は偽。** `cacheRetry()` の `backoffSec=0` はバグではなく、Go 側の自動指数バックオフを有効にする正しいデフォルト値。修正は不要。

### 監査文書の「修正案」も誤り

監査文書が提案する:
```typescript
const backoffSec = Math.min(300, Math.pow(2, item.attempts));
await this.rpcClient.cacheRetry(item.id, "narrative-worker", errMsg, MAX_CACHE_ATTEMPTS, backoffSec);
```

これは **TS 側で固定遅延を計算して Go 側に渡す** という二重計算になり、Go 側の自動指数バックオフを上書きしてしまう。実装してはならない。

### ✅ Pro Recommendation

> **修正不要。この文書の結論は正しい。**

理由:
1. `backoffSec=0`（デフォルト）→ Go側で `2^min(attempts, 8)` の自動指数バックオフが適用される
2. バックオフ進行: 2s → 4s → 8s → 16s → 32s → 64s → 128s → 256s（8回目以降キャップ）
3. `LeaseNext()` が `BackoffUntil` をチェックし、期限内のアイテムをスキップするため、ホットループは物理的に不可能
4. もし監査案の通りTS側で `backoffSec` を計算して渡すと、Go側の自動バックオフが上書きされて固定遅延になり、むしろ劣化する

**追加確認: `LeaseNext()` のバックオフフィルタリング**

`queue.go` L172-180 で、`BackoffUntil` が未来のアイテムは `continue` でスキップされる。バックオフ期限が切れたら `BackoffUntil = ""` にクリアされて通常リース可能になる。このフローは堅牢。
