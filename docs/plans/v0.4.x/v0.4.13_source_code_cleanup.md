## Part B: 監査偽陽性防止ノート追記

v0.4.12 Bug Regression Audit で**バグではない（設計上のトレードオフ・既知の制限）**と判定された項目について、次回の監査で再誤認されないよう該当コード付近に意図コメント `[AUDIT NOTE]` を追記する。

### B-1. `narrative-worker.ts` — `MAX_ECHO_SCAN_CHARS` 付近

**対象行**: `const MAX_ECHO_SCAN_CHARS = 5000;` の直後

**追記内容**:
```typescript
const MAX_ECHO_SCAN_CHARS = 5000; // Only scan first 5000 chars of input (echoes are near the beginning)
// [AUDIT NOTE] This is an intentional trade-off (v0.4.12 Phase E), NOT a bug:
// - Echoes always appear near the beginning of input (model echoes the first ~200 tokens)
// - Scanning beyond 5000 chars would catch tail echoes but at 85% more memory cost (192KB copy)
// - False negatives (tail echoes) are low-impact: output is still a narrative, just with some verbatim at the end
```

### B-2. `narrative-worker.ts` — `checkEchoDetection()` 内

**対象行**: `const inputPrefix = input.substring(0, MAX_ECHO_SCAN_CHARS).replace(/\s+/g, "");` の直前

**追記内容**:
```typescript
// [AUDIT NOTE] Whitespace collapse is intentional: CJK text has no whitespace so this is a no-op for CJK.
// MIN_ECHO_LENGTH=20 means 20 whitespace-collapsed chars ≈ 20 kanji for CJK — unlikely to false-positive.
// For Latin text, 20 chars ≈ 3-4 words — very unlikely to match by coincidence.
```

### B-3. `segmenter.ts` — `forceFlush()` の `this.pool.clear()` 付近

**対象行**: `this.pool.clear();` の直前

**追記内容**:
```typescript
// [AUDIT NOTE] this.pool.clear() runs AFTER await enqueueNarrativeChunks() (fixed in v0.4.13).
// If enqueue throws, pool data is preserved — the outer try/catch skips this line.
// Pre-v0.4.13 had pool.clear() before the await, which was a data-loss risk on enqueue failure.
// See L788 comment for the rationale of this ordering.
```

> **注**: プラン初版の監査ノートは「pool.clear()はawaitの前」と記載していたが、v0.4.13の修正でpool.clear()がawaitの後に移動された。本追記内容は修正後のコードに合わせている。

### B-4. `narrative-queue.ts` — `_chunkCounter` 付近

**対象行**: `let _chunkCounter = 0;` の直後

**追記内容**:
```typescript
let _chunkCounter = 0;
// [AUDIT NOTE] _chunkCounter resets to 0 on process restart. This is NOT a bug:
// - The orderKey format includes ISO timestamp + 4-digit counter + agentId prefix
// - Collision requires: same second + same counter value + same agent — near-zero probability
// - Global monotonic counter ensures uniqueness within a single process lifetime
```

### B-5. `utils.ts` — `estimateTokens()` JSDoc 強化

**対象行**: JSDoc コメントの末尾に追記

**追記内容**:
```
 * [AUDIT NOTE] The ~20-30% overshoot for CJK is intentional and NOT a bug:
 * - It provides a safety margin against API 400 errors from underestimation
 * - Premature chunk splitting (hitting 48K target early) is safer than exceeding the 64K hard cap
 * - Approximation (text.length * factor) was rejected in v0.4.12 Pro Engineer Review as YAGNI risk
```

### ⚠️ B-3 の問題: 監査ノート本文が v0.4.13 修正後のコードと矛盾

プランが提案するB-3の監査ノート:
```
[AUDIT NOTE] this.pool.clear() runs before the await enqueueNarrativeChunks below.
```

しかし実際のコード（v0.4.13 poolAndQueue修正後）:
```typescript
// L789: await enqueueNarrativeChunks(...)  ← 先にawait
// L792: this.pool.clear();                  ← 後にクリア
```

`pool.clear()` は `await` の**後**に実行される。プランの監査ノートをそのまま挿入すると、**事実と異なるコメント**がコードに残る。

さらにL788, L791に既にv0.4.13の意図コメントがある:
```
L788: // [v0.4.13] Await enqueue first, then clear pool — consistent with poolAndQueue semantics
L791: // [v0.4.13] Clear pool AFTER successful enqueue (data preserved on failure)
```
