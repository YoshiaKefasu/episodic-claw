### Critical Judgment: 適応型 Lambda は YAGNI か？

**判断: シンプル版のみ実装する。**

現在の固定 Lambda (2.0) の問題は「短い会話で感度が足りない」こと。本格的な適応型 Lambda（会話長に応じて連続的に変化）は複雑すぎて KISS 違反。

代わりに:
```typescript
// 2段階分岐（KISS）
const effectiveLambda = this.segCount < 10 ? 1.5 : this.segmentationLambda;
```

これだけで短い会話の感度問題は解決する。

**File:** `src/segmenter.ts`

```typescript
// processTurn() 内、surprise 計算の前に追加:
private isTimeGapBoundary(msgs: Message[]): boolean {
  if (msgs.length < 2) return false;
  const lastUserMsg = [...msgs].reverse().find(m => m.role === "user");
  const prevUserMsg = [...msgs].reverse().find((m, i) => m.role === "user" && i > 0);
  if (!lastUserMsg?.timestamp || !prevUserMsg?.timestamp) return false;

  const gap = new Date(lastUserMsg.timestamp).getTime() - new Date(prevUserMsg.timestamp).getTime();
  const thresholdMs = (this.segmentationTimeGapMinutes ?? 15) * 60 * 1000;
  return gap > thresholdMs;
}
```

```go
// parseFooterMetadata は末尾の <!-- episodic-meta ... --> からメタデータを抽出する
func parseFooterMetadata(content string) (*EpisodeRecord, string, error) {
    const marker = "<!-- episodic-meta"
    idx := strings.LastIndex(content, marker)
    if idx < 0 {
        return nil, content, nil // footer なし
    }
    endIdx := strings.Index(content[idx:], "-->")
    if endIdx < 0 {
        return nil, content, nil // 不完全な footer
    }

    jsonStr := strings.TrimSpace(content[idx+len(marker) : idx+endIdx])
    var meta FooterMetadata
    if err := json.Unmarshal([]byte(jsonStr), &meta); err != nil {
        return nil, content, fmt.Errorf("footer parse: %w", err)
    }

    body := strings.TrimRight(content[:idx], "\n")
    return meta.toEpisodeRecord(), body, nil
}
```

**5b. 3パターン読み取り対応:**

```go
func parseEpisodeFile(content string) (*EpisodeRecord, string, error) {
    // 1. Invisible Footer (v0.4.0+)
    if rec, body, err := parseFooterMetadata(content); rec != nil {
        return rec, body, err
    }

    // 2. YAML Frontmatter (v0.3.x)
    if rec, body, err := parseFrontmatter(content); rec != nil {
        return rec, body, err
    }

    // 3. メタデータなし — ファイル名から ID 生成
    return nil, content, nil
}
```

#### [P3-F3] `isTimeGapBoundary` が `dedupedMessages` のみで判定 — 注意

**場所:** `segmenter.ts` L230

**観察:** `isTimeGapBoundary(dedupedMessages)` は新しいメッセージ群のみで timestamp gap を判定する。つまり dedup 後のメッセージが 2 件以上の user メッセージを含む場合にのみ発火する。

1 回の `processTurn` で user メッセージが 1 件だけ来た場合（通常パターン）、`dedupedMessages` 内に user は 1 件しかないので `isTimeGapBoundary` は常に false を返す。

**本来の意図:** 前回の最後の user メッセージと今回の user メッセージの間のギャップを検出すべきでは?

**影響:** 時間ギャップ検出が、1 ターンごとに `processTurn` が呼ばれる通常フローでは **全く発火しない可能性が高い**。バッチ送信 (複数メッセージ一括) の場合のみ動作する。

**修正案:**
```typescript
private isTimeGapBoundary(newMsgs: Message[]): boolean {
  // 新メッセージの最初の user msg と、buffer の最後の user msg を比較
  const lastBufferUser = [...this.buffer].reverse().find(m => m.role === "user" && m.timestamp);
  const firstNewUser = newMsgs.find(m => m.role === "user" && m.timestamp);
  if (!lastBufferUser?.timestamp || !firstNewUser?.timestamp) return false;

  const gap = new Date(firstNewUser.timestamp).getTime() - new Date(lastBufferUser.timestamp).getTime();
  const thresholdMs = this.segmentationTimeGapMinutes * 60 * 1000;
  return gap > thresholdMs;
}
```

**Severity:** **高** — 時間ギャップ検出が通常の会話フローで機能しない。Phase 3 の主要機能が空振り。
