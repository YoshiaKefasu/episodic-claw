## 1. 問題定義

### 現状の実行順序

`segmenter.ts` `poolAndQueue()` L725-752:

```
1. L728: this.pool.add(this.buffer.slice(), ...)  — バッファをプールにコピー
2. L730: this.buffer = []                          — ★ バッファクリア（enqueue前）
3. L731: this.lastProcessedLength = 0               — ★ カウンタリセット（enqueue前）
4. L732: this.clearIdleFlushTimer()                 — タイマークリア
5. L736: this.pool.clear()                          — ★ プールクリア（enqueue前）
6. L745: splitIntoChunks(...)                       — チャンク分割（同期）
7. L747: enqueueNarrativeChunks(...)               — 非同期 enqueue（fire-and-forget）
   └── .catch(err => console.error(...))            — エラーログのみ、データ消失
```

### 何が起きるか

`enqueueNarrativeChunks` が失敗した場合（Goサイドカーソケット未準備、ペイロード過大など）:

1. `this.buffer` — 既にクリア済み（L730）
2. `this.pool` — 既にクリア済み（L736）
3. ローカル変数 `chunks` — `.catch()` の後、スコープ外に消失
4. エラーログだけ出力され、データは永久に失われる

### 影響

- Goサイドカーが一時的に利用不能な場合（起動レース、ソケットエラー）に会話データが消失
- 実稼働では稀（サイドカーはプラグインより先に起動する）だが、スタートアップ時のレースコンディションで発生可能

## 2. 修正内容

### 2.0 前提修正: `enqueueNarrativeChunks` のエラー伝播

**⚠️ この修正がなければ、後続の `poolAndQueue` 修正が機能しない。**

現状の `enqueueNarrativeChunks()`（`narrative-queue.ts` L155-175）はエラーを内部キャッチして再スローしない:

```typescript
// 現状 (narrative-queue.ts L162-174):
try {
    const result = await (rpcClient as any).request("cache.enqueueBatch", { items: chunks });
    // ...
} catch (err) {
    console.error("[Episodic Cache] Failed to enqueue narrative chunks:", err);
    // Fallback: don't lose the data — the caller's buffer still holds it  ← 偽。poolAndQueueは呼び出し前にバッファをクリアする
}
```

この関数は常に resolve し、決して reject しない。したがって:
- 現行の `poolAndQueue()` L748-749 の `.catch()` は**デッドコード**（発火しない）
- プランの `.then()/.catch()` パターンも機能しない（`.then()` が失敗時でも実行される）

**修正: `throw err` を追加してエラーを再スローする**

```typescript
// 変更後 (narrative-queue.ts):
} catch (err) {
    console.error("[Episodic Cache] Failed to enqueue narrative chunks:", err);
    throw err; // [v0.4.13] Re-throw so callers can detect failure and preserve data
}
```

**他の呼び出し元への影響確認:**

| 呼び出し元 | ファイル | try/catch保護 | 影響 |
|-----------|---------|--------------|------|
| `poolAndQueue()` | segmenter.ts L747 | `.catch()` あり（本修正で有効化） | ✅ 影響なし |
| `forceFlush()` | segmenter.ts L783 | try/catch（L770-794）あり | ✅ 影響なし |
| `archiver.ts` | archiver.ts | **要確認** | ⚠️ try/catchで保護されていなければ追加 |

### 2.1 方針

`poolAndQueue` はfire-and-forget（非同期enqueueをawaitしない）なので、単純にawaitに変えるとセグメンターのメインループがブロックされる。代わりに、バッファ/プールのクリアをenqueue成功後に遅延させる。

前提修正（2.0）により、`.then()/.catch()` パターンが正しく機能する。

### 2.2 変更対象

1. `src/narrative-queue.ts` — `enqueueNarrativeChunks()` のエラー再スロー
2. `src/segmenter.ts` — `poolAndQueue()` L725-752 のクリア順序変更
3. `src/segmenter.ts` — `forceFlush()` L769-796 の `pool.clear()` 移動
4. `src/archiver.ts` — `enqueueNarrativeChunks` 呼び出しのtry/catch確認（必要に応じて追加）

### 2.3 変更後: `poolAndQueue()`

```typescript
private async poolAndQueue(agentWs: string, agentId: string, surprise: number, reason: string): Promise<void> {
    if (!this.pool || !this.narrativeWorker) return;

    const item = this.pool.add(this.buffer.slice(), surprise, agentWs, agentId);
    // Clear idle flush timer; messages are now in the pool.
    this.clearIdleFlushTimer();

    if (item) {
      // Flush occurred: split and enqueue to cache DB
      const cacheReason = (
        reason === "idle-timeout" ? "idle-timeout" :
        reason === "surprise-boundary" ? "surprise-boundary" :
        "size-limit"
      ) as "idle-timeout" | "surprise-boundary" | "size-limit";
      const chunks = splitIntoChunks(item.rawText, agentWs, agentId, "live-turn", cacheReason, surprise);

      // Defer ALL buffer/pool clears until after enqueue confirmation
      // (requires enqueueNarrativeChunks to re-throw errors — see 2.0)
      enqueueNarrativeChunks(this.rpc, chunks, () => this.narrativeWorker?.wake())
        .then(() => {
          // Enqueue succeeded — safe to clear
          this.pool!.clear();
          this.buffer = [];
          this.lastProcessedLength = 0;
        })
        .catch(err => {
          console.error("[Episodic Memory] Cache enqueue failed:", err);
          // Data remains in pool AND buffer for potential retry
          // The next pool flush or forceFlush will re-attempt with the data still intact
        });
    } else {
      // No flush — pool is still buffering. Clear segmenter buffer only (data is in pool now).
      this.buffer = [];
      this.lastProcessedLength = 0;
    }
  }
```

### 2.4 変更後: `forceFlush()` の `pool.clear()` 移動

```typescript
// 変更前:
this.pool.clear();                          // L779 — await の前
await enqueueNarrativeChunks(...);          // L783
this.buffer = [];                           // L785

// 変更後:
await enqueueNarrativeChunks(...);          // 先にenqueue成功を確認
this.pool.clear();                          // 成功後にクリア
this.buffer = [];
```

`forceFlush` では `item`（PoolFlushItem）がデータのスナップショットを保持しているため、`pool.clear()` を `await` 前に実行してもデータ消失は起きない。しかし `poolAndQueue` と一貫したセマンティクス（「成功までプールデータを保持」）にするため、移動する。

### 2.5 設計判断

- **`enqueueNarrativeChunks` の `throw err` 追加（前提修正）**: `.then()/.catch()` パターンが機能するための必須前提。これなしでは `.catch()` が発火せず、失敗時でも `.then()` が実行されてデータが消失する
- **`this.buffer = []` を2箇所に分割**: flush時はenqueue成功後にクリア、非flush時は即時クリア（データは既にpool.add()にコピー済み）
- **pool.clear() を .then() 内に移動**: enqueue成功までプールデータを保持
- **.catch() でデータを保持**: 次回の `pool.add()` や `forceFlush()` で再試行の機会がある
- **重複データリスク**: enqueue失敗後にbufferがクリアされないと、次回の `pool.add()` で同じデータが再送される可能性がある。しかし「データ消失よりマシ」

### ⚠️ 基準2の詳細分析: `this.buffer = []` の配置

プラン Section 2.3 では `this.buffer = []` と `this.lastProcessedLength = 0` を `.then()` 内に配置すると記載していたが、実際の実装ではL730-731（enqueue前、`if (item)` の前）に配置されている。

**判定: ✅ 実装の方が正しい**

理由:
1. `pool.add(this.buffer.slice(), ...)` でデータは**スライスコピー**としてプールに渡される
2. `this.buffer.slice()` が呼ばれた時点で、プールはバッファデータの独立したコピーを持つ
3. したがって `this.buffer = []` を即時実行しても、データはプール内に安全に保持されている
4. プランの「`this.buffer = []` を `.then()` 内に」という記載は過剰防衛だった
5. 即時クリアの方が有利: 新しいメッセージが到着したときに古いバッファと混ざるリスクを防げる

**ただし**: enqueue失敗時に `this.buffer` は既に空なので、セグメンター側の再試行は「プールに残ったデータが次回フラッシュで再送される」経路に依存する。これはプランの設計判断と一致する。

### 🎯 `poolAndQueue()` 実装のフロー整理

```
成功時:
  pool.add(buffer.slice()) → buffer=[], lastProcessedLength=0 →
  enqueueNarrativeChunks → .then() → pool.clear() ✅ データはGo側に送信済み

失敗時:
  pool.add(buffer.slice()) → buffer=[], lastProcessedLength=0 →
  enqueueNarrativeChunks → .catch() → pool.clear() は呼ばれない
  → データはプール内に保持 ✅
  → 次回 pool.add() / forceFlush() で再送される ✅
```

### 🎯 `forceFlush()` 実装のフロー整理

```
成功時:
  pool.add(buffer.slice()) → pool.forceFlush() →
  await enqueueNarrativeChunks → pool.clear(), buffer=[], lastProcessedLength=0 ✅

失敗時:
  pool.add(buffer.slice()) → pool.forceFlush() →
  await enqueueNarrativeChunks → throw → catch(err) → pool.clear() はスキップ
  → データはプール内に保持 ✅
```

### ✅ 総合判定

> **v0.4.13 実装は完了。全受け入れ基準を満たす。** `this.buffer = []` の配置がプランと異なるが、実装の方が正しい（即時クリアでデータ安全性を損なわない）。`archiver.ts` も既に try/catch + フォールバック付きで保護されていたため、追加変更不要だった。
