# v0.2.6 個別プラン 2 — `episodes/episodes` 二重パスの徹底修正

## 目的
- `agentWs` がすでに `.../workspace/episodes` を指しているのに、さらに `episodes` を足してしまう経路を根絶する。
- `episodes/episodes/YYYY/MM/DD/...` の二重構造を新規生成しない。
- 既存の二重構造が索引・watcher・recall の挙動を濁らせないように整理する。

## 現時点の証拠
- 実データで `/home/kasou_yoshia/.openclaw/workspace/episodes/episodes/2026/03/...` が存在している。
- Historical evidence (pre-fix): `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\compactor.ts` の
  旧実装では `path.join(agentWs, "episodes", YYYY, MM, DD)` を使っていた。
  これは現在のコード状態では解消済みだが、当時の二重パス発生源として残す。
- 一方で Go 側の background index 出力は `filepath.Join(agentWs, YYYY, MM, DD)` で、こちらは二重化していない。
- つまり pre-fix では path 契約が TypeScript 側だけずれていたが、現在のコードでは compactor 側も含めて新規生成は止まっている。ここは歴史的証拠と現在状態を分けて読む。

## 調査で確定した事実
### 1. `agentWs` の意味はすでに「`.../workspace/episodes`」になっている
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:146-159`
- `resolveAgentWorkspaces()` は `agentRoot = resolveWorkspaceRoot(...)` のあとに `path.join(agentRoot, "episodes")` を返している。
- つまり `agentWs` は workspace root ではなく、episode root である。
- この前提を忘れてさらに `episodes` を足すと、`episodes/episodes` が発生する。

### 2. 二重パスを直接作っていたのは TypeScript 側の compactor（pre-fix evidence）
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\compactor.ts:131-136`
- `unprocessed.length > 50` の massive gap 分岐で、`dumpDir = path.join(agentWs, "episodes", today.slice(0, 4), today.slice(4, 6), today.slice(6, 8))` を使っている。
- この `agentWs` はすでに `.../workspace/episodes` なので、実際の保存先は `.../workspace/episodes/episodes/YYYY/MM/DD` になる。
- `dumpFile` はその配下に `legacy_backlog_YYYYMMDD_*.json` として作られる。

### 3. Go 側の background indexing は二重化していない
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\background.go:27-35`
- `ProcessBackgroundIndexing()` は `.md` を `ProcessMDFileIndex()`、それ以外を `processBacklogFile()` に分けている。
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\background.go:288-327`
- `.md` の保存先は `filepath.Join(agentWs, YYYY, MM, DD)` で、二重の `episodes` は足していない。
- backlog の JSON も `processBacklogFile()` で読み込まれるだけで、保存先の二重化は Go 側では起きていない。

### 4. 実データで二重構造が見えている
- SSH で `find /home/kasou_yoshia/.openclaw/workspace/episodes -maxdepth 3 -type d` を見たところ、`/home/kasou_yoshia/.openclaw/workspace/episodes/episodes` が実在していた。
- 同じく `find ... -path '*/episodes/*'` で、`/home/kasou_yoshia/.openclaw/workspace/episodes/episodes/2026/03/31/legacy_backlog_20260331_1774945154502.json` が確認できた。
- つまり二重化は仮説ではなく、実ファイルとしてすでに起きている。

### 5. 3/31 時点の通常 episode は別の階層に正しくある
- SSH 上では `/home/kasou_yoshia/.openclaw/workspace/episodes/2026/03/31/eju-essay-training.md` のような正規の episode も存在していた。
- つまり「全部が壊れている」のではなく、`compactor.ts` の massive gap 経路だけが余計な一段を作っている。

### 6. 既存ストアの recall 自体は動いている
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\\internal\\vector\\store.go:1050-1159`
- `RecallWithQuery()` は `queryString` を lexical search に回し、候補がなければ HNSW にフォールバックする。
- 実データを直接開いて `RecallWithQuery(\"EJU特訓\", zero-vector)` を実行すると 5 件ヒットした。
- したがって「記憶が無い」のではなく、「どの経路がどの store を見ていたか」を切り分ける必要がある。

### 7. `/tmp/episodic-core.log` では indexing と recall が実際に走っている
- `Method: ai.triggerBackgroundIndex` が 2026-04-01 20:29 頃に連続して出ている。
- その直後に `Method: ai.recall` も出ている。
- `Watcher event: CREATE /home/kasou_yoshia/.openclaw/workspace/episodes/2026/04/01/...` も同時に記録されている。
- つまり watcher / background index / recall は同じ時間帯に動いており、path の見え方だけが崩れると観測がかなり難しくなる。

## 追加で確認できた補助事実
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts:19-120` の auto recall は `rpcClient.recall(query, k, agentWs, [], undefined, calibration)` を使っており、workspace は `agentWs` 固定。
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:584-607` の `ep-recall` も `rpcClient.recall(..., agentWs, ...)` を使っており、explicit tool も同じ workspace 契約に立っている。
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:381-409` の watcher 起動は `ensureWatcher(agentWs)` で、これも `agentWs` 単位で回している。
- だから path 契約の崩れは、recall の空振りや注入ログの見え方にも波及しうる。

## 根本問題
- いまの runtime 契約では `agentWs = <workspaceRoot>/episodes`。
- この前提なのに、TypeScript 側の一部実装が `agentWs` を「workspaceRoot」だと思って扱っている。
- そのせいで compaction backlog だけ保存先が一段深くなり、索引・監視・保守運用の見通しを悪くしている。

## 調査対象
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\compactor.ts`
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts`
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\background.go`
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go`
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\docs\v0.2.5_fix_plan.md`

## 実装方針
### 1. `agentWs` の契約を一本化する
- `agentWs` は常に「episode root」だと明文化する。
- 日付ディレクトリを作る箇所はすべて `path.join(agentWs, YYYY, MM, DD)` に統一する。

### 2. 二重化する書き込み経路を修正する
- `Compactor.compact()` の massive gap dump 先を修正する。
- 同種の `path.join(agentWs, "episodes", ...)` が残っていないか全件点検する。

### 3. 既存データの移行方針を決める
- 既存の `episodes/episodes/...` はそのまま放置すると、保守時にまた混乱する。
- v0.2.6 では少なくとも次のどちらかを決める。
  1. 自動移行する
  2. 明示的な repair command / one-shot script を提供する
- 競合ファイルがある場合の扱いも先に決める。

### 4. 索引の再構築手順をセットで持つ
- パス移動後は `p2i` 逆引きや lexical / vector の古い SourcePath を残さない。
- 必要なら `rebuildIndex` を案内し、stale path を掃除する。

## 検証手順
1. massive gap compaction を再現し、backlog dump が `agentWs/YYYY/MM/DD` に出ることを確認する。
2. `episodes/episodes` が新規に増えないことを確認する。
3. 既存の二重パスを移行したあとも `ep-recall` と watcher が正常に動くことを確認する。
4. `SourcePath` ベースの処理で古い二重パスが残っていないことを確認する。

## 完了条件
- 新規書き込みで `episodes/episodes` が一切作られない。
- path 契約が TypeScript / Go / docs で一致する。
- 既存の二重パスデータを扱う運用方針が明文化される。
- path 修正後に recall と watcher の挙動が安定する。
