# v0.2.6 個別プラン 3 — `prependSystemContext` 注入ログの追加

## 目的
- 「自動注入されたのか、されなかったのか」をログで即判定できるようにする。
- `prependSystemContext` が空だった理由を、あとから追えるようにする。
- `ep-recall` 問題と注入問題の切り分けを速くする。

## 背景
- 現在は `assemble()` が `prependSystemContext` を返しているが、注入成否を一発で示す専用ログがない。
- そのため、空注入だったのか、recall が空だったのか、token budget に弾かれたのかが見えにくい。
- 今回の `EJU特訓` 事象でも、「記憶がない」のか「注入されていない」のかの判別に時間がかかっている。

## 実装結果
- `prependSystemContext` の runtime ログを追加した。
- ログは `status=injected|skipped|truncated` を先頭に置き、`queryHash` と `estimatedTokens` を軸に追えるようにした。
- `max_tokens_zero` は「入力時点で注入不能」、`budget_truncated_to_zero` は「候補はあったが reserve 後 budget で 0 に切られた」と、コードコメントとノートの両方で分けた。
- `retrieveRelevantContext()` は理由付きの outcome を返すようになり、`prependSystemContext` の空/非空だけでなく、理由コードでも切り分けられる。
- runtime log の形はだいたい次の3通り。
  - `prependSystemContext injected agentId=... agentWs=... queryHash=... estimatedTokens=... injectedEpisodeCount=... truncatedEpisodeCount=... firstEpisodeId=...`
  - `prependSystemContext skipped reason=max_tokens_zero agentId=... agentWs=... queryHash=... estimatedTokens=0 injectedEpisodeCount=0 truncatedEpisodeCount=0`
  - `prependSystemContext truncated agentId=... agentWs=... queryHash=... estimatedTokens=... injectedEpisodeCount=... truncatedEpisodeCount=... firstEpisodeId=...`

## すでに確認できた事実
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:490-548` の `assemble()` は、最終的に `{ prependSystemContext: episodicContext, estimatedTokens: estimateTokens(episodicContext) }` を返す。
- 同じ `assemble()` では `queryHash = sha1(fullQuery)` を作っている。`fullQuery` は直近 5 メッセージから組み立てている。
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:512-527` では、`maxEpisodicTokens <= 0` の場合に空注入で即 return する。
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts:19-123` の `retrieveRelevantContext()` は理由付き outcome を返す。
- `retrieveRelevantContext()` には `currentMessages.length === 0`、`maxTokens <= 0`、`query === ""`、`sourcedResults.length === 0`、`recall` 例外という空経路がある。
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts:43-45` では、`rpcClient.recall(query, k, agentWs, [], undefined, calibration)` を呼び、explicit `ep-recall` とは違って topics を送らない。
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\rpc-client.ts:14,171,434-445,479-491` の `SOCKET_ADDR_FILE` は reconnect 補助線にはなるが、空注入の一次原因かどうかは別に切り分けるべきだと見える。

## コードから確認できた事実
### `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts`
- `resolveAgentWorkspaces()` は `agentId` を解決したうえで、`agentWs = path.join(agentRoot, "episodes")` を返す。
- `assemble()` は `agentWs` を前提に `prepareWorkspaces(resolution)` を呼び、そのあと `retriever.retrieveRelevantContext(msgs, agentWs, k, maxEpisodicTokens)` を実行する。
- `assemble()` の空注入は少なくとも 2 箇所で起きる。
  - `agentWs` が無いとき
  - `maxEpisodicTokens <= 0` のとき
- `assemble()` の `queryHash` は「自動注入がどの会話を見たか」を追うための自然な相関キーになっている。
- `ep-recall` と `prependSystemContext` は、同じ `agentWs` を共有しつつも、前者は明示 query、後者は会話から組み立てた query を使う。

### `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts`
- `retrieveRelevantContext()` は、会話メッセージから query を作る auto inject 側の経路。
- ここでは query 本文を `rpcClient.recall()` に渡し、結果を `reason` 付き outcome として返す。
- したがって `prependSystemContext` が空でも、`no_messages` / `max_tokens_zero` / `empty_query` / `recall_empty` / `recall_failed` のどれだったかを後から追える。
- `queryHash` は `recallFeedback` 側の相関に使えるので、`ep-recall` 調査と揃える意義がある。

### `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\rpc-client.ts`
- `SOCKET_ADDR_FILE = path.join(os.tmpdir(), "episodic-claw-socket.addr")` があり、socket 復元はファイル経由で起こる。
- ただし、socket 復元の不一致はたいてい RPC 失敗として見えやすく、空注入そのものとは分けて追ったほうがいい。
- なので注入ログの主戦場は `assemble()` と `retrieveRelevantContext()` 側でよい。

## ログ要件
### 1. 注入成功ログ
- `prependSystemContext` が非空なら、次を出す。
- `agentId`
- `agentWs`
- `queryHash`
- `estimatedTokens`
- `injectedEpisodeCount`
- `firstEpisodeId` か、先頭数件の id

#### 観測メモ
- `agentId` / `agentWs` / `queryHash` は、`ep-recall` 調査側と同じ相関軸でそろえると比較しやすい。
- `estimatedTokens` は、`prependSystemContext` が空でない場合でも budget による切り詰めがあったかを残すのに向いている。
- `injectedEpisodeCount` は、実際に system prompt に載せた episode 数として扱うのが自然だが、本文そのものはログしない。

### 2. 注入スキップログ
- 空文字を返した理由を分類して出す。
- `no_messages`
- `max_tokens_zero`
- `empty_query`
- `recall_empty`
- `recall_failed`
- `budget_truncated_to_zero`

#### 観測メモ
- `no_messages` は `currentMessages.length === 0`。
- `max_tokens_zero` と `budget_truncated_to_zero` は、見た目は似ているが位置が違う。前者は auto inject の入力が空、後者は `assemble()` の budget で 0 以下になったケースだ。
- もう少し正確に言うと、`max_tokens_zero` は「入力時点で注入不能」、`budget_truncated_to_zero` は「候補はあったが reserve 後 budget で 0 に切られた」。
- `empty_query` はメッセージはあるが query が組めなかったとき。
- `recall_empty` は recall 自体は成功したが、返却結果が 0 件だったとき。
- `recall_failed` は transport / store / sidecar の問題も含む、結果取得前の失敗。

### 3. プライバシー / ノイズ制御
- memory 本文の全文はログしない。
- query 本文も原則そのまま出さず、`queryHash` か短い preview に留める。
- 連打時にノイズが多すぎるなら debug flag で制御できるようにする。

#### 観測メモ
- `queryHash` を使えば、`ep-recall` と `prependSystemContext` の双方を同じ query 系列として追える。
- 本文を出しすぎると、ログが原因調査より読みにくくなる。少なくとも全文は不要。

## 実装方針
### 1. `retrieveRelevantContext()` に理由コードを返せる形を用意する
- いまは `string` だけ返しているので、必要なら戻り値を構造化する。
- 例: `text`, `reason`, `injectedIds`, `tokenCount`, `queryHash`

### 2. `assemble()` 側で最終判断ログを出す
- `prependSystemContext` に実際に何を渡したかを、返却直前で記録する。
- これで「retriever では取れていたのに assemble で消えた」ケースも追える。

### 3. `ep-recall` 調査と同じ相関 ID を使う
- explicit recall と auto inject を並べて見比べられるように、query hash などの識別子を揃える。

## 実装に入る前に絞れたこと
- まず見るべきなのは `prependSystemContext` の空/非空だけではなく、`assemble()` の `queryHash` と `estimatedTokens`。
- 次に `retrieveRelevantContext()` の戻りを理由付きにするかどうかを決める。
- `SOCKET_ADDR_FILE` は補助線として残すが、主因としては追いすぎない。

## 検証手順
1. recall がヒットする入力で、注入成功ログが出ることを確認する。
2. `maxTokens=0` や空クエリで、スキップ理由ログが正しく出ることを確認する。
3. `ep-recall` が空だったケースと照合し、注入側のログだけで切り分けできるか確認する。

## 完了条件
- 「注入された / されなかった」がログだけで判定できる。
- 空注入の理由がコード化され、再現時の初動が早くなる。
- `ep-recall` 問題との関連を、推測ではなくログで追える。
