## 1. HIGH — Confirmed Bugs (Direct Impact on Runtime Behavior)

### 1.1. `narrativePreviousEpisodeRef` Config Ignored

**Severity:** MEDIUM  
**Location:** `narrative-worker.ts:257`, `config.ts:105`, `openclaw.plugin.json`  
**Introduced:** v0.4.0-alpha.1  
**Status:** Unfixed in v0.4.12

**Description:**  
The `narrativePreviousEpisodeRef` configuration field is fully wired through the config pipeline — it has a schema entry in `openclaw.plugin.json`, is parsed in `config.ts` (default: `true`), and the type is defined in `types.ts`. However, `NarrativeWorker.resolveUserPrompt()` **never checks this setting** and always passes the previous episode body to the LLM:

```typescript
// narrative-worker.ts:257 — previous is always fetched, regardless of config
const previous = this.lastNarrativeByAgent.get(item.agentId);
// resolveUserPrompt always includes previous?.body
const userMessage = this.resolveUserPrompt(previous?.body, conversationText);
```

The `resolveUserPrompt()` method simply interpolates `previousEpisode` into the template without any conditional check:

```typescript
const DEFAULT_USER_PROMPT_TEMPLATE = (previousEpisode, conversationText) =>
  `${previousEpisode ? `Previous episode:\n${previousEpisode}\n---\n` : ""}...`;
```

**Impact:**  
- Users who set `narrativePreviousEpisodeRef: false` will still have previous episode context injected into every LLM call.
- Wasted tokens on free-tier models (previous episode can be 200-800 chars per call).
- Not a data corruption issue, but the config is effectively a no-op — **a silent misconfiguration**.

**Fix:**  
```typescript
// In narrativizeWithRetry(), check config before passing previous:
const prevBody = this.config.narrativePreviousEpisodeRef
  ? this.lastNarrativeByAgent.get(item.agentId)?.body
  : undefined;
const userMessage = this.resolveUserPrompt(prevBody, conversationText);
```

### 1.2. `cacheRetry()` Called with `backoffSec=0` on Quality Gate Exhaustion

### 1.2. `cacheRetry()` Called with `backoffSec=0` — Hot-Loop Risk

**Severity:** HIGH (potential hot-loop; depends on Go sidecar behavior)  
**Location:** `narrative-worker.ts:231` (quality gate exhaust), `narrative-worker.ts:238` (catch block), `rpc-client.ts:628`  
**Introduced:** v0.4.12 (Phase A: fallback removal)  
**Status:** Unfixed

**Description:**  
There are **two** `cacheRetry()` call sites in `processNextFromCache()` that omit the `backoffSec` parameter, both defaulting to `0`:

**Call site A — Quality gate exhaustion (line 231):**
```typescript
await this.rpcClient.cacheRetry(
  item.id, "narrative-worker",
  "Quality gate exhausted: all retries failed",
  MAX_CACHE_ATTEMPTS  // maxAttempts=20
  // ⚠️ backoffSec not specified → defaults to 0
);
```

**Call site B — Exception in narrativizeWithRetry (line 238):**
```typescript
await this.rpcClient.cacheRetry(item.id, "narrative-worker", errMsg, MAX_CACHE_ATTEMPTS);
// ⚠️ Same issue: backoffSec defaults to 0
```

The `cacheRetry` signature is:  
`cacheRetry(id, workerId, error, maxAttempts=20, backoffSec=0)`

With `backoffSec=0`, the Go sidecar will re-enqueue the item as immediately leaseable. The Go sidecar's automatic exponential backoff (`2^attempt`, cap 300s) was introduced in v0.4.2, but **only applies when `BackoffUntil` is set by the retry call**. If `backoffSec=0` overrides or bypasses the auto-backoff, the same failing item can be immediately leased again, causing:

- Rapid consumption of OpenRouter API rate limits
- CPU spinning on the same unprocessable chunk
- Log spam until `MAX_CACHE_ATTEMPTS` (20) is reached

**Uncertainty:**  
Whether this is an actual hot-loop depends on the Go sidecar's `cache.retry` handler implementation. If the Go sidecar ignores `backoffSec=0` and applies its own `2^attempt` formula, there is no issue. If it respects the `0` value, the hot-loop occurs.

**Fix (defensive):**  
```typescript
// Apply to BOTH call sites (A and B)
const backoffSec = Math.min(300, Math.pow(2, item.attempts));
await this.rpcClient.cacheRetry(item.id, "narrative-worker", errMsg, MAX_CACHE_ATTEMPTS, backoffSec);
```

## 2. MEDIUM — Logic Gaps & Potential Regressions

### 2.1. `checkEchoDetection` Substring Matching with Whitespace Collapse

**Severity:** MEDIUM  
**Location:** `narrative-worker.ts:checkEchoDetection()`  
**Introduced:** v0.4.11 (Layer 1 quality gate)  
**Optimized:** v0.4.12 (MAX_ECHO_SCAN_CHARS = 5000)

**Description:**  
The echo detection function collapses ALL whitespace (`output.replace(/\s+/g, "")`) before comparing. This is intentional — to catch echoes with different formatting. However:

1. **CJK text with no whitespace**: A CJK string like `今日は良い天気ですね` has no whitespace to begin with. The collapse is a no-op. The function then checks if the first 80 collapsed characters appear in the first 5000 collapsed characters of the input. This works correctly for CJK echoes.

2. **Short non-echo passes**: If the LLM generates a novel narrative that happens to share the first 20 whitespace-collapsed characters with the input (e.g., same proper noun at the start), the quality gate will **falsely reject it**. With `MIN_ECHO_LENGTH=20`, this is unlikely for Latin text but possible for CJK where 20 chars ≈ 20 kanji.

3. **Scan truncation at 5000 chars**: Echoes that appear after the first 5000 characters of input will be **missed entirely**. This is a trade-off for memory efficiency (v0.4.12 Phase E), but means long-preamble inputs could have echo content in the tail that passes the gate.

**Assessment:** Acceptable trade-off for free-tier model filtering. The 5000-char scan window covers the most common echo pattern (models echoing the beginning of input). False negatives (tail echoes) are low-impact since the output is still a narrative, just with some verbatim content at the end.

### 2.2. `poolAndQueue` Fire-and-Forget: No Error Recovery for Enqueue Failure

**Severity:** MEDIUM  
**Location:** `segmenter.ts:poolAndQueue()`, line ~740  
**Introduced:** v0.4.2

**Description:**  
```typescript
// Fire-and-forget with .catch() — errors are logged but data is lost
enqueueNarrativeChunks(this.rpc, chunks, () => this.narrativeWorker?.wake())
  .catch(err => console.error("[Episodic Memory] Cache enqueue failed:", err));
```

When `enqueueNarrativeChunks` fails (e.g., Go sidecar socket not ready, Unix socket oversized payload), the chunk data is **silently lost** because:
1. The segmenter buffer was already cleared (`this.buffer = []`) before the async enqueue
2. The pool was already cleared (`this.pool.clear()`) before the async enqueue
3. The error is only logged, not recovered

The `enqueueNarrativeChunks` function itself notes: *"Fallback: don't lose the data — the caller's buffer still holds it"* — but the caller (poolAndQueue) has already cleared both buffers.

**Impact:** Data loss when Go sidecar is temporarily unavailable during a boundary flush. Rare in production (sidecar is started before plugin), but possible during startup race conditions or socket errors.

**Fix:** Defer ALL buffer/pool/position clears until after enqueue confirmation:
```typescript
// Move buffer=[], lastProcessedLength=0, pool.clear() AFTER confirmed enqueue
enqueueNarrativeChunks(this.rpc, chunks, () => this.narrativeWorker?.wake())
  .then(() => {
    this.pool!.clear();
    this.buffer = [];
    this.lastProcessedLength = 0;
  })
  .catch(err => {
    console.error("[Episodic Memory] Cache enqueue failed:", err);
    // Data remains in pool AND buffer for potential retry
  });
```

**Note:** The current code clears `this.buffer = []`, `this.lastProcessedLength = 0`, and `this.pool.clear()` synchronously before the async enqueue. ALL three must be deferred, not just `pool.clear()`.

### 2.3. NarrativePool `forceFlush()` Data Loss on Enqueue Failure

**Severity:** MEDIUM  
**Location:** `segmenter.ts:forceFlush()`, line ~775  
**Introduced:** v0.4.2

**Description:**  
```typescript
async forceFlush(agentWs: string, agentId: string = ""): Promise<void> {
  // ...
  this.pool.add(this.buffer.slice(), 0, agentWs, agentId);
  const item = this.pool.forceFlush(agentWs, agentId);
  this.pool.clear();  // ← Cleared before await
  if (item) {
    const chunks = splitIntoChunks(item.rawText, ...);
    await enqueueNarrativeChunks(this.rpc, chunks, ...);  // Can fail
  }
  this.buffer = [];
  this.lastProcessedLength = 0;
}
```

Unlike `poolAndQueue` (fire-and-forget), `forceFlush` **awaits** the enqueue. However, `this.pool.clear()` is called **before** the await, so if enqueue fails, the pool data is gone. The `this.buffer = []` is also executed unconditionally after the if-block.

**Impact:** Data loss during compaction/context-reset if the Go sidecar socket fails.

## 3. LOW — Minor Issues

### 3.1. `_chunkCounter` Module-Level Variable Reset on Process Restart

**Severity:** LOW  
**Location:** `narrative-queue.ts:62`  
**Introduced:** v0.4.2

**Description:**  
`_chunkCounter` is a module-level variable (`let _chunkCounter = 0`) used in `orderKey` generation:
```typescript
const orderKey = `${now}-${String(_chunkCounter).padStart(4, "0")}`;
```

On process restart, `_chunkCounter` resets to 0. If two chunks are generated within the same second after a restart, they could collide with chunks generated before the restart that had the same timestamp-second + counter. The `padStart(4, "0")` format means collisions are possible but extremely unlikely (requires same millisecond + same counter value).

**Impact:** Near-zero in practice. The `id` field also includes `agentId:` prefix, providing additional uniqueness.

### 3.2. `estimateTokens()` Overshoot for CJK Text

**Severity:** LOW  
**Location:** `utils.ts`  
**Introduced:** v0.3.0

**Description:**  
The function uses a fixed 1.5x multiplier for CJK characters and 0.25x for Latin. The JSDoc already notes ~20-30% overshoot compared to BPE tokenizers. This is a **known limitation**, not a bug, but can cause:
- Premature chunk splitting (48K token target is reached earlier than necessary)
- Under-utilization of the 64K token cap

**Assessment:** Acceptable. The overshoot provides a safety margin against API 400 errors.
