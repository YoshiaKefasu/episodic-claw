**Date:** 2026-04-19

**Baseline:** v0.4.14 (post Fix A/B/C)

## 0. Background: The Recurring Bug Pattern

v0.4.13 and v0.4.14 both fixed the same class of bug: **a config field defined in schema + type but not parsed by `loadConfig()`**, causing user settings to be silently ignored.

| Version | Missing Field | Root Cause | Symptom |
|---------|--------------|------------|---------|
| v0.4.13 | `narrativePreviousEpisodeRef` | `loadConfig()` omitted the field | Previous episode always injected regardless of config |
| v0.4.14 Fix A | `recallQueryRecentMessageCount`, `queryExcludedKeywords` | `loadConfig()` omitted both fields | User recall query config ignored, hardcoded defaults used |

This pattern prompted a full cross-source audit to find all similar mismatches. The audit revealed that **v0.4.14 Fix B introduced a new instance of the same pattern as a regression**.

## 1. 🔴 CRITICAL — Confirmed Regression: `openrouterConfig.maxTokens` Silently Dropped

**Severity:** CRITICAL (functional regression)  
**Location:** `config.ts:loadConfig()`, `index.ts:register()` OpenRouterClient constructor  
**Introduced:** v0.4.14 Fix B (narrativeMaxTokens removal)  
**Status:** Unfixed

### 1.1. Description

v0.4.14 Fix B removed the deprecated `narrativeMaxTokens` legacy alias. The removal itself was correct — `narrativeMaxTokens` was a flat-field alias for `openrouterConfig.maxTokens`. However, Fix B also **deleted the only code path that forwarded `openrouterConfig.maxTokens` to the OpenRouter API call**:

**Before v0.4.14 Fix B:**
```typescript
// config.ts — loadConfig()
narrativeMaxTokens: rawConfig?.openrouterConfig?.maxTokens ?? rawConfig?.narrativeMaxTokens,

// index.ts — OpenRouterClient constructor
maxTokens: cfg.narrativeMaxTokens,
```

**After v0.4.14 Fix B:**
```typescript
// config.ts — loadConfig()
// (line deleted entirely — narrativeMaxTokens no longer exists in the return object)

// index.ts — OpenRouterClient constructor
// (maxTokens parameter removed)
new OpenRouterClient({
  apiKey: cfg.openrouterApiKey,
  model: cfg.openrouterModel,
  temperature: cfg.narrativeTemperature,
  reasoning: cfg.openrouterReasoning,
  // maxTokens: ???  ← NO PATH EXISTS
})
```

### 1.2. Impact Analysis

`openrouterConfig.maxTokens` is a **valid, non-deprecated, actively documented** configuration option:

- **types.ts L127:** `openrouterConfig?: { maxTokens?: number; ... }` — still defined
- **index.ts PluginConfigSchema L419:** `maxTokens: Type.Optional(Type.Integer({ minimum: 256, maximum: 32768 }))` — still in TypeBox schema
- **openclaw.plugin.json L171:** `"maxTokens": { "type": "integer", ... "description": "Optional. Max tokens cap..." }` — still in user-facing schema

A user who sets `openrouterConfig.maxTokens: 4096` will observe:
1. `loadConfig()` extracts `openrouterConfig?.model` and `openrouterConfig?.temperature` into flat fields — but **not** `openrouterConfig?.maxTokens`
2. `cfg.openrouterConfig` is always `undefined` (see BUG 2 below)
3. `OpenRouterClient` is constructed without `maxTokens` → `undefined`
4. `openrouter-client.ts:89`: `if (this.config.maxTokens !== undefined)` — condition is false, `body.max_tokens` is never set
5. OpenRouter/model uses its own default (often the model's maximum output), which may be much larger than intended

**This is the exact same bug class as v0.4.13's `narrativePreviousEpisodeRef` and v0.4.14 Fix A's `recallQueryRecentMessageCount`** — schema + type define a field, but the config pipeline doesn't extract it. However, in this case the bug is **worse** because Fix B actively destroyed the previously-working transmission path.

### 1.3. Recommended Fix

Add `openrouterConfig.maxTokens` extraction to `loadConfig()` and forward it to `OpenRouterClient`:

```typescript
// config.ts — loadConfig() — add after openrouterModel line:
openrouterMaxTokens: rawConfig?.openrouterConfig?.maxTokens,

// types.ts — EpisodicPluginConfig — add field:
/** Max tokens cap for narrative generation (from openrouterConfig.maxTokens). */
openrouterMaxTokens?: number;

// index.ts — OpenRouterClient constructor:
new OpenRouterClient({
  apiKey: cfg.openrouterApiKey,
  model: cfg.openrouterModel,
  maxTokens: cfg.openrouterMaxTokens,   // ← restore the path
  temperature: cfg.narrativeTemperature,
  reasoning: cfg.openrouterReasoning,
})
```

**Note:** The field should be renamed from `narrativeMaxTokens` to `openrouterMaxTokens` to clearly indicate its source (the `openrouterConfig` nested object), not the deleted legacy flat alias.

## 2. 🟡 MEDIUM — Type Contract Violation: `openrouterConfig` Always `undefined`

**Severity:** MEDIUM (type contract violation, no runtime impact)  
**Location:** `config.ts:loadConfig()`, `types.ts:EpisodicPluginConfig`  
**Status:** Unfixed

### 2.1. Description

`EpisodicPluginConfig.openrouterConfig` is defined in `types.ts` as:

```typescript
openrouterConfig?: {
  model?: string;
  maxTokens?: number;
  temperature?: number;
  reasoning?: OpenRouterReasoningConfig;
};
```

However, `loadConfig()` **never sets `openrouterConfig:` on the return object**. It only reads individual sub-fields:

```typescript
openrouterModel: rawConfig?.openrouterConfig?.model ?? rawConfig?.openrouterModel ?? "openrouter/free",
narrativeTemperature: Math.max(0, Math.min(1, rawConfig?.openrouterConfig?.temperature ?? rawConfig?.narrativeTemperature ?? 0.4)),
openrouterReasoning: normalizeOpenRouterReasoning(rawConfig?.openrouterConfig?.reasoning ?? { enabled: true, effort: "high" }),
```

As a result, `cfg.openrouterConfig` is always `undefined` at runtime. Code search confirms **zero references** to `cfg.openrouterConfig` or `this.config.openrouterConfig` — so there is no runtime impact. However, the type contract says the field exists, which could mislead future developers into adding code that reads `cfg.openrouterConfig?.maxTokens` (which would always be `undefined`).

### 2.2. Recommended Fix

Either:
- **Option A (minimal):** Add `openrouterConfig: rawConfig?.openrouterConfig` to `loadConfig()` return. This makes the type contract honest. However, sub-fields would exist in BOTH the nested object and the flat fields, creating ambiguity about which is the source of truth.
- **Option B (preferred):** Remove `openrouterConfig` from `EpisodicPluginConfig` entirely. The nested object is only used as an input format that `loadConfig()` destructures into flat fields. It should not be on the output type. Add `openrouterMaxTokens` as a new flat field (see Fix 1.3) to complete the destructuring.

## 3. 🟡 MEDIUM — Documentation Mismatch: `segmentationWarmupCount` Default

**Severity:** MEDIUM (user-facing documentation incorrect)  
**Location:** `openclaw.plugin.json` vs `config.ts`  
**Status:** Unfixed

### 3.1. Description

| Source | Default Value |
|--------|--------------|
| `openclaw.plugin.json` `"segmentationWarmupCount"` | `"Default: 20"` |
| `config.ts` `loadConfig()` | `rawConfig?.segmentationWarmupCount ?? 10` (comment: `// Phase 3: was 20`) |
| `index.ts` `PluginConfigSchema` | No default specified |
| `types.ts` | JSDoc: `動的セグメンテーション: ウォームアップに必要な観測数` (no default mentioned) |

The code intentionally changed the default from 20 to 10 (noted in the `// Phase 3: was 20` comment), but the user-facing schema in `openclaw.plugin.json` still claims "Default: 20". Users who read the plugin documentation will believe the default is 20, but the actual runtime default is 10.

### 3.2. Impact

- User expectation mismatch: a user who doesn't set `segmentationWarmupCount` expects 20 but gets 10
- Earlier warmup (10 vs 20) means dynamic scoring activates sooner, which may cause premature boundary detection on small conversations
- The change from 20→10 may have been intentional (faster warmup for short sessions), but the documentation lag creates confusion

### 3.3. Recommended Fix

Update `openclaw.plugin.json` line for `segmentationWarmupCount`:

```json
"description": "How many messages the AI needs to observe to learn your 'chat style' before using dynamic sensitivity. Default: 10 (was 20 in pre-Phase 3).\n⚠️ [Blast Radius] If too high: The AI stays in a dumb 'fixed threshold' mode for way too long. If too low: It finishes learning on skewed data and its slicing logic gets permanently bugged."
```

## 4. 🟢 LOW — Dead Code: `recallReInjectionCooldownTurns` Double Fallback

**Severity:** LOW (dead code, no runtime impact)  
**Location:** `config.ts:loadConfig()` vs `index.ts:register()`  
**Status:** Unfixed

### 4.1. Description

`recallReInjectionCooldownTurns` has two default fallback values that can never both be exercised:

```typescript
// config.ts — loadConfig()
recallReInjectionCooldownTurns: Math.max(0, rawConfig?.recallReInjectionCooldownTurns ?? 24),

// index.ts — register()
const recallReInjectionCooldownTurns = Math.max(0, cfg.recallReInjectionCooldownTurns ?? 10);
```

Since `loadConfig()` always returns a number (minimum 0, default 24), the `?? 10` in `index.ts` is **unreachable dead code**. The effective default is always 24, not 10.

This creates two problems:
1. **Developer confusion:** A reader of `index.ts` might believe the default is 10
2. **Inconsistent with schema:** `openclaw.plugin.json` says "Default: 24" and the actual default IS 24 — but the dead `?? 10` could mislead someone into changing `loadConfig()` default thinking it's redundant

### 4.2. Recommended Fix

Remove the dead fallback in `index.ts`:

```typescript
// Before:
const recallReInjectionCooldownTurns = Math.max(0, cfg.recallReInjectionCooldownTurns ?? 10);

// After:
const recallReInjectionCooldownTurns = cfg.recallReInjectionCooldownTurns;
```

The `Math.max(0, ...)` is also redundant since `loadConfig()` already clamps it.

## 6. Schema Gaps (index.ts PluginConfigSchema vs Other Sources)

Several fields consumed by the plugin are **not in the TypeBox `PluginConfigSchema`** in `index.ts`:

| Missing from PluginConfigSchema | Present In | Impact |
|-------------------------------|-----------|--------|
| `tombstoneRetentionDays` | types.ts, config.ts, plugin.json | Field not validated at runtime by TypeBox; still works via rawConfig passthrough |
| `enableBackgroundWorkers` | types.ts, config.ts, plugin.json | Same as above |
| `lexicalPreFilterLimit` | types.ts, config.ts, plugin.json | Same as above |
| `anchorInjectionAssembles` | types.ts, config.ts | Not in plugin.json either — completely undocumented for users |

These gaps exist because `loadConfig()` receives the raw JSON object (`rawConfig`) before TypeBox validation, and reads fields directly. TypeBox validation would reject unknown fields (`additionalProperties: false`), but these fields are defined in `openclaw.plugin.json` which is the validation source, not `PluginConfigSchema`. The TypeBox schema in `index.ts` appears to be **legacy/unused** — the actual validation happens via `openclaw.plugin.json`.

**Assessment:** Not a bug, but a maintainability concern. Two parallel schema definitions (`PluginConfigSchema` in `index.ts` and `configSchema.properties` in `openclaw.plugin.json`) can drift apart, as they have here.

## 8. Root Cause Analysis: Why This Pattern Keeps Recurring

The v0.4.13 → v0.4.14 config pipeline bugs all share the same structural root cause:

### 8.1. The 4-Source Consistency Problem

Config fields must be defined in **4 separate locations** with no mechanical verification between them:

1. **`types.ts`** — TypeScript interface (compile-time type safety)
2. **`config.ts`** — `loadConfig()` parser (runtime extraction)
3. **`index.ts`** — TypeBox schema (runtime validation, appears unused)
4. **`openclaw.plugin.json`** — JSON Schema (user-facing validation + documentation)

When a developer adds a new config field, they must manually update all 4 sources. Missing any one produces the bugs we've seen:

- Missing from `loadConfig()` → user setting silently ignored (v0.4.13, v0.4.14 Fix A)
- Missing from `types.ts` → TypeScript doesn't catch downstream usage errors
- Missing from `openclaw.plugin.json` → user can't set the field, TypeBox rejects it
- Missing from `PluginConfigSchema` → runtime validation gap (lower impact)

### 8.2. The Nested Object Extraction Problem

`openrouterConfig` is a nested object that `loadConfig()` destructures into **flat fields**:

```
openrouterConfig.model      → openrouterModel
openrouterConfig.temperature → narrativeTemperature
openrouterConfig.reasoning   → openrouterReasoning
openrouterConfig.maxTokens   → (DROPPED — was narrativeMaxTokens)
```

This destructuring is fragile — adding a new sub-field to the nested object requires:
1. Adding it to `types.ts` `openrouterConfig` interface
2. Adding a new flat extraction line in `loadConfig()`
3. Adding it to `PluginConfigSchema` nested object
4. Adding it to `openclaw.plugin.json` nested object
5. Updating the `OpenRouterClient` constructor to use the new flat field

Step 2 was missed for `maxTokens` in v0.4.14 Fix B because the fix focused on removing the old path rather than ensuring the new path was complete.

### 8.3. Recommended Structural Fix

Add a **build-time or test-time** verification that all fields in `types.ts:EpisodicPluginConfig` have corresponding entries in `loadConfig()`:

```typescript
// test_config_pipeline.ts
// For each key in EpisodicPluginConfig, verify loadConfig() returns a defined value
const testConfig = { /* all fields set to non-default values */ };
const loaded = loadConfig(testConfig);
for (const key of Object.keys(testConfig)) {
  if (key === 'openrouterConfig') continue; // nested, destructured
  assert(loaded[key] !== undefined, `loadConfig() missing field: ${key}`);
}
```

This would have caught both v0.4.14 Fix A (missing `recallQueryRecentMessageCount`) and the v0.4.14 Fix B regression (missing `openrouterMaxTokens`).
