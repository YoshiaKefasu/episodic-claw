# v0.2.6 マスタープラン — recall / path / injection の統合修正

## この版で潰すもの
- `ep-recall` が空を返すのに実ファイルには記憶がある問題
- `episodes/episodes` の二重パス問題
- `prependSystemContext` の注入成否が追えない問題

## 基本認識
- この3件は別々に見えて、実際はつながっている。
- path 契約が崩れると索引や保守運用が濁る。
- recall の観測点が弱いと、空結果の原因を切り分けられない。
- 注入ログがないと、「recall 失敗」と「注入未実行」の区別がつかない。

## 効率優先の実施順
### Phase 1. 二重パスの是正
- まず保存構造の土台を直す。
- `agentWs` の意味を episode root に一本化し、`episodes/episodes` を今後増やさない。
- `episodes/episodes` の新規生成停止、nested tree の隔離、rollback、startup smoke まで含めて収束済み。
- これを先にやることで、以後の recall 調査結果が path 汚染に引っ張られない。

関連プラン:
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\docs\v0.2.6_duplicate_episode_paths_plan.md`

### Phase 2. `ep-recall` 空結果の修正
- path 契約を固定したあとで、空結果の真因を詰める。
- まず見るべきは `handleRecall()` に入った実 payload で、`agentWs` / `query` / `topics` / `strictTopics` を揃える。
- 次点で `workspaceCachePath(agentId)` の stale workspace を疑う。`SOCKET_ADDR_FILE` はその補助線として扱う。
- `result` 整形は原因候補から外す。空判定は整形より前で終わっている。
- 自動注入も同じ recall 経路を使うので、ここを直すと波及効果が大きい。

関連プラン:
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\docs\v0.2.6_ep_recall_empty_results_plan.md`

### Phase 3. `prependSystemContext` 注入ログの追加
- 上の2件を直したうえで、以後の監視を楽にする。
- ここで「注入された / されなかった / 空だった理由」を見える化する。
- `queryHash` / `estimatedTokens` / reason code を軸に、`max_tokens_zero` と `budget_truncated_to_zero` の境界を見分けられるようにする。
- 今後また recall 系の不具合が出ても、初動の切り分けが速くなる。

関連プラン:
- `D:\GitHub\OpenClaw Related Repos\episodic-claw\docs\v0.2.6_prepend_system_context_logging_plan.md`

## 依存関係
- Phase 2 は Phase 1 完了後に進める。
- Phase 3 は単独でも入れられるが、v0.2.6 では Phase 2 後の最終観測層として扱う。
- ただし、Phase 2 の切り分けで必要なら、Phase 3 の最小ログだけ先に一部前倒ししてよい。
