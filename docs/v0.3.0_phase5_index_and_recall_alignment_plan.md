# v0.3.0 Phase 5 プラン - 索引と recall の整合を揃える

- 状態: 収束済み

## 目的
- recall diagnostics と RPC surface を固定し、検索理由を観測できるようにする。
- compaction 後の Anchor flow と recall の責務境界を維持したまま、TypeScript 側で理由を潰さない形へ揃える。

## このフェーズで決めること
- `matchedBy` をどの RPC surface に残すか
- `fallbackReason` を `ScoredEpisode` に載せるか、別 shape にするか
- `rpc-client.ts` と `retriever.ts` でどこまで型を起こすか
- compaction 直後の Anchor 注入と recall diagnostics をどう住み分けるか

## 現行コードとの関係
- [retriever.ts](/D:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/src/retriever.ts) は auto recall 側の主戦場。
- [go/internal/vector/store.go](/D:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/go/internal/vector/store.go) は hybrid recall の土台。
- compaction 直後の Anchor は recall の結果ではなく、局所的な bridge memory として扱いたい。

## 実装タスク
- Go の `ScoredEpisode` に source metadata を追加する。
- `matchedBy` と `fallbackReason` の RPC surface を固定する。
- `rpc-client.ts` の `recall()` を最小型へ寄せる。
- `retriever.ts` と `index.ts` に recall diagnostics を通す。
- compaction 直後の Anchor は recall と別 source のまま維持する。

## 設計方針
- recall は長期記憶検索の責務を持つ。
- Anchor 注入は compaction 直後の局所 bridge の責務を持つ。
- 両者は連携しても、同じ source として扱わない。
- 検索 policy 変更はこのフェーズで一気に入れず、まず observability を先に固定する。

## 受け入れ条件
- `matchedBy` の記録で原因追跡がしやすい。
- `fallbackReason` の保持で lexical-only / topics fallback / embed fallback を追える。
- `rpc-client.ts` と `retriever.ts` が `any[]` 依存を減らし、score/source metadata を保持する。
- compaction 直後の Anchor は recall 不調でも働ける。
- recall と Anchor の責務分離が崩れない。

## 非対象
- `after_turn` 契約
- Anchor 生成 prompt
- raw history の保存
- `ep-save` の入口整理
- `/compact` command ownership（Phase 4 で確定した境界を前提に扱う）
- lexical-first 候補取得そのものの policy 変更
- `len(candidates) < N` backfill や常時並列 hybrid
- `lexicalLagMs` / `lexicalSeqLag` などの新規 freshness メトリクス

- 現状の責務分離はこうです。`Anchor` は compaction 直後の bridge memory、`recall` は長期検索。ここはすでにコード上で分離されています。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\index.ts:689-815`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\compactor.ts:275-331`
- 一方で `semantic/lexical の整合` と `matchedBy 記録` はまだ未実装です。Go 側には導出材料があり、TS 側はまだ捨てています。根拠: `D:\GitHub\OpenClaw Related Repos\episodic-claw\go\internal\vector\store.go:90-107`, `D:\GitHub\OpenClaw Related Repos\episodic-claw\src\retriever.ts:9-16`
- したがって Phase 5 は、**まず observability と source metadata の追加**、その次に candidate policy の見直し、の順が自然です。ここはまだ open です。

### Recommendation
- 最小変更で行くなら、Phase 5 の初手は「rank policy を変える」ではなく「candidate source / fallback reason を残す」です。
- `matchedBy` は TS 側の推定ではなく、Go 側で明示した値を RPC で返す方が安全です。
- Anchor 注入はすでに独立しているので、Phase 5 実装ではそこに触らず、recall 経路だけを対象にした方が枝が増えません。

### 今すぐ実装すべき最短順
1. Go `ScoredEpisode` に `matchedBy` と fallback reason の最小 field を追加し、candidate source を明示する。
2. `rpc-client.ts` の `recall()` を `any[]` から最小型へ寄せ、TS 側で score/source metadata を捨てないようにする。
3. `retriever.ts` の `RecallInjectionOutcome` に、ログ用の軽い recall diagnostics を追加する。
4. `index.ts` の `prependSystemContext` ログへ `matchedBy` 集計や fallback reason を足し、Anchor ログとは別 channel を維持する。
5. そのあとで必要なら、lexical hit 時も semantic candidates を混ぜるかどうかを別差分で判断する。

## Architecture Addendum — Retrieval Policy Options (2026-04-03, Future / Phase 6 Boundary)

### 目的
- lexical 優先の早期固定で recall を取り逃がすリスクを下げる。
- 非同期 lexical 更新（eventual）を維持しつつ、鮮度ギャップの実害を軽くする。
- strict topics fallback の挙動を「見える化」して、運用で原因追跡できるようにする。

### 現状の再確認（コード事実）
- 候補集合は lexical で 1 件以上ヒットすると HNSW 探索へ進まない。`len(candidates)==0` のときのみ HNSW へ進む。  
  根拠: `go/internal/vector/store.go:1132-1157`
- 最終順位は `semantic + lexical + freshness + surprise + usefulness + exploration` の重み付き和。  
  根拠: `go/internal/vector/store.go:999-1006`, `go/internal/vector/store.go:1226-1231`
- lexical index 更新は非同期 worker 前提。直後ズレは設計上起こりうる。  
  根拠: `go/internal/vector/lexical.go:50-204`
- strict topics は reverse index の cold 状態だけでなく、legacy data や sparse topics でも soft fallback へ落ちうる。  
  根拠: `go/internal/vector/store.go:1102-1114`, `go/internal/vector/store.go:1345-1359`
- TS 側は理由を粗く丸め、fallback 理由を詳細保持していない。  
  根拠: `src/retriever.ts:143-154`, `src/rpc-client.ts:491-500`

### 推奨アプローチ（優先順）
1. **Observability-first（最小リスク）**  
   Go で `matchedBy` と `fallbackReason` を返し、TS で保持・ログ出力する。初手では top-level RPC shape を増やさず、`ScoredEpisode` に載せる方針を優先する。  
   例: `matchedBy=semantic|lexical|both`, `fallbackReason=embed_timeout|embed_rate_limit|topics_fallback|none`
2. **候補不足時の段階的ハイブリッド**  
   `len(candidates)==0` ではなく、`len(candidates)<N` なら HNSW 候補を追加取得して union。  
   既存スコア式は維持し、まず recall 欠落だけ減らす。
3. **常時並列ハイブリッド（第2段階）**  
   lexical と HNSW を毎回並列で取得し、rank fusion（RRF）または正規化線形結合へ移行。  
   この段階で初めて policy 自体を拡張する。

### 非同期 lexical 更新の扱い
- 方針: **非同期更新は維持**（スループット優先）  
- 改善案（軽量）:
  - `lexicalLagMs` / `lexicalSeqLag` を返す（可視化）
  - ingest 直後の短時間だけ `candidateBackfill=true` で semantic 候補を補完
  - 直近更新を軽量 hot overlay で検索対象に含める（必要時）

### strict topics fallback の扱い
- 方針: fallback は残すが、必ず理由を返す。  
- 実装ガイド:
  - `topicsFallback=true` を返却
  - strict→soft へ落ちた場合にログへ明示
  - cold reverse index だけでなく、legacy data / sparse topics も原因候補として扱う
  - 必要なら fallback 時だけ弱い減衰係数を適用（過剰優遇を防止）

### 「semanticScore + bm25」説明（実装に即した定義）
- `semanticScore = 1/(1 + L2^2)`  
- `bm25 = rawBM25/maxBM25`（候補集合内正規化）
- これに freshness/surprise/usefulness/exploration を加えた重み付き和が最終 `finalScore`。  
  既定重み: `semantic 0.60`, `lexical 0.10`, `freshness 0.15`, `surprise 0.05`, `usefulness 0.08`, `exploration 0.02`  
  根拠: `go/internal/vector/store.go:999-1006`, `go/internal/vector/store.go:1226-1231`

### 参考（設計比較）
- OpenSearch Hybrid search（正規化 / rank fusion）: `https://docs.opensearch.org/3.0/vector-search/ai-search/hybrid-search/index/`
- Elasticsearch Linear retriever（正規化付き線形結合）: `https://www.elastic.co/docs/reference/elasticsearch/rest-apis/retrievers/linear-retriever`
