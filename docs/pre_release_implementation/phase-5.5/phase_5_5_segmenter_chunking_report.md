# Phase 5.5: Segmenter Chunking (巨大ダンプの自動分割) レポート

## 1. 背景と課題

Episodic-Claw におけるエピソード生成（Segmenterによるコンテキストのダンプ）において、以下のような深刻な課題が発生していました。

1. **一括ダンプによるペイロードの肥大化**
   セッションの初期化時や、ユーザーとの会話が長時間におよんでから初めてフラッシュ（ダンプ）処理が走った際、数万文字に及ぶ巨大な配列が一度に `generateEpisodeSlug` （および ベクトル保存）に送られていました。
   
2. **API上限オーバーとMD5フォールバックの誘発**
   Gemini API (Gemma) に一度に送信されるテキスト長が長すぎるため、タイムアウトやトークン制限エラー（HTTP 429/500 等）が頻発しました。結果として、API呼び出しに失敗し、名前が生成されないまま `episode-[MD5ハッシュ].md` になってしまう最大の原因の一つとなっていました。

3. **コンテキストの希釈（検索精度の低下）**
   148 KiB にも及ぶような巨大なMarkdownファイルが1つのチャンクとしてVectorDBに登録されてしまうと、後からRAG（検索）で思い出す際に、関連性の薄い様々なトピックが1つに混ざってしまい、LLMへのコンテキスト注入時（Assemble時）に重大なノイズを引き起こします。

## 2. 解決策：チャンク分割処理の実装

ユーザーからの「長すぎるダンプを分割して複数のMarkdownとして書き出せないか」というアイデアに基づき、`src/segmenter.ts` に以下の改良を行いました。

### A. 自動チャンク分割保存 (`chunkAndIngest`) の導入
バッファ全体を1つの大きなサマリーにして Ingest する既存の仕様を廃止し、**約10,000文字ごと（数千トークン程度）** の塊に切り分けて順次 Ingest する `chunkAndIngest` 関数を新設しました。
- これにより、どんなに巨大なコンテキスト・ダンプが飛んできても、内部で「サイコロ状」に切り分けられ、それぞれ独立した複数のMarkdownエピソード（ファイル群）として生成＆DB登録されるようになります。

### B. バッファ文字数ベースの強制フラッシュ（安全装置）
これまでの `processTurn` では、「Surprise（乖離度）」のしきい値を超えなければ、ずっとエピソード境界を跨がずバッファに溜まり続ける仕様でした。
これを改良し、**全体の文字数が `12,000文字` を超えた時点で、Surprise判定に関わらず強制的にフラッシュ（チャンク分割保存）が発動**するロジックを追加しました。これにより、メモリ上のバッファが無限に肥大化することを防いでいます。

## 3. 期待される効果

- **API安定性の劇的な向上**
  1リクエストあたりのテキスト量が一定（最大10,000文字程度）に抑えられるため、LLM APIでのエラーや制限に引っかかる率が激減します。
- **MD5フォールバックのさらなる減少**
  Deferred Queueの廃止や再試行ロジックの改善に加え、入力サイズの最適化が行われたことで、より確実に綺麗なAIスラッグが生成されるようになります。
- **RAGの検索精度向上**
  大きすぎたエピソードが複数の独立したMarkdownファイルに分散することで、キーワードごとのベクトル検索精度が高まり、記憶の引き出し（Recall）がよりシャープになります。

## 4. 適用状況
- `src/segmenter.ts` を修正し、TypeScript をビルド済み。
- コンパイル成果物（`dist/segmenter.js`）は既にWSL側のプラグインディレクトリ (`/root/.openclaw/extensions/episodic-claw/dist/`) に適用済みです。
- 次回 Gateway 起動時より、新しいチャンク分割ロジックが有効になります。
