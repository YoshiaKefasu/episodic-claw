# 容赦ないコードレビュー：Episodic Memory Phase 1
（Reviewer: Staff Software Engineer, Google）

## TL;DR
アイデアの方向性自体は悪くない。I/Oとベクトル処理の重たいプロセスをNode.jsからGoへ外出しし、シングルスレッドの限界を突破しようとするアプローチは間違っていない。Markdownファイルを正（Source of Truth）としてDBを再構築可能にする思想も、システムの寿命を延ばす良い仕組みだ。

だが、今のPhase 1の設計は完全に**「ハッピーパス」しか考えていないプロトタイプ**だ。すべてが正常に動く前提で書かれている。この設計のまま本番環境に投入すれば、1ヶ月もしないうちに謎のパースエラー、ゾンビプロセス、そして起動時のハングアップに悩まされることになる。

システムは必ず壊れる。この設計には「壊れた時のフェイルセーフ」が存在しない。以下に、本番投入前に必ず修正すべき致命的な欠陥を列挙する。

---

## 🚫 致命的な設計ミス (P0 レベル)

### 1. IPC（プロセス間通信）の脆さ
**問題:** [Phase_1_implementation_plan.md](file:///d:/GitHub/OpenClaw%20Related%20Repos/episodic-claw/docs/Phase_1_implementation_plan.md) に書かれている、`stdio`（標準入出力）を使ったJSON-RPC通信。これは最悪の選択だ。
Go側で意図せずパニックが起きたり、サードパーティの依存ライブラリが警告を吐いて `stdout` にログが混ざった瞬間、TS側のパーサーは死ぬ。JSONストリームの汚染は防ぎきれない。また、プランには「or localhost TCP」とあるが、これもポート競合やローカルファイアウォールの問題を引き起こす。
**解決策:** `stdio` は今すぐ捨てろ。Unix Domain Sockets（WindowsならNamed Pipes）を使え。ログ出力とコントロールプレーン（RPC通信）は物理的に分離するのが、信頼できるインフラ設計の基本だ。
✅ **[FIXED]** `main.go` と `rpc-client.ts` をTCP Sockets（Windowsネイティブ互換）およびUnix Domain Sockets（POSIX用）に切り替え。stdio通信を完全に廃止しました。

### 2. プロセスライフサイクルとゾンビ化
**問題:** `rpc-client.ts` がGoサイドカーを起動・停止するとある。だが、TS側（OpenClaw）がOOMなどでクラッシュし、綺麗に終了処理（Kill）できなかったらどうなる？ Goプロセスはゾンビ化して裏で動き続ける。次回起動時にポートやファイルロックの競合でシステム全体が立ち上がらなくなるのは目に見えている。
**解決策:** Go側にTS親プロセスへのハートビート監視、または親プロセスID（PPID）の死活監視を仕込め。親が死んだらGoプロセスも自動で道連れ終了（Suicide）する仕組みが必須だ。
✅ **[FIXED]** `main.go` で `-ppid` フラグを受け取り、2秒ごとに `os.FindProcess` で親プロセスの死活を監視して親の終了と共に自殺する Watchdog を実装しました。

## ⚠️ スケーラビリティとパフォーマンス (P1 レベル)

### 3.起動時の O(N) フルキャン（スケーラビリティの無視）
**問題:** 初期コンテキストを作るため、`indexer` が起動時に `episodes/**/*.md` をすべてトラバースする設計。エピソードが10件なら動くだろう。では、半年後に数千件のファイルになったら？ 起動するたびに毎度全ファイルをパースしてブロックする気か？ 起動時間がファイル数に比例して線形に延びる設計は、システムとして破綻している。
**解決策:** メタデータの軽量なローカルキャッシュ（SQLiteや単なるJSONマニフェスト）を持て。起動時は高速にキャッシュを読み込み、前回のプロセス終了後から変更された分だけを差分同期フェーズでパースしろ。
✅ **[FIXED]** `indexer.go` に `.episodic_index.json` キャッシュ機構を追加。変更や追加があったMarkdownのみパースし直す差分同期機能を実装しました。

### 4. Debounce制御の甘さと競合
**問題:** `fsnotify` 経由のイベントを1500msのDebounceで処理する実装。ファイルのロック中や、1.5秒以上の断続的な巨大書き込みが発生した場合、イベントはいつ発火するのか？ 不完全な状態のファイルをパースしようとしてクラッシュする危険がある。
**解決策:** 単純なタイマーではなく、キューとワーカーを用いた一元化されたイベントバッチ処理を組め。ファイルの書き込み完了（クローズ）を確実に検知する防御的なロジックがないと、Race Condition（競合状態）の温床になる。
✅ **[FIXED]** `watcher.go` の Timer を廃止。代わりにステートを持つ Queue と定期ワーカー（Ticker）によるスレッドセーフなバッチ処理イベントディスパッチャに変更し、競合を排除しました。

## 盲点：オブザーバビリティの欠如 (P2 レベル)

### 5. ブラックボックス化するサイドカー
**問題:** Goプロセスが異常終了したとき、あるいは処理が詰まったとき、どうやってデバッグする？ Phase 1のタスクリストには「ロギング」の責務が一行も書かれていない。
**解決策:** Go側で構造化ログ（JSONLines等）を専用のログファイル（例: `/tmp/episodic-core.log`）に非同期で書き出す仕組みを最初に入れろ。これが無いと、本番環境で問題が起きた時に誰も原因を追えなくなる。
✅ **[FIXED]** TSへのstderr出力と並行し、`/tmp/episodic-core.log` へ非同期かつ安全に構造化JSONロギングを行う `InitLogger()` を導入しました。

## 総評
厳しいことを言ったが、コアのアイデア（Markdown=Source of Truthのハイブリッド構成）には大いに賛同する。だからこそ、この「通信・監視の土台」となるPhase 1で手を抜いてはいけない。
ハッピーパスのプロトタイプ思考を捨てて、インフラのプロとして異常系（エラーハンドリング）に全力を注いだ設計に書き直してほしい。
